# 生成文章摘要、总结与译文（一次性）

为下面这篇文章一次性生成三样东西：(1) 原标题的中文翻译，(2) 索引页中文摘要，(3) 同时包含中文总结和完整中文译文的独立页面。

所有中文输出遵循以下写作要求：

- 不要生硬地直译英文，改用中文习惯的、更顺畅的表达。
- 使用短句和短段。每句话只表达一个主要意思，复杂因果关系要拆开写。
- 避免使用连续多个逗号或分号串联的长句。

**严格按下面的格式输出**：四个分隔行（`===TITLE_ZH===`、`===SUMMARY===`、`===VALUE===`、`===DIGEST===`）各自独占一行、原样照抄、各只出现一次；整段不要用 markdown 代码块包裹，不要加任何额外说明。

```
===TITLE_ZH===
（原标题的中文翻译，一行，不要加书名号）
===SUMMARY===
（索引摘要正文，直接写中文，不要用 JSON）
===VALUE===
（一句话）
===DIGEST===
<!doctype html> ... </html>
```

## TITLE_ZH 段（标题翻译）

- 只翻译原标题，不要扩写、点评或改写成摘要标题。
- 保留模型名、产品名、机构名、benchmark 名和专有名词；必要时只翻译连接词和说明性短语。
- 一行中文即可，不要加句号，不要加“中文总结”。

## SUMMARY 段（索引摘要）

- 写成一段连续中文文本，必须包含“结论：”和“关键数据：”两段。
- 关键数据优先使用原文中的数字、样本量、benchmark 结果、日期、比例或其他可验证测量。
- 简洁、具体、非宣传化；优先解释发现、数据集、方法、政策意义或实践影响，不要只复述标题。
- 如果正文为空或抓取失败，不要凭标题编造内容。
- 避免把背景、方法、数据、结论和影响塞进同一个长句；需要时拆成两三句。

## VALUE 段

- 一句话说明这篇文章为什么值得读。
- 不要以“这篇文章值得读，因为”“它值得读，因为”“值得读的原因是”等固定模板开头；直接从具体阅读价值切入，句式尽量自然变化。

## DIGEST 段（独立总结与译文页）

输出一个完整、自包含的 HTML 文档（从 `<!doctype html>` 到 `</html>`，`<html lang="zh-CN">`，`<meta charset="utf-8">`，样式写在文档内的 `<style>` 里）。页面先提供“更短、更易读”的中文总结，再提供完整中文译文。

### 总结部分

- 开头必须有“一句话总括”，说明文章真正的贡献，而不是复述标题。
- 页面开头必须同时展示中文标题和英文原标题。中文标题用 `TITLE_ZH` 段内容，英文标题用原文标题。
- 英文原标题要靠近中文标题显示，样式可以弱化，但不能省略。
- 每个关键数据和结论用一两句话讲清楚，并解释它为什么改变读者对 AI 影响的理解。
- 每个小段的标题是总结，中间是关键数据和趋势，最后一句话给出结论和洞见。
- 原文中的关键数据和结论，不要为了总结刻意增减，如实地总结，最终段落的数量和文章长度不是质量标准。
- 只依据原文内容写总结；如果正文为空或抓取失败，不要凭标题编造内容。
- 结尾用 3-5 条“最值得带走的洞见”综合全文。

### 完整译文部分

- 总结部分之后必须紧接一个 `<section id="translation">`，并在其中使用 `<h2>完整译文</h2>` 作为标题。这个结构是生成器校验译文是否存在的固定标记，不能改名或省略。
- 完整翻译输入中“正文”字段提供的全部内容。保留原文的信息和顺序，不要总结、缩写、合并或省略段落。
- 保留原文的章节层级、列表、表格、图注、注释和脚注。可以按中文阅读习惯调整语序和分段，但不能改变事实、数字和限定条件。
- 模型名、产品名、机构名、benchmark 名、代码、命令、URL、变量名和其他专有名词按原文保留；常见术语可在首次出现时采用“中文（English）”。
- 对原文中的链接，保留可用的 `href`；不要虚构正文里不存在的链接、图表或说明。
- 译文使用自然、准确的中文。不要插入译者评论，也不要把总结部分的观点重复写进译文。

### 原文图片

- 输入会在正文后提供“原文图片”JSON。图片按 `index` 编号，`after_block` 表示它在原文中出现前已有多少个正文内容块。
- 每个 `index` 表示一张逻辑图片，必须在 HTML 中出现一次且只出现一次。请在对应位置输出 `<figure data-source-image="N">...</figure>`，其中 `N` 原样使用图片的 `index`。如果图片含 `theme_sources`，它仍只需要一个 `figure`；生成器会在其中注入亮色/暗色资源并随页面主题切换。
- 有 `caption` 的图片必须在该 `figure` 内输出 `<figcaption>`，内容是原图注的完整中文翻译。没有图注的题图可以省略 `figcaption`。
- 不要自行输出 `<img>`，不要复制、改写或猜测图片 URL。生成器会根据 `data-source-image` 确定性注入原始图片地址、尺寸和替代文字。
- 所有原文图片都只能出现在 `<section id="translation">` 内。总结部分禁止出现 `data-source-image`、原文图片或替代图片。
- 题图放在“完整译文”标题之后、译文正文之前。正文图表按 `after_block` 和图注语义放回译文的对应段落附近，不能把所有图片集中堆在页面末尾。
- 不得用文字描述替代图片，也不要遗漏表格截图、图表或题图。

### 视觉与主题

- 充分使用不同的字号、颜色、背景和布局帮助阅读，避免单调。
- 句子中的关键数字、结论词等关键部分用高亮标注（如 `--hl`、`--hl2` 背景或加色加粗），让读者扫一眼就能抓住重点，减少阅读疲惫感。
- 总结部分不能把所有正文区块都做成同一种卡片。必须有明显不同的区块形态。
- 使用多种不同的内容展现形式，例如：大号 hero、横向关键数字带、左右分栏解释、时间线/流程线、对比表格、重点引语/洞见条、马赛克式 takeaway。
- 相邻正文区块的布局、背景或色彩必须有明显变化。不要连续使用相同的 `section` 样式。
- 色彩必须有层次。至少使用 4 个有区分度的强调色，并分别服务于数据、风险、机会、结论等不同语义。
- 避免整页只有同一色系的浅色卡片。可以使用局部深色区块、浅色信息条、带色左边框、数字徽章、分隔带等方式形成节奏。
- 所有颜色都必须通过 CSS 变量使用。不要写 inline `style="background:..."`。
- 每一个浅色背景变量都必须在 `@media (prefers-color-scheme: dark)` 和 `:root[data-theme="dark"]` 中给出暗色对应值。
- 同一组亮色变量必须显式复制到 `:root[data-theme="light"]`，保证系统处于暗色时手动选择亮色仍能完整恢复页面专用颜色。
- `@media (prefers-color-scheme: dark)` 内的暗色变量只能写在 `:root:not([data-theme="light"])` 下，不能使用无条件的 `:root` 覆盖显式亮色主题。
- 暗色模式下不能出现刺眼的纯白、浅黄、浅蓝、浅粉或浅绿大面积色块；需要保留色彩变化时，使用暗色底配高饱和边框、文字或小面积徽章。
- 样式优先使用 CSS 变量（如 `--bg`、`--card`、`--ink`、`--muted`、`--line`、`--accent`、`--accent2`、`--accent3`、`--hl`、`--hl2`），不要自己实现主题切换按钮或脚本；生成流程会统一注入亮色、暗色和跟随系统主题。
- 完整译文部分以连续阅读为主，使用舒适的正文宽度、行距和清晰标题层级，不要把每个译文段落都包装成卡片。
- `.source-figure` 应有清晰的上下间距；图片按原始宽高比响应式显示，图注与正文有明显但不过度的视觉区分。
