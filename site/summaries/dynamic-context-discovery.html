动态上下文发现：用"文件"让 Agent 按需取用上下文

作者：Jediah Katz 等多位 Cursor 员工 · 原文：cursor.com/blog/dynamic-context-discovery
一句话总括：Cursor 把"塞满上下文窗口"的传统做法，换成让 agent 自己按需去文件里拉取上下文——更少前置信息、更高 token 效率，并用一组真实工程案例证明，文件是当下最简单也最稳妥的 LLM 工具接口原语。

核心转变：从"静态注入"到"动态发现"

模型越强，越不需要喂得太满

随着 agent 模型变强，Cursor 发现前置信息给得越少，agent 反而越能自己找到相关上下文。他们把这种"按需拉取"称为 dynamic context discovery，与始终全量包含的 static context 相对。好处有二：只把必要数据放进窗口，token 更省；减少互相矛盾或干扰的信息，回答质量更高。

洞见：上下文工程的优化方向，正从"如何塞得更全"转为"如何让模型自己取"——这是一种对模型能力增长的顺势设计。

最硬的证据：MCP 工具按需加载省下近一半 token

每个服务器一个文件夹，只在提示里留工具名

−46.9%在调用了 MCP 工具的运行中，agent 总 token 下降幅度（A/B 测试，统计显著，方差随 MCP 数量增大）
很多 MCP 服务器自带大量工具和冗长描述，却始终全量进提示、大部分从不被用，多个服务器叠加更糟。Cursor 把工具描述同步到文件夹（每服务器一个文件夹而非扁平索引），系统提示里只留工具名，需要时再让 agent 用 rg、jq 去查找。额外收获：文件还能传达工具状态——例如某 MCP 需要重新认证时，agent 可主动提醒用户，而不再"直接忘掉"。

洞见：减少上下文占用应是"编码 agent 的责任"，而非指望每个 MCP 服务器自我优化——这把架构责任划清了。

把"会撑爆上下文的东西"统统变成文件

长响应、聊天历史、技能、终端输出，一律落盘按需读

  • 长工具响应 → 文件：第三方 shell/MCP 的超长 JSON 不再被截断（截断会丢关键数据），而是写入文件，agent 先 tail 看结尾、按需再读，减少了触及上限时不必要的总结。
  • 聊天历史 → 文件：上下文写满触发总结时，总结是有损压缩、易丢细节；把历史作为文件提供，agent 可回查找回被遗漏的关键信息。
  • Agent Skills 开放标准：技能的名称+描述作静态上下文，正文用 grep 与语义搜索按需发现；技能还能打包可执行脚本，因为本质都是文件。
  • 集成终端会话 → 文件：终端输出同步到本地文件系统，问"我的命令为什么失败"时 agent 能 grep 长日志里相关片段，效果对齐 CLI 类 agent，但属动态发现而非静态注入。

洞见：五个看似不同的问题，被同一个抽象（文件）统一解决——这正是好原语的标志。

为什么选"文件"这个朴素抽象

不确定它是终局，但它是最安全的赌注

Cursor 坦承文件未必是 LLM 工具的最终接口，但在 agent 快速演进的当下，文件简单、强大，且比"又发明一个无法预判未来的新抽象"更稳妥。相关改进将在数周内对所有用户上线。

洞见：在技术高速变化期，押注通用、低风险的原语，往往胜过押注精巧但脆弱的专用抽象。

最值得带走的洞见

  1. 范式转变：模型越强，越该"少喂、让它自己取"——动态发现取代静态注入。
  2. 可量化收益：MCP 工具按需加载使相关运行的 token 降低 46.9%，证明这不是玄学而是可测的工程改进。
  3. 统一抽象：长响应、聊天历史、技能、终端输出——四类上下文膨胀源都用"文件 + 按需 grep/read"一招解决。
  4. 责任归属：降低上下文占用是编码 agent 的职责，而非依赖每个 MCP/工具提供方自觉优化。
  5. 务实赌注：在不确定终局接口时，选择文件这种通用原语，是对未来最稳健的押注。
