Cursor Research · 2026-01-06

动态上下文发现：把资料放在文件里，让智能体按需读取

一句话总括：这篇文章的真正贡献，是把编码智能体的上下文管理从“提前塞满提示词”改成“把可检索资料放在文件系统中按需发现”，用更少 token 保留更多可恢复信息。

5 类 Cursor 已使用的动态上下文发现做法。
46.9% 涉及 MCP 工具调用的 A/B 测试中，总 agent tokens 的降幅。
数周内 文章称相关改进将面向所有用户上线。

少给静态上下文，反而能让智能体更稳

Cursor 认为，前沿模型变强之后，不必把大量规则、工具说明和历史输出一开始就放进上下文窗口。更好的方式是只给少量入口信息，再让智能体自己搜索需要的文件。

这改变了对 AI 编程影响的理解：瓶颈不只是模型能力，也包括上下文工程是否能减少噪声、冲突信息和无效 token 消耗。

长工具输出不截断，而是写成可读取文件

传统做法会截断过长的 shell 或 MCP 返回结果，可能丢掉真正关键的错误和数据。Cursor 的做法是把输出写入文件，让智能体先看结尾，再按需读取更多内容。

这个设计的意义在于：AI 不必在“塞满上下文”和“丢失细节”之间二选一，长日志和大 JSON 可以保留完整性，同时只消耗必要上下文。

摘要不再是单向压缩，历史可以被重新检索

当上下文窗口满了，Cursor 会触发总结，让智能体获得新的上下文窗口；但总结本身是有损压缩，可能漏掉任务细节。

把聊天历史作为文件提供后，智能体可以在摘要之后回查遗漏信息。结论是，长任务的可靠性不只靠一次完美总结，而要靠总结后的可恢复机制。

Agent Skills 和 MCP 工具都适合文件化发现

Skills 用名称和描述作为少量静态上下文，具体规则、脚本和可执行文件则按需读取。MCP 工具也被同步到按服务器分组的文件夹，而不是把所有长描述都塞入提示词。

最直接的数据是：在调用 MCP 工具的运行中，这种方式让总 agent tokens 降低 46.9%，且统计显著。它说明企业环境里接入更多内部工具，并不必然意味着上下文成本线性膨胀。

终端会话变成文件后，用户不用再解释“刚才那个错误”

Cursor 会把集成终端输出同步到本地文件系统，用户问“为什么命令失败”时，智能体可以直接检索相关终端历史。长时间运行的服务日志也可以用 grep 定位，而不是整段注入上下文。

这让图形 IDE 中的智能体更接近 CLI 编码智能体的工作方式：它能看到真实开发痕迹，但这些痕迹是动态发现的，不是永久占用上下文窗口。

文件是朴素但耐用的智能体接口

文章没有宣称文件一定是最终形态，而是把文件视为当前最简单、最强韧的抽象。相比再设计一个新索引或新协议，文件天然支持分组、搜索、tail、rg、jq 等成熟操作。

核心洞见是：AI 工具的进步可能来自更普通的工程接口，而不是更复杂的专用抽象。

最值得带走的洞见

  • 更大的上下文窗口不是唯一答案，按需发现上下文往往更省、更清晰。
  • 长日志、工具响应和聊天历史不应被简单截断，文件化能保留完整信息并降低即时 token 成本。
  • MCP 工具越多，按需加载越重要；Cursor 报告的 46.9% token 降幅说明收益已经可测。
  • 把终端、技能和工具描述统一成文件，让智能体更像真实开发者一样查资料、翻日志、恢复上下文。
  • AI 编程体验的关键竞争点，正在从“模型会不会写代码”扩展到“系统能不能让模型拿到正确上下文”。
