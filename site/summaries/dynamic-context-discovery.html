动态上下文发现：少给前置信息，让 Agent 自己按需取

来源：Cursor  ·  作者：Jediah Katz  ·  2026-01-06  ·  原文链接
一句话总括

Cursor 提出"动态上下文发现"——不再把一切静态塞进提示词，而是把数据落成文件、让 Agent 按需检索；并用一个真实 A/B 实验证明这能把调用 MCP 时的总 token 量砍掉近一半。

静态上下文
始终被注入提示词，无论是否用到——token 一直在烧，还可能引入矛盾信息。
动态上下文发现 →
只把"入口"放进上下文，真正的数据存为文件，Agent 用 grep / tail / 语义搜索按需拉取。
-46.9%
在 A/B 测试中，对于调用了 MCP 工具的运行，把 MCP 工具描述按服务器同步到文件夹后，Agent 总 token 量下降 46.9%。 统计显著；方差较大，取决于安装的 MCP 数量。

1过长的工具响应不截断，而是写成文件

截断 shell / MCP 的长结果会丢失你本想要的信息。

Cursor 把超长输出写入文件，Agent 先用 tail 看结尾，需要更多再继续读。第一方工具（编辑、搜索代码库）本就用精简响应格式控制膨胀，但第三方工具拿不到同等待遇，文件化补上了这一环。

洞见：用"读文件"替代"截断"，触达上下文上限而被迫总结的次数明显减少。

2总结时保留聊天历史作为可回查的文件

上下文填满触发的总结是一种有损压缩，Agent 可能忘掉任务的关键细节。

当上下文窗口触顶（或用户手动总结）时，Cursor 给 Agent 一个历史文件的引用。一旦发现摘要里缺了关键信息，它能回到历史中搜索找回。

洞见：不必把"记忆"全压进摘要，给一条回查通道就能对冲压缩损失。

3支持 Agent Skills 开放标准

Skill 用文件定义如何完成某类专门任务。

Skill 的名称与描述作为"静态上下文"进系统提示，正文、可执行脚本则是文件；Agent 用 grep 和语义搜索动态拉取相关 Skill。因为它们本质就是文件，找到与某个 Skill 相关的内容很容易。

洞见："放入口、藏正文"成为扩展能力的统一范式——能力可无限增长而不撑爆提示词。

4MCP 工具按需加载（省 token 的核心来源）

MCP 适合访问 OAuth 背后的资源（生产日志、设计文件、企业文档），但很多服务器工具又多又啰嗦，且总被全量注入。

Cursor 给每个 MCP 服务器建一个文件夹同步工具描述（而非散落的扁平索引，便于用 rg、jq 过滤）。系统提示里只留工具名等少量静态上下文，用时再查找。文件化还顺带解决了一个体验问题：服务器需要重新鉴权时，Agent 不再"忘掉"这些工具，而能主动提示用户去重新认证。

洞见：团队认为降低上下文占用是编码 Agent 的责任，不该指望每个 MCP 服务器自我优化——这正是 -46.9% 的来源。

5集成终端会话全部当作文件

不必再把终端输出复制粘贴进 Agent。

Cursor 把集成终端的输出同步到本地文件系统，于是"为什么我的命令失败了？"这类提问，Agent 能直接理解你在指什么。终端历史可能很长，Agent 用 grep 只取相关片段，对长跑服务的日志尤其有用。

洞见：这复刻了 CLI 编码 Agent 自带历史上下文的优势，但改为"动态发现"而非"静态注入"。

最值得带走的洞见

  1. 模型越强，越该少喂。随着 Agent 能力提升，"前置信息给得越少、让它自己取"反而效果更好——这是贯穿全文的方向性判断。
  2. 省 token 不是噱头，有硬数据。MCP 工具按文件夹同步带来 46.9% 的 token 下降（统计显著），证明动态发现的实际收益。
  3. 动态发现同时省钱又提质。更少 token 之外，剔除无关/矛盾信息还能直接改善 Agent 的回答质量。
  4. "文件"是当下最好的通用原语。它简单、可被 grep/tail/jq 操作，比再造一层抽象更安全、更面向未知的未来——尽管它未必是 LLM 工具的最终形态。
  5. 这是横切所有模型的改进。harness 会针对每个前沿模型单独调优，但这类上下文工程改进对所有模型通用，且将在数周内向全体用户上线。
