动态上下文发现

Dynamic context discovery

一句话总括：Cursor 把大量上下文转化为可搜索的文件，让编程智能体在真正需要时自行查找，从而减少 token 消耗、信息冲突和压缩造成的遗忘。

来源：Cursor 分类：Research 作者：Jediah Katz

核心发现

少给一点，反而可能做得更好。 Cursor 不再把所有工具、日志和历史记录预先塞进提示词，而是只提供发现入口。智能体随后按任务需要检索内容。这改变了上下文工程的重点：关键不只是选择放进什么，还包括设计一条可靠的取用路径。
46.9%

MCP 场景中的 token 降幅

在确实调用 MCP 工具的 A/B 测试运行中，按需加载工具说明使智能体的总 token 消耗降低了 46.9%。

结果具有统计显著性。不过，降幅会随安装的 MCP 数量产生较大波动。它说明静态注入大量工具说明的成本并非边缘问题，而是可以被架构性消除的开销。

静态上下文的问题

所有说明都常驻上下文，会挤占窗口空间。未使用的信息也可能互相矛盾，让模型难以判断重点。

截断超长输出虽然节省空间，却可能丢掉决定任务成败的内容。上下文摘要同样属于有损压缩，智能体可能忘记关键约束。

动态发现的做法

先给智能体少量索引，再把完整信息保存在文件中。智能体可通过 tail、grep、rg 或 jq 精确读取。

这样既保留全部数据，又只让相关片段进入上下文。文件因此成为工具输出、历史记录和能力说明之间的共同接口。

五种落地方式

  1. 保存长输出把过长的 shell 或 MCP 返回值写入文件，避免截断和数据丢失。
  2. 恢复聊天历史摘要后仍可搜索历史文件，找回压缩时遗漏的关键细节。
  3. 加载 Agent Skills静态提供名称与描述，具体规则、脚本和资源按需读取。
  4. 发现 MCP 工具按服务器组织工具说明，只有任务需要时才加载对应定义。
  5. 检索终端会话把集成终端输出同步到本地文件，按需定位报错和长时间运行日志。
文件的价值不在于新奇，而在于它是一个简单、可搜索、可组合的抽象。模型和工具仍在快速变化时，这种基础接口比过早设计复杂的新协议更稳妥。

最值得带走的洞见

上下文工程正在从“预先塞满”转向“提供发现能力” 优秀的智能体框架不必预测任务所需的一切。更可扩展的方式，是保留完整信息并提供高效检索入口。
文件避免了截断与常驻之间的二选一 大型工具输出可以完整落盘，同时只把相关部分送入模型。
摘要不再是不可逆的遗忘 聊天历史文件让智能体在上下文压缩后仍能恢复原始细节。
MCP 工具发现有可测量的收益 46.9% 的降幅表明，工具描述本身就是重要的 token 成本来源。
简单抽象更能适应模型迭代 文件可由成熟命令行工具直接处理，也便于按服务器和能力组织。

编程智能体正在迅速改变软件的构建方式。它们之所以进步得如此之快，一方面源于智能体模型能力的提升，另一方面也得益于更好的上下文工程。后者可以更有效地引导模型。

Cursor 的智能体框架由我们提供给模型的指令和工具组成。我们会针对支持的每一种新前沿模型分别优化这套框架。不过，有些上下文工程改进适用于框架内的所有模型。例如，如何收集上下文，以及如何在漫长的任务轨迹中优化 token 使用。

随着模型越来越擅长充当智能体，我们发现，预先提供更少的细节往往效果更好。这样可以让智能体更容易自行提取相关上下文。我们把这种模式称为“动态上下文发现”，以区别于始终包含在上下文中的“静态上下文”。

用于动态上下文发现的文件

动态上下文发现的 token 效率要高得多，因为只有必要的数据才会被拉入上下文窗口。它还可能提升智能体的响应质量，因为上下文窗口中容易造成困惑或相互矛盾的信息减少了。

Cursor 通过以下方式运用动态上下文发现：

  • 将过长的工具响应转为文件
  • 在摘要过程中引用聊天历史
  • 支持 Agent Skills 开放标准
  • 只高效加载所需的 MCP 工具
  • 将所有集成终端会话视为文件

1. 将过长的工具响应转为文件

工具调用返回的大型 JSON 响应，可能会使上下文窗口迅速膨胀。

对于 Cursor 的第一方工具，例如文件编辑和代码库搜索工具，我们可以通过设计合理的工具定义和精简的响应格式来避免上下文膨胀。不过，第三方工具并不会自然获得同样的处理，例如 shell 命令或 MCP 调用。

编程智能体通常会截断过长的 shell 命令输出或 MCP 结果。这可能造成数据丢失，而被删掉的内容中也许就包含你希望放入上下文的重要信息。Cursor 采用了另一种做法：我们把输出写入文件，并让智能体能够读取它。智能体会调用 tail 查看末尾内容。如果还需要更多信息，再继续读取。

这种做法减少了触及上下文限制时不必要的摘要操作。

2. 在摘要过程中引用聊天历史

当模型的上下文窗口被填满时，Cursor 会触发一次摘要操作。它会总结智能体截至当时完成的工作，并为智能体提供一个新的上下文窗口。

不过，摘要是对上下文的有损压缩。摘要完成后，智能体掌握的信息可能会退化。它也许已经忘记了任务中的关键细节。在 Cursor 中，我们把聊天历史作为文件，用它来提高摘要过程的质量。

当上下文窗口达到限制，或者用户决定手动执行摘要时，我们会向智能体提供历史文件的引用。如果智能体发现摘要缺少自己需要的细节，它就可以搜索历史记录并恢复这些信息。

3. 支持 Agent Skills 开放标准

Cursor 支持 Agent Skills。这是一项开放标准，可以用专门能力扩展编程智能体。Skills 与其他类型的 Rules 类似，都通过文件来定义。这些文件会告诉智能体，应该如何完成特定领域的任务。

Skill 还包含名称和描述。这些信息可以作为“静态上下文”放入系统提示词。智能体随后可以进行动态上下文发现，并利用 grep 和 Cursor 的语义搜索等工具提取相关 Skill。

Skill 还可以附带与任务相关的可执行文件或脚本。由于它们本身就是文件，智能体可以轻松找出某个 Skill 所需的内容。

4. 只高效加载所需的 MCP 工具

MCP 很适合访问受 OAuth 保护的资源。这些资源可能是生产环境日志、外部设计文件，也可能是企业内部的上下文和文档。

有些 MCP 服务器包含大量工具，而这些工具往往带有很长的说明。它们会显著挤占上下文窗口。即使大多数工具从未被使用，其说明也会一直包含在提示词中。如果使用多个 MCP 服务器，问题还会进一步叠加。

要求每个 MCP 服务器都自行完成这类优化并不现实。我们认为，减少上下文用量是编程智能体应该承担的责任。在 Cursor 中，我们把工具说明同步到一个文件夹，从而为 MCP 提供动态上下文发现能力。1

现在，智能体只会收到少量静态上下文，其中包括工具名称。这些信息会提示它在任务需要时查找相应工具。在一项 A/B 测试中，我们发现，在调用了 MCP 工具的运行中，这种策略让智能体使用的总 token 减少了 46.9%。这一结果具有统计显著性。不过，根据已安装 MCP 的数量不同，结果存在较大方差。

这种文件方案还让我们可以把 MCP 工具的状态传达给智能体。例如，过去某个 MCP 服务器需要重新认证时，智能体会彻底忘记那些工具，用户也会因此感到困惑。现在，智能体可以主动提醒用户重新认证。

5. 将所有集成终端会话视为文件

用户不再需要把某个终端会话的输出复制并粘贴到智能体的输入中。Cursor 现在会把集成终端的输出同步到本地文件系统。

这样一来，用户可以直接询问“为什么我的命令失败了？”，智能体也能理解用户指的是哪段输出。终端历史可能很长，所以智能体可以用 grep 只查找相关内容。这对于服务器等长期运行进程的日志尤其有用。

这种方式与命令行编程智能体看到的内容相似。后者会在上下文中看到之前的 shell 输出，但 Cursor 会动态发现这些内容，而不是把它们静态注入上下文。

简单的抽象

文件是否会成为基于大语言模型的工具最终采用的接口，目前还不明确。

不过，编程智能体正在快速进步。在这个过程中，文件一直是一种简单而强大的基础构件。与再造一种无法充分适应未来的新抽象相比，选择文件也更加稳妥。敬请关注，我们还会分享这一领域更多令人期待的工作。

这些改进将在未来几周内向所有用户开放。本文介绍的技术凝聚了多位 Cursor 员工的工作，其中包括 Lukas Moller、Yash Gaitonde、Wilson Lin、Jason Ma、Devang Jhabakh 和 Jediah Katz。如果你有兴趣利用 AI 解决最困难、最具雄心的编程任务，我们很乐意与你交流。请发送邮件至 hiring@cursor.com 联系我们。

1 我们考虑过采用工具搜索方案，但那会把工具分散在一个扁平索引中。我们最终选择为每台服务器创建一个文件夹，让同一服务器的工具在逻辑上保持成组。当模型列出文件夹内容时，它可以同时看到该服务器的所有工具，并把它们理解为一个完整单元。

文件还能支持更强大的搜索。智能体可以使用 rg 的完整参数，甚至可以用 jq 筛选工具说明。
