一句话总括

动态上下文发现

Dynamic context discovery

Cursor 的真正贡献，是把编码智能体的上下文从“预先塞满”改成“按需发现”。模型不必一开始看到所有内容，而是在需要时从文件化的工具输出、聊天历史、技能、MCP 描述和终端记录中检索。

Cursor Research 2026 年 1 月 6 日 作者：Jediah Katz
46.9% MCP 工具运行中的总 agent tokens 降幅。
5 文章列出的动态上下文实现路径。
1 个原则 少放静态上下文，多给可搜索入口。

更多上下文不等于更强智能体

传统做法倾向于把工具说明、历史记录和命令输出都放进提示词。这样会消耗 token，也会把无关或互相矛盾的信息带进模型视野。

Cursor 的做法更像给智能体一个可搜索的工作目录。默认只给少量线索，真正需要时再读取具体文件。

这改变了对 AI 编程能力的理解：长任务的质量不只取决于模型窗口大小，也取决于上下文是否能被可靠找回。

五种文件化上下文

1

长工具输出写入文件

第三方命令和 MCP 调用可能返回大量 JSON。Cursor 不简单截断，而是写入文件，让智能体用 tail 或继续读取来定位细节。这样减少了到达上下文上限时的无谓总结。

2

总结时引用聊天历史

上下文窗口满了以后，摘要会压缩信息，也会丢失细节。把聊天历史作为文件提供，允许模型在摘要后搜索遗漏内容，降低长任务失忆风险。

3

Skills 作为可发现能力

Agent Skills 用文件描述特定任务的做法。系统提示只保留名称和描述，具体说明、脚本和可执行文件按需搜索。

4

MCP 工具描述按服务器分组

许多 MCP 服务器有大量工具和长描述。Cursor 将工具描述同步到文件夹，只把工具名等少量静态信息给模型。A/B 测试中，这让调用 MCP 的运行减少 46.9% 总 agent tokens。

5

集成终端会话也变成文件

终端输出同步到本地文件系统后，用户可以直接问“命令为什么失败”。智能体能 grep 长日志，而不是依赖用户复制粘贴。

静态上下文与动态发现的差别

维度 静态上下文 动态上下文发现
Token 使用 提前注入大量信息，很多内容不会被使用。 只加载任务需要的部分，MCP 场景实测 token 降低明显。
信息完整性 长输出常被截断，关键错误可能消失。 完整输出留在文件里，模型可以继续读。
长任务连续性 摘要后容易忘记早期约束和细节。 聊天历史可搜索，摘要不再是唯一记忆。
工具生态 工具描述堆在提示词里，安装越多负担越重。 按服务器分组存放，状态和认证问题也能被模型发现。
文件不一定是 LLM 工具的最终界面，但它现在是一个简单、强韧、可搜索的抽象。 这说明编码智能体的进步不只是模型更强，也来自更朴素的工程接口。

这项设计解决的真正风险

截断造成的隐性数据丢失

当命令输出太长时，简单截断可能刚好删掉错误栈、失败原因或用户真正关心的行。文件化输出让模型可以回头查证。

摘要压缩造成的任务漂移

摘要会节省窗口，但它是有损的。把历史保留为可检索文件，让模型在不确定时恢复原始细节。

工具描述污染提示词

企业 MCP 可能连接日志、设计文件和内部文档。所有工具常驻提示词会迅速膨胀，也让模型更容易被无关工具干扰。

用户上下文需要手工搬运

终端历史同步后，用户不必复制粘贴失败输出。智能体能直接理解“刚才那个命令”的上下文。

最值得带走的洞见

上下文工程正在产品化

Cursor 把提示词外的文件系统变成智能体记忆层。这会影响所有长程编码任务。

省 token 也能提质量

少加载无关信息，模型更不容易被噪声带偏。46.9% 的 token 降幅说明收益不只是理论。

不要把截断当优化

截断节省窗口，却可能删掉最关键证据。可回读的文件比丢弃信息更稳妥。

未来的智能体会更像会搜索的工程师

它们不会记住一切，而是知道到哪里找。文件、历史、工具目录和终端记录共同构成工作记忆。

简单抽象仍然有优势

在工具接口尚未定型时，文件提供了兼容搜索、分组、脚本和状态传达的低成本方案。
