把"小时级"的代码库索引压成"秒级"，还不让你看到自己没有的代码

来源：Cursor 博客 · 作者 Jeremy Stribling · 2026-01-27

一句话总括：Cursor 发现"同一组织内同一代码库的克隆平均有 92% 相似"，于是不再为每个新人从零重建索引，而是用 Merkle 树 + 相似度哈希 + 内容证明安全复用队友的现成索引——把首次可查询时间从最长 4 小时压到 21 秒，同时用密码学保证客户端绝不会看到它本地没有的文件。

为什么值得做：语义搜索是 agent 性能的杠杆，但慢

除了准确率，语义搜索还让生成的改动更容易被代码库长期保留、整体请求满意度更高。但代价是：大仓库朴素索引要跑几小时，且要等索引完成至少 80% 才能用——新人入职、换机器时这段空窗反复出现。

真正要解决的不是"索引快不快"，而是"如何不为每个人重复支付同一份昂贵成本"。

1建立首个索引：用 Merkle 树只同步"变了的分支"

Cursor 给每个文件算加密哈希，目录哈希再由其子节点哈希推导，组成一棵 Merkle 树。本地小改只会改动该文件及其一路到根的父目录哈希；与服务端对比即可精确定位差异——哈希不同的同步、相同的跳过、客户端缺的从服务端删、服务端缺的补上，且同步过程从不修改客户端文件。

5 万文件工作区 仅文件名 + SHA-256 ≈ 3.2 MB 每次只走哈希不同的分支

变更文件被切成语法块，转成驱动语义搜索的 embedding；生成 embedding 是最贵的一步，所以放到后台异步做，并按块内容缓存，未变的块直接命中缓存。

Merkle 树把"全量传输"变成"差量遍历"，是后面一切复用与安全机制的密码学地基。

2找到可复用的索引：用 simhash 在向量库里"搜索一个代码库"

新人加入时，客户端从 Merkle 树派生出一个相似度哈希（simhash）——它是整库文件内容哈希的单值摘要。客户端上传 simhash，服务端把它当向量，在"同团队（或同用户）所有现有索引"的 simhash 向量库里检索；命中且超过阈值，就直接把那个索引当作新代码库的初始索引。

拷贝在后台进行；与此同时客户端已经可以对被拷贝的原索引发起语义搜索，于是首次查询几乎瞬时返回。前提是两条约束都成立：结果必须反映用户本地代码库（即使有差异），且客户端绝不能看到自己没有的代码。

"代码库相似度"被工程化成一个可向量检索的数，复用从经验直觉变成了可计算的匹配。

3证明你有权访问：用内容证明堵住跨副本泄露

安全性同样建在 Merkle 树上：树里每个节点都是其下内容的加密哈希，只有真正持有文件才能算出对应哈希。从拷贝索引启动时，客户端连同 simhash 一起上传完整 Merkle 树，为每个加密路径关联一个哈希；服务端把这棵树存成一组内容证明（content proof）。

检索时服务端拿客户端的树去过滤结果——证明不了持有某文件，该结果就被丢弃。客户端因此能立即查询、且只看到与拷贝索引共享的代码；后台同步补齐剩余差异，待两端树根一致后服务端删除内容证明，之后查询直接走完全同步的索引。

同一棵 Merkle 树既是提速工具又是访问控制凭据——安全不是事后补丁，而是复用机制自带的副产品。

4更快的上手：仓库越大，收益越夸张

复用队友索引对各种规模都有效，且效果随仓库规模放大：

分位原首次查询复用后
中位数仓库7.87 秒525 毫秒
P902.82 分钟1.87 秒
P994.03 小时21 秒

越大的代码库原本越痛，这套方案的收益越大——它消除的是一类反复发生的重复工作，而非一次性优化。

最值得带走的 5 条洞见

  1. 语义搜索是 agent 的性能杠杆：平均把回答准确率拉高 12.5%，并让改动更易被保留——值得为它的"可用速度"投入工程。
  2. "92% 相似"是整套设计的支点：同组织内代码库克隆高度相似，意味着为每人从零索引本质上是重复劳动，可以安全省掉。
  3. 一棵 Merkle 树身兼三职：差量同步（5 万文件只传约 3.2 MB 哈希）、相似度匹配（派生 simhash）、访问控制（内容证明），密码学结构被一物多用。
  4. 安全与提速并非取舍：用"只有持有文件才能算出其哈希"这一性质做内容证明，客户端能秒级查询却永远看不到自己没有的代码。
  5. 收益随规模指数级放大：P99 仓库从 4.03 小时降到 21 秒——对超大代码库，这是"能不能用"而不仅是"快不快"的区别。
