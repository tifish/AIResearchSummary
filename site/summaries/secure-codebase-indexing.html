安全地为大型代码库建立索引

Securely indexing large codebases

作者：Jeremy Stribling 分类：Research 阅读英文原文

一句话总括：Cursor 把团队成员已有的代码索引变成可安全复用的基础设施，让大型代码库的语义搜索在数秒内可用，同时避免泄露用户本地不存在的代码。

12.5% 语义搜索带来的平均回答准确率提升
92% 同一组织内代码库副本的平均相似度
80% 传统索引至少完成到这一比例后，语义搜索才可用

重复索引才是真正的瓶颈

大型仓库可能包含数万份文件。若每位成员都从零开始处理，首次查询会等待数小时。

Cursor 观察到，同一组织内的代码库副本平均有 92% 相似度。大部分计算其实是在重复处理相同内容。

因此，优化重点不只是让单次索引更快，而是识别最接近的已有索引并安全复用。这个变化把代码索引从个人任务转变为团队可以共享的计算成果。

复用流程：从相似度判断到完整同步

1. 构建 Merkle 树客户端计算文件和目录的加密哈希，只定位真正发生变化的分支。
2. 计算 simhash系统把整棵树压缩成代表代码内容的相似性哈希。
3. 选择已有索引服务器在同一团队或同一用户的索引中寻找超过阈值的匹配项。
4. 边查询边同步客户端立即查询共享内容，后台再补齐双方差异。

速度不能以代码泄露为代价

复用索引带来一个核心风险：旧索引可能包含新用户没有的文件。Cursor 要求客户端用 Merkle 树中的哈希证明自己确实持有文件。无法证明的搜索结果会被服务器直接丢弃。

这意味着用户可以提前使用正在复制的索引，却只能看到自己代码库中已有内容。这里的关键洞见是，内容证明让即时查询与访问隔离可以同时成立。

仓库越大，收益越明显

在普通仓库中，优化已经能被用户感知。到了超大型仓库，变化则是从“当天稍后才能用”变为“打开项目后很快就能用”。这改变了 AI 编程工具在大型企业代码库中的可用性边界。

Merkle 树在这里承担了两项工作：它既是高效增量同步的差异检测结构，也是防止跨副本泄露文件的访问证明。复用同一套加密基础，降低了系统复杂度。

语义搜索是提升代理表现的主要因素之一。在我们最近的评估中，它让回答准确率平均提高了 12.5%。借助语义搜索生成的代码修改也更有可能被保留在代码库中，用户对整体请求结果的满意度同样有所提升。

为了支持语义搜索，Cursor 会在你打开项目时为代码库建立可搜索的索引。小型项目几乎可以立即完成这项工作。但大型仓库可能包含数万份文件。如果采用简单直接的索引方式，整个过程可能耗时数小时。至少完成 80% 的工作后，语义搜索才可使用。

我们注意到，大多数团队成员使用的其实是同一代码库中近乎相同的副本。因此，我们开始寻找加快索引的方法。事实上，在同一组织的不同用户之间，同一代码库的副本平均有 92% 的相似度。

这意味着，当新成员加入团队或有人更换电脑时，我们不必每次都从零重建索引。系统可以安全地复用队友已有的索引。在规模最大的仓库中，这能把首次查询的等待时间从数小时缩短到数秒。

构建首个索引

Cursor 使用 Merkle 树建立对代码库的初始视图。它可以精确识别发生变化的文件和目录，无需重新处理全部内容。Merkle 树会记录每个文件的加密哈希。每个文件夹也有自己的哈希，而这个值由其子项的哈希计算得出。

客户端上的小范围编辑只会改变被编辑文件的哈希，以及从该文件所在目录一直到代码库根目录之间各级父目录的哈希。Cursor 会把这些哈希与服务器上的版本进行比较，从而准确找出两棵 Merkle 树出现分歧的位置。哈希不同的条目会被同步，相同的条目则会跳过。客户端缺少的条目会从服务器删除，服务器缺少的条目会被添加。同步过程绝不会修改客户端上的文件。

Merkle 树能显著减少每次同步所需传输的数据量。在一个包含五万份文件的工作区中，仅文件名和 SHA-256 哈希就约有 3.2 MB。如果没有这棵树，每次更新都需要传输这些数据。有了 Merkle 树，Cursor 只需遍历哈希发生变化的分支。

当文件发生变化时，Cursor 会按照语法结构把它拆成多个内容块。随后，系统会把这些内容块转换为支持语义搜索的嵌入向量。创建嵌入向量是成本较高的步骤，因此 Cursor 会在后台异步完成这项工作。

大多数编辑不会改变文件里的大部分内容块。Cursor 会根据内容块本身缓存嵌入向量。没有变化的内容块可以直接命中缓存，代理无需在推理时再次承担这项计算成本，回答速度也能保持稳定。最终得到的索引更新迅速，维护成本也很低。

寻找最适合复用的索引

对于 Cursor 从未见过的代码库，前述索引流程会上传其中的每一份文件。不过，同一组织中的新用户不需要再次经历整个过程。

新用户加入时，客户端会为新的代码库计算 Merkle 树，并由这棵树生成一个称为相似性哈希（simhash）的值。它是一个单独的值，用来概括代码库中各个文件的内容哈希。

客户端会把 simhash 上传到服务器。服务器随后把它作为向量，在一个向量数据库中执行搜索。这个数据库包含 Cursor 中同一团队其他索引的当前 simhash，也包括同一用户其他索引的当前 simhash。对于向量数据库返回的每项结果，我们都会检查它与客户端相似性哈希的匹配程度是否超过指定阈值。如果达到阈值，系统便把对应索引作为新代码库的初始索引。

复制会在后台进行。在此期间，客户端可以针对正在复制的原始索引发起新的语义搜索。因此，客户端几乎可以立即执行首次查询。

不过，这套方法成立需要满足两个条件。即使本地代码库与复制的索引存在差异，结果也必须反映用户的本地代码。客户端也绝不能看到自己尚未持有的代码所对应的结果。

证明访问权限

为了确保文件不会在代码库的不同副本之间泄露，我们再次利用了 Merkle 树的加密特性。

树中的每个节点都是其下方内容的加密哈希。只有持有对应文件，才能计算出这个哈希。当工作区以复制的索引作为起点时，客户端会把完整的 Merkle 树和相似性哈希一并上传。这样，代码库中的每条加密路径都会与一个哈希建立关联。

服务器会把这棵树保存为一组内容证明。在搜索过程中，服务器会对照客户端树中的哈希来过滤结果。如果客户端无法证明自己持有某份文件，相应结果就会被丢弃。

这样一来，客户端可以立即发起查询，但只能看到自身代码库与复制索引共有的代码。后台同步会继续协调剩余差异。当客户端与服务器的 Merkle 树根哈希一致后，服务器会删除这些内容证明。后续查询将直接针对已经完全同步的索引运行。

加快初次使用速度

复用队友的索引可以缩短各种规模仓库的设置时间。仓库越大，效果越明显：

  • 对于中位数仓库，首次查询时间从 7.87 秒缩短到 525 毫秒。
  • 在第 90 百分位，首次查询时间从 2.82 分钟缩短到 1.87 秒。
  • 在第 99 百分位，首次查询时间从 4.03 小时缩短到 21 秒。

这些变化消除了大量重复工作。即使面对非常庞大的代码库，Cursor 也能在数秒内理解其内容，而不再需要等待数小时。
