一句话总括

Cursor 把大型代码库索引从“重新构建”变成“安全复用”

文章真正的贡献是提出一套可落地的代码索引复用机制：用 Merkle tree 判断差异，用 simhash 找到相似索引，用内容证明保证搜索结果只来自用户已经拥有的文件。

Merkle tree 让索引更新只处理变化部分

Cursor 为每个文件和目录计算加密哈希，客户端和服务器只沿着哈希不同的分支同步。一个 5 万文件工作区，仅文件名和 SHA-256 哈希就约 3.2 MB，如果每次全量传输会浪费大量带宽。

这说明 AI 代码理解的基础设施不能只优化模型调用，还要把文件变化检测做成低成本、可验证的增量系统。

embedding 成本通过内容缓存被摊薄

文件变化后会被切成语法块，再转成支持语义搜索的 embeddings；这是昂贵步骤，所以 Cursor 在后台异步执行，并按 chunk 内容缓存结果。多数编辑不会改变多数 chunk，因此缓存命中能避免重复计算。

结论是，语义搜索的速度来自“少算”和“晚算”，而不是把所有索引工作都塞进用户等待时间。

simhash 利用团队代码库的高度相似性

文章指出，同一组织内同一代码库的克隆平均有 92% 相似度。新用户打开项目时，客户端从 Merkle tree 派生 simhash，服务器只在同团队或同用户的已有索引中寻找相似副本。

这改变了 onboarding 的假设：新机器不必从零理解仓库，可以先借用可信相似索引，再补齐本地差异。

内容证明把速度收益限制在权限边界内

复用索引的风险是用户可能看到自己没有的代码。Cursor 让客户端上传完整 Merkle tree，将每个加密路径与内容哈希绑定；搜索时服务器检查这些证明，客户端不能证明拥有的文件会被过滤。

关键洞见是，安全不是索引复用的附加层，而是让复用成立的前提。

最大的收益出现在最大仓库

原本语义搜索要等至少 80% 索引完成才可用，复用索引后，用户可以在后台复制和同步时先对共享部分发起查询。

中位数仓库 time-to-first-query7.87 秒 → 525 毫秒
90 分位仓库2.82 分钟 → 1.87 秒
99 分位仓库4.03 小时 → 21 秒

趋势很清楚：仓库越大，全量重建越不可接受，安全复用索引带来的体验改善越明显。

最值得带走的洞见

  • AI 编程工具的性能瓶颈很大一部分在上下文基础设施，而不只在模型本身。
  • 团队代码库天然高度相似，索引复用可以把重复计算变成一次构建、多次共享。
  • Merkle tree 同时解决增量同步和访问证明，是这套方案的核心结构。
  • 对大型仓库而言，把首次可查询时间从小时级降到秒级，会直接改变 agent 是否可用。
