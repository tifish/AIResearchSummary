使用 Auto-review 治理智能体自主性

Governing agent autonomy with Auto-review

一句话总括：Auto-review 把安全控制从固定权限开关变成上下文判断机制，让智能体在低风险任务中保持自主，同时在越过重要边界前主动减速。

12 小时用于建立正常智能体工作基线的内部开发会话。
6,122经过筛选、去重后的标注评测记录。
约 4%分类器运行时被阻止的受审操作比例。
约 7%Auto-review 对话中至少发生一次用户中断的比例。

真正的问题不是“危险吗”

同一条命令可能在一个工作流中完全正常，在另一个工作流中却不可接受。

风险取决于操作、用户意图以及出错后果之间的关系。孤立地查看命令，无法做出可靠判断。

关键变化：把权限开关改造成风险刻度盘

专用分类器会在工具调用执行前检查上下文。安全影响较小时，它会更宽松。后果较严重时，它会更谨慎。

这改变了我们理解 AI 安全控制的方式。安全不再等同于频繁弹出许可窗口，而是让审查强度随实际后果变化。

反复请求许可也会制造安全问题：当提示过多时，用户会停止认真阅读，批准动作也就失去了意义。

分类器如何进入执行链路

01 · 先理解意图

分类器判断下一步操作是否符合用户请求，而不是只匹配一份危险命令清单。

02 · 必要时检查工作区

对于 python script.py 这类无法只凭命令判断的操作，它可以使用 ReadFile、Grep、Glob 和 ListDir 检查内容。

03 · 把反馈交给父智能体

发生阻止时，父智能体可以缩小操作范围、换用其他工具，或绕开高风险步骤。用户不一定会被打断。

评测覆盖正常工作与罕见高风险场景

团队从约 12 小时的内部开发会话入手，最终得到 6,122 条标注记录。这些数据用于确认分类器不会妨碍常规开发。

内部数据很少包含最严重的失败，因此团队还生成了合成案例。它们覆盖读取机密、操作生产数据、遵循不可信指令和造成大范围副作用等情况。

这说明安全评测不能只看平均任务。低频但高后果的行为必须被主动纳入测试。

稳定性也是指标

如果同一案例运行十次，其中六次允许、四次阻止，通常说明策略或提示仍不明确。

重复运行用于发现这种“摆动”，并推动团队收紧规则。

阻止率低，不代表缺少保护

很多命令已经由允许列表或沙箱处理。分类器只需要审查那些必须结合上下文判断的操作。

分类器运行时约阻止 4% 的操作。大多数情况下，父智能体会依据反馈找到更安全的执行路径。

用户实际感受到多少中断

全球数据中，约 7% 的 Auto-review 对话至少发生一次中断。

部分合作企业此前约有 40% 的操作遭到阻止。两者口径不同，但对比显示，上下文审查有望显著减少无差别拦截。

最值得带走的洞见

自主性与安全不是二选一审查强度可以随风险变化。低后果操作应保持流畅，高后果操作才需要更强信号。
用户意图是判断核心命令本身不足以说明风险，授权范围和错误后果同样重要。
阻止不必等于打断先把原因反馈给父智能体，常能在不中断用户的情况下修正路径。
安全策略必须持续重做评测政策分类一旦变化，旧标签也可能过时。数据集、生产链路和重复运行需要同步更新。

要让智能体在编程和其他任务中发挥最大生产力，就需要给予它们适度的自主性。它们应该能够独立行动，发挥创造力，并完成工作，而不是频繁停下来请求许可。

但是，如果智能体采取了非预期行动，更大的自主性也会带来安全风险。对本地智能体来说尤其如此。它们通常运行在文件、凭据、环境变量和 MCP 工具附近，还可能访问生产系统。

最简单的办法，是要求智能体在采取任何行动之前都询问用户。但过于频繁地请求许可，也会造成另一种安全问题。反复看到提示后，人们会逐渐不再认真阅读，审批流程也会因此失去实际意义。

本周，我们推出了 Auto-review。它让智能体自主性的决策更像一个刻度盘，而不是一个开关。其核心思想是：风险较低时，智能体可以自由行动；下一步操作将越过重要边界时，它就应该放慢速度。

我们使用一个专门的分类器智能体，判断操作处于这条连续风险曲线上的什么位置。它会在操作执行前结合上下文进行审查。构建这个系统，意味着要把我们对智能体自主性治理的直觉，转化为一套可以实际运行的后果、意图与反馈模型，并用真实的智能体行为进行测试。

结合上下文判断风险

智能体操作是否存在风险，取决于具体情境。同一条命令在一个工作流中可能无害，在另一个工作流中却可能完全不可接受。真正重要的是操作、用户请求以及判断错误所造成后果之间的关系。

基于这一认识，我们开始开发一个负责治理智能体整体自主性的“分类器”智能体。我们希望使用小模型，让它保持快速且运行成本低。同时，它还需要细致判断下一步操作是否符合用户意图。

我们为分类器设定的核心规则是：安全风险较低时可以更宽松，风险较高时则要更谨慎。在这一总体原则下，我们开始把分类器构建成快速、可理解上下文的审查器，并将它直接放入智能体的执行链路。

构建分类器

第一个技术决策是选择模型。分类器会在工具调用执行前运行，因此它直接处于智能体循环之中。它既要准确，也必须足够快。作为一家采用多模型的公司，我们可以尝试多种模型和推理模式，再选择一个在速度与判断能力之间达到适当平衡的方案。

早期出现了一个出乎意料的发现：推理能力较弱的模型并不一定更快。如果模型难以理解策略或工具调用，它可能会花费更多时间和 token 寻找答案，最终得到的结果反而更差。更合适的取舍，是选择一个具备足够推理能力的小模型，让它可以干净利落地作出决定。

我们还赋予了分类器智能体式能力，因为有些操作无法只根据命令本身判断。以 python script.py 为例，它是否安全取决于文件内容。因此，分类器可以在决定前使用 ReadFile、Grep、Glob 和 ListDir 等工具检查工作区。

我们没有使用独立的分类接口，因为额外的往返请求会在每次受审工具调用前直接增加延迟。分类器改为在与父智能体相同的 RPC 流中运行，其架构与子智能体相似。

设计反馈闭环

下一个需要决定的问题是：阻止一次操作后，系统应该做什么。我们不希望分类器变成另一个不断生成审批提示的工具。当它阻止操作时，会向父智能体返回解释。父智能体通常可以利用这些反馈选择更安全的路径，而不必打断用户。

用户意图让这些反馈真正有用。需要判断的不是某项操作孤立看来是否危险，而是用户的请求能否证明该操作合理。这让正常开发工作可以继续推进，同时让后果更严重的操作需要用户给出更明确的信号。

只有使用那些应该放行和应该阻止的操作来调校分类器，这套设计才能奏效。因此，我们需要同时覆盖这两类行为的评测。

测试分类器

第一批评测来自内部使用数据，目的是了解智能体正常工作的基本形态。分类器必须发现危险操作，同时避免阻止日常开发工作。内部会话最适合帮助我们建立这条基线。我们最初收集了约 12 小时的内部开发会话，随后进行筛选，并对常见操作去重，最终形成 6,122 条标注记录。

我们还需要合成数据，因为最严重的情况在正常使用中出现得不够频繁。我们生成了一些案例，其中智能体可能读取机密、接触生产数据、遵循不可信指令，或执行具有大范围副作用的操作。这些样本覆盖了我们最希望分类器拦截的失败类型。

随着认识不断加深，策略也在变化，这让数据处理变得更加复杂。当我们调整分类器需要识别的行为类别时，就必须重新标注或重新生成评测集。否则，我们就是在使用已经过时的问题定义测试当前的分类器。

我们通过生产环境所使用的同一套后端分类器循环运行评测。这样可以测试完整链路，包括工具使用、最终分类、模型覆盖规则和解析失败。评测既会检查最终的允许或阻止决定，也会检查分类器在作出决定前需要查看工作区时所使用的上下文。

我们还会寻找判断摆动。如果同一个案例运行十次，其中六次允许、四次阻止，通常说明策略或提示定义得不够明确。重复运行可以帮助我们发现这些不稳定案例，并持续收紧分类器的规则，直到行为更加一致。

尽量减少直接阻止

实际使用中，只有一小部分智能体操作需要分类器审查。许多命令已经由允许列表或沙箱覆盖，因此分类器主要处理那些需要结合上下文判断的操作。

分类器运行时，目前会阻止约 4% 的操作。不过，一次阻止并不会立刻变成面向用户的提示。分类器会把解释发送给父智能体。父智能体通常可以缩小操作范围、选择其他工具，或者完全避开风险步骤。

分类器的部分阻止最终会打断用户。但从全球数据看，Auto-review 模式下只有约 7% 的全部对话至少发生过一次中断。作为对比，我们合作的一些企业客户此前在其组织内约有 40% 的操作会被阻止。

这些早期数据与我们希望实现的核心产品行为一致。分类器很少直接打断用户。在多数发生阻止的案例中，父智能体都能利用反馈，以更安全、更受限的方式继续工作。

持续完善智能体自主性

Auto-review 仍处于早期阶段。随着智能体能力增强，我们对自主性连续区间的理解也会继续变化。目前，它主要面向桌面应用中的本地智能体。我们预计，同样的思路将逐渐用于更多场景中的智能体自主性治理。

我们希望智能体拥有真正的自主性。同时，是否让它们放慢速度，应该取决于上下文，而不是某个统一的全局权限设置。分类器让我们能够提升安全性，又不必把自主性重新变成连续不断的审批提示。它会发现需要进一步审查的操作，向父智能体提供反馈，并在存在更安全路径时让智能体继续工作。

Auto-review 现在已成为新用户的默认设置。现有用户可以前往 Settings > Agents 启用它。

Auto-review 现在已成为新用户的默认设置。现有用户可以前往 Settings > Agents 启用它。
