一句话总括

用 Auto-review 治理智能体自主性

Governing agent autonomy with Auto-review

Cursor 的贡献不是再加一个确认弹窗，而是把本地 agent 的工具调用放进一套可评测的上下文风险回路里。

6,122 由约 12 小时内部开发会话去重后形成的标注评测行数。
4% 分类器运行时当前大约阻止的动作比例。
7% Auto-review 模式下产生至少一次用户中断的总聊天比例。
40% 部分企业客户过去在组织内看到的动作阻止比例。

风险不是命令本身，而是命令和意图的关系

同一条命令在不同任务里可能完全不同。读取脚本、运行脚本、访问文件，都要看用户到底要求 agent 做什么。

这改变了安全治理的重心。系统不只判断“这是不是危险动作”，而是判断“这一步是否被当前用户意图充分授权”。

为什么这很重要

本地 agent 靠近文件、凭证、环境变量、MCP 工具和生产系统。过度放权会带来真实风险。

但每步都询问用户也会失败。提示过多后，人会停止认真阅读，审批流程反而失去安全意义。

1

先看上下文

分类器在工具调用前运行，并可使用 ReadFile、Grep、Glob、ListDir 检查工作区。

2

判断后果

它把用户请求、下一步动作和错误后果放在一起评估，而不是只看命令字符串。

3

反馈给父 agent

被阻止时，分类器返回解释。父 agent 可以缩小范围、换工具或避开高风险步骤。

4

少打断用户

只有部分阻止会变成用户中断。目标是让低风险工作继续流动。

Auto-review 把权限从开关改成拨盘

三种做法的差异

治理方式 好处 代价 文章给出的方向
全局允许 agent 很顺畅。 高后果动作可能在用户没注意时发生。 不适合靠近本地文件和生产系统的 agent。
每步审批 表面上更可控。 提示疲劳会削弱真正的安全判断。 不能把安全完全外包给重复弹窗。
上下文分类器 低风险时放行，高风险时减速。 需要评测、标注、合成数据和稳定性调优。 更接近可持续的 agent 自主性治理。
核心洞见

真正有用的安全层不只是“挡住 agent”，而是教父 agent 找到更窄、更安全、仍能完成任务的路径。

评测让治理从直觉变成工程问题

Cursor 用内部使用数据建立正常开发基线，又用合成数据覆盖少见但高风险的场景，例如读取秘密、接触生产数据、遵循不可信指令或制造大副作用。它还反复运行同一案例来检查 flapping，因为同一案例时放行、时阻止通常说明政策或提示还不够明确。

小模型不等于低推理

分类器在工具调用前执行，所以速度很关键。文章的一个反直觉发现是，低推理模型并不总是更快。

当模型难以理解策略或工具调用时，它会花更多 token 搜索答案，并给出更差判断。因此更好的折中是小而有足够推理能力的模型。

架构选择

Cursor 没有使用单独的分类端点。额外往返会在每次被审查的工具调用前增加延迟。

分类器运行在父 agent 的同一 RPC 流里，架构类似 subagents。这说明安全机制必须嵌入执行路径，而不是作为外部慢服务。

最值得带走的洞见

自主性需要刻度

agent 不应只在“全放开”和“全审批”之间切换。更合理的方式是按后果调节速度。

上下文是安全边界

同一动作的风险取决于用户意图、当前任务和错误后果。

少打断也是安全能力

把阻止解释反馈给父 agent，可以减少无意义审批并保留控制力。

评测集必须跟政策一起演进

当分类类别变化时，旧标签会误导评测。安全策略需要持续重标或重建数据。

本地 agent 是高风险前沿

文件、凭证、环境变量和生产系统让本地执行比普通聊天更需要细粒度治理。
