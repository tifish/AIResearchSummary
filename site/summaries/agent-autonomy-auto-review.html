用 Auto-review 把 Agent 自主权做成"旋钮"，而不是"开关"

作者：David Gomes & Travis McPeak · 原文
一句话总括：Cursor 的真正贡献，不是又加了一道安全检查，而是用一个能读工作区、带轻量推理的小型"分类器智能体"， 在动作执行前判断它是否符合用户意图——从而在几乎不打断用户的情况下给本地 agent 装上可调节的安全闸。

01问题：自主与安全的两难，靠"多问一句"解决不了

Agent 要高效，就需要自主；但本地 agent 紧挨着文件、密钥、环境变量、MCP 工具甚至生产系统，自主越多风险越大。 最朴素的办法是"每个动作都先问用户"，但弹窗太多反而制造新的安全问题：

问得太频繁，人就不再认真看，审批流于形式。

洞见：安全感不等于安全。把"打断频率"本身当成一个要被优化的成本，是这套设计与传统权限弹窗的根本分野。

02核心判断：看的不是"动作危不危险"，而是"配不配得上用户的请求"

同一条命令在一个工作流里无害，在另一个里就不可接受。分类器被赋予的中心规则是： 安全代价低时更宽松，代价高时更谨慎。关键不是动作孤立看是否冒险，而是它是否被用户的请求所正当化。

洞见：把"意图一致性"作为裁决标准，让日常开发畅通无阻，只有真正高后果的动作才需要用户给出更明确的信号。

03工程取舍：小模型、能动手、零额外往返

分类器跑在每次工具调用之前，必须又快又准。一个反直觉的发现是——

低推理模型并不总是更快：它读不懂策略时会反复纠结，花更多 token 得到更差答案。

模型策略：小模型 + 足够推理
可调用工具：ReadFile / Grep / Glob / ListDir
部署：同一 RPC 流，无独立端点

它是"agentic"的：像 python script.py 这种命令，安全与否取决于文件内容，所以分类器会先检查工作区再裁决； 并与父 agent 共用 RPC 流（类似 subagent 架构），避免每次审查前都多一次网络往返带来的延迟。

洞见：安全组件必须"隐形"——延迟和成本一旦显现，就会被绕过或关掉。把它塞进 agent 主循环而非旁路，是能默认开启的前提。

04评测：真实数据定基线，合成数据补极端

来源：约 12 小时 内部开发会话
去重标注后：6,122 行
合成场景：读密钥/碰生产数据/跟随不可信指令

评测直接跑生产同款后端分类器回路，覆盖工具使用、最终判定、模型回退与解析失败的完整路径。 团队还专门盯"抖动"（flapping）：同一案例六次放行四次拦截，往往说明策略或提示词定义不清，需要收紧到行为一致。

洞见：策略一变就得重新标注/重建评测集，否则是在用过时的问题定义考核新分类器——安全策略的"数据债"是持续工程成本。

05效果：拦截很少，打断更少

分类器运行时拦截：约 4% 动作
含中断的对话：仅约 7%
对照（部分企业旧方案）：约 40% 动作被拦

多数动作早被允许列表或沙箱覆盖，分类器只在需要上下文判断时才出手。即便拦截，也是先把解释返回给父 agent， 后者通常能收窄动作、换个工具或干脆避开风险步骤，而不必惊动用户。

洞见：从约 40% 拦截到约 7% 对话才中断，量级差异说明"反馈给 agent 自我修正"比"反馈给人审批"在体验上是数量级的改进。

最值得带走的 5 条洞见

  1. 自主权应是连续旋钮：由上下文（后果×意图）决定是否减速，而非一个全局权限开关。
  2. 判据是意图一致性：问"这动作配不配得上用户的请求"，而不是"这动作看起来危不危险"。
  3. 小而能推理 + agentic：低推理未必更快；让分类器能读工作区、并嵌进主 RPC 流，才能既准又无额外延迟。
  4. 把打断当成本来优化：拦截先反馈给父 agent 自我修正，约 4% 拦截最终只让约 7% 对话产生中断。
  5. 安全策略有数据债：策略迭代必须同步重标评测集，并用重复跑测找"抖动"来保证行为稳定。
