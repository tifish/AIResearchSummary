Introducing GPT-5.3-Codex-Spark

一句话总括：OpenAI 用小型模型、专用低延迟硬件和端到端传输优化，把 Codex 的编程协作速度提升到接近即时响应。

真正的变化不是单纯“跑得更快”

Codex 多了一种实时协作模式

主线模型擅长持续数小时、数天甚至数周的自主任务。Codex-Spark 面向另一类工作：小范围改代码、调整逻辑、打磨界面，并立即看到结果。

它允许用户在执行过程中随时打断或改变方向。延迟降低后，人与模型之间的循环更像结对编程，而不是提交任务后等待结果。

结论：Codex-Spark 没有取代长时间自主执行。它补上了快速试验和即时反馈这一种工作方式。

速度也改变了默认行为

模型默认采用轻量工作方式。它倾向于进行最少且有针对性的修改。

除非用户明确提出要求，否则它不会自动运行测试。这能减少交互等待，但也意味着开发者需要主动要求验证。

因此，速度优势伴随着新的使用习惯：快速迭代之后，仍要安排测试和完整检查。

端到端延迟优化

模型速度只是实时体验的一部分

OpenAI 同时优化了客户端、服务器、推理栈和会话初始化。相关改进会惠及所有模型，而不仅是 Codex-Spark。

今天，我们发布 GPT‑5.3‑Codex‑Spark 的研究预览版。它是 GPT‑5.3‑Codex 的较小版本，也是我们首个专为实时编程设计的模型。Codex-Spark 是我们与 Cerebras 合作的首个里程碑。这项合作已于今年 1 月公布。Codex-Spark 针对超低延迟硬件进行了优化，力求带来接近即时响应的体验。它每秒可以生成超过 1000 个 token，同时仍能胜任真实的软件开发任务。

我们通过 Cerebras 向 ChatGPT Pro 用户提供 Codex-Spark 研究预览版，让开发者尽早开始试用。与此同时，我们将与 Cerebras 一起扩大数据中心容量，增强端到端用户体验的稳定性，并部署规模更大的前沿模型。

我们最新的前沿模型特别擅长执行长时间任务。它们可以在无人干预的情况下自主工作数小时、数天，甚至数周。Codex-Spark 则是我们首个专门面向 Codex 实时协作的模型。用户可以让它进行有针对性的修改、重塑逻辑或打磨界面，并立即看到结果。有了 Codex-Spark，Codex 既能承担长期而宏大的任务，也能处理当下需要立即完成的工作。我们希望了解开发者会如何使用它，并在持续扩大开放范围的过程中吸收反馈。

Codex-Spark 首发时提供 128k 上下文窗口，并且仅支持文本。在研究预览期间，它采用独立的速率限制。相关使用量不会计入标准速率限制。不过，当需求较高时，我们为了平衡所有用户的使用可靠性，可能会限制访问或临时安排排队。

速度与智能

Codex-Spark 针对交互式工作进行了优化。在这类工作中，延迟和智能同样重要。你可以实时与模型协作，在它工作时打断它或改变方向，并通过接近即时的响应迅速迭代。由于模型以速度为优化目标，Codex-Spark 的默认工作方式较为轻量。它会进行最少且有针对性的修改，也不会自动运行测试，除非你明确提出要求。

编程

Codex-Spark 是一款能力很强的小型模型，针对快速推理进行了优化。SWE-Bench Pro 和 Terminal-Bench 2.0 是两个评估代理式软件工程能力的基准测试。GPT‑5.3‑Codex‑Spark 在这两项测试中展现出强劲表现，同时只用了 GPT‑5.3‑Codex 完成相同任务所需时间的一小部分。

持续时间估算包含四部分：第一，输出生成时间，也就是输出 token 数除以采样速度；第二，预填充时间，也就是预填充 token 数除以预填充速度；第三，工具执行的总时间；第四，网络额外开销的总时间。

所有模型的延迟改进

在训练 Codex-Spark 的过程中，我们发现，模型速度只是实时协作的一部分。我们还需要降低整个请求与响应链路的延迟。为此，我们在运行框架中实施了端到端延迟优化。这些改进将惠及所有模型。具体来说，我们简化了响应在客户端与服务器之间的双向流式传输方式，重写了推理栈的关键部分，并重新设计了会话初始化流程。这样可以让第一个可见 token 更早出现，也能让 Codex 在连续迭代时保持响应灵敏。通过引入持久 WebSocket 连接，并对 Responses API 进行有针对性的优化，我们把每次客户端与服务器往返的额外开销降低了 80%，把单 token 的额外开销降低了 30%，并把首个 token 的等待时间降低了 50%。Codex-Spark 默认启用 WebSocket 通道。它很快也会成为所有模型的默认通道。

由 Cerebras 提供算力支持

Codex-Spark 运行在 Cerebras 的 Wafer Scale Engine 3 上。这是一款专为高速推理打造的 AI 加速器，为 Codex 提供以延迟为优先的服务层。我们与 Cerebras 合作，把这条低延迟通道加入与其他算力资源相同的生产服务栈。因此，它可以无缝用于 Codex，也为支持未来的模型奠定基础。

“GPT-5.3-Codex-Spark 最令我们兴奋的地方，是能够与 OpenAI 和开发者社区携手探索快速推理带来的可能性。这包括新的交互模式、新的使用场景，以及一种根本不同的模型体验。这次预览只是开始。”

GPU 仍是我们训练和推理管线的基础。在面向广泛用户提供服务时，GPU 也能以最具成本效益的方式生成 token。Cerebras 是对这一基础的补充。它擅长处理要求极低延迟的工作流，可以缩短端到端反馈循环，让你在迭代时感到 Codex 的响应更加灵敏。GPU 和 Cerebras 也可以在同一个工作负载中组合使用，以取得最佳性能。

使用范围与详情

Codex-Spark 于今天开始以研究预览形式逐步开放。ChatGPT Pro 用户可以在最新版 Codex 应用、CLI 和 VS Code 扩展中使用它。由于模型运行在专用低延迟硬件上，其使用量采用独立的速率限制。在研究预览期间，该限制可能根据需求进行调整。此外，我们还通过 API 向少量设计合作伙伴提供 Codex-Spark，以了解开发者希望如何把它集成到自己的产品中。未来几周，我们会继续在真实工作负载下调优集成方案，并逐步扩大开放范围。

Codex-Spark 目前仅支持文本，并提供 128k 上下文窗口。它是超高速模型系列中的第一款产品。随着我们和开发者社区进一步了解快速模型最适合哪些编程场景，我们将引入更多能力。这些能力包括更大的模型、更长的上下文窗口和多模态输入。

Codex-Spark 采用与主线模型相同的安全训练，其中包括与网络安全相关的训练。我们按照标准部署流程对 Codex-Spark 进行了评估。该流程包含针对网络安全和其他能力的基线评估。评估结果表明，它没有合理可能达到《Preparedness Framework》中有关网络安全或生物领域高能力的门槛。

下一步

Codex-Spark 是迈向双模式 Codex 的第一步。这两种模式可以相互补充：一种负责更长周期的推理和执行，另一种负责实时协作与快速迭代。随着时间推移，两种模式会逐渐融合。Codex 可以让你保持紧密的互动循环，同时把耗时更长的工作交给后台子代理。需要兼顾覆盖面和速度时，它也可以把任务分发给许多模型并行处理。这样一来，你不必事先只选择一种模式。

随着模型能力不断增强，交互速度正成为一个明显的瓶颈。超高速推理可以缩短反馈循环，让 Codex 用起来更加自然，也能为每一位希望把想法变成可用软件的人拓展可能性。
