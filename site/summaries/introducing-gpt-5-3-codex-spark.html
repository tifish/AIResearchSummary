GPT‑5.3‑Codex‑Spark：当编码 AI 开始把“快”当成核心能力

来源：OpenAI · 原文链接
一句话总括：Codex‑Spark 不是又一个“更聪明”的模型，而是 OpenAI 第一个为“实时编码”而生的模型——它把人机协作的瓶颈从智能转向延迟，用每秒超 1000 token 的速度，让 AI 编码从“提交任务、等待结果”变成“边写边改、即时迭代”。

它快到能让你实时打断和改方向

Codex‑Spark 是 GPT‑5.3‑Codex 的小型化版本，运行在 Cerebras 的 Wafer Scale Engine 3 上，输出速度 超过 1000 tokens/秒。这意味着你可以在它工作时随时打断、重定向，像和同事结对编程一样实时协作。

1000+tokens / 秒
128k上下文窗口
仅文本暂不支持多模态

结论：速度本身成为一种产品特性——当响应近乎瞬时，交互方式就会发生质变。

小模型，但编码能力没掉队

在评估 agentic 软件工程能力的两项基准 SWE‑Bench Pro 和 Terminal‑Bench 2.0 上，Codex‑Spark 表现强劲，而完成任务的耗时只是 GPT‑5.3‑Codex 的一小部分。它默认走轻量路线：只做最小化、有针对性的编辑，除非你要求，否则不自动跑测试。

结论：这是一次“用可接受的智能换取数量级速度”的取舍，且事实证明在真实编码场景里这笔交易划算。

真正的提速不只在模型，更在整条管线

OpenAI 发现，光让模型变快还不够，于是重写了推理栈、改造了会话初始化，并引入持久 WebSocket 连接和 Responses API 内部优化。这些改进会惠及所有模型，而不只是 Codex‑Spark。

‑80%每次往返开销
‑30%每 token 开销
‑50%首 token 时间

结论：实时体验是“端到端工程”而非单点突破——这提醒我们衡量 AI 体验时，延迟和智能同样关键。

Cerebras 与 GPU 互补，而非取代

这是 OpenAI 与 Cerebras 一月合作后的首个成果。GPU 仍是训练与推理的基础，提供最具成本效益的 token；Cerebras 则擅长极低延迟的工作负载，两者甚至可在单个工作负载中组合以达到最佳性能。Codex‑Spark 被接入与其余机群相同的生产服务栈。

结论：未来的推理基础设施是“异构组合”——按任务需求在成本与延迟之间动态选择硬件。

谁能用、以及下一步

目前作为研究预览，向 ChatGPT Pro 用户在最新版 Codex app、CLI 和 VS Code 扩展中开放，并向少量 API 设计合作伙伴提供。使用独立速率限制，高需求时可能排队。OpenAI 评估认为它在网络安全和生物领域均未达到 Preparedness Framework 的高能力阈值。它是“超快模型家族”的第一个成员，后续将带来更大模型、更长上下文和多模态输入。

结论：终局是 Codex 拥有两种互补模式——长时程自主推理与实时协作，并最终融合：主循环保持紧凑交互，把长任务委派给后台子代理。

最值得带走的洞见

  1. 编码 AI 的竞争维度正在从“更聪明”扩展到“更快”——当模型足够强，交互速度成为新的瓶颈。
  2. >1000 tokens/秒 改变的不是数字，而是交互范式：从异步“下任务等结果”变为同步“实时结对”。
  3. 体验提速是系统工程问题：WebSocket、推理栈重写带来的 80%/30%/50% 降幅，证明“快”要靠整条管线，而非单一模型。
  4. 异构硬件成为常态：GPU 管成本、Cerebras 管延迟，二者组合而非互斥，是大规模推理的新现实。
  5. OpenAI 正在押注“双模式 Codex”：长时自主 + 实时协作并最终融合，意味着开发者无需预先在“快”与“深”之间二选一。
