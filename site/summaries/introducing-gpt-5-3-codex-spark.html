一句话总括

Codex-Spark 的真正贡献，是把代码助手的体验从等待回复推进到接近实时协作。

OpenAI 发布 GPT-5.3-Codex-Spark 研究预览版，目标不是做最强的长任务模型，而是让开发者在改代码、调逻辑、迭代界面时能更快看到结果。

OpenAI 2026-02-12 Product ChatGPT Pro 研究预览

它补齐的是“即时编码”这一类工作流

OpenAI 将 Codex-Spark 描述为 GPT-5.3-Codex 的较小版本，也是首个专为实时编码设计的模型。它适合针对性修改、重塑逻辑和快速微调界面，而不是默认承担持续数小时、数天甚至数周的自主任务。

这改变了代码模型的使用重点：开发者不只是在等待代理完成任务，也可以像结对编程一样频繁打断、重定向和即时验证想法。

核心指标显示，低延迟本身成了产品能力

Codex-Spark 在 Cerebras 低延迟硬件上可达到每秒 1000+ token，当前支持 128k 上下文窗口，并且是 text-only。研究预览阶段有独立速率限制，不计入标准速率限制，但高需求时可能排队或受限。

1000+每秒 token，强调近实时生成
128k当前上下文窗口
text-only预览期暂不支持多模态输入

结论是，模型智能之外，交互速度正在成为开发工具是否好用的关键变量。

它在软件工程基准上强调“更快完成”，但原文没有给分数

OpenAI 称 Codex-Spark 在 SWE-Bench Pro 和 Terminal-Bench 2.0 上表现强劲，并能用 GPT-5.3-Codex 的一小部分时间完成任务。文章解释了耗时估算方式：输出生成、prefill、工具执行和网络开销共同构成总时长。

这里最重要的趋势不是某个未披露的分数，而是评估标准开始同时关注能力和端到端耗时；对实际开发者来说，能否快速完成可验证修改，比单纯回答得更长更重要。

OpenAI 优化了整条响应链路，而不只是换了更快模型

训练 Codex-Spark 后，OpenAI 发现实时协作的瓶颈不只在模型推理速度，也在客户端、服务器、推理栈和会话初始化。通过持久 WebSocket 连接和 Responses API 内部优化，OpenAI 降低了多项端到端延迟。

-80%每次客户端/服务器往返开销
-30%每 token 开销
-50%首 token 时间

这说明实时 AI 编程不是单一模型发布，而是模型、传输协议和产品运行时共同改造后的结果。

Cerebras 提供低延迟路径，GPU 仍承担广泛负载

Codex-Spark 运行在 Cerebras Wafer Scale Engine 3 上，用作 Codex 的低延迟服务层。OpenAI 同时强调，GPU 仍是训练和推理管线的基础，适合大规模、成本有效的通用 token 供给。

结论是，未来的 AI 基础设施可能不会由一种硬件包办所有任务，而是按工作负载组合 GPU 与专用低延迟硬件。

发布范围很窄，定位仍是研究预览

Codex-Spark 首先面向 ChatGPT Pro 用户，在新版 Codex app、CLI 和 VS Code extension 中开放；API 只提供给少量设计合作伙伴。OpenAI 表示会随着真实负载调优逐步扩大访问。

它还通过了标准部署流程中的网络安全和其他能力评估，OpenAI 判断其没有合理可能达到 Preparedness Framework 中网络安全或生物领域高能力阈值。

最终方向是实时协作与后台代理融合

OpenAI 设想 Codex 未来有两种互补模式：一边保持紧密的实时交互循环，一边把更长任务交给后台 sub-agents，或并行分发给多个模型。用户不必一开始就在“快”和“深”之间二选一。

这篇文章的核心洞见是：随着模型能力继续提高，人机互动速度会成为新的瓶颈，低延迟推理会直接扩大开发者能用 AI 完成的软件工作类型。

最值得带走的洞见

  • Codex-Spark 的重点是实时编码体验，而不是替代长时间自主执行的前沿模型。
  • 每秒 1000+ token、128k 上下文和 text-only 说明它优先服务低延迟文本代码场景。
  • 80%、30%、50% 的链路优化显示，AI 产品速度取决于完整系统，不只取决于模型。
  • Cerebras 与 GPU 的组合暗示未来推理基础设施会按延迟、成本和任务形态分层。
  • Codex 的长期方向可能是前台实时协作与后台多代理执行并存。
