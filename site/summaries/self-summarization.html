训练 Composer 应对更长的任务视野：把“压缩”变成模型自己学会的本事

来源：Cursor 博客 · 2026-06-29 收录 · 原文 2026-03-17 · 作者 Federico Cassano & Sasha Rush
一句话总括：Cursor 不再把“上下文压缩”当作 harness 在外部硬塞的补丁，而是把它折叠进强化学习训练循环，让编程模型 Composer 学会在合适时机自己总结上下文——这使它能从远超上下文窗口的长轨迹中获得训练信号，并更省 token 地完成数百步的复杂编程任务。

问题压缩是必需的，但传统压缩会“失忆”

CursorBench 内部基准显示：在真实困难编程任务上，更多的思考与代码库探索 = 更好的表现。但智能体的轨迹长度增长得比模型上下文窗口更快。

现有 harness 通常用两种方式压缩：文本空间里用提示式总结模型，或滑动窗口直接丢弃旧上下文（也有人探索更慢的隐空间向量压缩）。它们的共同缺陷是会丢掉关键信息，任务越长效果越差。

结论：压缩躲不掉，但“在模型外面做压缩”天生会遗忘——这正是要在训练里解决的。

方法自我总结：让模型在训练中就学会“该记住什么”

Composer 是在 Cursor 智能体 harness 内用强化学习训练的专用编程模型，因此可以做到“压缩在循环内”。运行流程是：

  1. 从提示开始生成，直到触及固定 token 长度阈值；
  2. 插入一个合成查询，要求模型总结当前上下文；
  3. 给模型“草稿空间”思考最佳摘要，再生成压缩后的上下文；
  4. 带着摘要 + 会话状态（计划、剩余任务、已总结次数等）回到第 1 步。

关键在于：同样的总结流程被放进训练。一次 rollout 可由多段生成靠摘要串联，摘要本身也会被奖励或惩罚——用最终奖励回传给链上所有 token，好摘要被加权、丢信息的差摘要被降权。技术上无需大改训练框架。

结论：压缩不再是外挂技巧，而成了模型可被训练、可被优化的内在行为；难题上它常会多次自我总结。

对照的是一个高度调优的提示式压缩基线：它的总结提示长达数千 token、含近十几个精心措辞的段落，输出压缩上下文平均超过 5,000 token。而 Composer 因为是被“训练出来”的，只需极短提示——内容差不多就是“请总结这段对话”——输出摘要平均仅约 1,000 token，因为它学会了按情境判断哪些是高价值信息。

在 80k 和 40k 两种触发阈值（后者总结更频繁）的受限环境下，自我总结在 CursorBench 上都明显更好且更省 token。

结论：把压缩交给模型自己，不仅更聪明，还更便宜、更快（可复用缓存）。

实证170 个回合、10 万 token 压成 1 千，解出 DOOM 难题

案例来自 Terminal-Bench 2.0 的 make-doom-for-mips：把 DOOM 源码编译成能在给定 vm.js 上运行的 MIPS 程序、并把每帧写成 bmp。题目描述简洁却极难，多个强力模型在官方成绩里都做不对。

一个早期研究版 Composer 正确解出了它：工作了 170 个回合，过程中写出紧凑、人类可读、结构化的自我摘要——把超过 100,000 token 压缩到它认为最有用的约 1,000 token。

结论：这正是压缩的终极承诺——让模型靠长推理链“一次性”啃下需要数百步的硬问题。

展望通往更长任务视野

把压缩折叠进训练循环，Composer 获得了一个显式的、把关键信息高效带向未来的机制。Cursor 把它视为迈向更长、更复杂流程（如多智能体协调）的一步，并预告 Composer 下一版本即将公布。

最值得带走的洞见

  1. 压缩应该是“训练目标”而非“外部补丁”。当摘要本身进入奖励信号，模型才会真正学会保留关键信息，而不是机械裁剪。
  2. “训练出来”的能力极其省事。基线要数千 token 的精雕提示，Composer 只需一句“请总结”就能产出更好、约 1/5 长度的摘要。
  3. 质量与效率可以同时拿。误差减半、token 用 1/5、还能复用 KV 缓存——并非通常的取舍关系。
  4. 长程能力是“硬题一次过”的前提。170 回合解出连强力模型都失败的 DOOM-MIPS 难题，证明把 10 万 token 压成 1 千仍能保住解题所需信息。
  5. 方向指向多智能体。自我总结只是起点，真正目标是让模型在更长、更复杂的协作流程中保持智能。
