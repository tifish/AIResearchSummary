面向更长任务周期训练 Composer

Training Composer for longer horizons

一句话总括：Cursor 把上下文压缩本身变成可受奖励的训练行为，使 Composer 学会在跨越上下文窗口的长期任务中主动保留决定成败的信息。

50%与精心设计的基线相比，自我总结把压缩造成的错误稳定降低一半。
1/5自我总结仅使用基线五分之一的 token，并可复用 KV cache。
170 轮Composer 为解决 make-doom-for-mips 执行的完整任务轮数。
100k → 1k超过 100,000 token 的经历被压缩成约 1,000 token 的高价值状态。

真正的瓶颈是遗忘

复杂编码任务需要更多思考和代码库探索，但代理轨迹增长得比模型上下文窗口更快。

传统压缩依赖外部总结模型、滑动窗口或尚且缓慢的潜在空间方法。这些方法可能丢失关键事实，使代理越接近任务后段，能力反而越弱。

结论：上下文长度并非唯一限制。压缩时保留了什么同样决定长期任务能否完成。

把总结训练成模型自己的能力

Composer 在达到固定 token 阈值时暂停，并总结自己的上下文。摘要连同计划状态、剩余任务和此前总结次数一起进入下一轮生成。

训练时，最终奖励会作用于整条生成链。成功轨迹中的回答和摘要都被提高权重。遗漏关键信息的摘要则会被降低权重。

洞见：摘要不再是代理外部的维护步骤，而是直接影响奖励的决策。

自我总结如何延长工作轨迹

1

Composer 从提示开始生成，直到触及固定的 token 长度阈值。

2

系统插入一条合成请求，要求模型总结当前上下文。

3

模型先在暂存空间思考，再生成凝练后的上下文。

4

摘要与会话状态一起回到生成流程，必要时重复多轮。

为什么重要：训练 rollout 可以由多段生成和多次摘要串联而成，不再局限于一次提示与一次回答。这让模型能从远长于最大上下文窗口的轨迹中获得训练信号。

“更长上下文”不只是塞入更多历史。真正有效的长期能力，是模型知道哪些历史必须带到下一阶段。

更短的摘要，反而保留了更多任务价值

提示式压缩基线

总结提示长达数千 token，包含近 12 个精心措辞的栏目。压缩后的上下文平均仍超过 5,000 token。

结构很全面，却未必能针对当前任务判断信息价值。

Composer 自我总结

提示基本只有“请总结这段对话”。输出平均约 1,000 token，由模型结合当前任务选择保留内容。

在 80k 与 40k 两种阈值下，它都取得更好的 CursorBench 结果。

结论：自我总结把压缩错误稳定降低 50%，token 用量只有基线的五分之一。这说明压缩质量取决于任务相关性，而不是摘要长度或栏目数量。

170 轮任务证明了什么

Terminal-Bench 2.0 的 make-doom-for-mips 要求代理改造 Doom 源码，生成 MIPS 可执行文件，并在给定虚拟机环境中正确运行。

多个强模型在官方结果中未能完成这道题。Composer 的早期研究检查点则通过大量编码、测试和替代方案探索，找到了准确解法。

关键轨迹

170 轮操作构成了一次连续解题过程。

模型多次自我总结，把超过 100,000 token 的历史压缩为它认为最有助于完成任务的约 1,000 token。

洞见：这不只是 benchmark 分数提升。它显示单个代理可以通过持续压缩状态，完成过去容易因上下文耗尽而中断的工程任务。

最值得带走的洞见

压缩应当参与训练。
只有当摘要也接受最终任务奖励，模型才会学会保留真正决定结果的信息。
更短不等于损失更大。
约 1,000 token 的任务感知摘要胜过超过 5,000 token 的固定结构摘要。
长期能力可以通过循环构建。
生成、总结、恢复状态并继续生成，使训练轨迹能够跨越单个上下文窗口。
下一步不只是一人一代理。
Cursor 将这种机制视为迈向多代理协作等更长、更复杂过程的一步。

我们通过一种名为“自我总结（self-summarization）”的强化学习过程，训练 Composer 执行长期任务。把自我总结纳入 Composer 的训练后，我们便能从远长于模型最大上下文窗口的轨迹中获得训练信号。这让 Composer 能够学习处理需要数百次操作的高难度编码任务。

压缩技术的局限

在内部 benchmark 套件 CursorBench 中，我们观察到：面对高难度的真实编码任务，更好的表现与更多思考和代码库探索直接相关。随着用户与代理合作处理难度更高、目标更宏大的任务，我们预计思考和探索带来的收益还会继续增加。

但这里存在一个主要挑战：代理轨迹的增长速度超过了模型上下文长度的增长速度。许多代理运行框架尝试把压缩作为工作流的中间步骤，以绕过这一限制。当代理触及上下文上限时，运行框架会把上下文转换成更短的形式，再让代理从中断处继续生成。

实际应用中的压缩通常由运行框架通过两种方式之一完成。第一种是在文本空间中使用带提示词的总结模型。第二种是采用滑动上下文窗口，让模型丢弃较早的上下文。研究人员也开始探索潜在空间中的压缩方法。在这种方法中，模型用向量而非文本记忆上下文。不过目前，这些方法比基于文本的方法慢得多。

这些压缩方法有一个共同缺点。它们可能导致模型遗忘上下文中的关键信息，从而降低模型在长期任务推进过程中的效能。

把自我总结训练成一种行为

Composer 是专门面向代理式编码设计的模型。它通过 Cursor 代理运行框架中的强化学习进行训练。因此，我们可以在训练期间就把压缩纳入循环，提升 Composer 判断哪些信息最关键、应当总结和保留的能力。

Composer 处理任务时，会逐渐接近一个固定的上下文长度触发点。到达该位置后，它会暂停并总结自己的上下文，然后继续工作。更准确地说，自我总结过程如下：

  1. Composer 从提示开始生成，直到达到固定的 token 长度触发点。
  2. 我们插入一条合成查询，要求模型总结当前上下文。
  3. 模型获得一块暂存空间，用于思考最佳总结方式，然后生成凝练后的上下文。
  4. Composer 带着凝练后的上下文回到第一步。该上下文包含摘要和会话状态，例如计划状态、剩余任务和此前总结的次数等。

为了让 Composer 在推理阶段做好这件事，我们在训练中采用相同的总结流程。每次训练 rollout 可以包含多段由摘要串联起来的生成，而不再只是一个提示与一个回答的组合。这意味着，自我生成的摘要本身也会参与奖励计算。

从技术角度看，这不需要对训练过程进行重大改动。我们把最终奖励用于生成链中由模型产生的所有 token。这会提高优质轨迹中代理回答的权重，也会提高那些帮助任务成功的自我总结的权重。与此同时，遗漏关键信息的劣质摘要会被降低权重。随着训练推进，Composer 会学会利用自我总结过程构建更长的上下文。面对困难样本时，它经常会进行多次自我总结。

高 token 效率的压缩

为了测试自我总结，我们将它与一套经过高度调优、由提示词驱动的压缩基线进行比较。我们在一组高难度软件工程任务上研究这个问题，同时改变压缩触发阈值。

在基线压缩方案中，总结提示本身长达数千 token。它包含近 12 个经过仔细措辞的栏目，用于说明摘要应当保留哪些内容。压缩后的上下文平均也超过 5,000 token，其中包含许多结构化栏目，用来描述上下文中的关键信息。

相比之下，因为 Composer 已经接受过自我总结训练，所以只需要非常短的提示。提示的内容基本不会比“请总结这段对话”多多少。它输出的摘要平均只有约 1,000 token，因为它已经学会结合当前上下文，判断哪些高价值信息值得保留。

我们在两种上下文受限的测试环境中测试 Composer，以衡量自我总结的影响。一种环境采用 80k token 触发阈值。另一种采用 40k token 触发阈值，这意味着总结会更频繁。在两种场景下，自我总结都以 token 效率高得多的压缩方式，在 CursorBench 上取得了显著更好的结果。即使与针对性很强的基线方案相比，自我总结仍能把压缩导致的错误稳定降低 50%。它只使用五分之一的 token，并且可以复用 KV cache，也就是此前 token 所产生并存储的中间计算结果。

解决高难度问题

压缩技术更大的前景，是让模型能够一次完成需要很长推理链的困难问题。在目前对 Composer 2 的训练中，我们经常看到这种情况。这里以 Terminal-Bench 2.0 中一道名为 make-doom-for-mips 的题目为例。这道题描述得很简洁，但难度很高：

我已经提供了 /app/doomgeneric/，其中包含 Doom 的源代码。我还编写了一个特殊的 doomgeneric_img.c，希望你使用它。这个文件会把绘制出的每一帧写入 /tmp/frame.bmp。最后，我还提供了 vm.js。它会查找名为 doomgeneric_mips 的文件并运行。请自行完成剩下的工作……

这个问题虽然很容易描述，但挑战性很高。根据官方公布的结果，多个强大的模型都无法正确完成它。

在测试 Composer 的一个早期研究检查点时，我们发现它能够正确解决这个问题。整个解法需要编写和测试大量代码，还需要探索若干替代实现。下面是解决问题期间渲染出的一张图像：

迈向长期任务的未来

把压缩纳入训练循环后，Composer 学会了一套明确机制，能够高效地把关键信息传递到后续阶段。这也让它更有能力应对高难度任务。自我总结研究是我们迈向更广泛目标的一步。我们希望训练 Composer 执行更长、更复杂的过程，例如多代理协调。我们持续观察到，更好的模型训练能够拓展这些代理系统的适用范围，并提升其智能水平。
