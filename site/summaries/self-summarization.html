把"上下文压缩"训进模型：Composer 如何学会自我总结

来源：Cursor ｜ 作者：Federico Cassano & Sasha Rush ｜ 2026-03-17 ｜ 原文
一句话总括

长程 Agent 的真正瓶颈是轨迹比上下文窗口长得更快。Cursor 的贡献不是又造了一个更聪明的压缩提示词，而是把"压缩"折进强化学习训练循环——让 Composer 把"如何总结自己的上下文"也当成被奖励的行为来学，从而能用远超窗口长度的超长轨迹做训练，并胜任需要数百步动作的硬任务。

01问题：轨迹膨胀比上下文窗口快，而现有压缩都在"丢信息"

在内部基准 CursorBench 上，Cursor 观察到一个直接相关：在高难度真实编码任务上表现更好，几乎总伴随更多的思考与代码库探索。任务越雄心勃勃，思考与探索的回报越高。但麻烦也随之而来——Agent 轨迹膨胀的速度快过模型上下文长度的增长。多数 harness 用"压缩"绕过这堵墙：撞到上下文上限时，把上下文变短再续写。常见做法有两类——文本空间里用提示式总结模型，或滑动窗口直接丢弃更早的上下文；也有研究者尝试在隐空间里用向量而非文本记忆上下文，但目前比文本法慢得多。它们有个共同硬伤：容易遗忘上下文里的关键信息，任务越长，效力越打折扣。
洞见：压缩本身不是创新点，"压缩会丢信息"才是真问题——当返回随思考与探索递增时，每一次遗忘都在直接吃掉模型在硬任务上的潜力。

02解法：把压缩当成"被训练出来的行为"，而非外挂步骤

Composer 是专为 Agentic 编码设计、在 Cursor 的 Agent harness 里用强化学习训练的模型，这让它能做到压缩在训练循环之内（compaction-in-the-loop）。具体的自我总结流程是：① 模型从提示开始生成，直到触及一个固定的 token 长度阈值；② 系统插入一个合成查询，要求模型总结当前上下文；③ 模型获得思考的草稿空间，先想清楚最佳总结，再产出一段浓缩上下文；④ 带着这段浓缩上下文（含总结加上会话状态：计划状态、剩余任务、已总结次数等）回到第①步循环。关键在于：训练时用的是同一套总结流程——一次 rollout 可由多次"生成—总结"链接而成，于是总结本身也进入了被奖励的范围。
洞见：把推理时要做的事原样搬进训练，模型学到的就不再是"听话照做总结"，而是在自己的轨迹上学会判断什么值得带走——这正是它能比人工提示更省、更准的根因。

03机制：用最终奖励同时给"答对"和"总结对"加权

技术上几乎不需要改训练框架。Cursor 把最终奖励施加到链条里模型产出的所有 token上——好轨迹里既给 Agent 的回答加权，也给那些让它成功的自我总结加权；反过来，丢失了关键信息的糟糕总结会被降权。随着训练推进，Composer 学会用这套自我总结去构建更长的上下文；对于难题，它常常多次自我总结。
洞见：不必给总结单独设计奖励信号——让最终成败回流到沿途每一次总结，好坏自然分化。这是把复杂行为训进模型的一种极简而有效的信号设计。

04结果：误差砍半，token 只用五分之一，还能复用 KV cache

对照组是一个高度调优的提示式压缩基线：它的总结提示词长达数千 token、包含近十几个精心措辞的小节来规定"该保留什么"，输出的压缩上下文平均也超过 5,000 token、塞满结构化小节。而 Composer 因为是被训练出来的，只需一句极短提示（内容不比"请总结对话"多多少），产出的总结平均仅约 1,000 token——它学会了在语境中自己判断哪些是高价值信息。在 80k 与 40k 两种 token 触发阈值（后者意味着更频繁地总结）的受限环境里测试，自我总结都在 CursorBench 上给出明显更好的结果，且压缩更省 token：将压缩误差稳定降低 50%，只用五分之一的 token，并能复用 KV cache（此前 token 的中间计算缓存）。
洞见：更短的提示 + 更短的输出 + 还能命中缓存——三者叠加意味着质量和成本不是此消彼长：训练出的内生能力同时赢下了准确度和效率两个维度。

05案例：170 回合一气呵成，把 10 万 token 浓缩进 1,000

压缩更大的承诺，是让模型一次性啃下需要长推理链的硬问题。在 Composer 2 的训练中这种情形已常出现。案例取自 Terminal-Bench 2.0 的 make-doom-for-mips：题面简短却极难——给定 doom 源码、一个把每帧写成 /tmp/frame.bmp 的特制 doomgeneric_img.c、以及期待 doomgeneric_mips 文件并运行它的 vm.js，"剩下的你自己想办法"。数个强力模型在官方数据里都做不对。而一个早期研究检查点的 Composer 解对了：它需要编写并测试大量代码、还探索了若干替代实现。最终它工作了 170 个回合求得精确解，途中以紧凑、人类可读、结构化的形式不断自我总结，把超过 100,000 token 压缩到它认为最有助于解题的约 1,000 token。
洞见："几个强模型都做不对、它做对了"的差距，恰恰落在能否在百回合尺度上把关键信息无损地搬下去——长程能力正在成为区分模型的新维度，而不只是单步推理强弱。

06方向：通往更长程的未来与多 Agent 协同

把压缩折进训练循环后，Composer 学到的是一套显式机制：高效地把关键信息向前携带，从而在硬任务上更强。Cursor 把自我总结视为通向更宏大目标的一步——在更长、更复杂的流程（如多 Agent 协同）上训练 Composer，并相信更好的模型训练会持续扩大这类 Agentic 系统的能力边界与智能。文末还预告：下一代 Composer 即将揭晓。
洞见：当单模型的长程能力被训练攻克，下一道墙就是多个 Agent 之间的上下文与协调——自我总结提供的"携带关键信息"原语，正是迈向那一步的可复用基础。

最值得带走的 5 条洞见

  1. 瓶颈是轨迹增长快过上下文窗口。而在 CursorBench 上，更强的硬任务表现几乎总伴随更多思考与探索——所以"如何长时间不丢信息地工作"成了核心约束。
  2. 把压缩从 harness 技巧变成内生能力。Composer 在训练时就用同一套"生成—总结"流程，总结本身被纳入奖励；模型学会的是判断"什么值得带走"，而非照搬一段提示。
  3. 极简的奖励设计就够了。把最终成败回流到链条中的每一个 token，好总结被上调、丢信息的坏总结被下调，无需为总结单设信号。
  4. 质量与成本可以同时赢。更短提示 + 平均约 1,000 token 的总结（基线 >5,000）+ 复用 KV cache，让压缩误差砍半的同时只花五分之一 token。
  5. 长程能力是模型的新分水岭。170 回合、10 万 token 压成 1,000 解出他模型做不对的难题——下一站是把同样的"携带关键信息"原语推广到多 Agent 协同。
