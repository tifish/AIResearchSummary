一句话总括

Composer 的贡献是把上下文压缩训练成模型自己的长期记忆策略

这篇文章真正要说明的是：长程编码代理不只需要更大的上下文窗口，还需要在训练中学会哪些历史信息会影响最终任务成败。

50%自我总结约减少一半压缩错误
1/5相比强提示基线使用约五分之一 token
170 轮案例任务中完成长链路工程求解

长任务失败的关键不是不会总结，而是总结不参与学习

传统代理通常在上下文快满时由外部系统压缩历史：要么用提示词让另一个摘要过程改写文本，要么滑动窗口丢掉旧内容。文章指出，这些方法都会在长任务中遗失关键线索，尤其是软件工程任务里那些早期调查、失败尝试和局部约束。

这改变了对 AI 编码影响的理解：模型的长期能力不是单纯由上下文长度决定，而是由它能否持续保留“会影响后续行动的信息”决定。

自我总结被放进强化学习轨迹后，摘要本身也会被奖励

Composer 在接近固定 token 阈值时暂停，收到一个合成摘要请求，先思考应保留什么，再生成压缩上下文；压缩后的状态包含摘要、计划状态、剩余任务和已总结次数等信息，然后继续生成。

训练时，Cursor 把多个由摘要串起来的生成步骤视为同一条 rollout，并用最终奖励作用到链条里的所有 token。好的摘要因为帮助最终成功而被上调权重，丢失关键事实的摘要则被下调。

结论是，压缩不再是推理时的补丁，而变成了模型可学习、可优化的行为。

更短的自我总结反而比复杂提示压缩更有效

文章将自我总结与一个高度调优的提示式压缩基线比较。基线提示有数千 token，包含近十个精心设计的保留项，输出压缩上下文平均超过 5,000 token；Composer 只需要接近“请总结对话”的短提示，平均输出约 1,000 token。

在 80k 与 40k token 触发阈值下，自我总结都在 CursorBench 上表现更好，约减少 50% 压缩错误，同时使用约 五分之一 token，并能复用 KV cache。

这说明更长、更细的摘要模板未必带来更可靠的长期能力；经过训练的模型可以用更少文字保留更高价值的信息。

真实工程案例显示，长程代理需要跨越大量探索与失败尝试

文章用 Terminal-Bench 2.0 的 make-doom-for-mips 任务作为案例：模型需要处理 Doom 源码、生成目标文件 doomgeneric_mips，并让 vm.js 运行输出帧图像。作者称该题在官方数字中难倒了多个强模型。

Composer 的早期研究检查点完成了这个任务，总共工作 170 轮，过程中把超过 100,000 token 的轨迹压缩成约 1,000 token 的人类可读结构化摘要。

这个例子强调，AI 编码代理的价值会越来越体现在“长时间保持问题状态并迭代工程方案”，而不是一次性写出短答案。

这项工作指向更长周期的代理训练，而不只是更大的上下文窗口

Cursor 将自我总结描述为走向长程代理的一步，未来目标包括更复杂过程和多代理协作。核心路线是把代理运行中真正会出现的上下文压缩、状态延续和错误恢复纳入训练，而不是只在推理框架外部修补。

最终洞见是：如果 AI 系统要承担更复杂的软件任务，训练目标必须覆盖“如何记住并继续工作”这一过程能力。

最值得带走的洞见

  • 长程编码能力的瓶颈之一是上下文压缩质量，而不是单纯的上下文窗口大小。
  • 当摘要参与强化学习奖励后，模型会学会为最终成功保留关键信息。
  • Composer 的自我总结平均约 1,000 token，却优于超过 5,000 token 的复杂提示式压缩基线。
  • 在 40k 和 80k token 触发环境下，压缩错误约减少 50%，说明该方法对更频繁压缩也有韧性。
  • 长程代理的下一步竞争点，是能否跨很多轮探索、测试和修正持续推进真实工程任务。
