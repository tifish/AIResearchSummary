Project Fetch：第二阶段

Project Fetch: Phase two

一句话总括：Claude 已从辅助非专家编写机器人程序，进展到能以远超人类团队的速度独立完成多数感知和控制开发任务，但精细的闭环物理控制仍未解决。

9 分 35 秒Opus 4.7 完成两支人类团队都完成的 4 项任务所需的平均时间。
37.7×相对于无 Claude 团队的速度提升。相对于 Claude 辅助团队则为 18.9 倍。
1,045 行Opus 4.7 生成的代码量。Claude 辅助团队生成了 10,309 行。

最大变化：人类不再负责结对编程

2025 年的第一阶段表明，Claude 能显著提升非机器人专家的表现。不到一年后，Opus 4.7 已能独立完成当时由人与模型共同完成的大部分工作。

凡是至少有一支人类团队完成的任务，Opus 4.7 都快了至少 10 倍。这意味着模型的角色正从“提供建议”转向“直接完成工具接入和程序开发”。

这种提升并非来自专门的机器人训练。研究者将其归因于更通用的模型规模扩展。因此，机器人能力可能会随着基础模型整体进步而继续快速变化。

能力迁移的三个阶段

  1. 模型帮助人类
    Claude 为缺少机器人经验的员工提供知识、方案和代码支持。
  2. 人类帮助模型
    研究者接好设备、输入提示、批准命令，并允许模型进入下一项任务。
  3. 模型独立完成
    Opus 4.7 自行选择传感器接口方案、编写代码，并完成多数实验目标。

速度突破没有消除物理控制瓶颈

Claude 可以让机器狗移动到球后方，也能摆好撞球的位置。但它无法根据球的偏移持续修正动作。最终的自主取球仍未成功。

这类任务需要快速形成闭环：感知误差，理解误差与上一条命令的关系，重新判断球的位置，再调整下一次输入。人类经过练习后擅长这种反馈控制，Claude 在第二阶段仍缺少同等的细腻程度。

能力实验表现带来的认识
传感器接入与程序开发快速且多次试验的用时较稳定现成硬件的高层操作正在进入模型能力范围
检测沙滩球成功，但曾选用过时算法方案并非总是最优，但模型能绕过问题
精确推动沙滩球控制不佳，未能完成高层代码能力不等于解决低层控制与实时反馈
Opus 4.7 在 5 项任务上的三次试验平均用时为12 分 7 秒。Claude 辅助团队用时为 264 分钟。速度差距说明，有限用途的“物理智能体 AI”可能已经进入早期阶段。

最值得带走的洞见

1. 能力跨越发生得很快不到一年，模型就从无法自行连接机器人，发展到在多数任务上远超人类团队。
2. 更少代码也能取得更好结果Opus 4.7 只生成 1,045 行代码，却达到或超过两支人类团队的成功程度。
3. 可靠性存在明确边界能力范围内的任务表现稳定，但涉及实时反馈和精细物理动作时仍会失败。
4. 通用扩展正在外溢到物理世界这些进步不是机器人专项研发的结果。模型使用软件工具的演进轨迹，可能正在硬件领域重现。

完整译文

Michael Ilie、C. Daniel Freeman 和 Kevin K. Troy

2025 年 8 月，我们开展了一项实验，想看看 Claude 能在多大程度上帮助 Anthropic 员工使用一台市售四足机器人，完成复杂而又有趣的任务。这些员工并不是机器人专家。下文将这台机器人称为“机器狗”。我们把这项实验称为 Project Fetch。

实验发现，当时最先进的模型 Claude Opus 4.1 帮助其中一支团队显著超过了另一支团队。后者只能依靠互联网和自己的聪明才智。使用 Claude 的团队完成了更多工作，而且速度更快。

在把同事们拉到仓库参加实验之前，我们反复确认过 Opus 4.1 能否完全独立完成这些任务。答案毫无疑问：不能。它与没有 Claude 帮助的人类团队很相似，一开始就卡在了如何连接机器人的准备工作上。

然而，AI 模型进步得非常快。它们的进步速度甚至超过了去年 8 月那只失控的机器狗。当时，它差点撞上一支人类团队。

我们认为，现在应该重新开展 Project Fetch，看看新模型能否超过上一代。结果不仅如此。Claude Opus 4.7 在没有人类协助的情况下，完成参与者不到一年前做过的全部可比任务时，速度约为最快人类团队的 20 倍。

这并不意味着 LLM 已经解决了机器人技术问题。远非如此。最新的 Claude 模型在使用机器人精确移动沙滩球时仍然困难重重。这正是 Project Fetch 中真正的“取回”环节。

而且，这些实验中的任务都没有涉及更困难的机器人低层控制问题，例如针对特定动作开发控制策略。不过，我们再次看到了同一种演进模式。首先，模型帮助人类。然后，人类帮助模型。最后，模型基本能够独立完成任务。我们已经在网络安全领域观察到这种现象。现在，同样的动态正开始出现在 AI 与物理世界的交叉地带。

我们做了什么？

最初的 Project Fetch 将 Anthropic 员工随机分成两组。一组可以使用 Claude，另一组不能使用。两组需要依次完成以下步骤：

  • 使用制造商提供的控制器操作机器狗；
  • 连接机器狗的视频和激光雷达传感器；
  • 编写并运行一个手动控制机器狗的程序；
  • 设计一种方法，监测机器狗在空间中的移动路径；
  • 编写检测沙滩球的程序；
  • 最后整合所有功能，让机器狗自主取回沙滩球。

在这次自主运行的后续实验中，我们无法要求 Claude 使用实体控制器。我们也没有评估研究者使用 Claude 编写的控制器取回沙滩球需要多长时间，不过我们确认了这个控制器能按预期工作。

对于剩余的任务子集，我们使用 Opus 4.7 进行了三次试验。模型在 Claude Code 中启用了自适应思考，并将 effort 设置为最高。我们测量了每项目标的实际耗时，同时对模型是否成功作出定性评估。

研究者的职责仅限于把运行 Claude Code 的笔记本电脑连接到机器狗，输入初始提示，批准命令，以及批准模型进入下一项任务。

Claude 擅长哪些任务？

答案很简单：对于 8 月实验中至少有一支人类团队完成的每项任务，Opus 4.7 完成相同任务的速度都至少快了 10 倍。1 如果只考察两支人类团队都完成的 4 项任务，Opus 4.7 的平均速度比无 Claude 团队快 37 倍以上，比 Claude 辅助团队快 18 倍以上。

下表比较了最初两支团队，也就是 Claude 辅助团队和无 Claude 团队，与 Opus 4.7 在第二阶段全部测试任务上的速度。

人类团队在面对多种机器狗传感器接口方案时，很难作出选择。Opus 4.7 则能迅速找到最佳路径。它编写的大部分代码第一次运行就有效，而最初实验中的 Claude 辅助团队和无 Claude 团队都没有做到这一点。

代码量也反映了 Opus 4.7 的效率。它取得的成功程度与两支人类团队相当或更高，但生成的代码量几乎比 Claude 辅助团队少 10 倍。

Opus 4.7 并不完美。例如，它默认采用了一种已经过时的目标检测算法。即便如此，它仍能绕过这个问题，最终找到有效的解决方案。

对于模型成功完成的步骤，我们观察到各次试验的完成时间差异很小，至少绝对差值不大。不过，前面提到的不理想算法选择，很可能导致其中一次沙滩球检测试验明显比另外两次更慢。总体而言，对于这项实验中处于 Claude 能力范围内的任务，它现在已经相当可靠。下一节将分析 Claude 仍然无法完成的任务。

值得再次强调的是，我们在上一篇文章中也说明过，这些进步并不是专门投入力量提升模型机器人能力的结果。与 LLM 发展史上的许多其他能力一样，这些提升来自更通用的规模扩展。

Claude 在哪里遇到了困难？

人类使用双手操作控制器并经过一些练习后，可以驾驶机器狗轻轻推动沙滩球，把球带回机器人出发的基地。基地是一块人造草坪。

这个过程要求操作者迅速察觉球是否偏离路线，判断这种误差与上一条指令有何关系，确认球现在的位置，然后调整后续输入，更精确地移动球。这是一种闭环控制。人类很擅长这类任务，至少在犯过一些错误并从中学习之后如此。

在第二阶段实验中，Claude 很难把握这种细微变化。与第一阶段中进入自主取球程序开发环节的人类一样，Claude 可以把机器人移动到球后方，并调整位置，准备把球撞回起点。但是，它对整个过程的控制很差。与人类参与者一样，它最终没有成功。

我们的一名研究者比第一阶段的志愿者拥有更多机器人经验。他成功编写程序，完成了自主取球任务。我们认为，如果有更多时间和额外的辅助框架，当前这一代 Claude 很可能也能做到。接下来，我们真正关注的是模型能否像完成 Project Fetch 其他环节一样，以同等的速度和可靠性完成最后这项任务。

这意味着什么？

在介绍第一阶段时，我们强调，LLM 可以提升非专业人士使用机器人的能力。现在，这一判断比之前更加成立。

过去需要人类与模型结对编程才能完成的工作，现在模型可以独立完成，而且速度快得多。这意味着人们能够更快进入实际控制和使用机器人的阶段。对于某些任务，让人类参与控制机器人，可能仍然胜过由 AI 模型把“虚拟的手”放在方向键上操作。

真正有趣而且有所不同的是，我们似乎已经更加接近这样一个世界：至少对于用途有限的任务，模型能够相对轻松地使用市售的实体工具。

这与 AI 模型向更具智能体特征的编程方式过渡时很相似。当时，模型开始使用 string-replace 等现成的软件编辑工具。如今，我们可能正在进入“物理智能体 AI”的早期阶段。

我们仍需开展更多研究，了解模型能否让这些物理工具更适合具体需求。例如，模型能否针对特定任务编写控制策略，或者设计机器人系统。要实现身体能力更强、适应性更好的通用语言模型，可能仍然存在重大障碍。

不过，正如我们已经看到的那样，模型能力之间看似遥远的距离，也可能在短时间内被跨越。不久前，模型自行构建软件工具或许还显得不可思议，但现在已经成为现实。如果认定硬件领域不可能出现相同的演进轨迹，将是不明智的。
