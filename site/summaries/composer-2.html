Composer 2：用更强基座把"前沿编码"做到便宜

来源 Cursor ｜ 作者 Cursor Team ｜ 2026-03-19

一句话总括：Composer 2 真正的贡献不是又发了一版编码模型，而是用"首次持续预训练 + 长轨迹强化学习"把前沿级编码质量和低价同时拿下——更强的预训练基座让后续 RL 能放大到"需要数百步动作"的长任务，而定价被压到 $0.50/$2.50 每百万 token，重新画了一条编码模型的"智能/成本"性价比线。

能力来源先把基座做厚，再用 RL 放大

官方把质量提升归因于一条清晰的训练链路，而不是某个单点技巧：

1
首次持续预训练（continued pretraining）：得到一个"远更强的基座"，为后续强化学习提供更高的起点。
2
在该基座上，对长轨迹编码任务做强化学习，把能力推向真实工程场景。
3
结果：模型能解决需要数百次动作（hundreds of actions）的高难度任务——这正是日常编码里"读—改—跑—修"反复循环的形态。

结论：编码能力的天花板更多由"基座有多厚 + RL 能否扩到长任务"决定，而非又调了一次提示或数据配方。

基准"全面提升"，但正文未给具体分数

官方称在其衡量的所有基准上都有大幅改进，并点名两项：Terminal-Bench 2.0（由 Laude Institute 维护的终端智能体评测）和 SWE-bench Multilingual。评测方法值得留意，因为它决定可比性：

  • Cursor 自评用官方指定的 Harbor 框架、默认设置，每个"模型-智能体"对跑 5 次取平均；
  • 对比的其他模型，取"官方榜单分"与"在 Cursor 基建上复跑分"中的较大者；
  • 对手用各自原生 harness（Anthropic 用 Claude Code、OpenAI 用 Simple Codex）。

结论："提升多少"无法从文字核实，但"怎么测"的透明度本身是有价值的可比性信号。

定价把"快"做成默认，且比别家"快速档"更便宜

两档智能完全相同，差别只是速度。官方强调快速版比其他模型的"快速档"更便宜，并把它设为默认选项；个人套餐里 Composer 用量走独立用量池，含较慷慨的额度。速度对照基于 2026-03-18 的 Cursor 流量快照，其中 Anthropic 的 token 约小 15%，TPS 与对手的输出价均做了归一化处理。

结论：把"同等智能的快速版"做成默认且更便宜，是一次面向真实开发体验的产品取舍——延迟比峰值分数更影响"好不好用"。

最值得带走的洞见

  1. "持续预训练"是这次跃升的真正引擎。更厚的基座才让长轨迹 RL 有放大空间——能力提升被明确归因到训练链路，而非单点技巧。
  2. 长任务能力是新战场。"数百步动作"对应真实工程里的反复循环，比一次性问答更能反映编码助手的实用度。
  3. 价格成了产品力的一部分。$0.50/$2.50 把前沿质量和低成本绑在一起，重画了智能/成本曲线。
  4. "快"被当作默认。同等智能下优先压低延迟、并定价低于同类快速档，说明真实开发体验里速度的权重在上升。
  5. 读基准先读方法。正文未给具体分数，但 Harbor 框架、5 次取均、对手取较大值这些口径，比一个孤立数字更值得信任。
