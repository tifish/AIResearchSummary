智能体编程与专业知识的持续回报

Agentic coding and persistent returns to expertise

一句话总括：Claude Code 正在接管更多技术执行工作，但决定结果的核心仍是人对问题本身的理解；智能体放大了领域知识，并没有取代它。

核心分工

人负责方向，智能体负责展开行动

研究把会话中的决策拆成规划和执行两类。由此可以看到，所谓“自主”并不等于人退出过程，而是双方承担不同层级的判断。

规划权仍主要在人手里

用户平均作出约 70% 的规划决策。这些决策包括要解决什么问题、采用什么思路，以及什么状态才算完成。

这说明人的主要贡献正从亲手实现，转向设定目标和判断结果。

执行权大多交给 Claude

Claude 平均作出约 80% 的执行决策。它选择要改哪些文件、写什么代码，以及运行哪些命令。

智能体首先替代的是实现过程中的密集操作，而不是对工作目标的理解。

一次指令可以触发一串工作

典型会话约有四轮交互。用户每发出一条提示，Claude 平均执行约 10 个动作，并产生约 2,400 个英文单词的输出。

衡量智能体价值时，不能只看回答次数，还要看每次指令实际展开了多少行动。

专业知识的回报并没有消失。真正发生变化的是：懂业务的人如今可以让智能体替自己完成更多技术工作。

七个月的结构变化

Claude Code 的用途正在从“修故障”走向端到端工作

2025 年 10 月

修复占据更大比重

修复损坏代码占会话的 33%。工具仍较多地被当作排错助手。

变化过程

调试占比近乎减半

到 2026 年 4 月，修复类会话降至 19%。下降不代表使用减少，而是工作范围扩大。

2026 年 4 月

运行、分析和写作增长

软件运营从 14% 升至 21%。写作和数据分析合计从约 10% 增至 20%。

结论是，智能体正在进入部署、运行、分析和文档生产等更完整的工作链条。

职业与任务价值

编程职业的优势缩小，领域能力的优势保留

研究并未发现只有软件从业者才能让 Claude Code 产出有效代码。不同职业之间的差距，远小于新手与熟练用户之间的差距。

非软件职业也能完成代码任务

在产生代码的会话中，软件相关职业的验证成功率为 34%，其他职业为 29%。至少部分成功率分别是 89% 和 88%。

编码背景仍有帮助，但已不再是成功完成编程任务的决定性门槛。

更专业的用户能调动更多智能体工作

新手每条提示通常触发约 5 个动作和约 600 个单词的输出。专家会话则达到约 12 个动作和 3,200 个单词。

领域专家不仅更会判断结果，也能用更少的沟通让智能体承担更长的工作链。

任务价值同步提高

按自由职业市场的相似职位粗略估算，平均会话价值增长 27%。构建、运营和修复类任务分别增长约 43%、34% 和 32%。

这些估值适合比较相对变化，不能当作精确的美元价值。

最值得带走的洞见

五个判断

1. 自主不等于无人负责

Claude 可以自主执行大量动作，但工作目标和完成标准仍主要来自人。

2. 领域知识比编码身份更关键

各主要职业的代码任务成功率彼此接近。理解具体问题，比是否以编程为职业更能解释结果。

3. 新手阶段的回报曲线最陡

从新手提升到中级带来的收益最大。深度专精仍有优势，但增量较小。

4. 专业能力的价值体现在纠偏

智能体遇到错误时，专家更能识别问题、补充约束并继续推进。新手则更容易放弃。

5. 未来的关键观察指标是专业知识回报是否下降

如果模型开始自行提供今天仍由用户贡献的关键判断，智能体的收益才会真正从领域专家扩展到更广泛人群。当前数据尚未显示这种替代已经发生。

完整译文

主要发现

在此前研究的基础上，我们提出了一个分析交互式智能体编程的框架。研究采用隐私保护方法，分析了 2025 年 10 月至 2026 年 4 月期间约 40 万次 Claude Code 会话。我们评估了任务构成、人机协作方式和成功率。

在典型会话中，人作出大部分规划决策，也就是决定做什么。Claude 作出大部分执行决策，也就是决定如何完成。一个人带入会话的领域知识越丰富，Claude 根据每条指令完成的工作就越多。在编程任务中，各主要职业完成用户既定目标的平均成功率都与软件工程师接近。这里的成功需要有测试通过或工作已提交等可验证证据。

一个人的领域知识越丰富，会话成功结束的概率就越高。不过，中级用户与专家之间的差距并不大。在我们观察的七个月里，用于调试的会话占比下降了近一半。使用方式也转向更完整的端到端智能体工作，包括部署和运行代码、分析数据，以及撰写非代码文档。

在这七个月中，我们通过与自由职业职位信息对比来估算典型任务的价值。几乎所有工作类别的任务价值都在上升，平均增幅约为 25%。

引言

智能体编程迅速普及。自 2025 年末以来，出现编程智能体活动的 GitHub 项目占比已经增长一倍以上。1 Claude Code 用户目前平均每周使用该工具 20 小时。2 没有正式编程经验的人，能否成功指挥智能体完成复杂的技术工作？这些工具的快速普及和改进，又会给广义的知识工作带来什么影响？我们还无法完整回答这些问题，但可以从 Claude Code 的使用数据中寻找早期信号。

本报告采用隐私保护方法，分析了 2025 年 10 月至 2026 年 4 月期间约 23.5 万人进行的约 40 万次交互式会话，以此呈现 Claude Code 的实际使用情况。此前的研究重点考察 Claude Code 会话中的自主程度，以及 Claude Code 如何改变 Anthropic 内部的工作。3 在此基础上，我们提出了一个描述交互式 AI 编程助手使用情况的框架：人们在做什么工作、谁在做这些工作，以及工作是否成功。我们重点研究通过命令行界面（CLI）、Claude.ai 或 Claude Code 桌面应用使用 Claude Code 的情况。4 通过追踪模型能力增强后智能体编程使用方式的变化，我们可以更好地理解这些工具如何影响编程专业人士和知识工作者的劳动力市场。

随着智能体进入非编程工作，Claude Code 上发生的事情可能预示着知识工作的未来。我们发现，Claude 正在处理更加复杂、价值更高的任务。与此同时，智能体编程中仍存在明确的分工：人决定要构建什么，智能体决定如何构建。

我们还发现，有效使用工具的放大器是领域知识，而不是编程熟练度。具体来说，领域专家的成功率更高，也更容易从错误和误解中恢复。不过，专家与中级用户之间的差距不大。这说明，只要在一个领域达到熟练水平，就能以接近深度专家的效率使用该工具。

这些发现让我们得以初步观察劳动力市场可能发生的转变。在我们的数据中，成功取决于一个人对待解决问题的理解程度，而不是他是否受过编程训练。如果这种模式在整个经济体系中同样成立，就意味着智能体编程工具虽然可能正在吸收部分以实现为主的工作，却也会奖励那些真正理解本职工作所解决问题的人。编程智能体并没有替代领域知识。劳动者带给智能体的理解越深入，智能体能够完成的高质量工作就越多。

工作如何分配

人们用 Claude Code 做什么

为了理解人们使用 Claude Code 的目的，我们把每次会话划分为九种工作模式之一。所选模式代表该会话试图完成的主要活动。5 其中四种模式直接涉及编写或维护代码：构建新内容、修复故障、测试代码，以及协调其他智能体或自动化流水线。另一类是运营软件，包括部署、配置、运行流水线和监控系统。还有两类更偏向于确定接下来该做什么：理解现有系统的工作方式，以及在实施之前规划变更。最后两类工作的最终成果与代码无关，或者代码只起辅助作用：分析数据，以及通过演示文稿和其他文字文档进行沟通。

约 56% 的会话用于编写代码（25%）、修复代码（26%），或者测试及协调代码（5%）。运营软件占 17%。规划或探索占 14%。生成分析或文字内容占 13%（图 1）。

我们先让模型阅读会话记录并进行分类，再使用隐私保护分析工具，将分类结果与每次会话自动记录的遥测数据核对。遥测数据包括是否增加或删除了代码行。两种数据源高度一致。例如，在分类器标为创建或修改代码的会话中，超过 90% 的会话也在遥测数据中显示了代码变更。详情请参阅附录。

谁作出哪些决定

Claude Code 的自主程度有多高？能力评估显示，它的上限很高，而且仍在提升。在 METR 的时间跨度评估等 benchmark 中，前沿模型如今可以自主完成原本需要人类花费数小时的软件任务，并在过程中自行克服障碍。但实际使用情况是什么样的？我们在这里考察真实会话中，人和 Claude 分别进行了多少引导。

我们从两个角度研究这个问题。第一，我们考察人们把多少决策交给 Claude。第二，我们考察人们交给 Claude 多少行动。为了理解会话中的决策分工，我们根据会话内容构建了一个保护隐私的决策归属分类器。我们让分类器列出会话中所有有实际意义的决策，再把这些决策分成规划和执行两类。规划包括要做什么、采用哪种方案，以及什么状态算完成。执行包括修改哪些文件、编写什么代码、使用什么编程语言，以及运行哪些命令。随后，分类器判断每项决策来自 Claude 还是用户。每次会话最终得到两个数字：用户在规划决策中的占比，以及用户在执行决策中的占比。

平均而言，人们作出约 70% 的规划决策，但只作出约 20% 的执行决策（图 2）。在实际的智能体编程中，工作分配十分明确：人决定构建什么，智能体决定如何构建。

为了理解一次会话中的行动委派程度，我们考察会话结构，而不是会话内容。Claude Code 会话由 Claude 与用户轮流交换提示和行动组成。用户写下一条提示，Claude 随后完成一些工作。用户再写下一条提示，如此反复。典型会话约有四轮这样的交互。在我们从 10 月到次年 4 月的历史数据中，用户每发出一条提示，平均会触发 Claude 执行约 10 个动作，有时会超过 100 个。6 每一轮中，Claude 都会读取文件、编辑代码和运行命令，平均输出约 2,400 个英文单词。

Claude 在两次用户确认之间完成多少工作，很大程度上取决于谁在作出决定。当用户保留执行控制权，也就是作出超过 80% 的执行决策时，Claude 每轮采取的行动较少，约为 8 个。当 Claude 掌握规划控制权，也就是作出超过 80% 的规划决策时，它采取的行动最多，约为 16 个。

专业水平

Claude 根据每份会话记录，在从新手到专家的五级量表上评定用户在当前任务中表现出的专业水平。专业水平分类器考察三类信号：用户给出指示时有多精确、用户要求 Claude 验证什么，以及用户更常纠正 Claude，还是 Claude 更常纠正用户。需要注意的是，这里的专业水平不同于职位或一般能力。更关键的是，它针对的是具体任务。一位资深工程师第一次询问 Rust 问题时，是 Rust 新手。一位从未使用 Python 的会计师，如果能准确告诉 Claude 一段 Python 脚本必须执行哪些对账规则，并发现脚本在月末结账时遗漏的边界情况，那么他就是这项任务的专家。

下表展示了分类器对各专业水平的定义，以及取自公开编程智能体会话数据集 SWE-chat 的请求示例。被归为“新手”的对话只给出笼统指示，没有体现特定领域知识。被归为“专家”的对话则体现了对代码库和技术环境的深入了解。

我们量化了用户专业水平与 Claude 每条提示所产生的输出和行动之间的关系。在典型的新手会话中，每条提示会触发 Claude 执行约 5 个动作，并输出约 600 个英文单词。专家会话触发的行动链长度超过其两倍，达到 12 个动作；输出量达到其五倍，约为 3,200 个英文单词（图 3）。在所有工作类型和各个任务价值区间内，都能看到新手与专家之间的这种差距。

这些指标补充了我们此前 Claude Code 报告中的自主性指标。此前的报告追踪智能体运行多长时间，以及用户多频繁地自动批准其行动。相比之下，决策归属指标反映一次完整会话中由谁作出实质性决定。每条提示的输出量和行动数则衡量一条人类提示会触发 Claude 进行多少自主活动。

谁在使用 Claude Code，他们用它做什么

用户

为了理解由谁完成这些工作，我们根据会话记录推断每位用户的职业，并将其映射到美国劳工统计局标准职业分类（SOC）体系中的 23 个主要职业组之一。分类器被要求只依据以下信号进行判断：智能体在会话开始时载入的项目背景、文件名称与结构、用户引用的材料，例如法律文件、临床数据、财务报告或课程，以及用户使用的词汇。7 分类器被明确要求，不得把编程行为本身视为用户从事编程职业的证据。只有在有明确信号表明软件或数据工作是用户职业的情况下，会话才会被归入编程对应的 SOC 类别，即计算机与数学职业。如果一位律师编写脚本，自动检查一批合同中缺失的条款，即使会话中的主要工作是开发软件，这次会话仍会被归入法律职业。没有任何职业信号时，会话不会被分类。

我们能在约 70% 的会话中推断出用户职业。在这部分会话中，包含大多数软件相关工作的计算机与数学职业毫无意外地成为最大群体。其后的主要群体依次是商业与金融运营职业、艺术设计与媒体职业、管理职业，以及生命科学、自然科学与社会科学职业。样本中增长最快的非软件职业群体是管理、销售和法律职业。

工作

从 2025 年 10 月到 2026 年 4 月，使用 Claude Code 完成的工作构成发生了显著变化。最明显的变化是，用于修复损坏代码的会话占比从 33% 降至 19%（图 4）。与此同时，围绕代码展开的其他工作占比增加。运营软件从 14% 增至 21%。写作和数据分析的占比大致翻倍，从约 10% 增至 20%。

任务本身的价值也有所提高。我们把每次会话中的工作与公开数据集中真实的自由职业职位信息进行校准，再询问这项工作在自由职业市场上需要花费多少，以此近似估算其经济价值。按照这一指标，从 10 月到次年 4 月，平均会话的估算价值增长了 27%。很多工作类型都出现增长。构建、运营和修复类任务的价值均增长约三分之一或更多，分别约为 43%、34% 和 32%。这些价格估算较为粗略，因此我们主要用它们比较不同任务及其随时间发生的变化，而不把估算出的美元金额当作精确数值。8 任务估值器的构建方法详见附录。

成功取决于用户带来了什么

任务估算价值可以从一个角度反映 Claude Code 如何帮助人们工作。另一个角度是观察多少会话取得成功，以及会话的哪些特征与成功有关。在所有成功指标中，我们都看到一种清晰模式：一个人在会话中表现出的专业水平越高，成功的可能性就越大。大部分增益集中在专业量表的较低一端。新手会话与中级会话之间的差距，大于中级会话与专家会话之间的差距。

在讨论成功会话的特征之前，我们需要准确说明如何衡量成功。我们无法观察用户在现实世界中的结果，也不能直接询问他们是否通过 Claude 得到了想要的成果。因此，我们采用两项互补的、基于会话记录的指标。第一项是“判断成功”。分类器阅读完整会话记录，再判断用户是否完成了原本想做的事情。可选结果包括成功、部分成功、失败和没有明确目标。随后，两个配套分类器评估这一判断的证据强度，以确定“验证成功”。成功信号分类器寻找可以验证成功的证据。它尤其关注与工作相符的 git 活动，例如提交和拉取请求，也会查找测试套件通过以及用户明确确认等证据。它把会话从“无信号”到“弱信号”（1），再到“多个硬信号”（5）进行评分。另一个并行的失败信号分类器则评估事情出错的证据，包括错误、测试失败、反复重试，以及用户反对或不满输出。只有当会话被判断为成功，并且至少存在一个可验证成功的硬信号时，才算“验证成功”。在以下分析中，我们重点研究会话成功或失败的程度，因此排除了被归为“没有明确目标”的会话。这些会话约占完整样本的 7.7%。

专业知识带来的回报

那么，哪些会话最容易成功？事实证明，前文所述的会话专业水平评分对成功与否影响很大。

有人可能担心，真正起作用的并不是专业水平。也许专家选择的任务不同，或者他们在其他方面有所不同。本节始终尝试部分消除这种顾虑。我们比较工作类型相同、任务估算价值相同、发生月份相同、主题相同，而且用户属于同一大类职业的会话，再考察不同专业水平对应的结果差异。

按照所有成功指标，一个人在会话中表现出的专业水平越高，会话成功的可能性就越大。被评为新手的会话有 15% 达到最严格的“验证成功”标准，77% 至少部分成功。被评为中级或以上的会话有 28% 至 33% 达到验证成功，91% 至 92% 至少部分成功（图 5）。

在每项指标中，大部分提升都来自从新手进步到中级。专业水平从中级提高到专家后，曲线斜率有所下降。图 5 所依据的回归分析详情见附录。

在过程中遇到困难的会话里，也能看到相似的梯度。如果失败信号记录了可验证的失败证据，我们就称该会话遇到了困难。这可能是出现错误、测试失败、多次尝试完成同一件事，或者用户表达挫败和不满。在遇到困难的会话中，最终达到验证成功的比例从新手会话的 4% 上升到专家会话的 15%。这一结果已经考虑了前文所述的全部控制因素（图 5）。采用较宽松的指标时，新手会话至少部分成功的比例为 60%，而中级至专家会话为 80% 至 81%。

我们也追踪相反的关系，即专业水平与多种失败指标之间的关系。需要注意的是，在这项分析中，被判断为失败的会话是连部分成功都没有达到的会话。如果一个遇到困难的会话被判断为失败，而且没有写入任何一行代码，我们就称其被放弃。在用户看起来是新手的会话中，有 19% 最终被放弃。其他所有专业等级的比例为 5% 至 7%。换句话说，经验最少的用户在难以获得想要的结果时，更容易放弃。专业知识的部分价值，似乎就在于能够把智能体引向正确方向。9

职业的影响可能小于专业水平

软件相关职业的用户在全部会话中达到验证成功的比例约为 30%，其他职业约为 26%。在产生代码的会话中，也就是增加或修改至少一行代码的会话中，这两个数字分别是 34% 和 29%（图 6）。采用较宽松的成功定义后，软件相关职业与其他职业之间的差距进一步缩小。在产生代码的会话中，两组至少部分成功的比例分别为 89% 和 88%。五个百分点的验证成功率差距很小。在七个月里，即使两组的成功率都在提高，这一差距也没有扩大或缩小。在产生代码的会话中，数据集里规模最大的十个职业群体，其成功率与软件工程师之间的差距都不超过七个百分点。管理职业的验证成功率最高，略高于软件工程相关职业。管理技能可能可以迁移到指挥智能体，因此管理者的验证成功率更高。但这也可能部分源自我们的测量方式。验证在一定程度上依赖会话中的明确确认，而管理者可能更习惯在得到所需结果后作出明确反馈。10

未来展望

本报告的结果初步呈现了这样一幅图景：智能体编程在放大某些知识和技能的同时，也在替代另一些技能。在产生代码的会话中，各主要职业的成功率都与软件相关职业相差不远。编程智能体似乎正在降低编程背景对成功完成软件工作的影响。

与此同时，成功会话更常体现出领域知识。被评为专家的会话达到验证成功的概率，是新手会话的两倍以上。当会话遇到困难时，新手放弃的比例是其他人的数倍。双方的协作方式进一步丰富了这幅图景。领域专家每给出一条指令，都能引导 Claude 完成更多工作。因此，把 Claude 引向成功所需的能力，更多来自对某个领域的掌握，而不是编写代码的能力。任何领域中掌握专业知识的人，现在都可能完成过去无法完成的技术工作。完全不具备相关专业知识的人，从同一工具中获得的收益会少得多。而且，大部分收益来自胜任，而不是精通。对领域具备实用理解，就能获得大部分好处。深度专精在此基础上只会带来少量额外收益。

这些发现仍属初步结果。与我们的多数研究一样，我们无法测量现实世界中的结果。例如，我们不知道会话中编写的代码随后被采用还是被丢弃，也不知道它是否产生了具有经济价值的成果。此外，本报告排除的非交互式使用占据了相当大的活动份额。为这种使用方式建立衡量框架，是未来工作的重点。我们对所有会话的分类都依赖模型对会话记录的理解。附录显示，我们的分类器与独立遥测数据之间的变化方向符合预期，而且在大多数会话上与一个强参考模型一致。但是，大规模验证分类器仍然很困难。Claude Code 会话可能很长，也很复杂，这进一步增加了难度，因为人类标签也未必适合作为真实标准。

随着模型、用户以及双方分工的改变，本报告所描绘的图景也会更新。我们希望利用这些指标，在重要变化发生时及时追踪。例如，如果专业知识带来的回报开始随时间下降，就可能说明模型开始提供目前仍由用户贡献的关键判断，也意味着这些工具带来的收益正从领域专家扩大到更广泛的人群。如果软件职业以外的用户成功完成编程会话的比例继续增长，则可能说明软件生产正在成为各领域日常工作的一部分，而不再是某一种职业的专属产物。这些变化会改变谁能从智能体编程中获益，也会改变获益程度，并进一步影响劳动力市场最看重什么。
