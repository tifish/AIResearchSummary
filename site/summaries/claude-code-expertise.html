智能体编程：真正持续获得回报的是"领域专业度"，而不是会不会写代码

基于约 40 万个 Claude Code 会话（约 23.5 万用户，2025-10 至 2026-04）的隐私保护分析
一句话总括：这份报告用大规模真实使用数据证明，智能体编程正在把"实现层"的工作交给 AI，但会不会成功、谁能拿到红利，取决于你对问题本身的理解（领域专业度），而非编程功底——而且这种专业度的回报真实存在、并随能力提升而递减。

分工已成定局：人决定"做什么"，AI 决定"怎么做"

报告把会话拆成"规划决策"（做什么、用什么方案、什么算完成）和"执行决策"（改哪个文件、写什么代码、跑什么命令）。结果出奇地一致。

~70%规划决策由人做
~20%执行决策由人做
~4 轮典型会话来回次数
~10 个每条 prompt 触发的动作（偶尔超 100）

谁掌控决策，AI 就放手到什么程度：当人牢牢握住执行权（>80%），Claude 每轮约做 8 个动作；当 Claude 接管规划（>80%），动作升到约 16 个。

结论：自主性不是一个固定旋钮，而是由人主动让渡的——这意味着用户的"指挥水平"直接决定了 AI 的产出规模。

专业度的杠杆：同样一句话，专家撬动五倍产出

Claude 按 5 级（新手→专家）评估用户在该任务上的专业度，且这是任务特定的：一个从没用过 Python、但能准确说出对账规则、还能抓出月末结算边界情况的会计，就是这个任务上的"专家"。

~5 → 12每 prompt 触发动作数（新手 → 专家）
600 → 3,200 词每 prompt 输出（新手 → 专家）

专家会话触发的动作链长一倍多，承载的输出多达五倍——这个差距在每一类工作、每一档任务价值里都成立。

结论：专业度不是"做得对"那么简单，它放大了每一条指令的功率——懂行的人让同一个工具干更多、更有质量的活。

成功率：胜负看你懂不懂问题，而非懂不懂代码

用两套基于转录文本的指标衡量成功："判定成功"（分类器读全程判断是否达成目标）与更严格的"已验证成功"（须有 git 提交、测试通过、用户明确确认等硬证据）。

已验证成功率
新手 15%
中级及以上 28–33%
至少部分成功率
新手 77%
中级以上 91–92%

关键在于收益集中在低端：从新手到中级的跨越，远大于从中级到专家的提升。遇到困难时差距更刺眼——被"放弃"（判失败且 0 行代码）的会话，新手占 19%，其他所有人仅 5–7%。

结论：专业度的一大价值是"把跑偏的智能体掰回正轨"的能力；新手不是写不出代码，而是更容易卡住后直接放弃。

职业不重要：各行各业的成功率都贴近软件工程师

在控制了工作类型、价值、月份、主题、职业大类后比较，差距小得惊人。

30% vs 26%软件职业 vs 其他职业（总体已验证成功）
34% vs 29%产出代码的会话中
≤7 分十大职业与软件工程师的成功率差距

这 5 个百分点的差距七个月里既没扩大也没缩小，而两组的成功率都在上升。管理类职业的已验证成功率甚至略高于软件工程——可能是"会管理 = 会指挥智能体"，也可能因为管理者更爱在对话里明确确认结果。

结论：编程出身正在变得"不那么相关"，软件生产可能正成为各行各业日常工作的一部分，而非某一种职业的专利。

七个月的演变：少修 bug，多端到端，活更值钱

33% → 19%修复代码（debugging）会话占比
14% → 21%操作软件（部署/运行/监控）
~10% → 20%写作与数据分析（约翻倍）
+25–27%平均任务估算价值上升

价值上升几乎遍及所有工作类型：building、operating、fixing 分别约升 43%、34%、32%。价值用与自由职业岗位的对比来估算，作者强调它用于相对比较而非绝对美元数。

结论：使用方式正从"救火式调试"转向"端到端地交付"——AI 承接的任务越来越复杂、越来越值钱，这正是知识工作可能的走向预演。

最值得带走的洞见

  1. 分工是稳定的：人定"做什么"（约 70% 规划权），AI 定"怎么做"（约 80% 执行权）——这是当前智能体协作的基本结构。
  2. 领域专业度才是杠杆：专家用同一句指令撬动两倍动作、五倍输出，已验证成功率（28–33%）是新手（15%）的两倍多。
  3. 回报递减：从新手到中级的跃升最大，中级到专家差距很小——"够用的理解"就能拿到大部分收益，深度精通只多加一点。
  4. 编程出身在贬值：各大职业产出代码的成功率都在软件工程师 7 分以内，且差距七个月未变——代码能力正与成功"脱钩"。
  5. 前瞻指标：若未来"专业度回报"开始下降，说明模型开始供给用户原本带来的判断力，红利将从领域专家扩散到更广人群——这是值得持续追踪的拐点。
