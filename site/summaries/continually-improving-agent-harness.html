一句话总括

Cursor 的核心观点是：AI 编程助手的质量，越来越取决于模型外层的 harness 工程。

这篇文章不是单纯介绍某个新模型，而是解释 Cursor 如何用上下文管理、真实使用评测、错误监控、模型定制和多智能体编排，把同一个模型变成更快、更准、更省的开发代理。

上下文窗口从“预塞材料”转向“按需取用”

Cursor 在 2024 年底刚做 coding agent 时，会向模型提供大量静态上下文，并加很多护栏，例如每次编辑后暴露 lint 和类型错误、改写过短的文件读取请求、限制单轮工具调用数量。

现在只保留操作系统、git 状态、当前和最近查看文件等少量静态信息，更多依赖 agent 在工作中动态拉取上下文。

这说明模型变强后，好的 harness 不再只是替模型提前喂资料，而是给它更可靠地探索环境和调用工具的能力。

质量评估必须同时看 benchmark 和真实用户行为

Cursor 使用公开 benchmark 和自研 CursorBench 做离线评估，但它也承认 benchmark 只能近似真实开发场景，因此还会做线上 A/B 实验。

线上指标包括延迟、token 效率、工具调用数、缓存命中率，以及更接近结果质量的 Keep Rate：agent 生成的代码在固定时间后有多少仍留在用户代码库中。

这改变了对 AI 编程质量的判断方式：真正重要的不是一次回答看起来是否合理，而是用户是否少改代码、少返工，并能继续推进下一个任务。

工具错误会污染上下文，因此必须像产品 bug 一样治理

文章强调，工具调用失败不只是一次失败；错误会留在上下文里浪费 token，并造成 context rot，让模型后续决策继续变差。

Cursor 将未知错误一律视为 harness bug；对预期错误则按 InvalidArguments、UnexpectedEnvironment、ProviderError、UserAborted、Timeout 等类型分类，并按工具和模型建立 baseline。

这个做法把“模型偶尔出错”变成可观测、可告警、可修复的工程问题，降低了 agent 在长任务中失控的概率。

自动化日志排查让 harness 改进进入软件工厂模式

Cursor 每周运行带有专门 skill 的 Automation，让模型搜索日志、发现新出现或近期激增的问题，并创建或更新 backlog 工单。

他们还用 Cloud Agents 并行启动多个修复任务，甚至可以从 Linear 直接触发；一次专项 sprint 中，unexpected tool call errors 被降低了一个数量级。

这说明 AI 工具自身也开始被 AI 维护，harness 的可靠性提升不再只依赖人工逐个排查。

不同模型需要不同工具形态和提示策略

Cursor 的 harness 抽象是模型无关的，但会为每个模型深度定制。例如 OpenAI 模型更适合 patch 格式编辑，Anthropic 模型更适合字符串替换格式。

给模型使用训练中不熟悉的工具形态，会消耗更多 reasoning token，并带来更多错误；新模型上线前，Cursor 通常会用数周时间从相近模型的 harness 开始迭代。

这提醒读者：模型能力不是孤立存在的，工具接口和提示方式如果不匹配，会直接浪费模型能力。

中途切换模型有真实代价，多智能体可能更自然

用户在对话中切换模型时，Cursor 会自动切换到对应 harness，但新模型必须接手另一个模型生成的历史记录，这会偏离它的训练分布。

模型切换还会导致 provider 和 model 级缓存失效，使第一轮更慢、更贵；摘要切换能减轻缓存惩罚，但复杂任务中可能丢失细节。

因此，除非有明确理由，文章建议一个复杂会话尽量保持同一模型；需要不同能力时，用 fresh context 的 subagent 可能更稳。

未来的软件开发代理会变成 harness 编排问题

规划快速编辑调试结果整合

Cursor 认为未来 AI 辅助软件工程会是多智能体系统：不同子任务交给不同专长的 agent 或 subagent，而不是让一个 agent 承担所有工作。

关键挑战会转移到 harness：决定派哪个 agent、如何描述任务、如何利用它的优势、如何把多个结果拼成连贯 workflow。

最终洞见是，模型越多、工具越多、任务越复杂，harness 就越像 AI 软件开发系统的操作系统。

最值得带走的洞见

  • AI 编程助手的质量不是模型单变量，harness 决定了模型如何看上下文、调用工具和修复错误。
  • Keep Rate 这类真实使用指标，比单纯 benchmark 更接近“代码是否真的有用”。
  • 工具错误会持续污染上下文，必须通过分类、baseline、告警和自动化修复来治理。
  • 同一个模型在不同工具格式和提示策略下表现会明显不同，模型适配是产品工程的一部分。
  • 多智能体开发的关键不是堆更多 agent，而是让 harness 学会分派、约束和整合它们。
