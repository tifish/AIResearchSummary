一句话总括

持续改进我们的智能体运行框架

Continually improving our agent harness

这篇文章的真正贡献，是把“更强模型”之外的工程层说清楚：编码智能体的质量，来自上下文、工具、评估、错误修复和模型适配组成的运行框架。

Cursor Research Apr 30, 2026 Stefan Heule & Jediah Katz
2024 末 Cursor 最早开发编码智能体时，仍大量依赖静态上下文和护栏。
2 类测试 离线 eval 与线上 A/B 测试共同判断 harness 是否真的变好。
每周 Automation 搜索日志、发现新问题，并把调查结果沉淀为工单。
10 倍级 一次集中 sprint 将 unexpected tool call errors 降低了一个数量级。
核心变化

上下文从“预装”转向“动态获取”

早期模型不擅长自己寻找信息。Cursor 因此把 lint、类型错误、文件结构、相关代码片段和压缩附件提前塞进上下文。

现在模型能力更强，许多护栏被拆掉。系统只保留少量稳定信息，例如操作系统、git 状态、当前文件和最近查看文件。

这改变了读者对 AI 编程的理解：好体验不是尽量把所有信息一次性给模型，而是让模型在工作中按需拉取正确上下文。

评估方法

benchmark 只是一层测量

Cursor 使用公开 benchmark 和自有 CursorBench 获得快速、标准化的质量读数。但它们不能完全代表真实使用。

线上 A/B 测试补上了这一层。指标包括延迟、token 效率、工具调用次数、缓存命中率，以及更接近结果质量的 Keep Rate。

Keep Rate 关注智能体生成的代码在一段时间后有多少仍留在用户代码库中。这个指标比单次答案分数更接近真实生产力。

运行框架的改进像软件产品迭代

文章反复强调一个模式：提出假设，部署实验，观测真实信号，再修复退化。模型只是其中一个变量。

1

形成体验假设

团队先定义理想 agent 体验，再决定需要改变哪一层 harness。

2

离线评估

CursorBench 和其他 eval 快速发现质量变化与模型困惑点。

3

线上 A/B

多个 harness 版本并行进入真实使用，观察用户是否真的受益。

4

定位退化

按工具和模型建立基线，区分未知错误、预期错误和供应商故障。

5

批量修复

Automation 与 Cloud Agents 发现问题、创建工单，并推动多个修复并行启动。

不同模型需要不同 harness

同一个模型放进不同运行框架，速度、聪明程度和成本都会变化。Cursor 的做法是让工具形态贴近模型训练经验。

维度 文章中的观察 为什么重要
编辑工具 OpenAI 模型更适合 patch-based 编辑格式。Anthropic 模型更熟悉字符串替换。 让模型使用不熟悉的工具，会消耗更多 reasoning tokens，并增加错误。
提示风格 OpenAI 模型更字面、更精确。Claude 对不精确指令更宽容，也更偏直觉。 通用提示不是最优解。harness 需要按 provider 和版本细调。
模型怪癖 团队曾观察到某个模型在上下文变满时出现“context anxiety”，开始拒绝或回避任务。 一些行为不是产品层误用，而是模型特性。合适的 prompt 可以缓解。
真正的风险不是单个工具调用失败，而是错误留在上下文里，继续污染后续决策。
错误会产生 context rot。

工具调用失败后，模型可能自我修正。但失败记录仍在上下文中，浪费 token，也会影响下一步判断。

未知错误被当作 harness bug。

Cursor 对 unknown error 设置固定阈值报警。对预期错误，则使用按工具和模型计算的异常检测基线。

这改变了质量管理方式。

智能体系统不能只看最终答案。工具错误率、超时、ProviderError 和 UserAborted 都是生产信号。

模型切换

中途换模型会破坏分布假设

用户切换模型时，Cursor 会切到对应 harness。新模型得到自己的 prompt 和工具，但必须接手另一个模型留下的历史。

这段历史对新模型来说可能是 out of distribution。Cursor 因此加入指令，告诉模型它正在中途接管，并避免调用自己没有的旧工具。

缓存也是问题。缓存通常按 provider 和模型绑定。切换模型意味着首轮更慢、更贵。

替代路径

子智能体比强行接管更干净

文章建议，复杂任务中最好持续使用同一个模型，除非确实有切换理由。

如果需要另一个模型完成特定子任务，更好的方式可能是启动 subagent。它拥有新的上下文窗口，不必继承旧会话的工具形状和缓存问题。

这说明多模型系统的难点不是“能不能切换”，而是如何保留任务连续性，同时避免上下文污染。

最值得带走的洞见

文章最后把方向推向多智能体软件开发。未来的系统会把规划、快速编辑和调试分配给不同 agent，而编排能力会留在 harness 中。

harness 是 AI 编程产品的操作系统

它决定上下文如何进入模型、工具如何暴露、错误如何处理，以及多个 agent 如何协作。模型越强，这一层越关键。

质量要从真实使用里测

benchmark 有用，但 Keep Rate 和用户后续反馈更接近“代码是否真的可用”。

小优化会叠加成体验差异

文章承认阶跃式进步少见。更多时候，进步来自持续压低错误、延迟和 token 浪费。

错误分类本身就是产品能力

InvalidArguments、UnexpectedEnvironment、ProviderError、Timeout 等分类，让团队能知道该修工具、调 prompt 还是等待供应商恢复。

多智能体时代更依赖编排

未来不是一个 agent 做所有事，而是系统选择合适 agent，并把结果拼成连贯工作流。

结论：这篇文章把 AI 编程的竞争焦点从“谁的模型更强”推进到“谁能把模型组织成可靠的软件生产系统”。
