持续打磨 Agent Harness：一家 Agent 公司到底在优化什么

来源：Cursor ｜ 作者：Stefan Heule & Jediah Katz ｜ 2026-04-30 ｜ 原文
一句话总括

Agent 的能力 = 模型 × harness（运行框架）。这篇文章把「harness 工程」拆开给你看：它不是靠灵感突变，而是把大量小优化反复叠加，并用一整套线上线下指标来回答那个最难量化的问题——「这个 agent 到底做得好不好」。

01上下文窗口：从「加护栏」到「让模型自取」

上下文窗口是和大模型交互的核心：系统提示 + 工具描述 → 对话状态 → 用户请求。2024 年底刚做 coding agent 时，模型还很弱，Cursor 砌了大量护栏——每次编辑后把 lint/类型错误喂回去、模型读文件读太少就替它改写、甚至限制一轮能调用的工具数；同时塞入大量始终在场的静态上下文（目录结构、语义匹配的代码片段、压缩后的附件文件）。如今这些大多已被拆掉：只留少量有用的静态信息（操作系统、git 状态、当前与最近浏览的文件），其余改为让 agent 在干活时自己动态拉取。
洞见：模型越强，harness 的活就越是「做减法」——少喂、少管，多给 agent 自取上下文与影响世界的能力。

02怎么判断一次 harness 改动是变好了

代理质量很难定义，Cursor 用多层测量来逼近它。离线有公开 benchmark + 自建套件 CursorBench，给出快速、标准化、可跨时间对比的读数；但 benchmark 终究只是近似，所以还要线上 A/B：同时部署两套以上 harness 变体跑真实流量。延迟、token 效率、工具调用数、缓存命中率这些好测但不触及「做得好不好」，于是用两个更硬的信号——Keep Rate（agent 提出的改动在固定时间后仍留在用户代码库里的比例，留不住说明用户在返工）和用语言模型判读用户回复（用户转去做下一个功能=满意；粘贴 stack trace=失败）。有时线上测试会反过来劝退看似不错的想法：他们曾试更贵的模型做上下文摘要，结果质量提升微乎其微、不值这笔成本。
洞见：「用户没再抱怨、改动留了下来」比任何 benchmark 分数都更接近真实质量——把人类的真实反应变成可测指标，是评测体系的关键一跳。

03把工具错误当 bug：分类、基线、自动修

模型和能力越加越多，harness 状态空间越大，bug 面也越大，很多只能在规模上才暴露。工具是最广的出错面，而工具错误特别伤——即便 agent 能自纠，错误仍留在上下文里浪费 token、造成「context rot」（累积的错误拖垮后续决策质量），严重时直接让 agent 卡死或跑偏。Cursor 把错误分两类处理：任何 unknown 错误一律视为 harness 的 bug，超过固定阈值即告警；而预期错误（如模型提了个错的编辑、读了个不存在的文件）按成因分类——InvalidArguments、UnexpectedEnvironment、ProviderError、UserAborted、Timeout 等——并按「每工具、每模型」分别算基线再做异常检测告警（因为不同模型搞砸工具调用的概率不同）。配合每周一次、带 skill 的 Automation 扫日志、建/更新 ticket，并用 Cloud Agents（甚至直接从 Linear 触发）批量修。
洞见：把「修 agent」做成一座自动化软件工厂——正是这套机制让他们在一次冲刺里把非预期工具错误压低了约 10 倍。

04一套 harness，按每个模型量身定制

所有 harness 抽象都是模型无关的，但可为每个模型深度定制。最典型的是工具格式：OpenAI 的模型训练时用 patch 式编辑，Anthropic 的模型训练时用字符串替换——两者其实都能用对方的格式，但给它不熟的那种会多耗推理 token、出更多错，所以 Cursor 按模型训练时的格式发工具。定制还深入到不同厂商、不同版本的提示词：OpenAI 的模型更字面、更精确地遵循指令，Claude 更凭直觉、对不精确的指令更宽容。拿到新模型早期访问权时，他们从最接近的现有 harness 出发，用离线评测 + 团队试用找出模型犯迷糊的地方再逐步调。有时还会撞上真实的模型怪癖——比如某个模型出现「context anxiety」：上下文越满越爱拒活、说任务太大——这类问题可用提示词调整缓解。
洞见：同一个模型放进「为它专门调过」的 harness 里，会明显更快、更聪明、更省——harness 的价值，很大一部分藏在这些按模型抠出来的细节里。

05对话中途换模型，与多 Agent 的未来

中途换模型很棘手：不同模型有不同行为、提示和工具形状。Cursor 会在切换时自动换上对应模型的 harness，但模型仍要面对一段由别的模型生成、对它而言 out-of-distribution 的对话历史。对策是加自定义指令，告诉它「你是中途接手的」，并引导它别去调那些只在历史里出现、却不属于自己工具集的工具。另一个代价是缓存按厂商/模型隔离，切换=缓存未命中→首轮更慢更贵；他们试过在切换时做摘要来减小惩罚，但用户若深陷复杂任务，摘要会丢细节——所以一般建议一条对话只用一个模型。更干净的绕法是用子 agent（从全新上下文起步），现在用户可以直接指定用某个模型跑子 agent。文章最后给出判断：软件工程的未来是多 agent——规划、快速编辑、调试各由擅长的 agent 承担，而「派谁、怎么交代、如何拼接结果」这套编排能力，恰恰住在 harness 里。
洞见：当系统从「单 agent」走向「多 agent 协作」，harness 不再只是模型的外壳，而成了调度与编排的大脑——harness 工程只会越来越重要。

最值得带走的 5 条洞见

  1. 能力 = 模型 × harness。同一模型放进专门调过的 harness，会明显更快、更聪明、更省；进步主要靠小优化的持续叠加，而非偶发的突变。
  2. 方向是做减法。模型越强，越要拆护栏、砍静态上下文，把「自取上下文、影响世界」的权力交还给 agent。
  3. 把真实质量变成可测指标。Keep Rate（改动留存率）+ 语言模型判读用户回复，比任何 benchmark 更接近「它到底做得好不好」。
  4. 把修 agent 做成软件工厂。错误分类 + 每工具/每模型基线 + 异常告警 + Cloud Agents 批量修，一次冲刺把非预期工具错误压低约 10 倍。
  5. 未来是多 agent，编排住在 harness 里。换模型、子 agent、派谁干什么——协调能力不属于任何单个 agent，而属于 harness，因此它只会更关键。
