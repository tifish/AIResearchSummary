持续改进我们的智能体运行框架

Continually improving our agent harness

一句话总括：Cursor 展示了一套把模型能力转化为可靠软件工程产出的持续优化体系，其核心是动态上下文、真实使用评测、故障治理、模型定制和多智能体编排。

2026 年 4 月 30 日 · Stefan Heule 与 Jediah Katz

更短、更易读的总结

模型决定能力上限，运行框架决定能力能否稳定兑现

Cursor 不把智能体优化理解为单纯更换模型。团队先提出理想体验，再通过离线评测和线上实验检验假设。真正的进步通常不是一次突破，而是许多小改进的累积。

早期：替模型预先做决定

系统会在每次编辑后主动显示 lint 和类型错误。它还会改写过窄的文件读取，并限制单轮工具调用数量。

问题在于：这类护栏会占用上下文，也可能限制更强模型的自主判断。

现在：让模型按需取用信息

框架仍提供操作系统、Git 状态和近期文件等基础信息，但更多上下文改为动态获取。

洞见是：模型越强，框架越应提供能力和接口，而不是提前替它选择所有信息。

评测必须越过 benchmark，观察真实产出

CursorBench 和公开 benchmark 提供快速、标准化的离线比较。真实使用中的 A/B 测试则同时观察效率和结果质量。

效率信号
延迟、令牌效率、工具调用次数和缓存命中率可以提示趋势，但不能单独证明任务完成得好。
代码保留率
系统追踪智能体生成的代码经过固定时间后还有多少留在代码库中。保留率低，往往意味着用户必须返工。
语义满意度
模型读取用户的后续回复。用户继续开发下一个功能是正面信号，贴出堆栈跟踪则是明确的失败信号。
成本约束
一次实验采用了更昂贵的上下文摘要模型，但质量改善可以忽略，因此没有抵消额外成本。

结论：智能体质量不能由一个分数概括。可靠判断需要把标准化评测、行为数据和用户反馈放在一起。

把故障治理做成持续运转的软件工厂

  1. 分类：未知错误一律视为框架缺陷。InvalidArguments、UnexpectedEnvironment、ProviderError、UserAborted 和 Timeout 等预期错误按原因归类。
  2. 检测：未知错误率越过固定阈值便触发告警。预期错误若显著偏离基线，也会触发异常检测。
  3. 细分：基线按工具和模型分别计算，因为不同模型调用工具的失误率并不相同。
  4. 修复：每周 Automation 搜索日志，发现新问题或异常增长，并创建或更新调查工单。Cloud Agents 可以并行启动修复。

理解上的变化：工具错误不只是一次失败。它们会留在上下文中，浪费令牌并造成“上下文腐化”，继而影响后续判断。

同一框架必须适配不同模型

OpenAI 模型接受过补丁式编辑训练，Anthropic 模型更熟悉字符串替换。Cursor 会为每个模型提供其训练期间熟悉的工具形式。

提示词也会细化到提供商和模型版本。因此，同一底层模型在不同运行框架中，速度、准确性和效率可能明显不同。

框架还能缓解模型怪癖

Cursor 曾观察到一种“上下文焦虑”：窗口越接近填满，模型越容易拒绝任务，并声称工作规模太大。

团队通过提示词调整减轻了这种行为。这说明部分模型缺陷可以在运行层得到控制。

中途换模型，会同时破坏上下文分布和缓存

切换后，Cursor 会加载新模型专属的提示词和工具，但新模型仍要接手另一模型产生的对话历史。框架会明确告知接管状态，并阻止它调用历史中出现、但自身并不具备的工具。

缓存由提供商和模型分别维护。切换模型会造成缓存未命中，使第一轮更慢、更贵。切换时生成摘要可以减轻代价，却可能丢失复杂任务的细节。

实践结论：没有明确理由时，最好在一段对话中坚持使用同一模型。若需要另一模型，使用拥有全新上下文的子智能体通常更合适。

未来的软件开发不会让一个智能体包办所有事情。运行框架将负责挑选规划、快速编辑和调试等专业智能体，并把各自结果组织成连贯工作流。多智能体能力的真正控制面位于运行框架，而不是某个单独模型。

最值得带走的洞见

01 · 减少过时护栏

模型能力提升后，静态上下文和强制规则可能变成负担。动态取用信息更能发挥模型判断力。

02 · 评测真实后果

benchmark 只能近似真实工作。代码是否被保留、用户是否继续下一项任务，更接近实际价值。

03 · 错误会持续污染上下文

失败的工具调用不仅浪费一次机会，还会造成上下文腐化。错误分类、分模型基线和自动调查因此非常重要。

04 · 编排是下一阶段的核心

未来竞争重点将从单个智能体的能力，转向框架能否正确分派专业智能体并整合结果。

我们打造 Cursor 智能体运行框架的方式，与开发任何一款雄心勃勃的软件产品并无不同。许多工作由愿景驱动。我们会先形成一个判断，设想理想的智能体体验应该是什么样子。

接着，我们提出能够接近这一愿景的假设，通过实验加以验证，再根据评测与真实使用中的定量和定性信号反复迭代。这个过程依赖合适的线上和离线观测手段。只有这样，我们才能知道一次改动是否真的改善了运行框架。

提前获得新模型的使用权限时，这些方法会汇聚到一起。我们会花费数周，根据模型的优势和特性定制运行框架。经过专门调优后，同一个模型会变得明显更快、更聪明，也更高效。

有时，我们会发现带来跨越式提升的改进。但多数时候，改善运行框架意味着专注地叠加许多细小优化。这些优化共同让智能体更擅长构建软件。

上下文窗口的演进

与大语言模型交互的核心是上下文窗口。让智能体构建某个功能时，上下文窗口先包含系统提示词和工具说明，随后是当前对话状态，最后是用户的请求。

在 Cursor 的发展过程中，我们填充和管理这个窗口的方式发生了很大变化。

2024 年末，我们刚开始开发编码智能体时，模型还不善于自行选择上下文。因此，我们投入大量上下文工程工作来建立护栏。例如，每次编辑后都向智能体显示 lint 和类型错误；当它请求读取的文件行数太少时，改写读取请求；甚至限制它在一轮中能够调用工具的最大次数。

我们还提供了大量静态上下文。每次会话开始时，这些信息始终对智能体可用。在不同阶段，它们包括代码库的目录结构、与查询在语义上相关的代码片段，以及用户手动附加文件的压缩版本。

这些做法如今大多已经退出历史舞台。

我们仍会包含一些有用的静态上下文，例如操作系统、Git 状态、当前文件和最近查看的文件。不过，随着模型能力增强，我们逐步撤掉护栏，并提供更多动态上下文。智能体可以在工作过程中自行获取这些信息。我们曾在一篇较早的文章中深入介绍动态上下文背后的部分技术，其中许多后来被其他编码智能体采用。如今，我们的大量工作都集中在为智能体提供更多动态拉取上下文和与外部世界交互的方式。

评估运行框架改动的两种方法

运行框架和模型共同决定智能体的表现，但“表现好”很难准确定义。为了找到答案，我们建立了多层测量体系。

除了公开 benchmark，我们还维护自己的评测套件 CursorBench。它可以快速、标准化地反映质量，也方便我们进行跨时间比较。然而，即使最好的 benchmark 也只能近似真实使用场景。完全依赖 benchmark，会让我们错过重要信号。

因此，我们还会开展线上实验。我们同时部署两个或更多运行框架版本，并基于真实使用情况进行 A/B 测试。在这些测试中，我们通过多种指标衡量智能体质量。部分指标很直接，例如延迟、令牌效率、工具调用次数和缓存命中率。这些指标能够提示总体方向，却仍无法回答一个更模糊也更重要的问题：智能体是否真的把工作做好了。对此，我们采用两种衡量方法。

第一种是智能体生成代码的“保留率”（Keep Rate）。针对智能体提出的一组代码变更，我们会跟踪经过若干固定时间间隔后，其中有多大比例仍然保留在用户的代码库中。通过这种方式，我们可以判断用户是否不得不手动调整智能体的输出，或者是否需要继续迭代并要求智能体修复问题。这些情况意味着智能体的首次响应质量较低。

第二种方法是使用语言模型阅读用户对智能体初次输出的回复，从语义上判断用户是否满意。用户开始处理下一个功能，是智能体完成工作的强烈信号。用户贴出堆栈跟踪，则是智能体没有完成工作的可靠信号。

有时，这些线上测试会促使我们搁置看似很有希望的想法。在一次实验中，我们尝试使用更昂贵的模型来总结上下文。结果显示，它对智能体质量的改善微乎其微，不值得为此支付更高成本。

跟踪并修复性能退化

随着模型和功能不断增加，运行框架会像其他软件一样变得更加复杂，并产生更多可能的状态。缺陷可能出现的范围随之扩大，其中许多问题只有在大规模运行时才能发现。

智能体的工具是最容易出现缺陷的领域之一。工具调用错误可能严重破坏 Cursor 中的一次会话。智能体通常能够自行纠正，但错误仍会留在上下文里。它们会浪费令牌，并造成“上下文腐化”（context rot）。随着错误累积，模型后续决策的质量也会下降。

一次工具调用失败后，智能体有时会陷入阻塞，甚至完全偏离任务。工具调用量和错误率等指标不能直接说明智能体是否完成了出色的工作，但它们可以作为指示器，帮助我们发现更广泛的问题。

任何未知错误都代表运行框架存在缺陷，我们会按缺陷处理。不过，许多错误属于“预期错误”。例如，模型偶尔会提出错误的编辑，或尝试读取不存在的文件。我们按照原因对这些预期错误分类。InvalidArguments 和 UnexpectedEnvironment 用来表示模型失误以及上下文窗口中的矛盾。ProviderError 则表示 GenerateImage 或 WebSearch 等工具的供应商发生服务故障。

我们还有 UserAborted 和 Timeout 等其他分类。它们合在一起涵盖了大多数预期错误。

我们根据这些指标设置告警，用于捕捉进入生产环境的重大退化。未知错误始终属于缺陷，因此，只要任何工具的未知错误率超过固定阈值，我们就会收到告警。但预期错误究竟代表运行框架的缺陷，还是正常行为，往往不容易判断。

例如，grep 搜索超时可能源于工具性能问题，也可能只是代码库规模太大，而模型又构造了低效查询。为了解决这个问题，我们设置了异常检测告警。当预期错误显著超过基线时，告警就会触发。我们会按工具和模型分别计算基线，因为不同模型在工具调用上的失误率可能不同。

我们每周还会运行一个配有 skill 的 Automation。这个 skill 会教模型如何搜索日志，发现新出现或近期大幅增加的问题，并在待办列表中创建或更新工单，同时附上调查结果。我们大量使用 Cloud Agents，同时启动许多问题的修复工作，甚至可以直接从 Linear 触发这些智能体。

这个流程是我们为智能体运行框架构建自动化“软件工厂”的一部分。在今年早些时候的一次集中冲刺中，我们将意外工具调用错误降低了一个数量级。

为不同模型定制运行框架

我们的所有运行框架抽象都与具体模型无关，同时也能针对支持的每个模型进行深度定制。例如，OpenAI 模型接受过使用补丁格式编辑文件的训练，而 Anthropic 模型接受的是字符串替换训练。两类模型都可以使用任一种工具，但让模型使用不熟悉的工具会消耗更多推理令牌，也会产生更多错误。因此，在我们的运行框架中，每个模型都会获得它在训练期间使用过的工具格式。

这种定制非常深入，包括针对不同提供商甚至不同模型版本设计专用提示词。OpenAI 模型在遵循指令时往往更为字面和精确。Claude 则更依靠直觉，对不够精确的指令也更宽容。

如果我们在新模型发布前获得提前使用权限，就会从最接近的现有模型运行框架开始迭代。我们运行离线评测，查找模型容易困惑的地方。团队成员也会亲自使用模型并反馈问题。我们再根据这些问题调整运行框架。这个过程不断重复，直到模型与运行框架的组合达到我们认为可以发布的水平。

这项调优工作大多是让运行框架适配新模型的优势。不过，我们有时也会遇到真正的模型怪癖，并通过运行框架进行缓解。例如，我们发现某个模型出现了一种被我们称为“上下文焦虑”（context anxiety）的行为。随着上下文窗口逐渐填满，它会开始拒绝工作，并含糊地表示任务似乎太大。我们通过调整提示词减轻了这种行为。

支持对话过程中切换模型

支持用户在对话中途切换模型尤其棘手，因为不同模型有不同的行为、提示词和工具形式。

用户切换模型时，Cursor 会自动切换到对应的运行框架，并加载为该模型定制的提示词和工具集合。然而，新模型仍要使用这些工具来处理另一个模型生成的对话历史。这种历史超出了它训练时的数据分布。

为了解决这个问题，我们会加入专门指令，告诉模型它正在对话中途接替另一个模型。这些指令还会阻止它调用对话历史中出现、但不属于其自身工具集合的工具。

第二个挑战是，缓存由提供商和模型分别维护。切换模型意味着缓存未命中，导致切换后的第一轮更慢、成本更高。我们尝试过在切换时总结对话，为模型提供一份干净的摘要，从而减轻缓存损失。但是，如果用户已经深入处理一项复杂任务，摘要可能丢失重要细节。除非有明确的切换理由，否则我们通常建议在一段对话中始终使用同一个模型。

另一种避开对话中途切换难题的方式，是改用拥有全新上下文窗口的子智能体。我们最近为运行框架增加了一项能力，用户可以直接要求使用指定模型运行一个子智能体。

运行框架与软件开发的未来

AI 辅助软件工程的未来将属于多智能体系统。系统不会再把每个子任务都交给同一个智能体，而是学会在专业智能体和子智能体之间进行委派。例如，一个负责规划，一个负责快速编辑，另一个负责调试。每个智能体只承担最适合自己的工作。

要让这种模式良好运转，本质上是一个运行框架问题。系统需要知道应该派出哪个智能体，如何根据该智能体的优势描述任务，以及如何把各项结果衔接成连贯的工作流。编排这类协作的能力将存在于运行框架中，而不是任何单个智能体中。这意味着，运行框架工程一直是决定智能体能否成功的关键，而且今后只会变得更加重要。

在 Cursor 的发展过程中，我们填充和管理这个窗口的方式发生了很大变化。

2024 年末，我们刚开始开发编码智能体时，模型还不善于自行选择上下文。因此，我们投入大量上下文工程工作来建立护栏。例如，每次编辑后都向智能体显示 lint 和类型错误；当它请求读取的文件行数太少时，改写读取请求；甚至限制它在一轮中能够调用工具的最大次数。

我们还提供了大量静态上下文。每次会话开始时，这些信息始终对智能体可用。在不同阶段，它们包括代码库的目录结构、与查询在语义上相关的代码片段，以及用户手动附加文件的压缩版本。

这些做法如今大多已经退出历史舞台。

我们仍会包含一些有用的静态上下文，例如操作系统、Git 状态、当前文件和最近查看的文件。不过，随着模型能力增强，我们逐步撤掉护栏，并提供更多动态上下文。智能体可以在工作过程中自行获取这些信息。我们曾在一篇较早的文章中深入介绍动态上下文背后的部分技术，其中许多后来被其他编码智能体采用。如今，我们的大量工作都集中在为智能体提供更多动态拉取上下文和与外部世界交互的方式。

评估运行框架改动的两种方法

运行框架和模型共同决定智能体的表现，但“表现好”很难准确定义。为了找到答案，我们建立了多层测量体系。

除了公开 benchmark，我们还维护自己的评测套件 CursorBench。它可以快速、标准化地反映质量，也方便我们进行跨时间比较。然而，即使最好的 benchmark 也只能近似真实使用场景。完全依赖 benchmark，会让我们错过重要信号。

因此，我们还会开展线上实验。我们同时部署两个或更多运行框架版本，并基于真实使用情况进行 A/B 测试。在这些测试中，我们通过多种指标衡量智能体质量。部分指标很直接，例如延迟、令牌效率、工具调用次数和缓存命中率。这些指标能够提示总体方向，却仍无法回答一个更模糊也更重要的问题：智能体是否真的把工作做好了。对此，我们采用两种衡量方法。

第一种是智能体生成代码的“保留率”（Keep Rate）。针对智能体提出的一组代码变更，我们会跟踪经过若干固定时间间隔后，其中有多大比例仍然保留在用户的代码库中。通过这种方式，我们可以判断用户是否不得不手动调整智能体的输出，或者是否需要继续迭代并要求智能体修复问题。这些情况意味着智能体的首次响应质量较低。

第二种方法是使用语言模型阅读用户对智能体初次输出的回复，从语义上判断用户是否满意。用户开始处理下一个功能，是智能体完成工作的强烈信号。用户贴出堆栈跟踪，则是智能体没有完成工作的可靠信号。

有时，这些线上测试会促使我们搁置看似很有希望的想法。在一次实验中，我们尝试使用更昂贵的模型来总结上下文。结果显示，它对智能体质量的改善微乎其微，不值得为此支付更高成本。

跟踪并修复性能退化

随着模型和功能不断增加，运行框架会像其他软件一样变得更加复杂，并产生更多可能的状态。缺陷可能出现的范围随之扩大，其中许多问题只有在大规模运行时才能发现。

智能体的工具是最容易出现缺陷的领域之一。工具调用错误可能严重破坏 Cursor 中的一次会话。智能体通常能够自行纠正，但错误仍会留在上下文里。它们会浪费令牌，并造成“上下文腐化”（context rot）。随着错误累积，模型后续决策的质量也会下降。

一次工具调用失败后，智能体有时会陷入阻塞，甚至完全偏离任务。工具调用量和错误率等指标不能直接说明智能体是否完成了出色的工作，但它们可以作为指示器，帮助我们发现更广泛的问题。

任何未知错误都代表运行框架存在缺陷，我们会按缺陷处理。不过，许多错误属于“预期错误”。例如，模型偶尔会提出错误的编辑，或尝试读取不存在的文件。我们按照原因对这些预期错误分类。InvalidArguments 和 UnexpectedEnvironment 用来表示模型失误以及上下文窗口中的矛盾。ProviderError 则表示 GenerateImage 或 WebSearch 等工具的供应商发生服务故障。

我们还有 UserAborted 和 Timeout 等其他分类。它们合在一起涵盖了大多数预期错误。

我们根据这些指标设置告警，用于捕捉进入生产环境的重大退化。未知错误始终属于缺陷，因此，只要任何工具的未知错误率超过固定阈值，我们就会收到告警。但预期错误究竟代表运行框架的缺陷，还是正常行为，往往不容易判断。

例如，grep 搜索超时可能源于工具性能问题，也可能只是代码库规模太大，而模型又构造了低效查询。为了解决这个问题，我们设置了异常检测告警。当预期错误显著超过基线时，告警就会触发。我们会按工具和模型分别计算基线，因为不同模型在工具调用上的失误率可能不同。

我们每周还会运行一个配有 skill 的 Automation。这个 skill 会教模型如何搜索日志，发现新出现或近期大幅增加的问题，并在待办列表中创建或更新工单，同时附上调查结果。我们大量使用 Cloud Agents，同时启动许多问题的修复工作，甚至可以直接从 Linear 触发这些智能体。

这个流程是我们为智能体运行框架构建自动化“软件工厂”的一部分。在今年早些时候的一次集中冲刺中，我们将意外工具调用错误降低了一个数量级。

为不同模型定制运行框架

我们的所有运行框架抽象都与具体模型无关，同时也能针对支持的每个模型进行深度定制。例如，OpenAI 模型接受过使用补丁格式编辑文件的训练，而 Anthropic 模型接受的是字符串替换训练。两类模型都可以使用任一种工具，但让模型使用不熟悉的工具会消耗更多推理令牌，也会产生更多错误。因此，在我们的运行框架中，每个模型都会获得它在训练期间使用过的工具格式。

这种定制非常深入，包括针对不同提供商甚至不同模型版本设计专用提示词。OpenAI 模型在遵循指令时往往更为字面和精确。Claude 则更依靠直觉，对不够精确的指令也更宽容。

如果我们在新模型发布前获得提前使用权限，就会从最接近的现有模型运行框架开始迭代。我们运行离线评测，查找模型容易困惑的地方。团队成员也会亲自使用模型并反馈问题。我们再根据这些问题调整运行框架。这个过程不断重复，直到模型与运行框架的组合达到我们认为可以发布的水平。

这项调优工作大多是让运行框架适配新模型的优势。不过，我们有时也会遇到真正的模型怪癖，并通过运行框架进行缓解。例如，我们发现某个模型出现了一种被我们称为“上下文焦虑”（context anxiety）的行为。随着上下文窗口逐渐填满，它会开始拒绝工作，并含糊地表示任务似乎太大。我们通过调整提示词减轻了这种行为。

支持对话过程中切换模型

支持用户在对话中途切换模型尤其棘手，因为不同模型有不同的行为、提示词和工具形式。

用户切换模型时，Cursor 会自动切换到对应的运行框架，并加载为该模型定制的提示词和工具集合。然而，新模型仍要使用这些工具来处理另一个模型生成的对话历史。这种历史超出了它训练时的数据分布。

为了解决这个问题，我们会加入专门指令，告诉模型它正在对话中途接替另一个模型。这些指令还会阻止它调用对话历史中出现、但不属于其自身工具集合的工具。

第二个挑战是，缓存由提供商和模型分别维护。切换模型意味着缓存未命中，导致切换后的第一轮更慢、成本更高。我们尝试过在切换时总结对话，为模型提供一份干净的摘要，从而减轻缓存损失。但是，如果用户已经深入处理一项复杂任务，摘要可能丢失重要细节。除非有明确的切换理由，否则我们通常建议在一段对话中始终使用同一个模型。

另一种避开对话中途切换难题的方式，是改用拥有全新上下文窗口的子智能体。我们最近为运行框架增加了一项能力，用户可以直接要求使用指定模型运行一个子智能体。

运行框架与软件开发的未来

AI 辅助软件工程的未来将属于多智能体系统。系统不会再把每个子任务都交给同一个智能体，而是学会在专业智能体和子智能体之间进行委派。例如，一个负责规划，一个负责快速编辑，另一个负责调试。每个智能体只承担最适合自己的工作。

要让这种模式良好运转，本质上是一个运行框架问题。系统需要知道应该派出哪个智能体，如何根据该智能体的优势描述任务，以及如何把各项结果衔接成连贯的工作流。编排这类协作的能力将存在于运行框架中，而不是任何单个智能体中。这意味着，运行框架工程一直是决定智能体能否成功的关键，而且今后只会变得更加重要。
