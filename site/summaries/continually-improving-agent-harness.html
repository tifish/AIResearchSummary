持续打磨 Agent Harness：好用的编码 agent，赢在模型之外

来源 Cursor ｜ 作者 Stefan Heule & Jediah Katz ｜ 2026-04-30
一句话总括：决定一个编码 agent 好不好用的，往往不是模型本身，而是包裹模型的那层 harness——上下文管理、质量度量、错误告警、按模型定制与多模型切换。本文是 Cursor 把 harness 当成"软件产品"来迭代的实操手册，核心结论是：进步靠痴迷式堆叠大量小优化，而非一次性突破。

01上下文窗口：从"加护栏"到"放手让 agent 自取"

早期模型不会自己挑上下文，Cursor 用大量护栏兜底；随着模型变强，护栏被逐一拆除，转向 agent 边干边拉取的动态上下文。这扭转了"喂得越多越好"的直觉——更强的模型反而需要更少的预置约束。
起点：2024 年末首个编码 agent 旧护栏：编辑后回灌 lint/类型错误 旧护栏：重写过少的文件读取 旧护栏：限制单轮工具调用数 现状：护栏"大多已成历史"
上下文工程的方向随模型能力反转——能力越强，越该给自由度而非加约束。

02怎么判断改动"更好"：Keep Rate + 语义满意度

benchmark 只是近似，真实质量靠线上 A/B 测。除了延迟、token 效率、缓存命中率这些直接指标，Cursor 用两个更接近"做得好不好"的信号：Keep Rate（agent 生成的代码在固定时间后仍留在用户代码库的比例，低=用户在反复返工），以及用 LLM 读用户回复判断满意度。
线下：自建套件 CursorBench + 公开 benchmark 满意信号：用户转向下一个功能 不满意信号：用户粘贴 stack trace 实验：贵模型做摘要，质量提升微乎其微→搁置
"代码留存率"把抽象的质量变成可量化指标——用户的后续行为本身就是最诚实的评分。

03追踪与修复退化：把工具错误降一个数量级

模型越多、harness 越复杂，bug 面越大。工具调用失败尤其有害：错误残留在上下文里造成"context rot"，拖累后续决策。Cursor 把错误分类管理，对未知错误（必为 bug）硬阈值告警，对预期错误按每工具/每模型算基线做异常检测，并用自动化 + Cloud Agents 批量修复。
分类：InvalidArguments / UnexpectedEnvironment ProviderError（厂商宕机）/ UserAborted / Timeout 每周自动化搜日志、建工单 成果：一个 sprint 内意外工具错误降低一个数量级（10×）
他们把可靠性工程做成了一座"软件工厂"——错误分类+基线告警+自动修复，让 agent 团队自己也用 agent 干活。

04按模型定制：连工具格式都对齐训练时的样子

harness 抽象是模型无关的，但会为每个模型深度定制。最具体的例子：OpenAI 模型训练时用 patch 格式编辑文件，Anthropic 模型用字符串替换——给错格式会多耗推理 token、多出错，所以各发各的。提示风格也分家：OpenAI 更字面精确，Claude 更直觉、更能容忍含糊指令。
OpenAI：patch 格式编辑 + 字面精确 Anthropic/Claude：字符串替换 + 直觉宽容 趣闻：某模型出现"context anxiety"，上下文将满时拒绝任务，靠改提示词缓解
"给模型它训练时熟悉的工具"是一条朴素却高杠杆的原则——契合训练分布，比工具本身强不强更重要。

05对话中途换模型：能做，但不推荐

中途切换会让新模型面对一段"别人写的、超出自己训练分布"的历史，还会触发缓存未命中、首轮更慢更贵。Cursor 加了接管指令、引导模型避开不属于自己的工具，也试过切换时摘要对话来省缓存，但复杂任务下摘要会丢细节。结论：一段对话尽量用同一模型，要换不如开 subagent（从干净上下文起步）。
问题：历史 out-of-distribution 代价：缓存provider/模型专属→必然 miss 缓解：接管指令 / 切换时摘要 建议：坚持单模型，或用 subagent
未来是多智能体的：调度哪个 agent、怎么裁剪任务、如何缝合结果——这套编排能力将住在 harness 里，而非任何单个 agent。

最值得带走的洞见

  1. 赢在模型之外。同一个模型放进精调过的 harness 会明显更快、更聪明、更省——harness 是被低估的竞争力来源。
  2. 进步是堆出来的。step-change 罕见，真正的提升来自痴迷式地叠加大量小优化。
  3. 能力越强，约束越少。护栏被逐步拆除，动态上下文取代静态预置——上下文工程的方向随模型变强而反转。
  4. 用行为衡量质量。Keep Rate 和"用户是转下一个功能还是粘 stack trace"，比 benchmark 更贴近真实好坏。
  5. 可靠性可工程化。错误分类 + 每模型基线告警 + 自动修复，让意外工具错误一个 sprint 降 10×；未来的多 agent 编排同样是 harness 的命题。
