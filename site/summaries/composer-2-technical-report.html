Composer 2 技术报告：把"真实开发环境"搬进模型训练

作者 Sasha Rush · 来源 Cursor 博客 / arXiv 技术报告

一句话总括：Composer 2 的真正贡献不是又刷高了一个分数，而是公开了一条"以代码为重的持续预训练 → 在真实 Cursor 会话里做大规模 RL → 用真实任务基准评测"的完整配方，并证明这条路线能在更低推理成本下逼近前沿模型。

01训练分两段，预训练打底决定 RL 上限

Composer 2 基于开源模型 Kimi K2.5，先做侧重代码的持续预训练以加深编程知识，再做大规模强化学习提升端到端的智能体表现。报告发现一个可复用的规律：降低预训练损失能可靠地提升下游 RL 的表现——更好的基础知识会稳定地翻译成更强的智能体。

别把预训练和 RL 当两件独立的事；地基质量直接设定了 RL 能达到的天花板。

02RL 在"和部署一模一样"的环境里进行

RL 训练发生在真实的 Cursor 会话中，使用与上线模型完全相同的工具和 harness，任务分布覆盖开发者真正会让 Composer 做的全部范围。结果是平均表现和 best-of-K 表现同时提升——这意味着模型在学习新的解题路径，而不只是把概率集中到已知答案上。

环境保真度本身就是能力来源：训练环境越接近真实使用，RL 越能拓展解空间而非收窄它。

03CursorBench：用真实代码会话取代失真的公开基准

公开基准常常脱离开发者的真实工作——任务被过度规定、解法狭窄、代码库太小。Cursor 用自家工程团队的真实编码会话构建了 CursorBench：提示词简短而模糊，解法往往需要跨多个文件、数百行的改动。它贯穿训练与评测，使模型持续对齐真实问题。

评测即指北针；当基准换成真实任务，"刷分"和"变得有用"才开始重合。

04同时赢下精度和成本

Composer 2 在 CursorBench 得 61.3（较 Composer 1.5 提升 37%，与最强前沿模型相当），公开基准上 SWE-bench Multilingual 得 73.7、Terminal-Bench 得 61.7。关键在于它以显著更低的推理成本达成这些分数，在精度—成本权衡上取得帕累托最优。

对交互式开发工作流，"够好且够便宜够快"比"绝对最强"更重要；低成本是产品级可用性的硬指标。

05支撑这一切的基础设施

训练需要大量底层工程：为 Blackwell GPU 定制的低精度 MoE 训练核、横跨多区域的全异步 RL 流水线，以及内部计算平台 Anyrun——可同时运行数十万个沙箱化编码环境。报告还覆盖权重同步、容错与环境保真度的完整技术栈，并附有训练配方的消融实验和智能体行为塑造方法。

前沿编程智能体的瓶颈正从模型架构转向"能否大规模、高保真地模拟真实开发环境"。

最值得带走的洞见

  1. 地基决定上限：降低预训练损失能可靠提升下游 RL，预训练与 RL 必须当作一条链来优化。
  2. 环境保真 = 能力：在与部署一致的真实 Cursor 会话里做 RL，让模型学到新解法（平均与 best-of-K 同涨）。
  3. 评测要对齐现实：CursorBench 用真实、多文件、模糊提示的任务取代失真公开基准，是对齐真实价值的关键。
  4. 低成本即护城河：61.3 分 + 显著更低推理成本的帕累托最优，比单纯的最高分更适合交互式开发。
  5. 基础设施是隐形主角：Blackwell 低精度核、跨区域异步 RL、数十万沙箱环境（Anyrun）才是这套配方能跑通的前提。
