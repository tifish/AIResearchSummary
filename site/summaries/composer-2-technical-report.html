一句话总括

Composer 2 的真正贡献，是把编码模型训练从静态题库推进到接近真实 Cursor 工作流的强化学习闭环。

报告展示了一个面向 agentic software engineering 的路线：先用代码密集数据继续预训练，补足模型基础能力；再在真实工具、真实 harness 和真实任务分布中做大规模强化学习，让模型学会完成端到端开发任务。

更好的基座知识，会放大后续 RL 的收益

Composer 2 先在强调代码的数据混合上继续预训练，再进入大规模强化学习。报告的核心观察是，预训练损失降低后，下游 RL 表现也会变好。

这改变了对编码 agent 的理解：RL 不是单独的魔法步骤，它依赖更扎实的代码知识作为起点。

真实会话比过度规整的公开题更能暴露能力差距

CursorBench 来自 Cursor 工程团队的真实编码会话，任务提示可能很短、含糊，解法可能需要跨许多文件修改数百行代码。

这说明评估重点从“能否答对小而清晰的问题”，转向“能否在不完整需求和真实代码库里推进工作”。

性能提升不是只看准确率，还要看交互成本

Composer 2 在 CursorBench 得分 61.3，比 Composer 1.5 提升 37%；在公开基准上，SWE-bench Multilingual 得分 73.7，Terminal-Bench 得分 61.7。

CursorBench 61.3 较 Composer 1.5 +37% SWE-bench Multilingual 73.7 Terminal-Bench 61.7

报告同时强调它的推理成本显著低于可比模型，因此在准确率和成本之间形成适合交互式开发的权衡。结论是，开发者工具里的模型不只要强，还要足够便宜、快速，才能高频嵌入日常工作。

RL 训练要模拟部署环境，才会学到可迁移行为

Composer 2 的 RL 训练发生在现实 Cursor 会话中，使用与部署模型相同的工具和 harness，并覆盖开发者实际会提出的多类问题。

平均表现和 best-of-K 表现同时改善，暗示模型不是简单收缩到已知答案，而是在学习新的解决路径。

基础设施成为模型能力的一部分

训练 Composer 2 需要面向 Blackwell GPU 的自定义低精度 MoE kernel、跨多区域的全异步 RL pipeline，以及用于运行数十万个沙盒编码环境的内部平台 Anyrun。

这表明高质量编码 agent 的瓶颈不只在模型结构，还在环境保真、权重同步、容错和大规模沙盒执行能力。

最值得带走的洞见

  • 编码模型的关键进步来自训练环境和真实开发流程的对齐，而不只是扩大模型或优化公开榜单。
  • 继续预训练仍然重要，因为更低的预训练损失会提高后续强化学习的上限。
  • CursorBench 的价值在于捕捉真实软件工程中的含糊需求、大范围修改和多文件协作。
  • 对交互式开发工具来说，准确率必须和推理成本一起看；低成本的强模型才更容易成为日常工作流的一部分。
  • 大规模沙盒环境、异步 RL 和高效 MoE 训练基础设施，正在成为编码 agent 竞争力的核心组成。
