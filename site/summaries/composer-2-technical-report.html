关于 Composer 2 的技术报告

A technical report on Composer 2

一句话总括：Composer 2 的贡献不是单一 benchmark 提分，而是把真实开发会话放进训练、评测和基础设施闭环，让代码代理更接近开发者日常使用场景。

真正的变化是训练对象变了

Composer 2 先在偏代码的数据混合上继续预训练。目标是补强基础代码知识。

随后，模型在接近真实 Cursor 的会话中做大规模强化学习。它使用部署时相同的工具和 harness。

这改变了读者对 AI 编程模型的理解。模型能力不只来自更大的参数或更多题库，也来自训练环境是否足够像真实工作。

基础知识
预训练 loss 降低后，下游 RL 表现也提高。更好的底座会转化为更好的代理。
解法探索
RL 同时提升平均表现和 best-of-K 表现，说明模型可能学到了新的解决路径。
真实约束
训练任务覆盖开发者会问 Composer 的多种问题，而不是只优化窄题。

从底座到代理的流程

继续预训练

Composer 2 基于开放模型 Kimi K2.5。Cursor 用更强调代码的数据继续训练，以深化编码知识。

真实会话 RL

强化学习发生在仿真的 Cursor 会话里。模型看到的是工具、交互和任务分布的组合。

端到端评测

CursorBench 贯穿训练和评估。它让模型优化目标更接近真实开发问题。

为什么 CursorBench 重要

维度常见公共 benchmarkCursorBench
任务描述 往往更明确，约束更完整。 来自真实工程会话，prompt 可能很短，也可能含糊。
代码改动 解法范围较窄，代码库较小。 可能需要跨很多文件修改数百行代码。
评测意义 适合衡量特定题型能力。 更能衡量交互式软件工程代理的可用性。
关键洞见：如果评测不包含模糊需求、跨文件修改和真实工具链，模型看起来会更强，但未必更能帮开发者完成工作。

性能背后是工程系统

低精度 MoE kernel

Cursor 为 Blackwell GPU 上的 MoE 训练做了自定义低精度 kernel。这说明模型训练收益依赖底层算子效率。

跨区域异步 RL

强化学习 pipeline 是全异步并跨多个区域运行。系统要同时处理吞吐、同步和容错。

Anyrun 沙盒环境

Anyrun 用来运行数十万个沙盒编码环境。它支撑了大规模、接近真实的代理训练。

成本与准确率

文章强调 Composer 2 的推理成本显著低于可比模型。对交互式开发工具来说，低成本会直接影响可用频率。

最值得带走的洞见

真实环境是能力的一部分Composer 2 的训练重点是模拟 Cursor 的真实使用场景。代码代理的强弱，越来越取决于环境、工具和任务分布是否一致。
RL 不是只会压榨旧答案平均表现和 best-of-K 同时提升，说明模型可能在学习更多解题路径。
benchmark 要贴近工作流短而模糊的需求、跨文件修改和大规模代码库，才更接近开发者每天遇到的问题。
基础知识仍然关键继续预训练降低 loss 后，后续 RL 更有效。代理能力仍要建立在可靠底座上。
低成本决定产品化在准确率之外，推理成本是交互式 AI 编程工具能否长期高频使用的核心指标。
