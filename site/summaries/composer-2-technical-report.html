Composer 2 技术报告：从开源基座到大规模 RL 的完整配方

来源 Cursor ｜ 作者 Sasha Rush ｜ 2026-03-27

一句话总括：这份技术报告真正的价值，不在于又一个高分，而在于把一个前沿 agentic 编码模型从头训出来的全流程摊开了——在开源基座 Kimi K2.5 上做以代码为主的继续预训练，再在与线上完全一致的 Cursor 环境里做大规模 RL，并用真实工程会话构成的私有评测来对齐"开发者实际要的东西"。结果是用显著更低的推理成本换来前沿级别的能力。

训练方法两阶段：先把代码知识喂厚，再用 RL 练成 agent

Composer 2 分两步走。第一步是继续预训练：在偏重代码的数据混合上，加深基座模型的编码知识。第二步是大规模强化学习，端到端提升 agent 表现。报告给出一个可靠的传导关系——降低预训练 loss，下游 RL 表现就更好，更扎实的基础知识能稳定地转化为更强的 agent。

1
在开源基座 Kimi K2.5 上做继续预训练，强化代码知识。
2
在真实 Cursor 会话里做 RL：用与线上部署完全相同的工具和 harness，问题分布覆盖开发者交给 Composer 的全部任务类型。

关键观察：RL 同时抬高了平均表现和 best-of-K 表现。后者尤其重要——它说明模型是在学到新的解题路径，而不是把概率单纯压向已知答案。

结论：把训练环境做到和部署环境一模一样，是让 RL 学到"真本事"而非"刷分技巧"的前提。

评测公开榜单不够真，就自己造 CursorBench

做编码模型的一个核心难题是：公开 benchmark 往往不像开发者真正做的活——任务被过度规定、解法狭窄、代码库还很小。Cursor 的应对是从工程团队的真实编码会话里构建 CursorBench，刻意保留两个真实特征：

  • 提示简短而含糊，不会把要求喂到嘴边；
  • 解法需要跨多个文件改动数百行，而非一个小函数。

这套 benchmark 贯穿训练与评测全程，用来持续把模型拉回到"真实问题"这条线上。

结论：评测集本身就是产品的一部分——它决定了模型被优化去解决"什么问题"。

在 CursorBench 上 61.3 分，与最强前沿模型相当；公开榜单同样站得住。但报告强调的重点不是单一分数，而是成本/精度的帕累托最优：它在显著更低的推理成本下达到这些成绩，对需要频繁交互的开发者工作流尤其划算——快、便宜、还够强，才是交互式编码真正在乎的指标。

结论：对 agent 模型而言，"在什么成本下达到什么精度"比孤立的榜单分数更有说服力。

基建支撑大规模 RL 的全栈：Blackwell 核、跨地异步流水线、Anyrun

训练 Composer 2 需要大量底层工程，报告把这套全栈也讲透了：

组件作用
自定义低精度核在 Blackwell GPU 上做高效的 MoE 训练
全异步 RL 流水线跨多个地域运行，覆盖权重同步与容错
Anyrun内部算力平台，运行数十万个沙箱化编码环境

报告还覆盖了权重同步、容错和环境保真度的具体做法——后者正是前面"在真实 Cursor 会话里做 RL"能成立的工程地基。此外还包含训练配方的消融实验、agent 行为塑造方法，以及评测套件的设计。

结论：agentic 模型的能力上限，越来越被"能否大规模、高保真地复现真实运行环境"这件基建活儿决定。

最值得带走的洞见

  1. 更好的基础知识可靠地换来更强的 agent。"降低预训练 loss → RL 下游更好"是一条被验证的传导链，意味着预训练阶段的投入不会在 RL 里白费。
  2. RL 在学新解法，而非收敛旧解。平均和 best-of-K 同时提升，是判断 RL "真的扩展了能力边界"而非单纯刷分的关键信号。
  3. 评测集决定优化方向。CursorBench 用真实、含糊、跨文件的任务替代过度规定的公开题，把模型持续对齐到开发者实际工作。
  4. 成本/精度的帕累托才是交互式工作流的真指标。用更低推理成本达到前沿水平，比孤立的高分更有商业意义。
  5. 能力越来越是基建的函数。Blackwell 低精度核、跨地异步流水线、Anyrun 的数十万沙箱环境——高保真复现真实环境的工程能力，正成为模型质量的隐形决定项。
