完整译文

Composer 的开发方式有一个值得注意的特点：我们会利用模型的旧版本，改进后续版本的训练过程。

环境配置是这种自举方式最明确的应用机会之一。强化学习（RL）训练需要可运行的环境。如果环境一开始就是坏的，模型就会浪费 token 调试配置，而不是学习如何解决问题。最糟糕时，错误的环境会让问题彻底无法解决。最终结果是消耗了算力，却得不到任何奖励信号。

为了解决这个问题，我们构建了 Composer autoinstall。该系统使用早期的 Composer 模型，从未经配置的仓库检出中自动创建可用的 RL 环境。训练最新版本 Composer 2 时，我们使用了它的前代 Composer 1.5 来管理这一过程。我们发现，现代编程模型不只会逐步执行说明。为了成功完成配置，它们会竭尽所能地模拟项目依赖，并通过测试确认配置确实有效。

更好的环境意味着更好的训练信号

与模型开发的许多部分一样，autoinstall 的灵感来自 Cursor 的生产系统。Cursor cloud agents 提供了一项功能，可以为用户自动配置云端环境，让代理在模拟环境中处理项目。代理从一次 git 检出开始，安装软件包、配置设置，并执行基础检查，确认代码可以运行且保持稳定。后续请求因此能从正确配置的环境开始。

对 RL 训练来说，环境问题更加核心，但也更具挑战。autoinstall 从一个仓库出发，目标是创建可运行的模拟基础版本，以便解决未来尚未出现的编程问题。这个基础环境十分关键，因为 Composer 会在一整套工具的支持下接受训练。这些工具包括编程语言的 lint 命令、搜索工具，以及在沙箱中使用 shell 的能力。如果环境没有正确配置，训练效率就会下降，还可能白白消耗算力而得不到奖励信号。

一个代理定义成功，另一个代理尝试实现

Autoinstall 分两个阶段进行。在第一个“目标设定”阶段，我们向 Cursor 代理提供固定检出版本的代码库。然后要求它提出 10 条命令，并概括说明环境正确配置后，这些命令应该产生什么输出。

代理会查看环境中的 README 或 Makefile，也会尝试符合相应编程语言习惯的命令。例如 uv 这样的项目管理器，或 clippy 这样的 lint 工具。代理通常会找出配置命令、可用的测试命令，以及可执行程序的启动命令。

第二阶段会启用一个独立的 Composer 代理。我们向它提供环境的初始状态，以及从前述 10 条命令中选出的 3 条目标命令。代理随后探索代码库，并调用工具配置环境，使这些命令能够运行。完成后，我们会检查这 3 条命令是否都能执行，也会确认输出是否符合第一个代理给出的目标描述。如果不符合要求，第二阶段就会重新开始。如果重复 5 次后，代理仍无法把环境配置到令人满意的程度，我们就会丢弃这个环境。

Composer 会通过 autoinstall 尽可能完整地配置环境。为此，它会模拟缺失文件、创建占位图片，甚至建立假的数据库表。有些项目需要额外组件才能运行测试，例如 S3 文件夹或缺失的边车容器。Composer 通常也会模拟这些资源。它可能创建 MinIO 配置或 Docker 容器，让测试得以运行。为了支持长期运行的进程，我们允许 autoinstall 创建启动脚本，在 RL 环境开始使用时启动这些进程。

面向真实项目的 autoinstall

为了说明 autoinstall 的工作过程，我们来看一次真实实验。在这次实验中，我们用 autoinstall 配置了一个复杂的真实项目。这个项目是由 celo-org/celo-monorepo 实现的 Celo。它是一个大型区块链项目，包含多个重要依赖。Celo 很适合用于测试 autoinstall，因为安装过程需要管理大量依赖，测试时还要模拟认证流程。

在 autoinstall 的第一阶段，我们观察到代理会浏览项目文档和代码，以寻找关键安装命令。不过，仓库自带的文档相对有限。因此，代理还使用 Web 命令搜索项目的文档网站，寻找更多配置命令。它找到的大多数命令都与安装或测试有关，同时也加入了文档中一个使用该软件的基础最小应用。

在第二阶段，代理需要真正让这些命令运行起来。任务范围虽然很明确，但模型事先并不知道会遇到哪些问题。在这个案例中，它发现还需要安装 Foundry 等其他依赖。Foundry 来自一个相关仓库。代理通过 Web 搜索阅读了这个必要项目的文档。它还需要在当前环境中运行一个最小应用。第二阶段第一次执行时，代理没能让这个测试应用成功运行。第二次执行时，它发现可以创建一个模拟用户，在本地启动应用并满足要求。

自举下一代模型

Autoinstall 是利用旧模型自举 RL 流程的一个有趣案例。值得注意的是，Composer 2 目前在 Terminal-Bench 上的得分明显更高。Composer 2 的成绩为 61.7%，Composer 1.5 则为 47.9%。这个 benchmark 包含对模型配置开发环境能力的测试。这说明 Composer 2 将能为 autoinstall 提供更好的基础。我们预计，在未来的训练中，旧版 Composer 会在训练流程的更多环节发挥重要作用，其中包括运行管理、数据预处理和架构调优。
