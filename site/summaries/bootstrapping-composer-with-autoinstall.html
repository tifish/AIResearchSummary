用 autoinstall 引导 Composer

Bootstrapping Composer with autoinstall

Cursor 的核心贡献是把旧模型变成训练管线里的环境工程师，让下一代模型从可运行仓库开始学习，而不是先把算力花在修安装问题上。

Composer 2 的 Terminal-Bench 分数高于 Composer 1.5 的 47.9%。更强模型会反过来改进下一轮 autoinstall。

坏环境会污染强化学习信号

强化学习需要可运行的开发环境。仓库一开始就坏掉时，模型会把 token 花在依赖、配置和启动问题上。

最坏情况不是慢，而是无解。 任务无法运行时，训练过程可能没有奖励信号，只是在消耗计算资源。

因此，环境搭建本身成了训练质量的一部分。

autoinstall 的位置

它从未配置的代码仓库出发，创建一个可运行的模拟基线环境。这个环境随后用于未来未见过的编码任务。

结论：模型训练不只依赖题目质量，也依赖题目前置环境是否可靠。

旧模型的新用途

训练 Composer 2 时，Cursor 使用 Composer 1.5 来执行 autoinstall。旧模型负责安装包、改配置、补 mock，并验证命令是否能跑。

结论：上一代模型不只是被替换，还能成为下一代模型的训练基础设施。

两阶段流程把“能运行”变成可判定目标

1

定义成功

Cursor agent 阅读 README、Makefile 和项目结构。它提出 10 条命令，并描述正确环境下的预期输出。

洞见：成功标准先由一个模型写出来，减少后续执行阶段的含糊空间。

2

尝试达成

另一个 Composer agent 接收初始仓库和 3 条目标命令。它探索代码库并调用工具完成安装、配置和补依赖。

洞见：模型不只是照文档执行，而是在真实约束下修出一个可工作的环境。

3

验收或丢弃

系统运行全部 3 条命令，并检查输出是否符合第一阶段描述。最多 5 次仍失败，就丢弃该环境。

洞见：自动化训练管线需要明确的淘汰机制，不能让坏环境进入强化学习。

模型会补齐真实项目缺口

遇到的问题 autoinstall 的做法 为什么重要
缺少文件或静态资源 创建 mock 文件或占位图片。 训练任务可以触达业务逻辑，而不是卡在资源缺失。
缺少数据库或外部服务 创建假数据库表、MinIO 配置或 Docker sidecar。 模型能在隔离环境里近似真实依赖，减少不可控外部因素。
需要长期运行进程 生成启动脚本，在 RL 使用前启动必要服务。 环境状态可以复用，未来任务不必重复从零搭建。

Celo 案例说明难点在真实依赖链

Celo monorepo 是一个大型区块链项目，依赖多，文档相对稀疏，还需要模拟认证流程。autoinstall 先从代码和文档中找安装命令，再通过文档站和 Foundry 等外部依赖补齐缺口。

第一次运行最小应用失败后，第二次通过创建 mock 用户让应用本地启动。结论是，自动环境搭建的价值不在完美复刻生产，而在构造足够真实、足够可验证的训练场。

最值得带走的洞见

训练基础设施也可以被模型改进

Composer autoinstall 展示了一条自举路径。旧模型负责准备环境，新模型从更干净的训练信号中学习。

可运行性是奖励信号的前提

仓库跑不起来时，强化学习可能没有有效反馈。环境质量直接影响训练效率。

mock 是工程策略

占位文件、假表、MinIO 和 Docker sidecar 让复杂项目在训练沙箱里可操作。

验收必须具体

10 条候选命令、3 条目标命令和 5 次重试构成了清晰边界。坏环境会被及时淘汰。

能力提升会形成循环

Composer 2 在 Terminal-Bench 上显著高于 Composer 1.5。更强的模型会成为更好的下一轮环境安装器。
