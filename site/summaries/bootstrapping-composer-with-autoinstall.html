用上一代模型，给下一代模型造训练环境

Composer Autoinstall ｜ 作者：Shomil Jain、Joshua Warner、Andrew Zhai
一句话总括：Cursor 让旧版模型（Composer 1.5）自动把"未配置的代码仓库"变成"可运行的强化学习训练环境"，从而消除了 RL 训练里最隐蔽的浪费——模型不是在学解题，而是在替坏掉的环境擦屁股；这套机制最终自举出了更强的 Composer 2。

①真正的瓶颈不是模型，而是环境

RL 训练需要能真正跑起来的环境。如果环境一开始就是坏的，模型会把宝贵的 token 花在调试安装上，而不是学习解决问题；最糟的情况下，坏环境让题目根本无解，白白烧掉算力却拿不到任何奖励信号。

洞见：当模型能力足够强时，训练效率的天花板往往不在算法，而在"喂给它的环境干不干净"——这是一个被低估的工程问题。

②两个 agent 的接力：一个定标准，一个去达成

3

5 次
配置失败重试上限，超过即丢弃该环境

第一阶段（"目标设定"）让 agent 翻 README、Makefile，尝试 uv、clippy 等语言特定命令，提出安装、测试、启动命令及预期输出；第二阶段另一个 Composer agent 拿到初始环境和 3 条目标命令，实际动手配置，跑通后验证输出是否匹配，不匹配就重来。

洞见：把"定义成功"和"实现成功"拆给两个 agent，本质是给环境搭建装上了一个可自动验证的奖励函数。

③模型会"不择手段"把环境补全

为了让命令真正能跑，Composer 不止照搬步骤，而是主动 mock 缺失文件、生成占位图片、伪造数据库表；遇到需要 S3 目录或缺失的 sidecar 容器时，它会配出 MinIO、起 Docker 容器；对长时运行的进程，还会写一个启动脚本在 RL 使用前拉起服务。

洞见：现代编码模型对"让东西跑起来"表现出近乎执拗的能动性，这正是它能胜任脏活的关键。

④真实考验：大型区块链项目 Celo

在 celo-org/celo-monorepo 这个依赖庞杂的区块链项目上，第一阶段 agent 因官方文档稀疏，转而用网页搜索去项目文档站找配置命令；第二阶段它发现需要额外安装 Foundry 等关联依赖，再次上网查文档，并在第二次迭代时用"创建 mock 用户"的办法本地启动了最小应用，满足了认证流程的测试要求。

洞见：能边查边补依赖、遇阻换策略，说明 autoinstall 已能应对真实世界项目的不确定性，而非只在玩具仓库上有效。

⑤自举闭环：今天的产物喂养明天的训练

61.7%
Composer 2 在 Terminal-Bench 的得分
47.9%
Composer 1.5 的得分（用于本轮 autoinstall）

Terminal-Bench 恰好包含对"搭建开发环境"能力的测试。Composer 2 大幅领先，意味着它将成为下一轮 autoinstall 更强的基础模型——上一代造环境、训出下一代、下一代再造更好的环境。Cursor 预期未来旧模型还会参与 run 管理、数据预处理、架构调参等更多训练环节。

洞见：这是一个可见的递归自我改进回路——模型的能力提升直接转化为它改进自身训练流程的能力。

最值得带走的洞见

  1. 环境质量是 RL 的隐形天花板。坏环境烧算力、无奖励信号，搭好环境本身就是提升训练效率的关键杠杆。
  2. "定义成功 + 达成成功"两阶段拆分，给本来主观的环境搭建装上了可自动验证的奖励机制。
  3. 能动性比照搬指令更重要：mock 文件、伪造数据库、起 Docker/MinIO，模型靠"想方设法跑起来"才能处理真实项目。
  4. 61.7% vs 47.9% 的 Terminal-Bench 差距证明：搭环境能力可量化、可提升，且能直接回流到训练流程。
  5. 自举正在成为常态——上一代模型造环境喂下一代，未来还将扩展到数据、调参、run 管理，AI 改进 AI 的循环已经落地。
