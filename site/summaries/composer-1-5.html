介绍 Composer 1.5

Introducing Composer 1.5

一句话总括：Composer 1.5 的真正贡献，是展示编码模型可以通过更大规模的强化学习后训练、按任务难度调节推理量，并用自总结延长复杂任务。

20x 强化学习后训练在同一预训练模型上继续扩大。
>预训练 Composer 1.5 后训练计算量超过基础模型预训练计算量。
2 次 Terminal-Bench 2.0 中每个 model-agent 组合运行 2 次并报告平均值。
核心变化

强化学习规模扩大后，困难编码任务受益最大

数据和趋势：Cursor 表示 Composer 1.5 在同一基础模型上把 RL 训练推到 20 倍更远，并在真实世界编码问题的内部基准上快速超过 Composer 1。提升在困难任务上最明显。

结论和洞见：这说明编码智能不只来自更大的预训练模型。持续扩展后训练本身，也可能带来可预测的能力增长。

推理方式

“会思考”不等于一直慢

Composer 1.5 会生成 thinking tokens，用来理解代码库和规划下一步。Cursor 的设计重点，是让模型在简单问题上少思考，在困难问题上多思考。

01

识别任务难度

简单问题需要低延迟。困难问题需要更多探索。

02

分配思考预算

模型被训练成在容易任务上快速响应。

03

深入复杂代码

复杂任务中，思考阶段用于推理代码库和计划动作。

04

保持交互体验

结论是速度和智能可以通过训练目标一起优化。

能力对比

从 Composer 1 到 Composer 1.5，重点不是换底座，而是把后训练做深

维度 Composer 1 Composer 1.5 对 AI 影响的理解
训练路径 首个 agentic coding model。 同一预训练模型上进一步扩大 RL。 模型改进可以来自后训练扩展，而不一定依赖全新基础模型。
交互体验 作为上一代基线。 强调日常使用中的速度和智能平衡。 编码助手的价值不只是最高分，还包括响应节奏是否适合真实工作流。
长任务 文章未强调自总结能力。 上下文耗尽时可生成摘要并继续探索。 长程任务的瓶颈从上下文窗口，转向模型能否保留有效任务状态。

自总结把“上下文不够”变成了可训练的问题，而不是单纯等待更长窗口。

评估口径

公开基准的重点在于 harness 和运行设置

文章说明 Terminal-Bench 2.0 是 Laude Institute 维护的终端使用 agent 评估。Cursor 分数使用官方 Harbor 评估框架和默认 benchmark 设置。

Anthropic 分数使用 Claude Code harness，OpenAI 分数使用 Simple Codex harness。除 Composer 1.5 外，其他模型取官方榜单分数和 Cursor 基础设施运行分数中的最大值。

结论是：比较 agent 能力时，模型本身、执行框架和评估环境会共同影响结果。读者应同时看分数和评估口径。

最值得带走的洞见

这篇短文给出的信号，比一个版本发布更重要

RL 扩展仍在产生收益

20 倍规模的后训练带来持续提升，说明编码模型的可扩展性不只发生在预训练阶段。

困难任务是分水岭

提升最明显的地方在挑战性任务上，这更接近真实开发中的高价值场景。

思考需要按需发生

交互式模型不能总是慢，也不能总是浅。关键是根据任务难度分配推理。

自总结是长任务基础设施

模型在上下文耗尽时总结状态，可以继续寻找解决方案，并维持准确率。

评估需要看实现细节

Terminal-Bench 2.0 的分数依赖 harness、默认设置和运行次数。比较时不能只看单个数字。
