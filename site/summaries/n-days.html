衡量 LLMs 对 N-day 漏洞利用的影响

Measuring LLMs’ impact on N-day exploits

一句话总括：这项研究用 Firefox 和 Windows 内核实验表明，前沿 LLM 已经把补丁差异到可用 exploit 的过程从专家周级工作压缩到小时级自动化。

18个 Firefox SpiderMonkey 安全补丁被测试。
21个 Windows 内核本地提权漏洞被测试。
8个 Firefox 代码执行 exploit 由 Mythos Preview 生成。
8条 Windows lowpriv 到 SYSTEM 的完整链被生成。

N-day 风险为什么变得更紧迫

N-day 不是未知漏洞。它是已经公开并发布补丁的漏洞。

补丁本身会暴露修复点。攻击者可以做 patch diff，再反推漏洞触发方式。

过去这一步慢，因为逆向和利用开发需要稀缺专家。模型把这个瓶颈大幅削弱了。

真正改变的是速度和规模

Anthropic 并不声称 exploit 开发是攻击活动的全部。

目标发现、投递和规避检测仍然需要工作。

但最耗专家时间的一环被压缩后，补丁窗口本身就变得危险。

从补丁到利用的流程被缩短

1. 补丁发布
模型获得公开 diff、组件信息和可验证构建。
2. 定位漏洞
它比较修复前后行为，并寻找能稳定触发的输入。
3. 生成 PoC
Firefox 中，Mythos Preview 对 18 个漏洞中的 14 个生成 PoC。
4. 串成 exploit
最强模型继续把崩溃转成代码执行或 SYSTEM 提权。

两组实验的核心结果

场景 输入条件 模型结果 读者应更新的判断
Firefox SpiderMonkey 18 个补丁。模型有公开 diff、组件名、严重性和修复前后 jsshell。 Mythos Preview 生成 14 个 PoC，并在约 12 小时内生成 8 个代码执行 exploit。 即使是自动更新较快的浏览器，补丁发布到用户重启之间也足够危险。
Windows 内核 21 个 2026 年 1-2 月本地提权漏洞。模型只有二进制、符号、反编译和公开公告。 Mythos Preview 触发 18 个 BSOD，并生成 8 条完整提权链。 闭源软件并不能阻止模型化 exploit 开发，只是提高了任务难度。
风险评级 Microsoft 将 14 个漏洞评为较不可能或不可能被利用。 Mythos Preview 对其中 13 个生成 PoC，并对一个“Exploitation Unlikely”漏洞完成提权。 面向人类研究者校准的利用可能性评级，需要重新纳入模型能力。
“N-day”这个词已经低估了现实风险。对强模型来说，许多补丁差异更接近“N-hour”。

最值得带走的洞见

结论

传统补丁节奏建立在“武器化需要专家数周”的假设上。这个假设正在失效。

防守方不能只依赖月度发布、慢速灰度和延迟重启。

数据

Mythos Preview 在 Windows 测试中生成 8 条完整提权链，总 API 成本约 15,700 美元。

风险

工业控制、医疗设备和 IoT 更脆弱，因为它们常有固定维护窗口和停机限制。

机会

更快部署补丁仍然必要，但更持久的做法是减少漏洞供给。

方向

迁移到 Rust 等内存安全语言，以及启用 CFG、硬件 shadow stack 等缓解措施，可以一次性淘汰部分漏洞类别。
