与 Mozilla 合作提升 Firefox 安全性

Partnering with Mozilla to improve Firefox’s security

一句话总括：Claude Opus 4.6 把复杂软件的漏洞发现速度推到了新水平，但实验也显示，发现漏洞与构造可用攻击之间仍有明显距离。

漏洞发现速度已经发生质变

Claude 在两周内发现了 22 个漏洞。其中 14 个属于高危，接近 Firefox 在 2025 年修复的全部高危漏洞的五分之一。

研究团队最终扫描了近 6,000 个 C++ 文件，并提交 112 份独立报告。多数问题已在 Firefox 148 中修复。

这说明前沿模型正在把高强度漏洞研究从低频人工活动，转变为可以大规模并行执行的工作。

发现漏洞不等于完成攻击

团队对这些漏洞进行了数百次利用测试，花费约 $4,000 API 额度。Opus 4.6 最终只在 2 个案例中构造出可运行的利用程序。

这些程序只能在移除了部分现代浏览器防护的测试环境中运行。Firefox 的沙箱和纵深防御仍能降低其影响。

当前优势仍在防守方手里，但自动化利用已经不再只是理论上的风险。

有效的 AI 安全流程不是“让模型报漏洞”

01 · 发现模型分析代码，并生成能触发异常的最小测试输入。
02 · 验证任务验证器确认漏洞能否复现，减少误报。
03 · 修复候选补丁既要消除漏洞，也要通过回归测试。
04 · 协作维护者负责分流报告、判断严重性，并协调发布修复。

结论：模型能力只有嵌入可靠的验证和维护流程，才能真正转化为安全收益。

Mozilla 最看重三类证据：最小测试用例、详细概念验证和候选补丁。这改变了“AI 报告越多越好”的直觉。对维护者而言，可复现、可验证的报告才有实际价值。

最值得带走的洞见

前沿模型已能在高度复杂、经过充分测试的代码库中发现新漏洞。这不再只是 benchmark 上的能力。
漏洞发现与漏洞利用之间仍有数量级上的成本差距，防守方暂时拥有时间窗口。
“任务验证器”是关键基础设施。它让模型能够反复检查漏洞是否消失，以及软件功能是否保持完整。
高质量协调披露比单纯扩大扫描规模更重要。否则大量模型输出会进一步压垮维护者。

完整译文

AI 模型如今已经能够独立发现复杂软件中的高危漏洞。我们最近曾披露，Claude 在经过充分测试的开源软件中找到了 500 多个零日漏洞。零日漏洞是软件维护者尚不知晓的安全缺陷。

本文将介绍我们与 Mozilla 研究人员的一次合作。在两周时间里，Claude Opus 4.6 发现了 22 个漏洞。其中 14 个被 Mozilla 定为高危。这接近 Firefox 在 2025 年修复的全部高危漏洞的五分之一。换句话说，AI 正在大幅加快严重安全漏洞的发现速度。

在此次合作中，Mozilla 接收了我们提交的大量报告。他们帮助我们理解哪些发现值得提交漏洞报告，还通过 Firefox 148.0 向数亿用户发布了修复。他们的合作方式以及我们学到的技术经验，为 AI 安全研究人员与软件维护者如何共同应对当前局面提供了范例。

从模型评估走向安全合作

2025 年末，我们注意到 Opus 4.5 已经接近解决 CyberGym 中的全部任务。CyberGym 是一项测试大语言模型能否复现已知安全漏洞的 benchmark。我们希望设计一种更困难、更贴近现实的评估，其中包含更多技术复杂度较高的漏洞，例如现代网页浏览器中的漏洞。因此，我们根据 Firefox 过去公开的常见漏洞与暴露（CVE）构建了一个数据集，用来测试 Claude 能否复现这些漏洞。

我们选择 Firefox，是因为它的代码库非常复杂，也是全球测试最充分、最安全的开源项目之一。与我们此前用于测试模型的开源软件相比，它更适合检验 AI 发现全新安全漏洞的能力。每天都有数亿用户依赖 Firefox。浏览器漏洞尤其危险，因为用户经常会接触不可信内容，并依赖浏览器保护自身安全。

第一步是让 Claude 在旧版 Firefox 代码库中寻找已经确认的 CVE。Opus 4.6 能够复现很大一部分历史 CVE，这让我们感到意外，因为每个漏洞最初都需要研究人员付出大量精力才能发现。不过，我们仍不确定这个结果有多可信。至少有一部分历史 CVE 可能已经存在于 Claude 的训练数据中。

于是，我们让 Claude 在当前版本的 Firefox 中寻找全新漏洞。按照定义，这些漏洞此前不可能被报告过。我们最初把重点放在 Firefox 的 JavaScript 引擎上，之后又扩展到浏览器的其他部分。JavaScript 引擎是一个适合起步的目标。它是 Firefox 代码库中相对独立的组成部分，可以单独分析。保障它的安全也非常重要，因为它的攻击面很广。用户浏览网页时，它会处理来自外部的不可信代码。

仅仅探索了 20 分钟后，Claude Opus 4.6 就报告称，它在 JavaScript 引擎中发现了一个释放后使用（Use After Free）漏洞。这类内存漏洞可能让攻击者用任意恶意内容覆盖数据。我们的一名研究人员使用最新版 Firefox，在独立虚拟机中验证了该漏洞。随后，他把漏洞转交给另外两名 Anthropic 研究人员，这两人也完成了验证。之后，我们在 Mozilla 的问题跟踪系统 Bugzilla 中提交了漏洞报告。报告包含漏洞说明和一份建议补丁。补丁由 Claude 编写，并由报告团队验证，以帮助 Mozilla 分析根本原因。

就在我们验证并提交第一个 Firefox 漏洞的这段时间里，Claude 已经另外发现了 50 个能够导致崩溃的独立输入。在我们分析这些崩溃时，一名 Mozilla 研究人员主动联系了我们。双方讨论了各自的技术流程。我们又分享了几个经过人工验证的漏洞。之后，Mozilla 鼓励我们批量提交全部发现，不必逐一验证。即使我们不确定每个崩溃测试用例是否都涉及安全问题，也可以提交。整个工作结束时，我们扫描了近 6,000 个 C++ 文件，共提交 112 份独立报告，其中包括前文提到的高危和中危漏洞。多数问题已在 Firefox 148 中修复，其余问题将在后续版本中修复。

在外部软件中开展这类漏洞研究时，我们始终清楚，自己可能遗漏了代码库中的某些关键信息。这可能导致相关发现成为误报。我们会尽力自行验证漏洞，但错误仍无法完全避免。我们非常感谢 Mozilla 公开透明地介绍他们的分流和分析流程。他们还帮助我们调整方法，确保提交的是他们关心的测试用例，即使其中并非每一个最终都与安全有关。此后，Mozilla 的研究人员也开始在内部尝试使用 Claude 开展安全工作。

从发现漏洞到编写初级利用程序

为了衡量 Claude 网络安全能力的上限，我们还设计了一项新评估，用于判断 Claude 能否利用我们发现的漏洞。换句话说，我们想知道 Claude 是否也能开发黑客会使用的工具，借助这些漏洞执行恶意代码。

为此，我们向 Claude 提供了此前提交给 Mozilla 的漏洞，并要求它逐一尝试构造利用程序。为了证明漏洞已被成功利用，我们要求 Claude 演示一次真实攻击。具体来说，它必须像攻击者一样，在目标系统中读取和写入本地文件。

我们从不同起点进行了数百次测试，消耗了约 4,000 美元的 API 额度。即便如此，Opus 4.6 最终也只在两个案例中成功把漏洞转化为利用程序。这说明了两件事。第一，Claude 寻找漏洞的能力远强于利用漏洞的能力。第二，发现漏洞的成本比构造利用程序低一个数量级。不过，即使只在少数案例中成功，Claude 能够自动开发初级浏览器利用程序这一事实仍然令人担忧。

这里必须强调“初级”这个限定词。Claude 编写的利用程序只能在我们的测试环境中运行。这个环境有意移除了现代浏览器中的部分安全功能，其中最重要的是沙箱。沙箱的作用正是降低这类漏洞造成的影响。因此，Firefox 的“纵深防御”原本可以有效缓解这些特定利用程序带来的风险。不过，能够逃逸沙箱的漏洞并非没有出现过。Claude 的攻击也是形成端到端利用链所需的一个组成部分。关于 Claude 如何开发其中一个 Firefox 利用程序，可以阅读我们的 Frontier Red Team 博客。

AI 赋能的网络安全将走向何方

AI 已经显现出开发漏洞利用程序的早期能力。这进一步说明，防守方必须加快发现和修复漏洞的流程。为此，我们希望分享在此次分析中总结出的一些技术与流程最佳实践。

首先，我们在研究“补丁智能体”时开发了几种方法，希望帮助维护者使用 Claude 等大语言模型，更快地分流和处理安全报告。补丁智能体会利用大语言模型开发并验证漏洞修复。1

根据我们的经验，当 Claude 能够使用另一种工具检查自己的工作时，它的表现最好。我们把这类工具称为“任务验证器”（task verifier）。它是一种可信方法，用来确认 AI 智能体的输出是否真正实现了目标。智能体探索代码库时，任务验证器可以提供实时反馈。智能体因而能够持续深入迭代，直到成功完成任务。

任务验证器帮助我们发现了上文介绍的 Firefox 漏洞。2在另一项研究中，我们还发现它们有助于修复缺陷。一个优秀的补丁智能体至少需要验证两件事。第一，漏洞是否确实已经消除。第二，程序原本的功能是否得到保留。在我们的工作中，我们构建了自动化工具，用于测试应用补丁后能否再次触发原始漏洞。我们还单独运行测试套件，以发现回归问题。回归是指某项改动意外破坏了其他功能。我们认为，各项目的维护者最清楚如何为自己的代码库构建这些验证器。关键在于，为智能体提供一种可靠方法，同时检查漏洞消除和功能保留，可以大幅提高输出质量。

我们无法保证所有通过这些测试的智能体补丁都可以立即合并。不过，任务验证器能增强我们的信心，让我们相信生成的补丁可以修复特定漏洞，同时保留程序功能。这达到了一个可信补丁应满足的最低要求。当然，在审查 AI 编写的补丁时，我们建议维护者采用与审查任何外部贡献者补丁相同的严格标准。

再从更广的视角来看漏洞和补丁提交流程。我们知道，维护者已经被大量工作压得喘不过气。因此，我们的方法是向维护者提供足够的信息，让他们能够信任并验证报告。Firefox 团队指出，我们提交的材料中有三个部分对建立信任至关重要：

  • 随附的最小测试用例
  • 详细的概念验证
  • 候选补丁

我们强烈建议使用大语言模型漏洞研究工具的研究人员，在提交基于这类工具输出的报告时，也提供类似的验证证据和可复现材料。

我们还发布了《协调漏洞披露运作原则》，介绍与维护者合作时会采用的程序。目前，我们的流程遵循行业标准惯例。不过，随着模型不断进步，我们可能需要调整流程，使其跟上模型能力的发展。

紧迫的时刻

前沿语言模型如今已是世界一流的漏洞研究者。除了在 Firefox 中发现的 22 个 CVE，我们还使用 Claude Opus 4.6 在其他重要软件项目中发现了漏洞，例如 Linux 内核。在接下来的几周和几个月里，我们将继续介绍如何使用模型，以及如何与开源社区合作提升安全性。

Opus 4.6 目前发现和修复漏洞的能力，远强于利用漏洞的能力。这给了防守方优势。随着 Claude Code Security 最近以有限研究预览形式发布，我们正在把漏洞发现和修复能力直接提供给客户及开源项目维护者。

不过，从能力进步的速度来看，前沿模型在漏洞发现和漏洞利用之间的差距不太可能长期存在。如果未来的语言模型突破了利用漏洞这道障碍，我们就需要考虑增加保护措施或采取其他行动，防止模型被恶意行为者滥用。

我们呼吁开发者利用当前的时间窗口，加倍努力提升软件安全性。Anthropic 计划大幅扩展网络安全工作。具体措施包括与开发者合作寻找漏洞，并遵循上文所述的协调漏洞披露流程。我们还将开发工具，帮助维护者分流漏洞报告，并直接提出补丁。

如果你有兴趣支持我们的安全工作，包括为开源软件漏洞发现编写新的脚手架，分流、修复和报告漏洞，以及为 AI 时代构建健全的协调漏洞披露流程，欢迎申请加入 Anthropic。
