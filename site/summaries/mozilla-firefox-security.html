一句话总括

Claude 在真实 Firefox 代码库中快速发现高危漏洞，说明 AI 已开始改变防御方寻找和修复漏洞的速度。

这篇文章的真正贡献不是宣布一个新 benchmark，而是展示 AI 安全研究如何进入真实开源项目：模型发现问题，研究员验证，维护者批量 triage，并最终把修复交付给数亿用户。

22两周内发现的 Firefox 漏洞
14被 Mozilla 评为高危
112最终提交的独立报告

AI 已能在高质量代码库里找到严重漏洞

Anthropic 称 Claude Opus 4.6 在与 Mozilla 的合作中，两周发现 22 个 Firefox 漏洞，其中 14 个是高危，约等于 2025 年 Firefox 已修复高危漏洞数量的近五分之一。

这改变了一个常见判断：AI 不只是能在练习题里复现旧漏洞，也能在复杂、成熟、测试充分的浏览器代码中发现新问题。

速度优势来自“模型探索 + 人类验证 + 维护者 triage”

Claude 在约 20 分钟探索后报告了 JavaScript 引擎中的 Use After Free；Anthropic 研究员在独立虚拟机中验证后提交到 Bugzilla，并附上说明和候选补丁。

在团队验证第一例漏洞期间，模型已经产生 50 个额外独立崩溃输入；最后研究覆盖近 6,000 个 C++ 文件，提交 112 份独立报告，多数问题已在 Firefox 148 修复。

关键变化是流程：当发现速度远超人工验证速度时，维护者需要可复现测试、详细 PoC 和候选补丁来提高 triage 效率。

发现漏洞比利用漏洞便宜得多

Anthropic 又测试 Claude 是否能把这些漏洞变成 exploit：数百次运行、约 4,000 美元 API 成本后，只成功了 2 次，并且只是能在目标系统读写本地文件的粗糙攻击。

这些 exploit 依赖去除现代浏览器部分安全防护的测试环境，尤其是没有完整沙箱；因此 Firefox 的纵深防御仍能缓解这些样例。

结论是防御方暂时仍有优势：当前模型更擅长发现和修复漏洞，而不是稳定武器化漏洞。

任务验证器会成为 AI 修复漏洞的核心工具

文章强调，Claude 在能“检查自己工作”的工具旁边表现最好。所谓任务验证器，就是可信地判断模型输出是否真的达成目标的机制。

对补丁代理来说，至少要验证两件事：漏洞是否被移除，以及原有功能是否没有回归。这个要求把 AI 生成补丁从“看起来合理”推进到“具备最低合并可信度”。

安全披露流程需要跟上模型能力

Mozilla 认为 Anthropic 报告中最有助于建立信任的三类材料是：最小测试用例、详细 PoC、候选补丁。

这说明 AI 漏洞报告不能只给结论；它必须给维护者足够证据，让对方能快速复现、判断严重性并部署修复。

窗口期不会一直存在

Anthropic 认为 Opus 4.6 当前仍明显更擅长发现和修复漏洞，而不是开发 exploit，但这种能力差距可能不会持续太久。

因此文章的政策含义很直接：现在应利用 AI 帮助开源维护者扩大漏洞发现、triage 和修复能力，同时为未来更强的自动化利用能力准备新的防护和使用限制。

最值得带走的洞见

  • AI 安全能力已经进入真实复杂软件，而不只是 benchmark。
  • 漏洞发现正在变得更便宜、更快，维护者的 triage 能力会成为瓶颈。
  • 自动化 exploit 仍困难，但已经出现早期成功信号。
  • 高质量 AI 漏洞报告应包含最小复现、PoC 和候选补丁。
  • 防御方应趁模型仍更擅长发现而非攻击时，加速修复体系建设。
