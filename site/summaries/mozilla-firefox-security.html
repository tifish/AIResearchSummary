Claude 两周在 Firefox 里找到 22 个漏洞：AI 已是世界级漏洞猎手，但还不擅长"利用"

来源：Anthropic · 2026年3月6日 · 原文链接 · 分类：Policy
一句话总括：这篇文章不是又一次"AI 能找 bug"的宣传，而是用一场与 Mozilla 的真实合作给出第一手的能力刻度——Claude Opus 4.6 两周内在全球最受信任的开源浏览器之一里找到 22 个漏洞（14 个高危），但花了约 $4,000、跑了数百次也只在 2 个案例里把漏洞做成可用的 exploit。"发现"已经成熟、"利用"仍然困难，这道差距决定了防守方此刻拥有一个不会持续太久的窗口期。

01两周、22 个漏洞、14 个高危：相当于 Firefox 全年高危漏洞的近五分之一

这不是在玩具代码上的演示。Mozilla 将其中 14 个评为高危——约占 Firefox 在 2025 年全年修复的所有高危漏洞的五分之一，修复已随 Firefox 148.0 推送给数亿用户。选择 Firefox 正是因为它既复杂、又是世界上测试最充分、最安全的开源项目之一，比此前用过的开源软件更难"挖到新洞"。

洞见：能在最难啃的代码上批量找到高危漏洞，说明前沿模型已经跨过"实验室能力"门槛，进入了对真实软件供应链有直接影响的阶段。

02从"复现历史 CVE"到"找全新漏洞"：JS 引擎里 20 分钟就报出第一个 Use-After-Free

研究始于一个评测动机：Opus 4.5 已接近解决 CyberGym 基准（复现已知漏洞）里的全部任务，于是团队想造一个更难、更贴近现代浏览器的评测。他们先让 Claude 在旧版 Firefox 里复现历史 CVE，复现率出乎意料地高——但这些 CVE 可能已在训练数据里。真正的检验是去当前版本找定义上不可能被报告过的新漏洞。结果，在 JS 引擎里 20 分钟，Claude 就报出一个 Use-After-Free（可被攻击者用任意恶意内容覆写数据的内存漏洞）。

在团队还在三人交叉验证、向 Bugzilla 提交这第一个漏洞的同时，Claude 已经又发现了 50 个独特的崩溃输入。Mozilla 随后主动联系，建议直接批量提交、不必逐个确认安全影响。最终团队扫描近 6,000 个 C++ 文件，提交 112 份独立报告，多数已在 Firefox 148 修复。

洞见：瓶颈已经从"模型能不能找到"转移到"人能不能审完"。Mozilla 让团队"先批量提交、再分诊"的做法，本身就是 AI 时代漏洞协作流程的一次现场改造。

03关键反差：找漏洞很在行，把漏洞变成武器却很难——$4,000 只换来 2 个 exploit

为了测能力上限，团队让 Claude 拿到这些已提交的漏洞、尝试写出真正的 exploit，并要求它在目标系统上读写一个本地文件来"证明攻击成功"。他们用不同起点跑了数百次、花掉约 $4,000 API 额度，Opus 4.6 最终只在 2 个案例中真正把漏洞做成了 exploit。

用于 exploit 测试的 API 额度

而且这 2 个还是"粗糙"的：它们只在被刻意去掉部分现代浏览器防护（最关键的是沙箱）的测试环境里有效。也就是说，Firefox 的"纵深防御"本可以挡住这些攻击。但能逃逸沙箱的漏洞并非闻所未闻，而 Claude 写出的攻击正是一条端到端利用链中的必要一环。

洞见：发现漏洞的成本比开发 exploit 低一个数量级。这条不对称是当下最重要的安全事实：它意味着同样的能力，给防守方带来的杠杆远大于给攻击者的。

04让 AI 可信地交付：task verifier 与"补丁代理"是关键工程

能让 Claude 在外部代码里稳定产出的核心，是"task verifier"（任务验证器）——一种可信的、能确认 AI 输出是否真的达成目标的工具，给智能体实时反馈、让它反复迭代到成功。它既帮助发现了上述漏洞，也用于修复：一个合格的"补丁代理"必须同时验证两件事——漏洞确实被移除、且程序原有功能没被破坏（不引入回归）。

面对"维护者已经忙不过来"的现实，团队的策略是给维护者足够的信任与验证材料。Firefox 团队点名三样东西最关键：

配套的最小复现用例
详细的概念验证（PoC）
候选补丁

团队也公布了《协同漏洞披露（CVD）操作原则》，目前遵循行业标准做法，但表示会随模型能力提升而调整。

洞见：AI 安全研究能否落地，不取决于模型多聪明，而取决于有没有"可验证的反馈回路"和"可被维护者信任的证据包"——这是把能力变成生产力的工程关口。

05窗口期：差距不会持续太久，呼吁开发者趁现在加固

除了 Firefox 的这些 CVE，团队还用 Opus 4.6 在 Linux 内核等重要项目中发现了漏洞，并以"限量研究预览"形式推出了 Claude Code Security，把漏洞发现与修复能力直接交到客户和开源维护者手中。当前"会找远胜于会用"的格局让防守方占优，但报告明确警告：以进步速度看，发现与利用之间的差距不太可能维持很久，一旦未来模型突破"利用"门槛，就需要额外的防护措施。

洞见：这篇文章本质是一份带数据的"倒计时通知"：现在的优势是暂时的，开发者应抓住这段窗口期主动加固，而不是等利用能力追上来。

最值得带走的 5 条洞见

  1. 能力已跨过门槛。在最难的开源代码上，Claude 两周找到 22 个漏洞、14 个高危，约占 Firefox 全年高危修复的五分之一——AI 已是世界级漏洞研究者。
  2. "找"与"用"严重不对称。发现漏洞比开发 exploit 便宜一个数量级；$4,000、数百次尝试只换来 2 个粗糙、且需关闭沙箱才生效的 exploit。
  3. 防守方现在占优，但有时限。这道差距是当前安全格局的核心红利，报告明确警告它"不会持续很久"。
  4. 流程比模型更决定成败。批量提交、task verifier、最小用例＋PoC＋候选补丁的"信任证据包"，以及 CVD 原则，才是让 AI 安全研究真正落地的关键。
  5. 这是一份倒计时通知。Claude Code Security 已进入研究预览，Anthropic 呼吁开发者趁窗口期主动加固软件，而非被动等待利用能力追上。

洞见：能在最难啃的代码上批量找到高危漏洞，说明前沿模型已经跨过"实验室能力"门槛，进入了对真实软件供应链有直接影响的阶段。

02从"复现历史 CVE"到"找全新漏洞"：JS 引擎里 20 分钟就报出第一个 Use-After-Free

研究始于一个评测动机：Opus 4.5 已接近解决 CyberGym 基准（复现已知漏洞）里的全部任务，于是团队想造一个更难、更贴近现代浏览器的评测。他们先让 Claude 在旧版 Firefox 里复现历史 CVE，复现率出乎意料地高——但这些 CVE 可能已在训练数据里。真正的检验是去当前版本找定义上不可能被报告过的新漏洞。结果，在 JS 引擎里 20 分钟，Claude 就报出一个 Use-After-Free（可被攻击者用任意恶意内容覆写数据的内存漏洞）。

在团队还在三人交叉验证、向 Bugzilla 提交这第一个漏洞的同时，Claude 已经又发现了 50 个独特的崩溃输入。Mozilla 随后主动联系，建议直接批量提交、不必逐个确认安全影响。最终团队扫描近 6,000 个 C++ 文件，提交 112 份独立报告，多数已在 Firefox 148 修复。

洞见：瓶颈已经从"模型能不能找到"转移到"人能不能审完"。Mozilla 让团队"先批量提交、再分诊"的做法，本身就是 AI 时代漏洞协作流程的一次现场改造。

03关键反差：找漏洞很在行，把漏洞变成武器却很难——$4,000 只换来 2 个 exploit

为了测能力上限，团队让 Claude 拿到这些已提交的漏洞、尝试写出真正的 exploit，并要求它在目标系统上读写一个本地文件来"证明攻击成功"。他们用不同起点跑了数百次、花掉约 $4,000 API 额度，Opus 4.6 最终只在 2 个案例中真正把漏洞做成了 exploit。

而且这 2 个还是"粗糙"的：它们只在被刻意去掉部分现代浏览器防护（最关键的是沙箱）的测试环境里有效。也就是说，Firefox 的"纵深防御"本可以挡住这些攻击。但能逃逸沙箱的漏洞并非闻所未闻，而 Claude 写出的攻击正是一条端到端利用链中的必要一环。

洞见：发现漏洞的成本比开发 exploit 低一个数量级。这条不对称是当下最重要的安全事实：同样的能力，给防守方带来的杠杆远大于给攻击者的。

04让 AI 可信地交付：task verifier 与"补丁代理"是关键工程

能让 Claude 在外部代码里稳定产出的核心，是"task verifier"（任务验证器）——一种可信的、能确认 AI 输出是否真的达成目标的工具，给智能体实时反馈、让它反复迭代到成功。它既帮助发现了上述漏洞，也用于修复：一个合格的"补丁代理"必须同时验证两件事——漏洞确实被移除、且程序原有功能没被破坏（不引入回归）。

面对"维护者已经忙不过来"的现实，团队的策略是给维护者足够的信任与验证材料。Firefox 团队点名三样东西最关键：

配套的最小复现用例
详细的概念验证（PoC）
候选补丁

团队也公布了《协同漏洞披露（CVD）操作原则》，目前遵循行业标准做法，但表示会随模型能力提升而调整。

洞见：AI 安全研究能否落地，不取决于模型多聪明，而取决于有没有"可验证的反馈回路"和"可被维护者信任的证据包"——这是把能力变成生产力的工程关口。

05窗口期：差距不会持续太久，呼吁开发者趁现在加固

除了 Firefox 的这些 CVE，团队还用 Opus 4.6 在 Linux 内核等重要项目中发现了漏洞，并以"限量研究预览"形式推出了 Claude Code Security，把漏洞发现与修复能力直接交到客户和开源维护者手中。当前"会找远胜于会用"的格局让防守方占优，但报告明确警告：以进步速度看，发现与利用之间的差距不太可能维持很久，一旦未来模型突破"利用"门槛，就需要额外的防护措施。

洞见：这篇文章本质是一份带数据的"倒计时通知"：现在的优势是暂时的，开发者应抓住这段窗口期主动加固，而不是等利用能力追上来。

最值得带走的 5 条洞见

  1. 能力已跨过门槛。在最难啃的开源代码上，Claude 两周找到 22 个漏洞、14 个高危，约占 Firefox 全年高危修复的五分之一——AI 已是世界级漏洞研究者。
  2. "找"与"用"严重不对称。发现漏洞比开发 exploit 便宜一个数量级；$4,000、数百次尝试只换来 2 个粗糙、且需关闭沙箱才生效的 exploit。
  3. 防守方现在占优，但有时限。这道差距是当前安全格局的核心红利，报告明确警告它"不会持续很久"。
  4. 流程比模型更决定成败。批量提交、task verifier、最小用例＋PoC＋候选补丁的"信任证据包"，以及 CVD 原则，才是让 AI 安全研究真正落地的关键。
  5. 这是一份倒计时通知。Claude Code Security 已进入研究预览，Anthropic 呼吁开发者趁窗口期主动加固软件，而非被动等待利用能力追上。
