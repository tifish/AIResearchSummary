当 AI 成为世界级漏洞猎手：Claude 与 Mozilla 合作加固 Firefox

速度才是真正的震撼点

关键不是"AI 能找漏洞"，而是它找的有多快、多密。

团队选 Firefox 正因为它是全球测试最充分、最安全的开源项目之一，是检验 AI 找"新"漏洞能力的高难度试金石。 在 JavaScript 引擎里，Claude 仅探索 20 分钟就报出第一个 Use-After-Free 内存漏洞；当研究员还在人工验证这一个漏洞时， Claude 已经又发现了 50 个独特的崩溃输入。

结论：瓶颈已经从"能不能发现"转移到"人类能不能跟上验证与修复的节奏"。

14 个高危，意味着 AI 一举顶上接近五分之一的年度工作量

这把"AI 找漏洞"从演示变成了有统计意义的产出。

Mozilla 将其中 14 个定级为高危，约等于 Firefox 在 2025 全年修复的全部高危漏洞的近 1/5。 大部分问题已在 Firefox 148 中修复，直接覆盖数亿用户。此前 Claude 还在其他经过良好测试的开源软件中发现过 500 多个零日漏洞，并已用于 Linux 内核。

结论：这不再是孤例，而是一种可规模化的新防御能力。

"发现"便宜，"利用"昂贵——防御者的窗口期

这是全文最具政策含义的一段。

为测能力上限，团队让 Claude 尝试把已发现的漏洞变成真正的攻击（读写目标系统本地文件）。运行数百次、花费约 4000 美元 API 额度后，Opus 4.6 仅在 2 个案例中真正成功，且这些 exploit 还很"粗糙"—— 只在刻意关闭了沙箱等现代浏览器防护的测试环境里有效。这说明：利用漏洞的成本比发现漏洞高一个数量级， Firefox 的"纵深防御"能挡住这些攻击。

结论：当下优势在防御方；但作者明确警告，发现与利用之间的差距"不太可能维持很久"。

给维护者的可复用方法论：让 AI 自检

真正能落地的，是协作流程而不只是炫技。

核心是"任务验证器（task verifier）"——给 agent 一个可信工具实时检验自己的输出是否达成目标，从而反复迭代。 对修复 agent 而言，必须同时验证两件事：漏洞确已消除，且原有功能未被破坏（跑测试套件防回归）。 Mozilla 也点明三样让报告"可被信任"的关键证据：最小测试用例、详细 PoC、候选补丁。Anthropic 并已发布其 协同漏洞披露（CVD）操作原则。

结论：AI 的产出要被维护者接纳，靠的是可验证、可复现的证据链，而非盲目信任。

最值得带走的洞见

  1. 速度是质变：20 分钟首个漏洞、验证一个的工夫又冒出 50 个崩溃输入——AI 把安全研究的瓶颈推向了人类的验证与修复端。
  2. 规模已有统计意义：14 个高危 ≈ Firefox 2025 全年高危的近 1/5，叠加此前 500+ 零日，证明这是可规模化能力而非个例。
  3. 攻防不对称是当前的关键事实：发现便宜、利用昂贵（数百次 + $4000 仅成功 2 例，且依赖关闭沙箱），优势暂在防御方。
  4. 窗口期会关闭：作者直言发现与利用的差距不会长久，呼吁开发者抓紧此刻加固软件、并预留未来的额外安全措施。
  5. 协作 > 自动化：透明的 triage 流程、task verifier、最小用例 + PoC + 候选补丁，才是让 AI 安全成果真正进入主线代码的前提。
