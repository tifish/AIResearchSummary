构建更好的 Bugbot

Building a better Bugbot

Cursor 的真正贡献不是又做了一个代码审查机器人，而是把 AI 审查质量转化为可测量、可实验、可持续改进的生产系统。

40发布后运行的主要实验数量。
52% → 70%+resolution rate 的提升，说明更多报告最终被作者修复。
0.4 → 0.7单次运行平均标记 bug 数提高。
200万+Bugbot 当前每月审查的客户 PR 数量。

指标改变了产品方向

早期 Bugbot 依靠工程师主观判断来减少误报。后来 Cursor 定义了 resolution rate：在 PR 合并时，用 AI 判断报告的 bug 是否真的被作者解决。

这个指标把“评论看起来有用”变成了“工程师是否采纳并修复”。它让团队能在线观察真实使用效果，也能离线用 BugBench 对真实 diff 和人工标注 bug 做对比。

为什么这很关键

AI 代码审查最容易被表面活跃度误导。评论多不等于有价值，点赞也不等于发现了真实缺陷。resolution rate 直接看最终代码是否改变，因此更接近业务价值。

从固定流水线到 agentic 架构

1

早期靠多轮并行审查

Bugbot 会用随机 diff 顺序运行八个并行 pass，再合并相似问题，用多数投票过滤只出现一次的发现。这降低了偶然误报。

2

生产化补上基础设施

Cursor 重写 Rust Git 集成，减少抓取数据，并加入速率限制监控、请求批处理和代理基础设施。没有这些工程层，模型能力无法稳定服务真实仓库。

3

最大增益来自代理式审查

新的 Bugbot 能围绕 diff 推理、调用工具、决定深入哪里。它不再只按固定流程走，而是在运行时拉取所需上下文。

旧方法和新方法的差别

维度 早期 Bugbot agentic Bugbot
上下文 更多依赖预先塞入的静态上下文。 按需调用工具，运行时补足动态上下文。
提示词 主要约束模型，避免误报。 鼓励更积极调查可疑模式，避免过度保守。
优化方式 依靠内部质检和直觉迭代。 用真实 resolution rate 和 BugBench 做爬坡实验。
真正影响 AI 审查质量的，不只是模型本身，也包括它能调用什么工具、何时获取上下文，以及团队用什么信号判断它是否有效。

下一步从发现走向验证和修复

Bugbot Autofix Beta

发现 PR bug 后自动启动 Cloud Agent 修复问题。这把审查从提醒推进到执行。

运行代码验证报告

未来 Bugbot 将能运行代码验证自己的 bug 报告。它有机会减少误报，并提高开发者信任。

常驻扫描代码库

Cursor 还在实验 always-on 版本。审查不再只等待 PR，而是持续观察代码库风险。

最值得带走的洞见

一AI 代码审查的核心指标应接近真实修复行为，而不是评论数量或主观反馈。
二40 次主要实验把每个 PR 解决的 bug 数从约 0.2 推到约 0.5，说明系统性爬坡能产生复利。
三agentic 架构让模型从被动读上下文，变成主动寻找证据。
四工具接口本身会塑造模型行为，小改动也可能影响结果。
五下一阶段的竞争点会从“发现问题”扩展到“验证问题并自动修复”。
