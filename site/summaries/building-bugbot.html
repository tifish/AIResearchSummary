把代码评审 Agent 做对：Bugbot 的质量爬坡之路

来源：Cursor 博客 · Jon Kaplan · 2026 年 1 月 15 日
一句话总括：Bugbot 的真正突破不在于「用 LLM 找 bug」，而在于发明了一个能在生产环境量化「找到的 bug 是否真被修复」的指标——解决率（resolution rate），从而把代码评审 agent 从「凭感觉调参」变成「有真实信号可以工程化爬坡」的产品。

01没有指标，就只有天花板

关键转折：从定性判断到可量化信号

最初的 Bugbot 完全靠内部工程师的主观反馈调优——哪个配置「看起来误报更少」就采用哪个。这种方式能上线，却无法回答最根本的问题：质量到底有没有在提升？没有度量，进步就有了天花板。

真正限制 AI 产品的往往不是模型能力，而是「你能不能客观衡量它是否变好」。

02解决率：用 AI 判断 bug 是否被真正修复

关键设计：在 PR 合并时回看最终代码

团队发明了「解决率」：在 PR 合并那一刻，用 AI 判断 Bugbot 报出的 bug 中有哪些被作者在最终代码里真正改掉了。为验证可信度，他们逐例与 PR 作者人工核对，发现 LLM 对「是否已解决」的分类几乎全部正确。这个指标直接回答了「Bugbot 是不是在找到工程师愿意修的真问题」。

比起评论里的点赞或轶事反馈，「被修复」才是代码评审价值的硬证据。

03早期靠并行投票，但很多直觉竟然是对的

关键方法：8 路并行 + 多数投票过滤误报

上线版本的流水线相当工程化：

  • 用随机化的 diff 顺序跑 8 路并行评审，逼模型走不同推理路径
  • 把相似 bug 归并成桶，多数投票过滤掉只出现一次的
  • 过滤无关类别（编译警告、文档错误），再交给验证模型抓误报
  • 与历史报告去重

有了 BugBench（带人工标注的真实代码 diff 基准）做离线评估、解决率做在线评估后，他们跑了数十次实验，不少改动反而让指标倒退——结果证明早期定性分析的许多判断其实是对的。

有了指标后最大的收获之一，是知道哪些「改进」其实是退步。

04转向全 Agentic 架构带来最大跃升

关键反转：从「克制以防误报」到「激进鼓励排查」

今年秋天，Bugbot 从固定流水线切换为完全 agentic 设计：agent 能推理 diff、调用工具、自己决定深挖哪里。这带来了最大的提升，也带来了相反的问题——agent 变得过于谨慎。于是提示词从「克制」彻底反转为「激进」，鼓励它排查每一个可疑模式、宁可多报。同时，信息从静态上下文转入动态上下文，模型在运行时按需拉取，工具接口的微小改动都对结果有放大效应。

从流水线到 agent，调优重心从「写死的步骤」转移到「工具设计与提示词姿态」。

05规模与下一步

关键体量：每月 200 万+ PR

如今 Bugbot 每月为 Rippling、Discord、Samsara、Airtable、Sierra AI 等客户评审超过 200 万个 PR，Cursor 内部所有代码也都跑 Bugbot。已推出能自动派生 Cloud Agent 修复 bug 的 Bugbot Autofix（Beta），接下来要让 Bugbot 能跑代码验证自己的报告、遇到复杂问题时做深度研究，并试验持续扫描代码库的「常驻版」。

底层基建同样关键：为可靠访问仓库，团队用 Rust 重写了 Git 集成并精简数据抓取。

最值得带走的 5 条洞见

  1. 可量化指标是上限的解锁器。「解决率」把代码评审从主观感受变成可工程化爬坡的目标，是质量翻倍的真正前提。
  2. 衡量「是否被采纳」，而非「是否被报出」。在合并时回看作者有没有真的改，比任何点赞都更接近价值本身。
  3. Agentic 让模型从被动执行变主动调查。但它会过度谨慎，需要把提示词姿态从克制反转为激进。
  4. 有了指标，才知道直觉哪里错、哪里对。数十次实验里很多「改进」反而是退步。
  5. 产品化 = 评审逻辑 + 基建。Rust 重写 Git、限流监控、批处理与代理基建，决定了它能否在真实约束下规模化运行。
