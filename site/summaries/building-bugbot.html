把 AI 代码审查从「能用」推到「生产级」的关键，不是更强的模型，而是一个能衡量「真修了」的指标

来源：Cursor 博客 · 2026 年 1 月 15 日 · 作者 Jon Kaplan · 原文链接

一句话总括：Cursor 的代码审查 agent Bugbot 真正的进步，来自先发明一个可量化真实价值的指标「解决率」，再围绕它做系统化「爬山」优化——并在切换到完全 agentic 架构后拿到最大一次跃升，把质量做到比发布时翻了几倍。

01没有指标，质量优化就有天花板

Bugbot 早期靠工程师内部「凭感觉」的定性评估迭代：哪种配置误报更少就采用哪种。它能做出一个超越市面工具的版本，但团队坦言——没有一个能量化的指标，就无法在真实环境里判断质量到底有没有变好，这给优化设了上限。

结论与洞见：LLM 产品的瓶颈常常不在模型，而在「你能不能测量它真的有用」。定性打磨能起步，但量化信号才能解锁规模化改进。

02「解决率」：让 AI 判断 bug 是否真被修

团队发明了核心指标解决率（resolution rate）：在 PR 合并时，用 AI 判断哪些被标出的 bug 在最终代码里真的被作者改掉了。开发该指标时，他们逐例与 PR 作者人工核对，发现 LLM 的「已解决/未解决」分类几乎全部正确，因此把它作为仪表盘上的主指标对客户公开。

为什么重要：比起「点赞数」或主观反馈，解决率直接回答了唯一关键问题——Bugbot 找到的是不是工程师愿意去修的真问题。这把评估从「噪声」变成了「信号」。

03有了真实信号，就能「爬山」而非靠直觉

有了解决率，团队开始双轨评估：在线用真实解决率、离线用 BugBench（人工标注真实代码 diff 的精选基准），围绕模型、提示词、迭代次数、验证器、上下文管理、类别过滤等跑了数十次实验。一个反直觉的发现是：很多改动反而让指标变差，而早期那些定性判断不少其实是对的。

洞见：系统化实验既能验证新点子，也能为旧直觉「正名」；没有度量时，你分不清进步和退步。

04最大跃升来自切换到完全 agentic 架构

早期 Bugbot 是固定流水线，最有效的招是并行多轮 + 多数投票去过滤误报：

  • 8 路并行、随机化 diff 顺序，逼模型走不同推理路径
  • 把相似 bug 归并到同一「桶」
  • 多数投票，剔除只在单轮出现的 bug
  • 每个桶合并成一条清晰描述
  • 过滤不想要的类别（如编译警告、文档错误）
  • 过验证器模型再抓一遍误报，并与历史结果去重

今秋切换到完全 agentic 设计后收益最大：agent 能在 diff 上自主推理、调用工具、决定往哪深挖，而不再走固定流程。这还带来一个反转——

固定流水线时代

需要克制模型，主要矛盾是压低误报。

Agentic 时代

模型反而太谨慎，于是改用「激进」提示，鼓励它调查一切可疑模式、宁可多标。

洞见：架构一变，提示策略要整体反转；同时把信息从静态上下文挪到动态上下文后，模型能在运行时自取所需，无需把一切预先喂给它。工具接口的微小改动也会对结果产生不成比例的影响。

05生产化靠的是不显眼的底层工程

要让 Bugbot 真正可用，团队用 Rust 重写 Git 集成、最小化拉取的数据量，并加上限流监控、请求批处理与代理基础设施以适配 GitHub 的约束；还引入 Bugbot rules，让各团队把「危险迁移、内部 API 误用」等代码库专属约束编码进来，而不必硬写进系统。如今 Bugbot 每月为 Rippling、Discord、Samsara、Airtable、Sierra AI 等客户审查超过 200 万个 PR，并已上线自动派 Cloud Agent 修 bug 的 Bugbot Autofix（Beta），下一步将让它运行代码自验报告、遇到复杂问题时做深度研究，并试验「常驻扫描」模式。

结论：智能体的「智能」要落地，离不开速度、可靠性与可定制的工程底座——产品力 = 模型 + harness 设计 + 审查结构的组合。

最值得带走的 5 条洞见

  1. 先造度量，再谈优化。「解决率」这个能衡量真实修复的指标，是把 Bugbot 从凭感觉推向系统化的转折点；没有它，质量提升就有天花板。
  2. 数字会说话：40 次实验把解决率从 52% 推到 70%+，每 PR 真修 bug 数从约 0.2 翻倍到约 0.5——验证了方法论而非单点运气。
  3. 架构换代带来最大跃升。从固定多轮流水线到完全 agentic，模型能自主调工具深挖，是收益最大的一步。
  4. 提示策略随架构反转。流水线时代要「克制防误报」，agentic 时代反而要「激进鼓励多标」，并把上下文从静态转向动态按需获取。
  5. 生产级 = 模型 + 工程 + 结构。Rust 重写 Git、限流/批处理、可定制 rules 等底座，才支撑起每月 200 万 PR 的真实规模。
