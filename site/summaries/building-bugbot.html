Bugbot 的真正进展，是把代码审查做成可度量、可实验、可迭代的系统

一句话总括：这篇文章的贡献不在于宣称 AI 能审查代码，而在于说明 Cursor 如何用“作者是否真的修复了问题”作为核心指标，把一个代码审查 agent 从经验调参推进到生产级持续优化。

质量提升来自真实反馈，而不是只看评论反应

Cursor 为 Bugbot 定义了 resolution rate：在 PR 合并时，用 AI 判断 Bugbot 提出的 bug 是否已经被作者在最终代码中修复。团队还与 PR 作者一起抽查样例，确认该分类在内部验证中几乎都能正确判断。

这改变了对 AI 代码审查价值的理解：真正有用的审查，不是发出更多评论，而是提出工程师会采纳并修掉的问题。

40 次主要实验把 Bugbot 从“可用”推向“明显更有用”

52% → 70%+resolution rate
0.4 → 0.7每次运行平均标记 bug 数
0.2 → 0.5每个 PR 被解决的 bug 数
40上线后的主要实验数

这些数字说明，Bugbot 的进步同时来自召回更多真实问题和过滤更少有价值的误报。对读者来说，这意味着 AI 工具的影响应看“最终改变了多少代码质量”，而不是看模型输出本身有多像审查意见。

早期有效方法是多路并行审查加投票过滤

早期版本会运行 8 个并行 bug-finding pass，并给每个 pass 不同的 diff 顺序；随后把相似 bug 聚类，用多数投票过滤只有一次出现的问题，再经过描述合并、类别过滤、验证模型和历史去重。

这个流程的洞见是：同一个模型在不同上下文顺序下会走出不同推理路径，多次独立发现同一问题时，信号强度更高。

生产可用性依赖底层系统，而不只是审查逻辑

Cursor 为 Bugbot 重建了 Rust 版 Git 集成，减少仓库数据获取，并增加 GitHub 约束下所需的限流监控、请求批处理和代理基础设施。随着客户采用增长，Bugbot rules 又让团队能表达代码库特定规则，例如危险迁移或内部 API 误用。

这说明 AI agent 要进入真实研发流程，必须同时解决权限、速度、规模、限流和团队自定义知识的问题。

最大的跃迁来自 agentic 架构和动态上下文

Cursor 在秋季把 Bugbot 切换为完全 agentic 的设计：agent 不再只按固定步骤跑 pass，而是能围绕 diff 推理、调用工具，并决定哪里需要深入调查。团队还发现，agent 版本反而过于谨慎，因此提示词要更鼓励它调查可疑模式并报告潜在问题。

动态上下文也变得更重要：模型不必一开始拿到所有信息，而是在运行时主动拉取需要的上下文。结论是，agent 的能力很大程度上由它能调用的工具和工具界面塑造。

下一步是从“指出 bug”走向“验证并修复 bug”

Bugbot 目前每月为客户审查超过 200 万个 PR，客户包括 Rippling、Discord、Samsara、Airtable 和 Sierra AI，Cursor 也把它用于内部所有代码。Cursor 已推出 Bugbot Autofix Beta，让 Cloud Agent 自动修复 PR 审查中发现的问题。

后续方向包括让 Bugbot 运行代码验证自己的报告、遇到复杂问题时做深度研究，以及持续扫描代码库的 always-on 版本。这意味着 AI 代码审查可能从被动评论逐步变成持续质量维护系统。

最值得带走的洞见

  • AI 代码审查的核心指标应接近真实工程结果：问题是否被作者修复。
  • 模型能力只是基础，评估指标、上下文管理、工具接口和生产基础设施共同决定最终质量。
  • 多路并行和多数投票能提高早期系统可靠性，但 agentic 架构为更深层推理和动态探索打开空间。
  • 很多看似合理的改动会让指标回退，所以在线真实数据和离线人工标注 benchmark 都不可缺。
  • AI 研发工具的下一阶段不只是发现问题，而是验证问题、修复问题，并持续守护代码库。
