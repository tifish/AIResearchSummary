打造更好的 Bugbot

Building a better Bugbot

一句话总括：Cursor 为 AI 代码审查建立了可量化的反馈闭环，并借助真实 PR 数据、离线 benchmark 和智能体架构，把 Bugbot 从经验驱动的原型变成了能够持续优化的生产系统。

40 次上线后的重大实验
52% → 70%+Bugbot 解决率
0.4 → 0.7每次运行平均报告的问题数
200 万+每月审查的 PR 数量

真正的突破是找到可优化的目标

早期团队主要依靠工程师的主观判断。某个配置看起来误报更少，就采用这个配置。

这种方法足以打造原型，却无法证明线上质量是否真的提高。团队因此定义了“解决率”。它衡量开发者最终是否修改了 Bugbot 指出的问题。

解决率为何重要

评论反应和零散反馈容易受到表达习惯影响。解决率直接观察合并后的代码，因此更接近实际价值。

Cursor 表示，内部逐例与 PR 作者核对时，LLM 对“已解决”或“未解决”的判断几乎全部正确。这个指标让团队首次能够依据真实信号进行爬坡式优化。

早期方法：固定流程抑制误报

Bugbot 并行执行 8 次检查。每次使用不同的 diff 顺序，再通过多数投票、聚类、验证和去重过滤结果。

核心思路是约束模型。只有多个检查过程独立发现同一问题时，系统才把它视为更可靠的信号。

智能体方法：主动调查可疑线索

全智能体版本可以分析 diff、调用工具，并自行决定深入检查的位置。团队发现，它面临的问题不再是过度报告，而是过于谨慎。

提示策略因此转向更积极的调查。模型会在运行时补充所需上下文，不必预先接收全部资料。

从原型走向持续优化系统

01 · 定性迭代 比较模型、流水线、过滤器和上下文策略。
02 · 生产基础设施 用 Rust 重建 Git 集成，并加入批处理、限流监控和代理设施。
03 · 双轨评估 线上观察真实解决率，线下使用人工标注的 BugBench。
04 · 智能体架构 让模型调用工具、动态寻找上下文，并决定调查路径。

数字显示的不只是“多报了一些问题”

解决率从 52% 提升到 70% 以上。与此同时，每次运行报告的问题从 0.4 个增至 0.7 个。

每个 PR 最终解决的问题因此从约 0.2 个增至约 0.5 个，提升超过一倍。这说明覆盖面扩大时，结果的可用性并未同步恶化。

许多看似合理的改动反而会退步

团队测试了模型、提示、迭代次数、验证器、上下文管理、类别过滤和智能体设计。大量改动让指标下降。

这改变了对 AI 产品迭代的理解。局部上更复杂或更“聪明”的设计，不一定能改善线上效果。可靠评估比直觉更重要。

工具不是智能体外围的辅助设施。工具的设计和可用性会直接塑造模型行为。Cursor 发现，即使是很小的接口变化，也可能对最终结果产生超出预期的影响。

最值得带走的洞见

评估指标决定优化上限

没有解决率，团队只能判断评论“看起来是否更好”。有了合并时的真实结果，优化才形成闭环。

质量与数量可以同时提升

报告量增加的同时，解决率也从 52% 升至 70% 以上。

智能体需要不同的提示策略

固定流水线需要抑制误报。自主调查的智能体反而需要鼓励它大胆追查异常。

动态上下文比一次性塞满信息更灵活

模型能够在运行时主动取得额外上下文。这降低了预先提供全部信息的必要性，也扩大了系统的实验空间。

下一步是验证与修复

Bugbot Autofix 已进入 Beta。后续方向包括运行代码验证报告、研究复杂问题，以及持续扫描整个代码库。

随着编程智能体越来越强，我们发现自己花在代码审查上的时间也越来越多。为了解决这个问题，我们打造了 Bugbot。它是一个代码审查智能体，可以在代码进入生产环境之前分析 pull request，找出逻辑错误、性能问题和安全漏洞。到去年夏天，它的表现已经相当出色，因此我们决定向用户发布。

Bugbot 的开发最初依靠定性评估，之后逐渐转向更系统的方法。我们设计了一个由 AI 驱动的自定义指标，用它对质量进行爬坡式优化。

上线以来，我们完成了 40 次重大实验。Bugbot 的解决率从 52% 提高到 70% 以上，单次运行平均报告的问题数也从 0.4 个升至 0.7 个。这意味着每个 PR 中最终得到解决的问题数量增加了一倍以上，从约 0.2 个升至约 0.5 个。

从不起眼的起点开始

我们第一次尝试开发代码审查智能体时，模型能力还不够强，给出的审查意见没有多少帮助。但随着基础模型不断进步，我们意识到，可以从多个方面提高问题报告的质量。

我们尝试了不同的模型、流水线、过滤器和巧妙的上下文管理策略，也在这个过程中不断征询内部工程师的意见。如果某种配置看起来能减少误报，我们就采用它。

早期最有效的质量改进之一，是并行执行多次问题查找，再通过多数投票合并结果。每次检查收到的 diff 顺序都不一样，这会推动模型采用不同的推理路径。如果多次检查分别发现了同一个问题，我们就会把它视为更强的信号，认为这个问题更可能真实存在。

经过数周的内部定性迭代，我们最终得到了一个优于市场上其他代码审查工具的 Bugbot 版本。这让我们有信心将它发布。它采用以下流程：

  1. 以随机的 diff 顺序并行执行 8 次检查。
  2. 把相似的问题归入同一个分组。
  3. 通过多数投票，过滤掉只在一次检查中发现的问题。
  4. 把每个分组整合成一段清晰的描述。
  5. 过滤不需要的类别，例如编译器警告或文档错误。
  6. 让验证模型检查结果，找出误报。
  7. 与此前运行已经发布的问题进行去重。

从原型走向生产环境

为了让 Bugbot 真正可用，我们不仅要改进核心审查逻辑，还必须建设一系列基础系统。为了快速、可靠地访问代码仓库，我们用 Rust 重写了 Git 集成，并尽量减少需要获取的数据量。我们还加入了限流监控、请求批处理和基于代理的基础设施，以便在 GitHub 的限制下运行。

随着使用规模扩大，各个团队也需要表达代码库特有的不变条件，例如不安全的迁移方式，或错误使用内部 API。为此，我们加入了 Bugbot 规则。团队可以通过规则执行这些检查，不必把它们硬编码进系统。

这些组件共同让 Bugbot 能够实际运行，也能适应真实的代码库。但它们无法告诉我们，质量是否真的有所提高。如果没有衡量进展的指标，我们就无法定量评估 Bugbot 在真实使用环境中的表现。这也限制了我们继续改进它的空间。

衡量真正重要的指标

为了解决这个问题，我们设计了一个名为“解决率”的指标。它会在 PR 合并时使用 AI 判断作者最终在代码中解决了哪些问题。开发这个指标时，我们在内部逐例抽查，并与对应的 PR 作者核实。结果显示，LLM 几乎总能正确判断问题是已经解决还是尚未解决。

团队经常询问，该如何评估 Bugbot 为他们带来的影响。因此，我们在仪表板的显眼位置展示了这个指标。对于评估实际效果的团队来说，它比零散反馈或评论反应更清晰。解决率直接回答了一个问题：Bugbot 找出的是否是真实存在、并且工程师愿意修复的问题。

爬坡式优化

定义解决率改变了我们开发 Bugbot 的方式。我们第一次可以依据真实信号进行爬坡式优化，而不再只凭感觉。我们开始通过实际解决率在线评估改动，同时使用 BugBench 进行离线评估。BugBench 是一个经过筛选的 benchmark，其中包含真实代码 diff 和人工标注的问题。

我们围绕模型、提示、迭代次数、验证器、上下文管理、类别过滤和智能体设计开展了数十次实验。令人意外的是，许多改动反而让指标下降。事实证明，我们早期在定性分析中做出的很多初始判断是正确的。

智能体架构

去年秋天，我们把 Bugbot 改成完全基于智能体的设计。正是在这次转换后，我们看到了最大幅度的提升。智能体可以分析 diff、调用工具，并自行决定应该深入检查哪些地方，不必再遵循固定的检查步骤。

智能体循环迫使我们重新思考提示设计。在早期版本中，我们需要约束模型，以尽量减少误报。但采用智能体方法后，我们遇到了相反的问题：它过于谨慎。于是，我们改用更积极的提示，鼓励智能体调查每一种可疑模式，并在判断边界不明确时倾向于报告潜在问题。

此外，智能体架构也提供了更丰富的实验空间。我们可以把更多信息从静态上下文移到动态上下文中。我们会调整模型预先获得的上下文数量，再观察它如何适应。模型总能在运行时取得自己需要的额外上下文，无须提前得到全部信息。

同一套架构也让我们可以直接迭代工具集。模型行为会受到可调用工具的影响，因此，即使只是工具设计或可用性上的小改动，也会对结果产生格外显著的影响。经过多轮迭代，我们不断调整和改进这个接口，直到模型行为能够稳定符合预期。

下一步

如今，Bugbot 每月为 Rippling、Discord、Samsara、Airtable 和 Sierra AI 等客户审查超过 200 万个 PR。Cursor 的所有内部代码也会使用 Bugbot 审查。

展望未来，我们预计新模型会持续出现。无论是其他供应商提供的模型，还是我们自行训练的模型，它们都会有不同的优势和弱点。要继续取得进展，就必须找到模型、运行框架设计和审查结构之间的正确组合。今天的 Bugbot 已经比刚发布时强了数倍。我们预计，再过几个月，它还会获得显著提升。

我们已经开始为这个未来做准备。我们刚刚发布了 Beta 版 Bugbot Autofix。它会自动启动一个 Cloud Agent，修复 PR 审查期间发现的问题。接下来的主要能力包括：让 Bugbot 运行代码，以验证自己的问题报告；在遇到复杂问题时进行深度研究。我们还在试验一个始终在线的版本。它会持续扫描代码库，而不是等到出现 pull request 后才开始工作。

我们至今取得的巨大进展，离不开一些核心团队成员的贡献，其中包括 Lee Danilek、Vincent Marti、Rohan Varma、Yuri Volkov、Jack Pertschuk、Michiel De Jong、Federico Cassano、Ravi Rahman 和 Josh Ma。我们的目标始终不变：随着团队扩大 AI 开发工作流，帮助他们继续维持代码质量。

我们至今取得的巨大进展，离不开一些核心团队成员的贡献，其中包括 Lee Danilek、Vincent Marti、Rohan Varma、Yuri Volkov、Jack Pertschuk、Michiel De Jong、Federico Cassano、Ravi Rahman 和 Josh Ma。我们的目标始终不变：随着团队扩大 AI 开发工作流，帮助他们继续维持代码质量。
