深入了解我们对 Model Spec 的处理方法

Inside our approach to the Model Spec

这篇文章的贡献，是把模型行为从抽象的“有帮助且安全”，拆成公开规则、优先级、评估和修订流程，让 AI 的影响更容易被用户、开发者和政策制定者检查。

OpenAI Research 2026-03-25 Alignment
2024 Model Spec 首版发布，此后随反馈和能力变化迭代。
3 高层目标：赋能用户和开发者、防止严重伤害、维持运营许可。
0-3 月 Spec 通常写在当前模型能力前方一小段距离，而不是远期理想。
4 类 更新来源：公开反馈、内部问题、政策变化、新能力和新产品。

Model Spec 是行为治理接口

它定义模型如何遵循指令、解决冲突、尊重用户自由并保持安全。重点不是展示训练配方，而是让预期行为可以被阅读、质疑和改进。

这改变了读者对 AI 影响的理解：模型行为不是自然生成的中性结果，而是需要公开选择和持续校准的制度设计。

它不是三件事

  • 不是声明模型今天已经完全做到这些要求。
  • 不是训练栈、内部 token 格式或产品功能的完整说明。
  • 不是让模型自主追求“造福人类”的总目标。

这让边界更清楚：Spec 约束模型行为，产品安全还需要政策、监控、功能和防御纵深。

Chain of Command 让冲突可判定

模型会遇到来自 OpenAI、开发者和用户的不同指令。Chain of Command 给这些指令分配权威层级，并要求模型在冲突时优先遵循更高权威指令的文字和精神。

1

硬规则

不可被用户或开发者覆盖，主要防止灾难性风险、直接身体伤害、违法行为和破坏指令层级。

2

系统与开发者

为应用场景设定边界，让产品和开发者在安全范围内控制模型行为。

3

用户请求

在安全边界内优先满足用户意图，包括语气、格式、深度和观点角度。

4

默认行为

当没有明确指令时，模型使用可预测的默认值，例如诚实、客观和合适风格。

硬边界和可覆盖默认值服务不同目的

文章最重要的区分，是少量不可覆盖规则与大量可调默认行为并存。这使安全边界和用户自由不必互相吞没。

机制 关键数据和趋势 为什么重要
硬规则 覆盖现实安全风险、法律边界、指令层级和未成年人额外保护。 AI 越像基础设施，限制自由的规则越需要少而明确。
默认值 语气、风格等 guideline 默认值可被隐式引导；真实性和客观性等 user-level 默认值需要明确指令才可改变。 这让模型既可控，又不至于悄悄漂移到迎合或不透明立场。
解释辅助 决策准则和少量高信号示例帮助处理灰区，例如控制真实世界副作用。 影响不只来自拒绝或回答，还来自模型如何在不确定场景中做取舍。

模型越强、越自主、部署越广，含糊规则的成本越高。

文章的核心洞见不是“规则越多越好”，而是重要价值判断不能被完全外包给模型自己猜。

Spec 是可修订的公共目标

OpenAI 把 Model Spec 写成现实可追赶的目标，通常领先当前行为 0-3 个月。这样它既能指导训练和评估，又不会变成遥远愿景。

2024：首版发布

Model Spec 开始作为公开参考点，帮助外部判断模型行为是 bug 还是有意设计。

持续迭代：公开和内部反馈进入流程

OpenAI 内部任何人可以评论或提议修改，最终更新由跨职能利益相关方批准。文章称已有数十人直接贡献文本。

2026：配套发布 Model Spec Evals

评估套件用代表性场景尽量覆盖 Spec 断言，帮助发现模型行为与 Spec 的不一致。

下一步：随能力扩展而扩展

多模态、自治代理和未成年人场景会继续推动规则澄清。结论是，AI 治理必须跟随部署现实更新。

评估让价值声明变成可检查对象

Model Spec Evals 不是唯一评估方法，但它给公开规则配上了可执行测试。文章还提到更具体的安全、真实性、迎合性、人格风格和能力评估。

这改变了 AI 影响的讨论方式：争论不只停留在原则，还可以追问模型在哪些场景偏离了公开目标。

最值得带走的洞见

公开规则降低治理黑箱

用户和研究者可以引用同一份文本来批评、验证和推动修改。

安全不是单层模型行为

Spec 只描述模型，产品政策、监控和防御纵深仍然必要。

智能不能替代价值选择

“有帮助且安全”在真实场景中充满取舍，需要人类可见的解释机制。

可修订性是核心能力

随着模型更强、更能行动，规则必须保持清晰、可测且能随反馈更新。
