Model Spec 背后的方法论：把"模型该怎么行为"变成公开制度

原文：Inside our approach to the Model Spec · openai.com
一句话总括：这篇文章不讲 Model Spec 写了哪些规则，而是公开它的"幕后操作系统"——为什么 OpenAI 要把模型的意图行为写成一份可被公众阅读、辩论、评估、修订的活文档，以及这份文档如何被结构化、撰写和迭代。真正的贡献在于：它把 AI 对齐从内部训练的黑箱，转化为一套有问责机制的公共治理实践。

01Model Spec 是写给人看的"接口"，不是写给模型的代码

OpenAI 明确：Spec 是 interface（接口）而非 implementation（实现），它描述"想要的行为"，刻意不绑定 token 格式或训练配方等会变的细节。它的主要受众是人——员工、用户、开发者、研究者、政策制定者，而不是模型本身。它也只描述"模型"，不等于整个产品（还有自定义指令、记忆、监控、策略执行等多层防御）。

首版：2024 年 定位：描述意图行为，非当前完美行为 理念：defense in depth 纵深防御
为什么改变认知：它否定了"AI 安全=调模型"的简化想象——安全首先是一份人类能共同审查的公开契约。

02开篇三目标定方向，但不直接命令模型

Spec 以高层意图开篇，声明三个系统级目标，并把它们之间的权衡讲具体。关键细节：造福人类是 OpenAI 的目标，而非让模型自主追求的目标——否则 OpenAI 就成了在广义上"裁决道德"的一方。当 Spec 应用出现歧义时，这段开篇用来兜底解释。

① 迭代部署，赋能开发者与用户 ② 防止严重伤害 ③ 维持 OpenAI 运营许可
洞见：把"对人类好"留给人类去辩论，让模型遵循指挥链，是出于尊重人类自主与思想自由的刻意选择。

03指挥链 + 硬规则/默认值：在安全边界内最大化自由

核心是 Chain of Command（指挥链）：每条策略和指令都有"权限级别"，冲突时模型优先遵守更高权限指令的字面与精神。要造炸弹 → 优先硬安全边界；要求"吐槽我一下" → 一般优先满足，覆盖低权限的反滥用默认。这让一小套不可覆盖的硬规则与一大套可覆盖的默认值并存。

  • 硬规则（root/system 级）：多为禁止性，针对灾难性风险、人身伤害、违法、破坏指挥链；并为 18 岁以下用户叠加额外保护。类比"互联网基础设施"，只在必要时才设限。
  • 默认值：可被显式指令覆盖的"最佳猜测"起点；语气风格类可隐式引导，真实性/客观性类只能被显式指令改变，以防"凭感觉悄悄漂移"。
为什么重要：用极少的不可逾越红线 + 大量可调默认，同时换来用户自由、开发者控制和可预测性——这是 Spec 的工程化核心。

04智能再高，也替代不了价值权衡

文章正面回应"先进 AI 难道不能自己想明白吗"：在数学等有客观对错的领域，智能可以替代规则；但模型多数时候处在没有唯一正确答案的价值地带。即便模型更聪明，也无法告诉你该做什么伦理取舍。若唯一规则是"有益且安全"，人类就失去了辩论边界的机制，一切决定都被甩给模型。

类比：成文宪法 vs 判例法——宪法给原则，但仍需解释机制、澄清与明确裁决来处理棘手个案。Model Spec 三者兼任。
反直觉结论：模型越强、越具自主性、部署越广，模糊的代价越高，清晰的行为框架因此更重要，而非更不重要。

05"现实的抱负"与跨职能共识写作

写作在"如实描述现状"与"描述遥远理想"之间取平衡，通常瞄准领先现状 0–3 个月，让 Spec 在若干活跃开发领域略微超前于模型。它由开放的内部流程产出：任何 OpenAI 员工可评论或提案，最终由跨职能利益相关方审批。

超前幅度：0–3 个月 数十人直接贡献文本 研究/工程/产品/安全/政策/法务等多职能审批

一个意外发现：只要逼自己把权衡写得足够精确，让分歧变具体，"真正的共识往往是可能的"。

洞见：大量 Spec 写作其实是"翻译"——把已有的安全与政策工作变得更简单、一致、可读，而不丢失原意。

06用 Evals 量化差距：模型与规范的对齐可被测量

OpenAI 承认生产模型尚未完全符合 Spec：训练可能滞后于规范更新、可能无意教出不一致行为（视为严重 bug）、无法覆盖所有长尾、泛化可能偏离意图。为此本文随附发布 Model Spec Evals——一套基于场景、力图用少量代表性样例覆盖尽可能多 Spec 断言的评估套件，用来追踪模型行为与规范何处脱节、模型是否按预期理解规范。

配套发布：Model Spec Evals 图表显示各章节合规度随迭代提升 提示：部分提升源于"用旧模型对照更新政策"的测量效应
为什么改变认知：对齐不再只是口号，而是有公开断言、可设计评估、可诊断事故的可执行工程。

最值得带走的洞见

  1. Spec 是给人的公共契约。它把"模型该怎么行为"从训练黑箱搬到公开桌面，让 bug 与 feature 可被区分、被批评、被改进。
  2. 结构即治理。指挥链 + 少量硬规则 + 大量可覆盖默认 + 决策准则与示例，在安全边界内同时最大化用户自由与开发者控制。
  3. 智能不替代价值。越强的模型越需要显式框架，因为伦理取舍没有唯一正解，人类必须保留辩论边界的机制。
  4. 现实的抱负。瞄准领先现状 0–3 个月，Spec 是一份会迭代的活文档，靠数十人跨职能共识与公众反馈驱动。
  5. 三条成功标准。可读性（Legibility）、可执行性（Actionability）、可修订性（Revisability）——既要能被理解和挑战，也要能落地评估并持续修订而不失稳。
