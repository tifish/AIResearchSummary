深入了解 OpenAI 制定 Model Spec 的方法

Inside our approach to the Model Spec

一句话总括：OpenAI 把模型行为从内部训练偏好转化为一套公开接口，让用户、开发者和社会都能看见、检验并推动规则改变。

更短、更易读的中文总结

Model Spec 不是一句笼统的“有帮助且安全”，也不是一份训练配方。它把价值目标、指令权限、硬性边界、默认行为、判断准则和实例组织成可执行的公共框架。这样一来，模型为什么服从或拒绝某项要求，就有了可以讨论的依据。

2024首版 Model Spec 发布，此后根据真实使用和公众反馈持续演进。
0—3 个月规范通常领先当前模型的目标区间，既保持进取，也必须具备近期实现路径。
数十人直接参与文本贡献，背后还有更多跨研究、产品、安全、政策和法律职能的评审者。

硬规则：少而不可覆盖

涉及灾难性风险、直接人身伤害、违法行为或指令层级完整性的边界，用户和开发者都不能取消。其意义在于，安全底线不随提示词变化。

但 OpenAI 表示，这类限制应尽量克制。因为 AI 可能成为类似互联网基础设施的基础技术，过度限制会直接压缩思想自由。

默认行为：可预测，也可引导

语气、深度、格式和观点表达通常可以由用户或开发者调整。真实性与客观性等信任基础，则要求更明确的覆盖指令。

这种设计说明，规范并非追求统一答案，而是要在安全范围内保留用户自由和开发者控制权。

Model Spec 如何进入改进循环

规范本身不会自动让模型合规。OpenAI 需要把公开目标接入训练、评估、问题发现和修订流程。

01把期望行为写成清晰、可争论的公开规则。
02通过训练和不同技术，让模型学习相应行为。
03用 Model Spec Evals 和专项评估寻找偏差。
04结合真实故障、公众反馈和新能力更新规范。

结论：这是一套持续治理机制，不是发布一次便结束的静态政策。

更强的模型，反而更需要明确规则

智能可以帮助模型解决有客观答案的问题，却不能自动决定伦理取舍。模型越强、越自主、部署越广，模糊指令造成的代价就越高。文章因此把 Model Spec 类比为成文宪法加判例机制：原则负责定方向，解释工具和明确裁决负责处理现实中的灰色地带。

评估带来可见差距

同期发布的 Model Spec Evals 用少量代表性场景，覆盖尽可能多的规范断言。它用于判断模型是否正确理解规范，并追踪模型行为与公开目标是否一致。

这改变了“发布规则就等于实现规则”的印象。OpenAI 明确承认，训练滞后、错误泛化和真实场景的长尾问题都会造成偏差。

规范不等于完整安全系统

Model Spec 描述的是模型行为，不是整个产品。自定义指令、记忆、监控、政策执行和其他产品层仍需共同发挥作用。

洞见：公开行为规范只是纵深防御的一层，不能替代部署环节的技术和治理措施。

最值得带走的洞见

公开规范是一种权力约束

明确写出模型应如何处理冲突，能让外界判断某种行为究竟是缺陷还是设计选择。任何规则变化也必须显性发生，不能悄然漂移。

价值判断不能外包给智能

模型能力再强，也无法从“有帮助且安全”自动推导出所有人都认可的伦理边界。

成功取决于三项标准

可理解、可执行、可修订。缺少其中任何一项，规范都可能沦为口号或成为不稳定的移动目标。

在 OpenAI，我们认为 AI 应当公平、安全，并且人人可用。这样，更多人才能用它解决难题、创造机会，并在健康、科学、教育、工作和日常生活等领域受益。

我们相信，让更多人平等地使用 AI，是最好的前进道路。AI 的利益和控制权不应集中在少数人手中。更多人应当能够接触 AI、理解 AI，并参与塑造 AI。

这正是 OpenAI Model Spec 存在的一个核心原因。Model Spec 是我们关于模型行为的正式框架。它规定了模型应当如何遵循指令、解决冲突、尊重用户自由，并在用户每天提出的极其广泛的问题中保持安全。

从更广的角度看，它是我们把预期模型行为明确表达出来的一次尝试。这些行为不应只存在于训练过程内部，还应以用户、开发者、研究人员、政策制定者和公众都能阅读、审视和讨论的形式呈现。

Model Spec 并不是在声称，我们目前的模型已经能够完美地按照这些要求行动。它在许多方面描述了现状，但也代表了我们希望模型行为达到的目标。

我们用它明确预期行为，进而据此训练模型、开展评估，并持续改进。

本文将介绍 Model Spec 本身没有写出的背景，包括它背后的理念和运行机制。我们会说明它如何组织、为什么采用这样的结构，以及我们如何编写、实施并持续演进这份规范。

一个公开的模型行为框架

Model Spec 是 OpenAI 构建安全且可问责 AI 的整体方法之一。Preparedness Framework 关注前沿能力带来的风险，以及风险上升时所需的防护措施。Model Spec 处理的则是另一个互补问题：模型在各种情境中应当如何行动。

再把视角拉远一些，AI resilience 试图解决一项更广泛的社会挑战：随着能力越来越强的系统投入部署，如何帮助社会获取先进 AI 的收益，同时减少冲击和新出现的风险。

这些举措共同希望让迈向 AGI 的过程渐进、迭代，并能被民主社会理解。人们和各类机构需要时间适应。与此同时，我们也要建立必要的防护措施、问责机制和公众认知，使强大的 AI 始终与人类利益保持一致。

公开说明模型行为，对公平和安全都很重要。

它关系到公平，因为人们需要理解 AI 为什么以某种方式对待自己。出现公平性问题时，人们也应当能够发现、质疑并解决这些问题。

它同样关系到安全。随着 AI 系统能力增强，个人和机构需要更清楚地知道系统被设计成怎样行动、其中包含哪些取舍，以及这些选择今后可以如何改进。

这种可理解性也有助于增强韧性，因为更多人可以据此检查、质疑和改进系统。

自 2024 年发布首个版本以来，Model Spec 已发生很大变化。我们持续了解用户的偏好和需求，也不断扩大覆盖范围，以适应更强的模型能力。公众对模型行为和 Model Spec 的反馈，同样推动了这些变化。

Model Spec 延续了迭代部署的理念。它是一份不断演进的文档，既涵盖基础价值观，也包含明确且易于理解的规则。随着我们从真实部署和反馈中获得新认识，还会有一套流程用于修改其中的具体内容。

我们也在投入建设 collective alignment 等公众反馈机制，希望让人类继续掌握 AI 使用方式和 AI 行为塑造方式的控制权。

在 OpenAI 内部，Model Spec 为预期行为提供了一个“北极星”，也是训练、评估和治理所共用的框架。在外部，它提供了一个公开参照。人们可以借此理解我们的方法、提出批评，并帮助它持续改进。

Model Spec 包含什么

Model Spec 由多种不同类型的模型指导内容组成。这是有意为之。

模型行为的不同部分需要采用不同方式处理。一份真正有用的公开文档，也不能只是罗列规则。

高层意图和公开承诺

Model Spec 以高层意图开篇。它清楚说明我们试图在整个系统层面优化什么，以及为什么这样做。

序言明确了我们在推进使命时的三个目标：

  • 以迭代方式部署能够赋能开发者和用户的模型。
  • 防止模型对用户或其他人造成严重伤害。
  • 维护 OpenAI 持续运营的社会许可。

随后，序言解释了我们在实践中如何平衡这些目标。它把相关取舍说明得足够具体，为后续更详细的原则提供支持。

需要强调的是，这段序言并不是直接给模型下达的指令。造福人类是 OpenAI 的目标，而不是我们希望模型自主追求的目标。

我们希望模型遵循一套指令层级，其中包括 Model Spec，以及 OpenAI、开发者和用户发出的适用指令。即使有人不认同某个具体结果，模型也应遵循这套层级。

我们认为这种平衡是正确的，因为我们重视人的自主权和思想自由。如果我们按照自己对社会福祉的理解，训练模型自行判断哪些指令值得服从，OpenAI 就会处于在极大范围内裁决道德问题的位置。

不过，序言仍然很重要。当 Model Spec 的具体应用方式存在歧义时，序言应当帮助解决这些歧义。

Model Spec 还包含一些公开承诺。它们不只涉及能够直接测量的模型行为，也涉及训练意图和部署限制。

例如，我们的 Red-line principles 承诺，在 ChatGPT 等第一方部署中，绝不会利用系统消息故意损害客观性或相关原则。No other objectives 则承诺，我们优化模型回答是为了用户利益，而不是为了收入，也不是为了增加对用户无益的停留时间。

指令层级

Model Spec 的核心是指令层级（Chain of Command）。这套框架用于判断，在某个具体情境中应当采用哪些指令。

它还规定了模型应如何处理不够明确的指令。这一点在智能体场景中尤其重要。此时，模型往往需要自主补足细节，同时谨慎控制对现实世界产生的副作用。

判断应当采用哪些指令的基本思路很简单。指令可能来自 OpenAI、开发者和用户等不同来源，而这些指令可能互相冲突。指令层级说明了模型应如何解决这些冲突。

Model Spec 中的每项政策和每条指令都会被赋予一个权限等级。发生冲突时，模型应优先遵循权限更高的指令，包括其字面要求和真实意图。

如果用户要求协助制造炸弹，模型应优先遵守硬性安全边界。如果用户要求模型尖刻地调侃自己，模型通常应优先满足这项要求，而不是遵循 Model Spec 中权限较低的反侮辱政策。

借助这种结构，我们可以设置一小组不可覆盖的规则，同时提供数量更多的默认规则。我们正是通过这种方式，尝试在安全约束内尽量扩大用户自由和开发者控制权。

硬规则是用户或开发者都不能覆盖的明确边界。按照 Model Spec 的术语，它们属于“root”或“system”级指令。

这些规则大多用于禁止某些行为。它们要求模型避免助长灾难性风险或直接人身伤害，避免违法，也不能破坏指令层级。

我们预计，AI 将成为社会的基础技术，类似于基本的互联网基础设施。因此，只有当我们认为一项规则对广泛接触 AI 的开发者和用户确有必要时，才会施加可能限制思想自由的规则。

在 Model Spec 中，Stay in bounds 包含处理现实世界具体安全风险的硬规则。Under-18 Principles 则为未满 18 岁的用户增加了额外保护。

默认规则是可以覆盖的起点。当用户或开发者没有说明偏好时，它们代表助手对合适行为的“最佳判断”。

我们利用默认规则，让大规模使用中的行为更加可预测、可控制。这样，人们不必每次都编写一套专门指令，也能预期模型会如何回应。

默认规则保留了可引导性。只要不越过安全边界，用户和开发者就能明确调整语气、深度、格式，甚至观点。

语气或风格等 guideline 级默认规则，可以被隐含地引导。真实性和客观性等 user 级默认规则，则是信任和可预测性的基础，只能通过明确指令覆盖。

这类默认行为不应仅凭氛围悄然变化。如果用户想采用另一种事实立场，要求其明确说明，可以使这种转变保持透明、易于理解。

这些默认规则体现在 Seek the truth together、Do the best work 和 Use appropriate style 等部分。其内容包括诚实和客观、避免谄媚，以及直接、适度温暖和专业等互动规范。

解释工具：决策准则和具体实例

除了指令层级本身，Model Spec 还使用解释工具，帮助模型和人类在灰色地带中保持一致。这些工具包括：

  • 决策准则。它们帮助模型在灰色地带作出一致选择，同时不假装存在唯一的机械规则。例如，Model Spec 关于控制副作用的指导要求考虑减少不可逆操作、让行动与目标相称、降低意外程度，以及优先选择可逆方案。模型还应把这些因素与快速、有效完成任务等目标进行权衡。
  • 具体实例。它们展示一项原则在实践中应如何应用。实例通常由简短的提示和回答组成，往往同时包含合规与不合规的回答。所选提示通常位于重要决策边界附近，处理难度较高。

这些实例并不是为了模拟一段完整而真实的对话。它们的目标是把关键区别讲清楚，同时展示理想的回答风格。

我们会把实例数量控制在较小范围内，只保留信息量最大的内容。更广泛的评估套件则负责覆盖更多长尾情况。

Model Spec 不是什么

Model Spec 是接口，而不是实现。它描述我们希望看到的行为，但不涵盖产生这些行为的每项技术细节。

我们尽量避免让它依赖内部 token 格式或某项行为的确切训练配方等实现细节。即使期望行为没有改变，这些细节也可能发生变化。

Model Spec 的主要读者不是模型，而是人。它旨在帮助 OpenAI 员工、用户、开发者、研究人员和政策制定者理解、讨论并决定预期的模型行为。

Model Spec 描述的也是模型，而不是整个产品。我们的使用政策对它形成补充，说明我们对人们使用 API 和 ChatGPT 的方式有何要求。

用户接触的系统不只包含模型本身。自定义指令、记忆等产品功能，以及监控、政策执行和其他层面都很重要。

安全远不止模型行为。我们相信纵深防御。

Model Spec 也不是整个训练技术栈的完整说明，更不会收录每一项内部政策区分。它的目标不是捕捉所有细节，而是以完全符合预期模型行为的方式，让最重要的行为决策变得可以理解。

我们如何形成这一结构

为什么要把这些内容写进 Model Spec？

我们把如此多的内容写入规范，而不是假定读者或模型能从少数高层目标中推断出一切，原因有几个。

首先，Model Spec 是提高透明度和问责能力的工具。它的设计目的之一，是推动有意义的公众反馈。

明确的公开目标可以帮助人们判断某种行为究竟是缺陷，还是有意设计的功能。它为批评和具体反馈提供了稳定的参照。

这也是我们将 Model Spec 开源并选择公开迭代的原因。首次发布后，我们根据公众反馈进行了多项修改。这些反馈来自反馈表单、公开批评，以及专门收集民主意见的工作等多种渠道。

其次，Model Spec 是 OpenAI 内部的协作工具。它为研究、产品、安全、政策、法律、传播和其他职能的员工提供了讨论模型行为的共同语言，也提供了提出和审核修改建议的机制。

第三，明确的政策可以弥补模型智能和运行时上下文的现实限制，并让行为更可预测。虽然这一点的重要性正在逐渐降低，但有些政策用于弥补智能不足的问题。模型未必总能从高层原则中可靠地推导出正确行为。

例如，Be clear and direct 曾建议早期模型在解决需要计算的难题时，先展示过程，再给出答案。但如今，我们的模型会通过强化学习自然学会这种行为。

另一些政策处理的是运行时上下文不足。助手只能依赖当前互动中可以观察到的信息。它很少了解用户的完整处境、真实意图、后续用途，或者模型之外还存在哪些保护措施。

在这类情况下，即使模型通过充分研究和思考可能判断出正确行为，更具体的规则仍能提高效率和可预测性。它把大量判断压缩成指导，减少相似提示之间的行为差异，也让用户和研究人员更容易理解模型。

最后，Model Spec 希望成为一份完整的高层政策清单，涵盖与评估和测量有关的主要行为。如果要判断模型是否按照预期行动，一份列出主要行为类别的公开清单会很有用。

先进 AI 难道不能自行判断吗？

人们很容易认为，能力足够强的模型应该能从“有帮助且安全”等少量目标中推断出正确行为。这种看法确实有一部分道理。

在数学等拥有客观成功标准的领域，智能往往可以取代详细规则。

但总体而言，模型行为并不像解决一道简单的数学题。模型经常需要在更棘手的空间中行动，而这些空间不存在所有人都认可的唯一道德答案。

例如，一个模型怎样才算“有帮助且安全”，高度依赖具体情境，也必然来自包含价值判断的决策过程。仅凭智能，无法决定伦理和价值方面应当作出怎样的取舍。

因此，即使模型越来越聪明，我们仍需要研究如何理解和引导价值判断，也要明确在特定情境中怎样做才算“合乎伦理”。

即使模型能力大幅提升，建立 Model Spec 的多数理由仍然成立。我们仍需要一个供公众协调的目标，需要一种判断行为是否符合意图的方法，也需要一套随着认识加深而修改规则的机制。

如果唯一规则只是“有帮助且安全”，人类就没有机制讨论模型应当拒绝提供哪些内容等边界问题。这相当于把所有决定都交给模型。

事实上，随着模型能力增强、自主性提高并得到更广泛部署，模糊性的代价还会增加。因此，清晰的行为框架只会变得更重要。

一个有用的类比，是成文宪法与判例法之间的区别。

成文宪法可以提供高层原则和具体规则，但无法预见所有可能需要其指导的案件。现实治理体系还需要解释机制、澄清和明确裁决，才能处理复杂案件和未曾预见的问题。

公开规则可以帮助不同利益相关者协调，即使他们彼此意见不一。它也能约束变化，因为任何修改都必须明确发生。

Model Spec 希望同时承担这些角色：它既是原则声明，也是公开的行为框架，还是一套持续修改规范的流程。

不过，我们并不认为，模型行为中所有重要事项最终都能化为明确规则。

随着系统自主性增强，可靠性和信任会越来越依赖更广泛的能力与倾向。例如，妥善表达不确定性、尊重自主权限范围、避免造成糟糕的意外、持续追踪用户意图，以及结合情境正确理解人类价值观。

我们如何编写和实施 Model Spec

保持现实基础的前瞻性

编写 Model Spec 时，我们面对一条连续光谱。一端是如实描述当前模型的实际行为，包括全部缺点；另一端则是描绘遥远未来的理想目标。

我们努力在两者之间取得平衡，通常把目标设在当前时间点之后约 0—3 个月。因此，至少在几个积极开发的领域，Model Spec 往往会领先于模型。

这反映了 Model Spec 描述预期行为的作用。它应当指向一个连贯方向，同时又必须立足现实。相关行为要么已经实现，要么已有近期实施的具体计划。

谁参与其中，以及这为什么重要

Model Spec 通过开放的内部流程制定。OpenAI 的任何人都可以发表评论或提出修改建议，最终更新则由广泛的跨职能利益相关者批准。

在实际工作中，已有数十人直接贡献文本。来自研究、工程、产品、安全、政策、法律、传播、全球事务和其他职能的更多人员也会参与讨论。

我们还会从公开发布和外部反馈中学习。这些反馈有助于在真实部署中检验相关选择。

这种广泛参与非常重要，因为模型行为及其现实影响极其复杂。没有任何一个人能够独自掌握全部行为、训练流程和下游影响。

通过众多跨职能贡献者和审核者的协作，我们可以提高质量，也能增强对决策的信心。

一个令人欣喜的意外是，真正的共识往往能够达成。尤其是当我们迫使自己把取舍写得足够精确，让分歧变得具体时，更容易形成共识。

Model Spec 也不是脱离其他工作凭空写成的。它的许多内容，都是对模型行为、安全和政策领域更广泛工作的总结。

大量 Model Spec 编写工作本质上是一种转译。我们把现有成果变得更简单、更一致、更有组织，也更容易理解，同时不丢失背后的原始意图。

我们如何发现缺口并推动更新

目前投入生产的模型还没有完全体现 Model Spec，原因有以下几项。

模型训练可能落后于 Model Spec 的更新。规范描述的是我们正在努力实现的行为，因此它可能领先于最新模型已经接受的训练。

训练可能无意中教会模型不符合 Model Spec 的行为。我们会努力避免这种情况。如果问题发生，我们会把它视为严重缺陷，并调整模型行为或 Model Spec，使二者重新一致。

训练永远无法覆盖所有可能行为。真实使用中存在大量只有在规模化运行后才会出现的长尾情境和边缘案例。任何训练流程都不可能涵盖一切。

模型的泛化方式可能偏离我们的意图。模型可能出于错误原因，在训练中产生“正确”的输出。遇到与训练场景不同的新情况时，这可能引发非预期行为。Deliberative alignment 等技术可以提供帮助，但无法彻底解决问题。

更广泛地说，Model Spec 描述了大量期望行为，但这不意味着存在一种方法，可以把所有行为一次性教给模型。

遵循指令、安全边界、个性、恰当地表达不确定性等不同方面，通常需要不同技术，也会出现不同的失败模式。

Model Spec 让预期行为更容易被理解和批评，但如何妥善实现这些行为，仍然既是一门艺术，也是一个活跃的研究领域。

在发布本文的同时，我们还推出了 Model Spec Evals。这套基于场景的评估，尝试用少量代表性实例覆盖 Model Spec 中尽可能多的断言。

它可以帮助我们追踪模型行为在哪些地方可能偏离 Model Spec，也可以检验模型是否按照我们的意图理解规范。

这些评估只是整体评估策略的一部分。其他部分还包括针对多种行为维度的专项评估，例如具体安全领域、真实性与谄媚、个性与风格，以及能力。

图表展示 OpenAI 各模型的 Model Spec 分章节合规程度随时间的变化。有关评估方式以及结果解读，请参阅配套博客文章。简而言之，我们认为这些结果体现了模型对齐水平随时间取得的真实、广泛进步。不过，用较新的政策衡量较旧模型，也带来了一小部分影响。

在实践中，大多数规范更新由以下几类反复出现的输入推动：

  • 公众提出的问题和反馈。包括 Model Spec 文本或模型行为中的疑惑、边缘情况和失败模式。
  • 内部发现的问题。包括开发和测试期间观察到的模式，以及不同合理解释会引发不同行为的歧义。
  • 行为和安全政策更新。当更高层的约束或承诺发生变化时，规范必须清楚反映新的结构。
  • 新能力和新产品。随着模型获得新的行为能力，以及我们推出新产品，Model Spec 的内容和覆盖范围也需要同步发展。例如，为多模态互动、自主智能体和未满 18 岁的用户增加规则。

怎样的规范内容才算优秀

我们在编写和修订 Model Spec 时，会遵循几项设计原则。

清晰和精确。“保持诚实”是一项好的价值观，但不是完整的决策程序。Model Spec 应当让分歧变得更明确，而不是用人人都赞同的措辞掩盖分歧。

在可行的情况下，我们应明确指出规则之间可能发生的冲突，并提供解决冲突的指导或实例。

例如，Do not lie 指出它可能与 Be warm 发生冲突。相关说明要求助手遵循礼貌规范，但不能为了礼貌而说善意谎言。因为这种做法可能构成谄媚，并损害用户利益。

有实质内容的规则。读者面对一个现实提示时，应当能够写出一份回答，让另一位读者清楚判断它是否符合边界。边缘情况仍然可能需要判断。

尽量提高信噪比的实例。优秀实例通常是高质量规范更新的核心。

实例应当直击模型行为规范中最难处理的问题，把棘手冲突带到表面，并明确说明应如何解决。

其次，实例还应尽量成为理想语气和风格的示范。这些特征有时很难仅靠说明文字表达。

稳健性。我们会尽量避免实例包含无关的歧义或复杂性，让核心冲突和预期解决方式保持清晰。

一致性和清晰的组织方式。我们努力使 Model Spec 中的规则彼此完全一致，也与我们的预期模型行为一致。同时，整份文档的组织应清楚、易于理解。

未来方向

Model Spec 并不是在声称，我们可以写下所有重要事项，也不是在声称模型总能达到目标。

它真正主张的是：预期行为足够重要，因此必须清楚、可执行，并且可以修订。

我们用三项成功标准指导它的演进。

可理解性。OpenAI 内外的人都能对模型行为形成准确预期。模型表现令人意外时，人们也能指出对应的规范文本。

可执行性。Model Spec 可以用于设计评估、诊断事故并作出一致的产品决策，而不只是表达价值观。

可修订性。Model Spec 能够随着我们的认识发展而变化，同时不会成为一个不稳定的移动目标。

随着模型和产品演进，我们预计 Model Spec 会配合新能力和新部署环境继续扩展并澄清。

我们的目标，是让这套行为规范保持连贯、可测试，并与我们的使命一致：确保 AGI 造福全人类。
