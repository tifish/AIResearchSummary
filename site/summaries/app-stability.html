让被全天重度使用的桌面应用「不崩溃」

来源：Cursor ｜ 作者：Andrew Chan & Kevin Nguyen ｜ 2026-04-21 ｜ 原文
一句话总括

Cursor 把「桌面应用稳定性」从玄学变成了工程：用可观测性 + 双向定位 + 防回归护栏三套系统，几个月内把内存崩溃率压下七八成——证明 Agentic 时代提速越快，越要靠扎实的旧工程基本功为新工具链「重新配套」。

01问题的根：多进程架构里的「内存耗尽」

Cursor 建立在 VS Code 与 Electron 之上，是多进程架构：崩溃可能发生在驱动编辑器与新 Agents 窗口的渲染进程，也可能发生在驱动扩展、存储、Agent 功能的工具进程。其中渲染进程崩溃最致命——它会让编辑器彻底不可用，而且多由命中 V8 内存上限引发，因此成为治理重点。随着 subagents、即时 grep、浏览器使用等「越来越有野心」的功能上线，加上用户全天重度使用，连罕见崩溃都会造成巨大干扰。
洞见：绝大多数崩溃的根因是 OOM（内存耗尽）——稳定性问题的本质，是一道内存管理题。

02先能「看见」：分钟级的崩溃可观测性

每一次致命崩溃都会被遥测上报，附带受影响进程、崩溃类型、设备与应用元数据，以及（在可获取时的）minidump 与堆栈。团队据此构建了可按应用版本拆分的指标，分别按「每会话」（多少会话遇到崩溃）和「每请求」（受影响会话的严重程度）口径计算。仪表盘在崩溃事件发生后数分钟内即更新，于是新版本一发布就能紧密追踪、快速发现潜在回归。
洞见：「每会话 vs 每请求」两个口径分别回答了「波及面多广」和「问题多严重」——治理前先把度量分清楚。

03双向夹击：自上而下找贡献者，自下而上挖根因

调试用两条腿走路。下面拆开看两套互补的方法。

Top-down 自上而下

对已知内存密集的功能，把崩溃指标关联到 Statsig 里的特征开关并做 A/B 测试，量化它对崩溃率的贡献；同时追踪与崩溃强相关、更易在开发期观察的代理指标——典型如超大消息载荷（进程间通道里超阈值的消息，附带 callstack 回溯到源码）；再为并行 Agent、工具调用、终端等加「面包屑」，让每次崩溃都带着事发前的活动记录。

Bottom-up 自下而上

把单个崩溃追到根因。主进程跑一个用 CDP（Chrome DevTools Protocol）的崩溃监视服务实时捕获 OOM 堆栈，并已向上游打补丁 Electron，使其无需笨重的 CDP 机制也能拿到堆栈；这些堆栈喂给每日自动化，分析后对高置信度问题自动提 PR 并逐版本验证。配合用户自愿提交的堆快照、以及低采样率的持续堆分配剖析（可做版本间 diff）。
洞见：代理指标（超大载荷）让问题在崩溃发生前就暴露，自动化 PR 让定位到修复形成闭环——这是把人力从「逐个救火」中解放出来的关键。

04对症下药：急性 OOM 与慢性 OOM

两套方法共同揭示出崩溃落入两种模式。急性 OOM：内存骤增、进程瞬死，多见于堆栈而少见于堆快照，常因一次性加载太多数据——应用大量处理用户工作区内容，会从磁盘或 IPC 加载完整文件，而某些工作区含超大文件能把应用噎死，对策是加 killswitch 或把大 blob 分块处理。慢性 OOM：内存在会话中缓慢爬升直至越限，多因手动管理的状态未被释放、或经游离强引用泄漏资源——它在堆快照中稳定可见，靠追踪 retainer、理清长生命周期对象来修复，部分修复已回馈上游 VSCode。扩展崩溃则部分靠进程隔离缓解（类似 Chrome 隔离标签页，代价是略多内存）。
洞见：「骤死」和「慢漏」需要完全不同的工具与心智模型——一个靠崩溃堆栈，一个靠堆快照；分清类型才能不浪费定位精力。

05修不难，难在「不让新坑出现」

修崩溃通常比防崩溃容易，因为修复是针对性的；而预防要让每位开发者都意识到自己对稳定性的影响，又不能牺牲 Agent 带来的速度，于是同时投入流程与工具：为每一大类 OOM/崩溃沉淀 Bugbot 规则；用 Skills 通过 Agentic 计算机使用轻松压测应用；消除「足枪」（用垃圾回收替换手动资源管理以避免泄漏）；每次改动后跑传统自动化性能测试；并用指标回归自动回滚闭合检测环路。
洞见：Agentic 开发让「上新功能」和「引入性能 bug」都更容易——稳定性靠的不是新魔法，而是把回归预防工程化、自动化。

最值得带走的洞见

  1. 数字说话。OOM 每会话率较 2 月底峰值降 80%，每请求率自 3 月 1 日降 73%——稳定性是能被度量、被持续压低的工程目标。
  2. 先看见，再治理。分钟级、可按版本拆分、分「每会话/每请求」两口径的仪表盘，是一切快速发现回归的前提。
  3. 两条腿走路。自上而下用特征开关 A/B 和代理指标量化贡献，自下而上用 CDP 崩溃堆栈与堆快照挖根因，并把高置信度修复做成每日自动 PR。
  4. 分型施治。急性 OOM 靠 killswitch 与分块处理，慢性 OOM 靠追 retainer 清理生命周期——部分泄漏修复已回馈上游 VSCode。
  5. 难点在预防。Bugbot 规则、压测 Skills、用 GC 消除足枪、性能测试与自动回滚，让提速不以稳定性为代价——Agentic 时代依旧要靠软件工程基本功，只是为新一代工具重新配套。
