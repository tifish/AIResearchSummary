让 Cursor 应用保持稳定

Keeping the Cursor app stable

一句话总括：Cursor 把崩溃遥测、内存分析、自动修复和回归防护连成闭环，在功能快速扩张的同时显著降低了 OOM。

突发风险

急性 OOM：一次加载太多数据

内存会在短时间内猛增，进程随即退出。这类问题常见于一次读取超大文件，或通过 IPC 传递完整的大块内容。

有效措施包括加入 killswitch，或将大块数据拆成多个小块处理。

结论：面对不可控的用户工作区，系统必须主动限制单次工作量。

累积风险

缓慢 OOM：资源没有正确释放

内存会在整个会话中持续爬升，最终越过进程上限。常见原因是手动管理的状态没有销毁，或多余的强引用长期保留对象。

堆快照可以稳定呈现这类问题。团队通过追踪 retainer 和修正长生命周期对象来消除泄漏。

结论：长期稳定性取决于清晰、可验证的资源生命周期。

真正改变理解的不是单一优化，而是把用户现场、全量趋势与自动修复连接起来。稳定性由此从事后救火变成持续反馈系统。

最值得带走的洞见

指标必须区分覆盖面和严重程度 每会话崩溃率反映多少会话受到影响。每请求崩溃率更接近受影响会话中的问题严重度。两个指标结合，才能避免单一数字掩盖真实体验。

许多用户整天都在使用 Cursor。这意味着，即使崩溃很少发生，也可能造成极大干扰。与此同时，随着用户数量增长，以及 subagents、instant grep、browser use 等更具雄心的功能陆续发布，让应用保持稳定也变得越来越困难。

这些崩溃大多源于应用内存不足（OOM）。过去几个月里，我们构建了一系列系统，用于观察崩溃和内存压力。我们也针对热点路径实施了高置信度修复和优化，并设置护栏，在回归问题发布前将其发现。

汇总 Cursor 应用所有版本后，我们的每会话 OOM 率较 2 月下旬的峰值下降了 80%。每请求 OOM 率自 3 月 1 日以来下降了 73%。本文将详细介绍我们为实现这些改进而构建的系统。

发现并衡量不稳定性

我们的桌面应用建立在开源的 Visual Studio Code 和 Electron 基础之上，因此采用多进程架构。崩溃可能发生在为编辑器和新 Agent 窗口提供支持的渲染进程中，也可能发生在为扩展、存储和 Agent 功能提供支持的实用进程中。

渲染进程崩溃最为严重，因为它会让用户完全无法使用编辑器。我们发现，这类崩溃大多是达到 V8 内存上限造成的，也是近期工作的重点。扩展崩溃也会中断语言服务等重要功能，但通常可以自行恢复，对用户造成的干扰没有那么严重。

每次致命崩溃都会通过遥测上报。上报信息包括受影响的进程、崩溃类型、设备和应用元数据，以及能够获得的 minidump 和堆栈跟踪。

基于这些崩溃事件，我们建立了可以按应用版本拆分的指标，并分别按会话和请求计算发生率。前者大致反映有多少会话遭遇崩溃。后者则反映崩溃问题对受影响会话有多严重。这些看板会在崩溃事件发生后的数分钟内更新，因此我们可以密切跟踪新版本发布，并迅速发现潜在回归。

两种调试策略

在调试应用崩溃和内存不足问题时，我们采用双管齐下的策略。

自上而下

第一种是自上而下的调查，重点关注内存消耗最大的功能。如果已知某项功能会大量占用内存，我们就可以将崩溃指标与实验平台 Statsig 中对应的功能标记关联起来。随后，我们通过 A/B 测试衡量它对崩溃率的影响。

我们还会跟踪与崩溃高度相关、在开发环境中也更容易观察的代理指标。超大消息载荷就是其中之一。由于应用采用多进程架构，数据会不断通过进程间通道和持久化层，在编辑器、扩展与 Agent 之间传递。我们在这两个环节加入监测，追踪超过一定阈值的消息。这类消息与内存问题高度相关。我们还会附上调用栈，以便将每条消息追溯到应用代码中的来源。

为了还原某次崩溃发生时的情况，我们会为并行 Agent 使用、工具调用和终端等功能加入 breadcrumbs。breadcrumbs 是附加到错误上的特殊元数据日志。这样，每个崩溃事件都会带有一份记录，说明崩溃前发生了哪些活动。

自下而上

在自下而上的调查中，我们会把单个崩溃事件追溯到根本原因。第一步是捕获进程死亡时发生的情况。我们在主进程中运行崩溃监视服务。它使用 Chrome DevTools Protocol（CDP）检测内存不足错误，并实时捕获崩溃堆栈。我们还向上游 Electron 提交了补丁，让系统无需使用笨重的 CDP 机制也能取得这些堆栈。这些崩溃堆栈会进入一套每日运行的自动化流程。该流程详细分析每个堆栈，为存在高置信度修复方案的堆栈创建优化 PR，并通过版本间对比验证问题是否已经解决。

为了了解内存在一次会话中如何逐渐累积，我们会查看堆快照。当检测到 Cursor 使用了过多内存时，我们会提示用户捕获并发送快照。这些快照可能包含敏感信息，例如已打开编辑器或聊天中的内容，因此提交完全由用户自行选择。不过，堆快照对于把不断累积的内存压力追溯到具体对象和 retainer 非常有价值。用户愿意参与时，我们对此十分感谢。

为了了解整个用户群体的内存使用模式，我们会以较低的采样率持续进行堆分配分析。我们按应用版本汇总这些数据，得到按调用栈划分的内存压力明细。这让我们能够从全局观察各次应用会话中的内存压力。我们甚至可以比较不同版本，从而了解新版本中某条分配路径相较旧版本有所改善还是发生了回归，以及变化幅度有多大。

针对性缓解措施

通过这两种调查方法，我们发现崩溃通常可以归入两种模式。

第一种是急性 OOM。内存会突然飙升，随后进程退出。这类问题通常通过崩溃堆栈发现，很少出现在堆转储或持续分析数据中。一个非常常见的原因是，某项功能一次加载了过多数据。我们的应用需要广泛处理用户工作区中的内容，因此经常从磁盘读取完整文件，或通过 IPC 加载完整文件内容。一些用户的工作区可能包含极其庞大的文件，导致应用无法承受。为相关功能加入 killswitch，或将大块数据拆分成多个小块处理，至关重要。

第二种是缓慢而持续的 OOM。内存会在整个会话过程中逐渐增加，直到把进程推过上限。当手动管理的状态没有正确销毁，或游离的强引用造成资源泄漏时，就会出现这种问题。它们能够稳定地出现在堆转储中。我们可以通过追踪 retainer，并清理长生命周期对象的生命周期来修复问题。我们已经向 VSCode 上游提交了几项内存泄漏修复，并计划继续提交更多修复。

扩展也可能因为内存不足而崩溃。我们使用进程隔离来缓解部分影响。粗略来说，把扩展分别放在独立进程中运行，可以避免某个扩展的崩溃或长时间任务影响其他扩展的功能。这与 Chrome 隔离不同标签页的方式相似，代价是系统内存占用会略微增加。

防止回归，同时保持速度

修复应用崩溃通常比防止新崩溃进入产品更直接，因为修复可以针对具体问题。预防则要求每位开发者都了解自己的工作对稳定性的影响，同时不能牺牲我们借助 Agent 获得的开发速度。这意味着我们必须同时投资流程和工具。

我们采取的方法包括：

  • 为遇到过的每一类主要 OOM 或应用崩溃编写 Bugbot 规则。
  • 创建 Skills，让我们能够通过 Agent 式计算机操作轻松对应用进行压力测试。
  • 消除容易误用的设计。例如，用垃圾回收取代手动管理的资源，避免内存泄漏。
  • 在每次代码变更后运行传统的自动化性能测试。
  • 完善从检测到处置的闭环。例如，在指标发生回归时自动回滚。

面向新一代软件的稳定性

Agent 式软件开发让新功能的交付比以往任何时候都更容易，但也让性能问题和缺陷更容易进入产品。与此同时，实现应用稳定性仍然依赖相同的软件工程基本功。只不过面向新一代软件，这些方法也需要演进。我们需要采用 Agent 式策略来修复问题，并防止问题发生。

打造高质量软件一直很难，如今它比以往任何时候都更加重要。如果你也对此充满热情，我们很希望收到你的消息。

Agent 式软件开发让新功能的交付比以往任何时候都更容易，但也让性能问题和缺陷更容易进入产品。与此同时，实现应用稳定性仍然依赖相同的软件工程基本功。只不过面向新一代软件，这些方法也需要演进。我们需要采用 Agent 式策略来修复问题，并防止问题发生。

打造高质量软件一直很难，如今它比以往任何时候都更加重要。如果你也对此充满热情，我们很希望收到你的消息。
