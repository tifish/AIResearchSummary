一句话总括

LLM 已能把 0-day 发现从专家手工研究推进到可规模化生产。

Anthropic 的核心贡献不是宣布模型“懂安全”，而是展示 Claude Opus 4.6 能在成熟开源项目中找到高危漏洞，并同时提出验证、补丁和滥用检测流程。

3 个案例GhostScript、OpenSC、CGIF 展示不同发现路径
90 天传统披露窗口可能难以承受批量发现速度

模型的变化在于“会读代码”，不只是会跑工具。

Anthropic 将 Claude 放进模拟计算机，提供常规命令行工具、调试器和 fuzzing 工具，但没有给专用 harness、特殊提示或任务脚手架。Opus 4.6 仍能在已有长期 fuzzing 的项目里找到高危漏洞。

这改变了对 AI 安全能力的理解：模型不只是替代随机输入测试，而是能像研究员一样阅读提交历史、识别危险模式，并推导触发漏洞的输入。

结论和洞见：当模型能主动决定该看哪里，漏洞发现的瓶颈会从“算力覆盖多少路径”转向“谁能验证、修复和协调披露”。

超过 500 个高危漏洞说明问题已经进入工程规模。

文章称 Anthropic 已发现并验证超过 500 个高危漏洞，并开始向维护者报告，其中一些补丁已经合入。研究流程优先关注内存破坏类漏洞，因为崩溃、地址清理器和复现样例能更可靠地区分真实问题与幻觉。

为降低开源维护者负担，Anthropic 让 Claude 先批判、去重、排序，再由内部和外部人类安全研究员验证并编写补丁。这个流程承认了 LLM 的双重现实：发现能力很强，但未经验证的报告会制造新的维护成本。

结论和洞见：LLM 漏洞发现的关键不是“模型报告了什么”，而是是否能形成低误报、可复现、带补丁的交付链。

三个案例显示 LLM 能覆盖 fuzzing 不擅长的漏洞形态。

在 GhostScript 中，Claude 先尝试 fuzzing 和手工分析，随后转向阅读 Git 提交历史，从安全相关修复中寻找未覆盖的相似调用点，并构造了能触发崩溃的 proof of concept。

在 OpenSC 中，Claude 识别连续 strcat 调用带来的缓冲区溢出风险。传统 fuzzers 很少触达这行代码，因为需要满足许多前置条件；模型则能先判断代码片段是否值得深入。

在 CGIF 中，Claude 利用对 GIF LZW 压缩机制的理解，发现“压缩后一定更小”的假设在符号表耗尽并插入 clear token 时可能失效，从而触发缓冲区溢出。即使有很高的行覆盖和分支覆盖，这类特定操作序列也可能被漏掉。

结论和洞见：LLM 的优势不只是速度，而是能把提交语义、危险 API、文件格式和算法机制组合起来推理。

防御价值越大，滥用治理越必须同步升级。

Anthropic 在 Claude Opus 4.6 发布时引入新的 cyber-specific probes，用模型生成过程中的激活信号帮助识别网络安全滥用，并更新执法流程以支持更快响应。

文章明确提到可能进行实时干预，包括阻断被检测为恶意的流量。Anthropic 也承认，这会给合法研究和防御工作带来摩擦，因此需要与安全研究社区协调。

结论和洞见：LLM 安全能力不是单向利好；同一套能力既能帮助维护者修复开源基础设施，也能降低攻击者寻找 0-day 的门槛。

漏洞披露规范需要适应更高发现速度。

文章指出，行业常用的 90 天披露窗口可能无法适应 LLM 批量发现漏洞的速度和数量。问题不只是发现更快，还包括维护者如何接收、复现、排序、修补和发布。

开源项目尤其受影响，因为许多基础项目由小团队或志愿者维护，却支撑企业系统和关键基础设施。模型发现漏洞后，真正的社会价值取决于补丁是否能及时落地。

结论和洞见：未来的安全治理会更像供应链协作问题，而不是单个研究员向单个项目提交一个漏洞报告。

最值得带走的洞见

  • Claude Opus 4.6 的关键进展是无需专用脚手架也能发现真实高危漏洞。
  • 超过 500 个已验证发现说明 LLM 漏洞挖掘已经具备规模化特征。
  • LLM 能补足 fuzzing 的盲区，尤其是需要语义理解、历史对照或特定算法路径的漏洞。
  • 误报控制、人工验证和补丁协作将决定这类能力对开源生态是减负还是增负。
  • 传统披露窗口和滥用检测机制都需要更新，否则防御方可能跟不上模型制造的信息量。
