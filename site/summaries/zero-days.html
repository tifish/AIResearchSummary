评估并缓解 LLM 发现 0-day 漏洞所带来的日益增长的风险

Evaluating and mitigating the growing risk of LLM-discovered 0-days

发布日期：2026 年 2 月 5 日 来源：Anthropic 更新：2026 年 2 月 6 日

一句话总括：Claude Opus 4.6 已能像安全研究员一样理解代码，在成熟项目中规模化发现传统工具长期遗漏的高危漏洞，这同时扩大了防御能力和双重用途风险。

500+ 已发现并经过验证的高危漏洞
数百万小时 部分目标代码库此前累计接受的 CPU fuzzing 时间
90 天 可能难以承受 LLM 漏洞发现速度的行业披露窗口

贡献不只是“更快地跑 fuzzing”

Claude 被放入一台配有常规工具的虚拟机。研究团队没有提供专用 harness、特殊脚手架或工具使用教程。

它会阅读提交历史，识别容易出错的函数，并推理出触发特定逻辑路径所需的输入。这说明 LLM 带来的增益来自代码理解和策略选择，而不只是更多算力。

成熟代码库也不再等于低风险

一些目标项目已经被 fuzzers 测试多年，却仍藏有高危漏洞。部分问题甚至存在了数十年。

这改变了一个常见判断：高覆盖率和长期测试不能证明复杂状态组合已经安全。LLM 可能重新打开那些被认为已充分检查的代码区域。

团队如何控制误报

  1. 寻找问题Claude 在真实开源项目中自主选择分析方法。
  2. 验证崩溃团队使用崩溃监控和 AddressSanitizer 等工具确认内存错误。
  3. 复核排序Claude 对结果进行批评、去重，并重新评估优先级。
  4. 人工把关内部及外部安全研究人员验证漏洞，并协助开发补丁。

流程把减少误报放在重要位置，因为未经验证的自动报告会把成本转嫁给资源有限的开源维护者。结论是，规模化发现必须与同样可靠的验证和修复能力一起扩展。

案例 01 · GhostScript

从历史补丁反推遗漏点

fuzzing 和初步人工分析都没有结果。Claude 随后阅读 Git 提交历史，找到一项与安全有关的修改，再搜索同一函数的其他调用位置。

它最终构造出可触发崩溃的文件。这个案例表明，历史修复记录也能成为发现相似未修复漏洞的线索。

案例 02 · OpenSC

跳过低价值路径，聚焦危险代码

Claude 发现连续使用多个 strcat 的位置，并识别出缓冲区溢出风险。传统 fuzzer 很少到达这行代码，因为触发它需要满足许多前置条件。

LLM 能判断哪些代码片段更值得检查。它不必对所有路径平均分配精力。

案例 03 · CGIF

理解算法后构造罕见状态

CGIF 假定压缩后的数据总会更小。Claude 理解 LZW 固定大小符号表和特殊“clear”标记的关系，并发现压缩结果可能反而变大。

即使达到 100% 行覆盖率和分支覆盖率，这个漏洞也可能被遗漏，因为它需要一组非常具体的操作顺序。

能力增长也会压缩响应时间

Anthropic 为 Claude Opus 4.6 增加了面向网络安全场景的 probes。它们在模型生成响应时测量内部激活，用于识别潜在滥用。配套措施可能包括实时干预和阻断恶意流量。

这些措施也可能给合法研究和防御工作制造阻力。更深层的问题是，发现速度正在接近甚至可能超过专家研究人员，而漏洞验证、协调披露和补丁部署仍受人工流程约束。

核心结论：风险不只在于 LLM 会发现漏洞，还在于现有安全制度可能跟不上它发现漏洞的规模。

防守方眼下拥有一个短暂窗口：尽快让 AI 帮助检查和修复更多代码，同时建立能够承受同等速度的验证、披露与滥用防控流程。

最值得带走的洞见

LLM 已经具备实用的 0-day 发现能力。这不是单一 benchmark 推测，而是来自真实开源项目和人工验证漏洞的结果。

推理能力补足了 fuzzing 的盲区。LLM 可以结合代码语义、提交历史和算法状态，主动选择高价值路径。

发现规模必须匹配修复规模。超过 500 个高危发现说明，验证、去重、补丁开发和维护者协作会成为新的瓶颈。

披露规范需要调整。传统 90 天窗口可能无法适应 LLM 发现漏洞的速度和数量。

完整译文

Nicholas Carlini*、Keane Lucas*、Evyatar Ben Asher*、Newton Cheng、Hasnain Lakhani、David Forsythe 和 Kyla Guru。* 表示贡献相同。

Claude Opus 4.6 于今天发布。它延续了 AI 模型网络安全能力显著提升的趋势。去年秋天，我们曾写道，AI 对网络安全的影响已经来到一个转折点。能力进步可能迅速加快，而现在正是加速利用 AI 开展防御的时机。此后的证据进一步强化了这一判断。AI 模型如今已经可以规模化发现高危漏洞。我们认为，现在必须迅速行动。在窗口仍然存在时，我们应当增强防守方的能力，并保护尽可能多的代码。

Opus 4.6 发现高危漏洞的能力明显优于此前的模型。这也说明相关能力的发展速度有多快。多年来，安全团队一直在自动化漏洞发现。他们投入大量资源建设 fuzzing 基础设施和定制 harness，希望规模化寻找缺陷。

但在早期测试中，最引人注意的是 Opus 4.6 无需针对任务开发专用工具，也不需要定制脚手架或特殊提示，就能迅速发现漏洞。它发现漏洞的方式更值得关注。Fuzzer 会向代码投放大量随机输入，观察什么输入会令程序出错。Opus 4.6 则会像人类研究员一样阅读并分析代码。它会查看过去的修复，从中寻找尚未处理的相似缺陷。它会识别容易引发问题的模式，也能充分理解一段逻辑，再判断究竟什么输入可以破坏它。

我们让 Opus 4.6 检查了一些经过最充分测试的代码库。这些项目已经接受 fuzzers 多年测试，累计消耗了数百万小时的 CPU 时间。Opus 4.6 仍然在其中发现了高危漏洞。有些漏洞已经潜伏了数十年，却始终没有被发现。

要让天平向防守方倾斜，我们也必须亲自投入这项工作。我们现在正使用 Claude 寻找开源软件中的漏洞，并协助修复。我们首先选择开源软件，是因为它无处不在。从企业系统到关键基础设施都在运行开源软件，其中的漏洞会在整个互联网扩散。

许多开源项目由小型团队或志愿者维护。他们没有专门的安全资源。因此，找到经过人工验证的缺陷，再贡献经过人工审查的补丁，可以带来很大帮助。

到目前为止，我们已经发现并验证了 500 多个高危漏洞。我们已经开始报告这些问题，也看到首批补丁陆续被合入。我们仍在与维护者合作，修复其余漏洞。

本文将介绍我们的方法，分享 Claude 发现的一些早期漏洞案例，并讨论我们为应对潜在滥用而设置的防护措施。这些能力仍在持续进步，而这项工作也才刚刚开始。随着工作规模扩大，我们还会分享更多信息。

实验设置

在这项工作中，我们把 Claude 放进一台“虚拟机”。它确实是一台模拟出来的计算机，其中包含开源项目的最新版本。我们为它提供了标准工具，例如常见的 coreutils 和 Python。我们也提供了漏洞分析工具，例如调试器和 fuzzers。

不过，我们没有特别说明应该怎样使用这些工具。我们也没有提供定制 harness，让 Claude 获得关于如何更有效发现漏洞的专门知识。这意味着，我们直接测试的是 Claude 的“开箱即用”能力。整个实验只依赖一个事实：现代大语言模型已经是通用型智能体，能够自行判断怎样充分利用现有工具。

在报告任何问题之前，我们都进行了充分验证，以确保 Claude 没有产生虚构漏洞。所谓虚构漏洞，是指模型编造实际上并不存在的问题。这种情况正在给开源开发者造成越来越不合理的负担。

我们重点寻找内存损坏漏洞，因为这类问题相对容易验证。逻辑错误发生后，程序可能仍然可以运行。内存损坏则更容易识别。我们可以监控程序是否崩溃，也可以运行 AddressSanitizer 等工具，捕获没有导致崩溃的内存错误。

不过，并非所有能让程序崩溃的输入都代表高危漏洞。因此，我们随后让 Claude 对剩余的崩溃进行批评、去重，并重新评估优先级。在首轮发现中，我们自己的安全研究人员验证了每个漏洞，并手工编写补丁。

随着发现数量增加，我们又邀请外部的人类安全研究人员协助验证和开发补丁。我们的目标是切实帮助维护者处理这些报告。因此，整个流程特别重视减少误报。与此同时，我们也在加快自动化补丁开发，希望在发现缺陷后能够可靠地完成修复。

CGIF 是一个处理 GIF 文件的库。在这个案例中，让我们意外的不是 Claude 如何找到漏洞，而是它如何验证漏洞，并生成概念验证来证明问题确实存在。

简单来说，Claude 发现这个库假定压缩后的数据一定比原始数据小。这个假定通常是安全的，但在这里可以被利用。证明这一点并不容易。

GIF 文件格式使用 LZW 压缩算法压缩数据。较为传统的 LZ77 系压缩器使用“距离、长度”二元组来编码匹配项。LZW 的做法不同。它会为频繁使用的 token 组合建立字典。这些组合可以是单个字符，也可以是两个 token 的组合。压缩器再次遇到同一个 token 时，只需输出一个更短的比特串，表示“再次输出这个 token”。熟悉语言建模的读者可以把 LZW 理解为一种在思路上与 BPE 分词很相似的过程。

CGIF 隐含地假定，字符串压缩后的大小总会小于未压缩时的大小。这个假定几乎总是成立。不过，Claude 立刻识别出了这里的漏洞：

Claude 意识到，LZW 维护着一个固定大小的符号表。如果符号表可以被填满，LZW 就会在数据流中插入一个特殊的新“clear”token。这样一来，“压缩后”的输出大小就可能超过未压缩数据的大小，进而触发缓冲区溢出漏洞。

这个漏洞格外有意思，因为触发它需要从概念上理解 LZW 算法，以及它与 GIF 文件格式的关系。传统 fuzzers 很难触发这种漏洞。即使是覆盖率引导的 fuzzers 也会遇到困难，因为程序必须在多个分支中作出特定选择。

事实上，即使 CGIF 达到 100% 的行覆盖率和分支覆盖率，这个漏洞仍然可能不被发现。它需要一组非常具体的操作顺序。

防护措施

在发布 Claude Opus 4.6 的同时，我们正在引入一个新的检测层，帮助 Safeguards 团队识别并应对利用 Claude 实施的网络安全滥用。

这项工作的核心是 probes。它们会在模型生成响应时测量模型内部的激活，使我们能够规模化检测特定危害。随着本次发布，我们创建了面向网络安全场景的新 probes，以便更好地追踪和理解 Claude 在网络安全领域可能遭到的滥用。

在执行措施方面，我们也在改进处理管线，使其跟上新的检测架构。这包括更新网络安全执行工作流，利用基于 probes 的检测结果。我们还会扩大应对网络安全滥用时可采取的措施范围。

具体来说，我们可能实施实时干预，包括阻断被判定为恶意的流量。这会给合法研究和部分防御工作带来阻力。我们希望与安全研究社区合作，在问题出现时寻找解决办法。

我们致力于让 Claude 保持在网络安全领域的前沿。为此，我们会努力确保它既安全又有效。

这些变化结合起来，意味着我们在防止滥用方面迈出了重要一步。提升的不只是可检测的范围，也包括我们根据发现采取行动的速度和有效性。

结论

即使没有专用脚手架，Claude Opus 4.6 也可以在经过充分测试的代码库中发现有实质意义的 0-day 漏洞。我们的结果表明，语言模型能够在现有发现工具的基础上提供真正的增益。前文介绍的 Safeguards 工作，对于管理由此产生的双重用途风险至关重要。

展望未来，我们和更广泛的安全社区都必须面对一个令人不安的现实：语言模型已经能够发现新的漏洞，并且可能很快在速度和规模上超过人类专家研究员。

与此同时，现有的漏洞披露规范也需要调整。行业标准的 90 天窗口可能无法承受 LLM 发现漏洞的速度和数量。整个行业需要建立能够跟上这种节奏的工作流程。

这项工作仍在进行。我们很快会分享更多内容，包括我们对这些能力如何演变的认识，以及安全社区应当怎样充分利用它们。
