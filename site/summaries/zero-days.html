评估并缓解 LLM 发现 0-day 的风险

来源 Anthropic ｜ 作者 Nicholas Carlini、Keane Lucas、Evyatar Ben Asher、Newton Cheng、Hasnain Lakhani、David Forsythe、Kyla Guru ｜ 2026-02-05

一句话总括：这篇文章真正的贡献不是宣布"AI 会找漏洞"，而是给出证据——Claude Opus 4.6 无需任何定制脚手架、专用 harness 或特制提示，就能"开箱即用"地像人类安全研究员一样阅读并推理代码，在被 fuzzer 跑了几十年、累积数百万 CPU 小时的成熟开源代码库里挖出高危 0-day，其中一些已潜伏数十年。Anthropic 据此已找到并人工验证 500+ 个高危漏洞，并主张：现在是趁攻防窗口还开着、用 AI 武装防守方的关键时刻。

为什么是拐点不是更快的 fuzzer，而是会"读代码"的研究员

去年秋天 Anthropic 判断 AI 对网络安全的影响已到拐点；Opus 4.6 把这个判断坐实了。关键差别在于怎么找：fuzzer 靠海量随机输入去撞崩溃；Opus 4.6 则像人一样读代码——翻历史补丁找同类未修的 bug、识别容易出问题的模式、把一段逻辑吃透到知道哪种输入会让它崩。当被指向那些已跑了数百万 CPU 小时 fuzzer 的代码库时，它挖出了沉睡数十年的高危漏洞。

结论：当发现方式从"穷举输入"换成"理解代码"，那些被测了几十年仍存活的漏洞，第一次被系统性触达。

方法不给专用 harness，只给一台虚拟机和常规工具

Claude 被放进一台虚拟机，配上最新版开源项目、coreutils/Python 等标准工具，以及调试器、fuzzer 等漏洞分析工具——但不给任何"怎么找漏洞"的特别指令或定制脚手架，直接测它的开箱能力。为防"幻觉式误报"（这类误报正给开源维护者造成沉重负担），团队聚焦易验证的内存破坏漏洞：用崩溃监控加 address sanitizer 就能坐实；再让 Claude 自我批评、去重、重排剩下的崩溃；首轮由 Anthropic 安全研究员逐一人工复核并手写补丁，量大后又引入外部安全研究员协助验证与打补丁。

三个案例它们的共同点是都需要"理解"，而非"撞运气"

三个已被维护者修补的漏洞，展示了 LLM 相对其它工具的独特优势——都不是靠随机输入撞出来的，而是靠对代码与算法的理解推出来的。

项目Claude 的方法为何 fuzzer 难做到
GhostScript
PostScript/PDF 处理		 先尝试 fuzz 与人工分析均无果，转而读 Git 提交历史，发现一个安全相关提交，再去找同一函数其它未修补的调用点，构造出 PoC 崩溃 fuzzer 不会"读历史补丁"举一反三
OpenSC
智能卡数据处理		 同样在 fuzz/人工分析失败后改换思路，搜索易错函数调用，发现连续多次 strcat 缺少长度检查导致缓冲区溢出 该行因前置条件太多，极少被传统 fuzzer 触达
CGIF
GIF 图像库		 理解 LZW 压缩：符号表占满时会插入 clear 标记，使"压缩后"体积反而超过原始大小，突破"压缩必更小"的隐含假设 → 缓冲区溢出 需要特定操作序列；即便 100% 行/分支覆盖率也测不出

结论：CGIF 案例尤其关键——它证明对算法的"概念理解"能触达 fuzzer 在理论上都够不到的盲区，这是 LLM 相对 fuzzer 的结构性优势。

安全护栏用"探针"在生成时实时识别网络滥用

与 Opus 4.6 同步，Anthropic 给 Safeguards 团队加了新一层检测：核心是 probes（探针），在模型生成回复时直接测量其内部激活，从而大规模检测特定危害；这次新增了网络安全专用探针。执行侧也在升级——可能对检测为恶意的流量做实时干预甚至直接拦截。Anthropic 坦承这会给正当研究和部分防守工作带来摩擦，并表示要与安全社区一起化解。

结论：能力是双刃的，所以检测必须从"事后看日志"前移到"生成时看激活"，并能即时行动。

趋势90 天披露窗口可能扛不住 LLM 的速度与规模

Anthropic 选择从开源入手，因为它无处不在——从企业系统到关键基础设施——漏洞会沿互联网层层扩散，而很多项目由缺乏安全资源的小团队或志愿者维护，"人工验证的 bug + 人工复核的补丁"价值最大。展望未来：LLM 已能发现新漏洞，且速度与规模可能很快超过专家级人类研究员；面对 LLM 产出 bug 的数量与速度，行业标准的 90 天披露窗口恐难维持，披露规范与工作流都需要随之进化。

结论：进攻与防御都将因此重写流程；当下的关键，是趁窗口还开着把 AI 的能力尽量倒向防守方。

最值得带走的洞见

  1. 真正的新闻是"会推理的发现方式"，不是更快的 fuzzer。Opus 4.6 读提交历史、识别易错模式、吃透算法逻辑，触达了 fuzzer 几十年都没碰到的漏洞类别。
  2. "开箱即用"是关键限定词。没有定制 harness、没有特制提示，仅靠通用智能体能力加标准工具就能找出高危 0-day，意味着发现漏洞的专家门槛正在塌。
  3. 500+ 高危漏洞且经人工验证。聚焦内存破坏类是为了可验证、降误报；流程刻意优化"真正帮到维护者"而非刷量，初步补丁已开始落地。
  4. CGIF 案例是 fuzzer 的能力天花板示例。即便 100% 覆盖率也测不出的 LZW 漏洞，证明"理解"带来的是结构性优势而非边际改进。
  5. 制度必须同步进化。能力的双用风险，要求检测前移到生成时的探针与实时拦截，也要求 90 天披露窗口等行业规范被重写。
