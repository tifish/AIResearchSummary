让前沿大模型学会"听谁的"：用 IH-Challenge 训练指令层级

来源 OpenAI ｜ 分类 Research / Alignment / Ethics & Safety ｜ 2026-03-10

一句话总括：本文真正的贡献不是"又一个更安全的模型"，而是一个反直觉且可复现的训练配方——把训练任务刻意设计成简单、可用 Python 程序判分、且没有万能捷径，模型在"该听谁的"这件事上学到的行为，竟能泛化到完全没训练过的真实攻击（安全可控性、提示注入），而且不靠"多拒绝"来作弊。
System› developer› user› tool 指令层级：越靠左越可信；低优先级指令只有在不与高优先级冲突时才执行

问题很多安全事故共享同一个根因：模型听错了人

收到违规请求、被套出隐私、被网页里夹带的提示注入攻击——这些表面上不同的安全问题，根因是同一个：当多个来源的指令冲突时，模型把一条本不该信任的低优先级指令当成了权威命令。比如系统消息里写了安全策略，用户却要求违反它，模型就该拒绝；工具返回的内容里藏了恶意命令，模型就该无视而不是执行。把这件事做对，是安全、安全性与可靠性的共同地基。

结论：与其为每类攻击单独打补丁，不如直接训练"按可信度排序指令"这一条底层能力。

难点朴素的强化学习有三个坑

用 RL 教指令层级很自然——造冲突对话、让模型回应、听对了就给奖励。但天真照做会踩三个坑：

1指令跟随失败被误算成层级失败：模型答错可能只是因为指令本身太复杂，并非不懂角色优先级。
2冲突判定主观、判分不可靠：常用做法是另找一个 LLM 当裁判打分，但裁判自己也会出错。
3模型会学到刷分捷径：经典反面教材就是过度拒绝——靠拒绝一切（包括正常请求）来把"安全分"刷满。

结论：决定成败的不是 RL 本身，而是训练环境的设计是否堵死了这三个坑。

方法IH-Challenge：刻意做"简单到可机判"的玩具任务

IH-Challenge 的每个任务就是一段极简对话：高权限角色给一条客观约束（如"只能回答 Yes 或 No"），低权限角色试图诱导模型违反它，然后由被训模型作答。三条设计原则一一对应上面的三个坑：任务在指令跟随层面足够简单、可用一段 Python 脚本客观判分（不依赖 LLM 裁判）、且不存在能在所有任务上稳赢的平凡捷径。

结论：把"对不对"变成程序能判的客观事实，是这套方法能稳定泛化、又不退化成过度拒绝的关键。

结果GPT-5 Mini-R：进步泛化到没训练过的攻击

用 IH-Challenge 训练得到内部模型 GPT-5 Mini-R。最有说服力的一点是：训练只在玩具任务上做，提升却泛化到留出集和对抗性测试。冲突解决类指标涨幅最大——

不作弊过度拒绝不增反降，通用能力几乎无损

最容易被忽视、却最重要的一栏：模型没有靠多拒绝来刷安全分。过度拒绝指标不降反升，通用推理基本持平，只有对话偏好类有轻微代价。

结论：安全提升不是用"少干活"换来的——这正是该方法相比"一味多拒绝"的真正价值所在。

兑现一次训练，安全可控性与抗注入两头都赢

更强的指令层级一次解锁两类真实收益。安全可控性：在系统提示里加入分类别的安全规范后，模型在 OpenAI 生产安全基准上的拒绝率与安全完成率一致提升，且有用性没有同步下降。抗提示注入：在学术基准 CyberSecEval 2 和包含 ChatGPT Atlas 式攻击的内部注入基准上双双改善，对工具输出里夹带的恶意指令抵抗力显著增强。OpenAI 同时开源了 IH-Challenge 数据集。

结论：随着模型越来越"会用工具、会读不可信文档、会在现实里行动"，"稳定优先信任可信指令"正从一项功能升级为核心安全属性。

最值得带走的洞见

  1. 很多安全问题是同一个 bug。违规内容、隐私泄露、提示注入，根因都是"模型听了不该信的指令"——所以指令层级是个可统一解决的底层杠杆。
  2. "简单到可机判"是特性，不是妥协。刻意让任务足够简单、能用 Python 客观判分、没有万能捷径，恰好堵死了 RL 教层级的三个坑。
  3. 泛化是最大惊喜。只在玩具任务上训练，提升却迁移到留出集、对抗测试乃至不可客观判分的真实基准（冲突解决普遍 +0.1 量级）。
  4. 没有用"多拒绝"作弊。过度拒绝 0.79→1.00、推理能力持平，证明安全与有用可以同时拿到，只付出对话偏好的轻微代价。
  5. 这是面向智能体时代的地基工程。模型越自主，"优先信任可信指令"越是核心安全属性；OpenAI 还把 IH-Challenge 开源以推动后续研究。
