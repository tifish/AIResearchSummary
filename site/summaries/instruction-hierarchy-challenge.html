一句话总括

改进前沿 LLM 的指令层级

Improving instruction hierarchy in frontier LLMs

OpenAI 的贡献不是只提出一个新榜单，而是把“模型该听谁的”变成可训练、可自动评分、并能提升真实安全性的任务设计。

真正的问题是权限判断

AI 系统会同时看到系统政策、开发者规则、用户请求和工具输出。安全失败常常不是模型不会回答，而是它把低可信来源当成了命令。

这改变了安全训练的视角。重点不只是拒绝危险请求，而是在冲突中稳定识别哪条指令有更高优先级。

OpenAI 采用的层级

1

System：最高优先级，承载安全政策和全局约束。

2

Developer：产品或应用规则，低于系统但高于用户。

3

User：用户目标，只有不冲突时才应执行。

4

Tool：工具返回的数据最不可信，尤其容易夹带恶意指令。

IH-Challenge 的设计避开三类捷径

任务要简单

指令本身不能太复杂。否则失败可能只是普通指令跟随失败，而不是层级判断失败。

评分要客观

每个环境都能用简单 Python 脚本检查结果。这样减少了 LLM judge 的主观误差。

不能靠拒绝刷分

任务设计避免“一律拒绝”成为高分策略。安全训练因此更接近真实使用需求。

最重要的信号是泛化：一个看似简单的训练集，让模型在留出测试、对抗测试、安全规范执行和 prompt injection 防御上都更稳。这说明指令层级可能是一种底层安全能力，而不是某个榜单技巧。

为什么这会影响 prompt injection

风险来自工具输出。 网页、文档或外部工具可能包含“忽略之前指令”之类的恶意文本。模型如果把这些文本当成命令，就会泄露信息或执行错误动作。

IH 训练让模型更倾向于把工具内容视为数据，而不是高权限指令。这直接提升了面对恶意工具指令时的鲁棒性。

为什么这会改善安全可控性

OpenAI 在系统提示中加入特定类别的安全规范，再用生产安全 benchmark 观察模型行为。IH 训练后的模型在违规类别上有更高的拒绝和安全完成率。

关键点是有用性没有同步下降。模型不是简单变得更保守，而是更会在冲突中执行上级安全规范。

最值得带走的洞见

指令层级是安全、可靠性和产品控制的共同基础。模型越像代理，越需要稳定判断信息来源的可信度。
好数据集不一定复杂。IH-Challenge 的价值在于任务简单、评分明确、难以靠拒绝取巧。
prompt injection 防御不能只靠过滤文本。模型本身必须学会把工具输出当作低权限信息。
安全训练的理想结果不是“更多拒绝”，而是在应该拒绝时拒绝，在安全请求中保持帮助能力。
这些结果支持一个更广的判断：未来 AI 安全需要训练模型理解权限边界，而不只是识别危险内容。
