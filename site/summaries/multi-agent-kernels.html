一句话总括

通过多智能体系统将 GPU kernels 加速 38%

Speeding up GPU kernels by 38% with a multi-agent system

Cursor 的实验把多智能体从常规编码推进到 GPU 底层性能工程，并证明它能在真实生产模型产生的 CUDA kernel 问题上自主搜索、测试和优化。

235个 CUDA kernel 优化问题，来自真实模型工作负载。
38%相对优化后 PyTorch 基线的几何平均加速。
63%149 个问题超过基线，说明收益不是个别样本。
19%45 个问题超过 2 倍加速，长尾任务里也有大收益。
为什么重要

这不是让 AI 补全代码，而是让它在硬件约束里找性能。

CUDA kernels 直接影响 AI 训练和推理的 GPU 利用率、延迟、能耗和每 token 成本。

传统优化通常拆成单个数学操作分别调优。这样可控，但容易错过跨系统的组合收益。

多智能体系统的价值在于扩大搜索空间。它可以把 benchmark 结果变成反馈，并持续重排任务优先级。

实验流程

整个系统靠一个规划智能体和多个工作智能体循环推进。

1

NVIDIA 用 SOL-ExecBench 生成 235 个问题，覆盖 LLM、扩散、视觉、音频、视频和多模态模型。

2

规划智能体按性能指标分配任务，并在运行中重新平衡工作。

3

工作智能体自主调用 benchmark，形成测试、调试和优化循环。

4

系统分别尝试 CUDA C 加 inline PTX，以及训练数据中较少见的 CuTe DSL。

三个代表案例

不同 kernel 触发了不同优化策略。

问题 系统做了什么 结果说明
BF16 Grouped Query Attention with Paged Prefill 使用特定硬件指令、persistent kernels 和针对输入尺寸的细调。 SOL score 达到 0.9722，相对 FlashInfer 人工优化基线几何平均加速 84%。替换 SGLang kernel 后，Llama 3.1 8B 的 TTFT 提升 3%。
NVFP4 MoE Linear with Gating 识别量化为瓶颈，把 scale 计算和 rounding 融合，并用预计算阈值桶把 FP32 映射到 16 个 NVFP4 值。 相对优化后 PyTorch 基线加速 39%，SOL score 为 0.58。它显示智能体能抓住低比特量化中的真实瓶颈。
BF16 Matrix Multiplication 从零生成专用 CUDA C++ GEMM kernel，并学习 Blackwell 指令、访存模式和特定形状优化。 达到 NVIDIA cuBLAS 人工优化基线的 86%。在小 M 测试中最高超过库实现 9%，这类形状对 LLM decode 很关键。
38% 的平均加速很强，但 median SOL score 只有 0.56。结论不是“已经替代专家”，而是“多智能体开始能进入专家级问题的搜索空间”。 这改变了读者对 AI 影响的理解：AI 的边界正在从代码生成，移动到可度量的工程优化。
最值得带走的洞见

这项实验最有价值的地方，是把多智能体能力放进了硬指标里。

真实负载比玩具任务更有说服力

问题来自 124 个以上生产级开源模型。结果更接近实际训练和推理瓶颈。

多智能体适合开放式优化

kernel 优化没有单一正确答案。系统能用 benchmark 反馈持续探索。

收益会影响成本结构

更快 kernels 意味着更低延迟、更高吞吐和更低每 token 成本。

专家经验仍然重要

median SOL score 说明系统离理论上限还有距离。多智能体目前更像放大搜索能力，而不是终结工程判断。

未来瓶颈可能是算力预算

实验只用 27 块 Blackwell 200 GPU。更多计算资源可能让系统探索更深的方案空间。
