多智能体系统把 GPU kernel 提速 38%

来源 Cursor ｜ 作者 Wilson Lin, Sahil Modi, Yuan Zhang & Edward Lin ｜ 2026-04-14

一句话总括：这篇文章真正的贡献，不是又快了 38% 这个数字，而是用一个无人知道标准答案的硬核任务（CUDA kernel 优化）当试金石——证明多智能体系统能在"训练分布之外、没有现成 diff 可抄"的开放式难题上，自主跑出过去需要资深工程师数月到数年才能达到的性能水平。

为什么是 kernel选一个"连出题人都不知道答案"的问题

评测长程多智能体系统，最好的题目是开放式优化问题：它有可量化的目标可以反复迭代，而不是去命中一个已知的标准修改。kernel 优化恰好满足——CUDA kernel 是 GPU 上训练和推理的核心软件，更快的 kernel 直接意味着更高利用率、更低能耗与时延、更低的每 token 成本。

今天工程师的做法是把模型拆成一个个数学算子逐个调优，这让问题可控，却把"跨整个系统联合优化"的收益留在了桌上。这次实验想看的，正是智能体能否跳出这种人工简化，去探索更大的解空间。

结论：用开放式、可度量的难题来考多智能体，比让它复现一个已知答案更能暴露真实能力。

怎么跑的235 道真实题，全靠一个 markdown 协议

NVIDIA 用 SOL-ExecBench 从 124+ 个开源生产模型（DeepSeek、Qwen、Gemma、Kimi、Stable Diffusion 等）生成 235 道题——全是 LLM、扩散、视觉、音频、视频、多模态等真实训练/推理负载的约束，不是玩具 kernel。评测在 27 张 B200 上进行，并会对比理论硬件极限；若智能体靠缓存等作弊跑出超过 B200 物理上限的成绩，结果会被判无效。

整套系统在一次运行里解完全部 235 题：一个 planner 智能体按性能指标把任务分发并动态再平衡给自主 worker。全部协调协议只写在一个 markdown 文件里（规定输出格式、规则和测试），系统自己学会调用基准管线，形成"测试→调试→优化"的闭环，全程无人干预。它还分两次、用 GPU 抽象谱系两端的语言各跑一遍：

CUDA C + PTX
直达寄存器与 ISA 级指令，考"能否在最底层推理硬件"
CuTe DSL
公开训练数据极少的高层抽象，考"能否纯靠文档学新 API"

结论：把复杂协调压进一个 markdown 文件就能驱动，说明能力瓶颈正在从"框架"转向"模型自身的自主性"。

结果63% 跑赢基线，近五分之一翻倍

性能用两种口径报告：相对"单智能体优化过的 PyTorch 代码"的几何平均加速比；以及 SOL 分数（0.5 = 优化后的 PyTorch 基线，1.0 = 理论硬件极限，对数曲线）。

指标数值
跑赢基线的题目149 / 235（63%）
几何平均加速1.38×（+38%）
提速超过 2× 的题目45 / 235（19%）
中位 SOL 分数0.56

结论：38% 是几何平均，背后是近两成题目直接翻倍——长尾上的大幅提速正是过去人工方法难以覆盖的部分。

三个案例同一系统，长出三套不同的优化策略

BF16 分组查询注意力（分页 prefill）：从 SGLang 为 Llama 3.1 8B 的推理中抽取。智能体用 CUDA C++ 动用特定访存/计算指令、用 persistent kernel 改进调度并针对输入尺寸超调，做出 SOL 0.9722、比 FlashInfer 人工基线快 84% 的方案；替换进 SGLang 后，TTFT 端到端提速 3%（该算子占 prefill 的 2–5%，算非平凡收益）。

NVFP4 MoE 线性层 + 门控：MoE 模型（如 Qwen3）常见的两 kernel 模式，输入与中间结果量化到 4-bit 浮点。智能体准确识别量化是主瓶颈，把缩放与舍入融合——利用 NVFP4 只有 16 个可能值，用预算好的阈值桶把 FP32 直接映射成 FP4 码，最终比 PyTorch 基线快 39%，SOL 0.58。

BF16 矩阵乘（GEMM）：历来是 kernel 专家的禁脔（需要内联 PTX、流水线、kernel 内分级）。系统从零写出专用 CUDA C++ GEMM，自学 Blackwell 专属指令并按精确形状超调，达到 cuBLAS 人工基线的 86%；在对解码尤为关键的 small-M 情形，甚至最高反超库 9%。

结论：面对量化、注意力、矩阵乘三类截然不同的瓶颈，系统不是套模板，而是各自"想"出对路的策略——这正是泛化能力的证据。

边界中位 0.56，且只跑在 27 张卡上

作者坦承：38% 之外，中位 SOL 仅 0.56，仍有巨大优化空间；而且数百道题、众多智能体却只共享 27 张 GPU，严重限制了系统的探索深度。他们认为给更多算力，多智能体方案还能大幅提升，去挖更深、更新颖的解。

更大的论点是：最有野心的软件任务都是开放式、没有明确解的；单智能体在这里吃力，因为模型最擅长训练里见过的窄任务。这次实验被视为又一次佐证——多智能体架构会很快成为构建软件的默认方式，因为它能啃下落在训练分布之外的新问题。

结论：38% 是"算力受限下的下限"而非上限；真正值得关注的是趋势——系统在最难的 GEMM 上已能局部超越专家。

最值得带走的洞见

  1. 它在量、而不只是在说。235 道真实题、27 张 B200、防作弊的 SOL 评测，把"多智能体很强"变成了可复核的数字：63% 跑赢、38% 几何提速、19% 翻倍。
  2. 价值在长尾，不在峰值。真正难得的是系统覆盖了一大批"人工优化不划算"的 kernel——这正是过去被留在桌上的性能。
  3. 泛化的证据是策略多样性。对量化、注意力、矩阵乘各自长出不同打法，且能自学 CuTe DSL 这种几乎没进过训练数据的 API，说明它不是在背答案。
  4. 端到端收益才算数。注意力 kernel 提速 84% 听着夸张，但替换进 SGLang 后 TTFT 只快 3%——文章诚实地把"算子级"和"系统级"分开，这种克制值得借鉴。
  5. 瓶颈是算力，不是方法。中位 SOL 0.56、仅 27 张卡，意味着这是"受限下的成绩"；它给出的不是终点，而是"多智能体即将在最硬的问题上超越专家"的方向标。

注：文中数据均来自 Cursor 官方博客（与 NVIDIA 合作）。SOL 分数为对数曲线，0.5 对应优化后 PyTorch 基线、1.0 对应理论硬件极限；加速比均以"单智能体优化的 PyTorch 代码"为基线。
