用多智能体系统将 GPU 内核提速 38%

Speeding up GPU kernels by 38% with a multi-agent system

NVIDIA Blackwell 200 · CUDA 内核优化 · 长时间自主智能体

一句话总括：这项工作的真正贡献，是让多智能体在数百个真实 GPU 内核问题上自主执行“测试—调试—优化”循环，并在三周内取得可测量的整体性能收益。

235个真实内核优化问题
38%整体几何平均提速
63%问题超过基线
19%问题提速超过 2 倍

评测对象不是玩具内核

NVIDIA 通过 SOL-ExecBench，从 124 个以上的生产级开源模型中生成 235 个问题。模型覆盖 LLM、扩散、视觉、音频、视频和多模态架构。

评测运行在 27 块 NVIDIA Blackwell 200 GPU 上。流水线会同时检查现有软件基线和硬件理论极限，也会拒绝缓存等作弊方案。

洞见：这些结果比合成 benchmark 更接近真实训练和推理工作负载。

38% 并不等于问题已经解决

系统的 SOL 中位数只有 0.56。0.5 代表优化后的 PyTorch 基线，1.0 才代表理论硬件极限。

作者认为，数百个问题和智能体共用 27 块 GPU，限制了搜索深度。但文章没有通过更多算力实验验证这一推测。

洞见：多智能体已能扩大优化覆盖面，但距离普遍逼近硬件极限仍然很远。

01 · 规划规划智能体分配任务。
02 · 测试工作智能体调用评测流水线。
03 · 调试根据错误和性能指标修改内核。
04 · 再平衡按进展重新分配计算资源。

同一套系统找到了三类不同策略

分页预填充注意力

系统使用底层加载和计算指令，引入持久化内核调度，并针对输入尺寸深度优化。

结果达到 0.9722 SOL，比 FlashInfer 人工基线几何平均快 84%。替换 SGLang 内核后，Llama 3.1 8B 的 TTFT 提升 3%。

结论：单个热点只占预填充过程的 2%–5%，仍能带来可见的端到端收益。

NVFP4 MoE Linear

系统判断量化是主要瓶颈，于是融合缩放计算和舍入。它利用 NVFP4 只有 16 个可能值的特点，通过预计算阈值桶直接映射 FP32。

最终取得 39% 几何平均提速，SOL 为 0.58。

结论：智能体不只调整参数，也能重写关键数据路径。

BF16 矩阵乘法

系统从零生成专用 CUDA C++ GEMM，并自行学习 Blackwell 指令、内存访问和流水线安排。

整体达到人工调优 cuBLAS 的 86%。在 small-M 测试中，它最多领先 9%。

结论：针对特定形状的专用内核，可能在局部场景超过成熟通用库。

真正改变认知的不是某一个 2 倍提速案例，而是系统能同时处理 235 个彼此不同的问题，并为不同瓶颈形成不同方案。这显示 AI 的价值正在从“写出已知代码”转向“搜索尚无标准答案的实现空间”。

我们的多智能体系统为 NVIDIA Blackwell 200 GPU 自主优化了 235 个 CUDA 内核。它只用了三周，就比基线取得了 38% 的几何平均提速。

过去几个月，我们一直在开发一个多智能体系统。它能够自主构建、维护和部署复杂软件。作为这项工作的一部分，我们一直在不同领域测试该系统。其中包括让它从零构建浏览器，以及在 First Proof benchmark 上解决一道研究级数学问题。

最近，我们开始与 NVIDIA 合作，应对一项新挑战：使用多智能体框架优化 CUDA 内核。这类技术问题难度很高，也会产生重要的现实影响。CUDA 内核是支撑 NVIDIA GPU 上 AI 模型训练和推理的核心软件。更快的内核意味着更高的 GPU 利用率、更低的能耗、更短的延迟，以及更低的单 token 成本。服务提供商因此可以同时向更多用户提供规模更大、能力更强的模型。

我们的多智能体框架在三周内自主处理了 235 个问题。系统从零构建并优化 Blackwell GPU 内核，一直深入到汇编层级，最终取得 38% 的几何平均提速。

这一级别的性能提升，通常需要经验丰富的内核工程师投入数月甚至数年。多智能体系统在几周内完成了这项工作。它还处理了大量过去用现有方式并不值得投入资源的长尾内核问题。

以内核优化检验智能体系统能力

评估长时间运行的多智能体系统时，一种很有效的方法是向它们提供开放式优化问题。这类问题连我们自己也不知道正确答案。内核优化符合这一条件。它提供可测量的目标，系统可以围绕目标反复优化，而不是仅仅追求一个已知且简单的代码差异。

如今，工程师会把模型拆成单独的数学运算，再分别调整每项运算，以此优化内核。这种方法降低了问题的复杂度，但也会损失潜在性能。分块优化无法发现同时优化整个系统可能获得的收益。到目前为止，我们之所以无法越过这些人工简化并探索完整解空间，也限制了 GPU 性能的进一步提升。

通过这项实验，我们想知道多智能体系统能否跳出这些限制。我们希望它探索更广阔的解空间，并生成更快的内核。

使用 SOL-ExecBench 生成问题并进行基准测试

NVIDIA 使用 SOL-ExecBench，从 124 个以上的生产级开源模型中生成了 235 个优化问题。这些模型包括 Deepseek、Qwen、Gemma、Kimi 和 Stable Diffusion。它们并非合成数据或玩具内核。每个问题都来自真实训练或推理工作负载中的约束，覆盖 LLM、扩散、视觉、音频、视频以及多模态混合架构。

我们还使用 SOL-ExecBench，在 27 块 NVIDIA Blackwell 200 GPU 上评测多智能体生成的内核方案。SOL-ExecBench 是一种有效的评估工具。它会把内核性能与现有软件基线及硬件理论性能极限进行比较。如果智能体采用缓存等作弊手段，提交了超出 B200 能力范围的性能结果，流水线就会判定结果无效。

实验的运行方式

多智能体系统通过一次运行解决了全部 235 个 GPU 内核优化问题。系统部署了一个规划智能体。它根据性能指标，在自主工作的智能体之间分配任务，并持续重新平衡工作量。

整个协调协议都写在一个 Markdown 文件里。文件规定了输出格式、规则和测试。运行过程中，多智能体系统自行学会了调用基准测试流水线。由此形成一个循环，系统可以持续测试、调试和优化内核，全程不需要开发者干预。

为了更准确地评估多智能体系统的能力，我们让它分别用两种语言编写方案，并进行了两轮独立运行。这两种语言位于 GPU 抽象层次的两端：

  • 带 inline PTX 的 CUDA C：它让智能体能够直接访问寄存器和 ISA 层级指令，用于检验系统能否在最底层理解硬件。

  • CuTe DSL：它提供高层、可组合的抽象，但在公开训练数据中很少出现，用于检验系统能否只依靠所提供的文档学习新 API。

整体提速 38%，其中 19% 的优化超过 2 倍

我们通过两种方式报告多智能体系统的性能：

  • 与经过单个智能体优化的 PyTorch 代码基线相比，计算几何平均加速比。

  • 使用 Speed-of-Light（SOL）分数，表示一个方案在对数曲线上相对于硬件理论极限的表现。0.5 分代表优化后的 PyTorch 基线，1.0 分代表性能极限。

多智能体系统在 235 个问题中的 149 个问题上成功超过基线，占 63%。其几何平均比率为 1.38 倍，也就是几何平均提速 38%。

在 235 个问题中的 45 个问题上，多智能体系统取得了超过基线 2 倍的优化，占 19%。你可以在公开代码仓库中查看该系统最终生成的方案。

针对不同问题采用不同优化策略

为了展示系统适应不同问题类型的能力，我们选出了三个问题。在这些问题中，系统自然形成了各不相同的优化策略。

采用分页预填充的 BF16 分组查询注意力

采用分页预填充的分组查询注意力，是现代 LLM 推理栈中常见的提示词阶段操作。经过良好优化的实现，可以在相同 GPU 显存条件下支持更长的上下文、更高的并发量和更好的吞吐量。

智能体使用 CUDA C++ 优化了这个注意力问题。该问题提取自 Llama 3.1 8B 的 SGLang 推理过程。在反复调整内核时，智能体成功采用了用于内存加载和数学运算的特定硬件指令。它还通过持久化内核改进调度，并针对输入尺寸进行深度优化。

我们把多智能体系统生成的自定义内核，与 FlashInfer 库中由人工优化的基线进行了比较。结果显示，该方案的 SOL 分数达到 0.9722，已经接近硬件极限。与基线相比，其几何平均提速为 84%。

我们还在 SGLang 中替换了现有内核，并观察到 Llama 3.1 8B 的首个 token 时间（TTFT）缩短了 3%。根据服务配置不同，这个注意力问题占预填充过程的 2%–5%。因此，我们认为这是一项不可忽视的端到端性能提升。

带门控的 NVFP4 MoE Linear

这个问题代表了 Qwen3 等混合专家模型中常见的双内核模式。不同之处在于，输入张量和中间乘法输出会被量化为 NVFP4，也就是 4 位浮点格式。

智能体正确判断量化环节是主要瓶颈，并据此融合了缩放计算和舍入操作。通常的量化会先缩放，再执行舍入。它改为使用预先计算的阈值桶，直接把 FP32 值映射到 FP4 编码。由于 NVFP4 只有 16 个可能值，这种做法是可行的。随后，它把这些优化应用到规模更大的测试用例。

智能体最终超过了优化后的 PyTorch 基线，取得 39% 的几何平均提速，SOL 分数为 0.58。

BF16 矩阵乘法

矩阵乘法一直是非常难以优化的问题。优化者必须深入理解不同硬件单元及其调度方式。要让矩阵乘法内核（GEMM）充分发挥性能，通常需要 inline PTX、流水线技术，以及内核内部的分阶段处理。因此，过去只有经验极其丰富的内核专家才能编写高速 GEMM。

Cursor 多智能体系统从零生成了一个专用 CUDA C++ GEMM 内核。它达到了 NVIDIA cuBLAS 库中精细人工调优基线的 86%，差距已经很小。为了取得这一结果，系统自行学会使用 Blackwell 专用指令。它还针对硬件优化内存读写，并围绕具体形状进行深度优化。

在对 LLM 推理解码尤其重要的 small-M 测试中，多智能体系统生成的内核最多比该库快 9%。这一结果表明，即使面对最困难的内核问题，多智能体系统也可能很快在部分场景中超过领域专家。

用于构建软件的多智能体系统

多智能体框架虽然比基线取得了 38% 的几何平均提速，但 SOL 中位数仍然只有 0.56。这说明仍有很大的进一步优化空间。我们认为，只要投入更多计算资源，多智能体方案就能得到显著改善。我们当时有数百个问题和智能体，却只有 27 块 GPU。这限制了我们充分发挥多智能体系统的能力。如果拥有更多 GPU，系统就能探索更深层、更新颖的方案。

软件开发中最具挑战性的任务通常是开放式的，没有明确解法。单智能体系统很难处理这类任务，因为模型最擅长的是训练期间已经见过、范围较窄的问题。我们认为，这次内核优化实验进一步证明，多智能体架构将很快成为构建软件的默认方式。它们能够解决远远超出训练数据分布的新问题。

我们在这里研究的技术很快会用于 Cursor 的核心产品。如果你有兴趣研究多智能体协调中的难题，Cursor 团队欢迎你通过 hiring@cursor.com 与我们联系。

多智能体系统在 235 个问题中的 149 个问题上成功超过基线，占 63%。其几何平均比率为 1.38 倍，也就是几何平均提速 38%。

在 235 个问题中的 45 个问题上，多智能体系统取得了超过基线 2 倍的优化，占 19%。你可以在公开代码仓库中查看该系统最终生成的方案。

针对不同问题采用不同优化策略

为了展示系统适应不同问题类型的能力，我们选出了三个问题。在这些问题中，系统自然形成了各不相同的优化策略。

采用分页预填充的 BF16 分组查询注意力

采用分页预填充的分组查询注意力，是现代 LLM 推理栈中常见的提示词阶段操作。经过良好优化的实现，可以在相同 GPU 显存条件下支持更长的上下文、更高的并发量和更好的吞吐量。

智能体使用 CUDA C++ 优化了这个注意力问题。该问题提取自 Llama 3.1 8B 的 SGLang 推理过程。在反复调整内核时，智能体成功采用了用于内存加载和数学运算的特定硬件指令。它还通过持久化内核改进调度，并针对输入尺寸进行深度优化。

我们把多智能体系统生成的自定义内核，与 FlashInfer 库中由人工优化的基线进行了比较。结果显示，该方案的 SOL 分数达到 0.9722，已经接近硬件极限。与基线相比，其几何平均提速为 84%。

我们还在 SGLang 中替换了现有内核，并观察到 Llama 3.1 8B 的首个 token 时间（TTFT）缩短了 3%。根据服务配置不同，这个注意力问题占预填充过程的 2%–5%。因此，我们认为这是一项不可忽视的端到端性能提升。

带门控的 NVFP4 MoE Linear

这个问题代表了 Qwen3 等混合专家模型中常见的双内核模式。不同之处在于，输入张量和中间乘法输出会被量化为 NVFP4，也就是 4 位浮点格式。

智能体正确判断量化环节是主要瓶颈，并据此融合了缩放计算和舍入操作。通常的量化会先缩放，再执行舍入。它改为使用预先计算的阈值桶，直接把 FP32 值映射到 FP4 编码。由于 NVFP4 只有 16 个可能值，这种做法是可行的。随后，它把这些优化应用到规模更大的测试用例。

智能体最终超过了优化后的 PyTorch 基线，取得 39% 的几何平均提速，SOL 分数为 0.58。

BF16 矩阵乘法

矩阵乘法一直是非常难以优化的问题。优化者必须深入理解不同硬件单元及其调度方式。要让矩阵乘法内核（GEMM）充分发挥性能，通常需要 inline PTX、流水线技术，以及内核内部的分阶段处理。因此，过去只有经验极其丰富的内核专家才能编写高速 GEMM。

Cursor 多智能体系统从零生成了一个专用 CUDA C++ GEMM 内核。它达到了 NVIDIA cuBLAS 库中精细人工调优基线的 86%，差距已经很小。为了取得这一结果，系统自行学会使用 Blackwell 专用指令。它还针对硬件优化内存读写，并围绕具体形状进行深度优化。

在对 LLM 推理解码尤其重要的 small-M 测试中，多智能体系统生成的内核最多比该库快 9%。这一结果表明，即使面对最困难的内核问题，多智能体系统也可能很快在部分场景中超过领域专家。

用于构建软件的多智能体系统

多智能体框架虽然比基线取得了 38% 的几何平均提速，但 SOL 中位数仍然只有 0.56。这说明仍有很大的进一步优化空间。我们认为，只要投入更多计算资源，多智能体方案就能得到显著改善。我们当时有数百个问题和智能体，却只有 27 块 GPU。这限制了我们充分发挥多智能体系统的能力。如果拥有更多 GPU，系统就能探索更深层、更新颖的方案。

软件开发中最具挑战性的任务通常是开放式的，没有明确解法。单智能体系统很难处理这类任务，因为模型最擅长的是训练期间已经见过、范围较窄的问题。我们认为，这次内核优化实验进一步证明，多智能体架构将很快成为构建软件的默认方式。它们能够解决远远超出训练数据分布的新问题。

我们在这里研究的技术很快会用于 Cursor 的核心产品。如果你有兴趣研究多智能体协调中的难题，Cursor 团队欢迎你通过 hiring@cursor.com 与我们联系。
