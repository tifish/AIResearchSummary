迈向自动驾驶代码库

作者 Wilson Lin · 原文：cursor.com/blog/self-driving-codebases
一句话总括：Cursor 不是发布了一个新模型，而是回答了一个工程问题——当你给一个编码智能体"多几个数量级的时间和算力"时，该如何组织它？答案是一套让数千个智能体协同自治的编排框架，它在一周内几乎无人干预地写出了一个可运行的浏览器引擎。这把"AI 写代码"从"补全一段函数"推进到"持续拥有并演进整个代码库"。

单个智能体撞墙，不是因为不会写代码，而是任务太大

关键数据 / 趋势

用 Opus 4.5 规划"从零写浏览器"很快失败：模型会跟丢进度、过早宣布成功、卡在实现细节——但它能把小块代码写好。问题出在任务规模，而非智能体智力。后来转向 GPT-5.1/5.2，因为它们更精确地遵循指令，更适合长程任务。

洞见：长程自治的瓶颈是"任务分解与协调"，不是单点代码能力。模型已经足够聪明，缺的是组织方式。

对等"自协调"彻底失败，加锁让 20 个智能体退化成 1-3 个

关键数据 / 趋势

第一版让平权智能体共享一个状态文件、自己抢锁协调。结果锁被长期占用、忘记释放、非法加解锁；竞争之激烈使 20 个智能体的实际吞吐跌到只剩 1-3 个。无结构还导致没人愿意扛复杂大任务，大家都挑安全的小改动。

洞见：把人类难写对的并发原语（锁）直接丢给智能体不可行；缺乏角色与归属感，群体会集体回避责任。

真正起作用的是角色分工与单向归属

架构演进的终点

root planner（只规划、不写码，拥有全局目标）→ 可递归派生 subplanner（全权负责一个切片）→ worker（在自己的仓库副本里独立干完一件事，完成后只回传一份 handoff）

worker 之间互不通信、不知道彼此存在；信息只沿链条单向上报给请求它的 planner。handoff 不只写"做了什么"，还包含顾虑、偏差、发现与反馈，让 planner 即便"做完了"也能持续收到更新、拉取最新仓库、继续决策。系统由此自收敛，无需全局同步。途中刻意移除了 judge 和 integrator——后者作为唯一合并入口成了几百个 worker 排队的"红头文件"瓶颈。

洞见：这套结构竟然自然长成了人类软件团队的样子（owner→分工→交接），而模型并未被专门如此训练，暗示这可能是组织软件工作的"涌现正解"。

反直觉的核心权衡：为吞吐量主动接受错误率

关键数据 / 趋势

要求"每次提交 100% 正确"会让系统严重串行化：一个小 typo 就让全体停摆，worker 越界乱修、互相踩踏。

放松后，智能体相信"别的错很快会被同伴修掉"（因为系统对整个代码库有有效归属），错误产生即被快速修复，错误率小而稳定、不爆炸也不恶化。代价是多花些 token 和局部湍流，但换来全局简单与高产。结论：理想系统应接受一个稳定错误率，再靠一条定期快照的"绿色分支"在发布前统一修复。

洞见：在多智能体规模下，"完美正确"是吞吐量的敌人；可靠性应在系统层（最终对账）而非每次提交处保证。

意外的瓶颈是磁盘，不是模型

关键数据 / 趋势

限制内存后，磁盘成了热点：单体项目里数百个智能体同时编译，产生 每秒数 GB 的构建产物读写，显著拖累整体吞吐。这揭示一个新视角——项目结构、架构与开发体验会直接决定 token 和提交吞吐，因为"和代码库打交道（编译）"的时间反而盖过了"思考和写代码"。后来一次把单体拆成多个自包含 crate 的重构，使编译等待大减，吞吐翻了数倍。

洞见：为人类单用户优化的工具（Git/Cargo 的共享锁、无去重的副本）在多智能体下成了短板；引入数据库式并发控制、写时复制与去重，可能带来巨大且廉价的效率提升。

规模放大一切，指令规格成了最大变量

提示工程的实战结论

  • 给数字："生成 20-100 个任务" 远胜 "生成很多任务"——后者会因保守而只产出寥寥几个。
  • 约束优于指令："不许留 TODO、不许半成品" 比 "记得写完" 更有效；模型默认会做好事，约束是在划边界。
  • 别教它已会的：把模型当"懂工程但不懂你代码库的聪明新人"，只讲它不知道的（多智能体协作、你的测试/部署流程）。
  • 避免打勾心态：列具体清单会让模型只盯清单、隐性忽略未列项，高层任务要传达意图而非任务表。

洞见：智能体会忠实地把指令执行到底——好坏皆然。坏输出常常不是 harness 的错，而是规格没写清。规模越大，"表达与理解意图（steerability）"越是核心研究方向。

最值得带走的洞见

  1. "自动驾驶代码库"已被验证可行：数千智能体在一周内以约 1,000 commits/小时、10M 次工具调用，无人干预地产出几乎可运行的浏览器引擎。
  2. 组织 > 智力：长程自治的胜负手是角色分工与单向归属（root planner→subplanner→worker + handoff 上报），而非单点模型能力；瓶颈型的中央角色（judge、integrator）该删则删。
  3. 用错误率换吞吐：放弃"每提交都完美"，接受小而稳定的错误，再用最终"绿色分支"统一修复——这是高吞吐的关键反直觉权衡。
  4. 基础设施决定上限：磁盘 I/O 与为单用户设计的工具锁成了新瓶颈，提示项目架构和开发工具需要为"多智能体"重新设计（写时复制、去重、数据库式并发）。
  5. 意图即代码：在巨大算力放大下，模糊或糟糕的指令会被等比放大成糟糕的产出；规格化、约束化、给定量范围，是驾驭这类系统的真正杠杆。
