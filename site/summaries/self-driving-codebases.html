迈向自动驾驶代码库

Towards self-driving codebases

一句话总括：Cursor 的贡献不是证明“很多智能体一起写代码”很热闹，而是给出了一套能持续运行、能收敛、能解释失败原因的多智能体代码库架构。

1 周
系统稳定连续运行，并承担研究浏览器项目的大多数提交。
~1,000
峰值每小时提交数。吞吐量成为显式设计目标。
10M
一周内累计工具调用量。小低效会被规模放大。
20 → 1-3
共享锁让 20 个智能体退化到约 1-3 个智能体的吞吐。

真正的问题是协调，而不是会不会写代码

单个模型能写好小块代码，但浏览器这种任务会让它丢失全局状态、提前宣布成功，并卡在复杂细节里。

文章从一个浏览器项目开始。浏览器足够复杂，渲染、架构、性能和工具链会互相牵制。

直接让一个模型“继续做下去”很快失败。把任务拆成并行子任务后，吞吐量提高了，但没有解决沟通、反馈和全局责任问题。

这改变了对 AI 编程能力的理解。瓶颈不是单个补丁的能力，而是系统能否让很多补丁朝同一个方向收敛。

系统设计一路从自由协作走向递归所有权

平等智能体加共享文件。 方案看似简单，但锁竞争、忘记释放锁和非法更新让系统快速失控。
planner、executor、judge 分工。 角色分离带来责任感，worker 能专注执行，系统开始有线性吞吐。
连续 executor。 去掉独立 planner 后更灵活，但一个智能体同时负责规划、派工、审查、合并和判断完成，最终被角色压力压垮。
最终架构。 root planner 拥有全局目标，subplanner 递归拥有窄切片，worker 在独立仓库副本中完成任务，并用单次 handoff 回传发现。

吞吐量来自接受不完美，而不是强行消灭所有错误

设计选择 直接后果 改变了什么理解
每次提交都要求 100% 正确 一个 API 变化或拼写错误就会让系统串行化。很多 worker 会越界去修同一个问题。 大规模自治编码不适合把每个中间状态都当成发布状态。
允许小而稳定的错误率 错误会出现，但其他智能体会继续修复。系统保持 steady state，而不是停摆。 高效系统更像持续收敛的流水线，需要最后的 green branch 或发布前修复通道。
接受局部文件冲突 会消耗额外 token，也会有短期波动。 过度同步会比局部混乱更贵。简单可观测的系统更容易扩展。
架构和指令会被规模放大。模糊目标不会被更多智能体自动修正，只会更快地扩散成错误方向。 这也是文章最实用的工程结论：提示词不是装饰，而是系统规格的一部分。

基础设施也会变成 AI 吞吐瓶颈

磁盘 I/O 成为热点。 限制 RAM 后，数百个智能体同时编译会产生每秒数 GB 的构建读写。代码库结构会直接影响 agent 的产出速度。
单机更易观察。 每次多智能体运行都放在一台大型机器上，避免过早引入分布式复杂性。
开发工具假设被挑战。 Git 和 Cargo 的共享锁对单用户合理，但在数百个 agent 同机运行时会放大成瓶颈。
仓库架构影响 token。 从单体迁到多个自包含 crate 后，系统减少等待编译的时间，吞吐量提升到原来的数倍。
潜在机会在底层原语。 copy-on-write、去重和数据库式并发控制可能让传统开发环境更适合多智能体工作负载。

给智能体的指令要像工程规格

模型会严格沿着指令走。坏规格不会因为算力增加而变好，只会更快地产生坏结果。

“实现规范”这种指令太宽泛，智能体会深入冷门特性，而不是按用户价值排序。

性能、超时、依赖哲学、禁止使用的库和资源回收策略，都需要明确写出来。否则系统会把沉默当成许可。

文章还给出提示经验：不要重复教模型会做的通用工程动作。重点说明协作规则、项目流程和领域边界。

约束比提醒更有效。“不要 TODO、不要半成品”比“记得完成实现”更能定义边界。

最值得带走的洞见

多智能体系统的核心是责任边界。递归 planner 和独立 worker 比平等协商更稳。
吞吐量需要被明确优化。要求每个中间提交完美，会把系统拖回串行模式。
可观测性是研究和调参的前提。日志、时间戳和回放能力让团队能发现重复失败模式。
提示词是规格。模糊目标、隐含性能要求和依赖边界都会在规模下变成真实工程问题。
开发工具链需要重新面向并发。仓库结构、构建缓存、文件锁和磁盘 I/O 会决定 AI 编程系统的上限。
