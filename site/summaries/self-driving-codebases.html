一句话总括

自驱代码库的关键不是更多 agent，而是让 agent 形成可观察、可委托、可收敛的工程组织。

Cursor 这篇研究记录了一套多智能体编码 harness 的演化：从失败的共享协调，到分层规划、独立执行、结构化交接，再到为了吞吐主动接受短期错误。

单个聪明 agent 无法自然承担整个复杂代码库

研究从“让模型持续写一个浏览器”开始。模型能写出小块好代码，但面对浏览器这种多子系统工程时，会丢失上下文、过早宣布成功，并卡在复杂实现细节上。

这改变了问题定义：瓶颈不是模型有没有工程知识，而是大任务如何被拆分、归属、观察和重新汇总。

结论是，长期自治编码首先是系统设计问题，其次才是单次补全质量问题。

平等 agent 的自协调很快被锁和责任真空拖垮

最初方案让多个平等 agent 通过共享状态文件协调工作，但锁被长时间持有、忘记释放、非法加锁或解锁，提示词也没有真正修好这些行为。

更关键的数据是，20 个 agent 会因为锁争用下降到约 1-3 个 agent 的吞吐；同时，没人愿意承担大而复杂的整体任务，agent 倾向于选择小而安全的改动。

这说明多智能体系统不能只靠“大家自己商量”，必须明确所有权、职责边界和信息流。

最终有效形态接近递归的软件团队结构

最终设计由根规划器拥有全局目标，子规划器递归拥有局部范围，worker 在自己的仓库副本中完成具体任务，并通过一次 handoff 把结果、偏差、担忧和发现交回上级。

worker 不互相通信，也不了解全局；规划器不亲自编码，只负责理解状态、继续拆分任务和吸收反馈。这样信息沿责任链向上汇聚，避免全局同步和无意义串话。

洞见是，大规模 AI 编码更像组织设计：好的结构让局部行动和全局目标同时存在。

吞吐来自接受短期不完美，而不是每一步都追求干净

系统在一周运行中跨 1000 万次 tool calls，峰值达到约每小时 1000 个 commits。为了达到这个速度，Cursor 放弃了“每个提交必须 100% 正确”的要求。

如果每个提交都必须完全绿色，一个小 API 变化或拼写错误就会让系统串行化，许多 agent 会越界修同一个问题并相互踩踏。允许小而稳定的错误率后，错误会出现也会很快被其他 agent 修掉。

真正的工程取舍是：高吞吐主线可以带一点噪声，但发布前需要一个专门的绿色分支或快照修复流程。

代码库结构会直接影响 token 和 commit 吞吐

系统多数运行在单台大型 Linux VM 上，避免过早引入分布式复杂度。限制 agent 内存后，磁盘成为热点；单体仓库中数百个 agent 同时编译，会产生每秒数 GB 的构建产物读写。

后来将浏览器项目重构为多个自包含 crate，即使仓库一度严重损坏，多智能体系统仍在几天内收敛回可工作状态，并且等待编译的时间明显减少、吞吐提升数倍。

这说明 AI 时代的开发体验不只是人类体验，构建系统、锁、缓存、去重和模块边界都会影响机器团队的生产率。

提示词要表达意图和边界，而不是替模型写流程清单

研究发现，初始指令会被规模放大：含糊的“实现规范”会让 agent 深入冷门特性，隐含的性能要求不会自动出现，依赖哲学不明确就会引入不该引入的库。

更有效的提示方式是告诉模型它不知道的上下文，例如测试、部署、依赖边界和协作机制；约束通常比提醒更有力，比如“不留 TODO、不做半成品”优于“记得完成实现”。

同时，数字范围能改变行为：“生成很多任务”往往很保守，而“生成 20-100 个任务”会明确传达期望规模。

最值得带走的洞见

  • 自驱代码库的核心能力不是单个 agent 更强，而是任务所有权、反馈路径和失败恢复机制更清楚。
  • 全局共享锁和全员互相协调会快速放大混乱，分层委托和结构化 handoff 更容易扩展。
  • 高吞吐系统需要接受短期错误和局部冲突，再用专门收敛流程保证发布质量。
  • 代码库架构、构建速度、存储锁和缓存策略会成为 AI 编码系统的实际瓶颈。
  • 提示词的真正作用是指定意图、边界和本地流程；过细的清单反而可能缩窄 agent 的判断空间。
