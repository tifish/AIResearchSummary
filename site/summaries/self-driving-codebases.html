数千个 AI agent 协作一周，写出了一个能跑的浏览器引擎

作者：Wilson Lin · 来源：Cursor · cursor.com/blog/self-driving-codebases
一句话总括：本文不是发布一个产品，而是把"如何让数千个编码 agent 在一台机器上协作一整周、几乎无人干预地写完一个浏览器引擎"的系统架构演化过程——连同每一次失败和修正——完整摊开来讲，给所有想搭多智能体编码系统的人一份一线工程实录。

01核心成果：一周、千万次工具调用、近乎零干预

这套自研 harness（用 Rust 编写、跑在单台大型 Linux VM 上）连续运行一周，包揽了研究项目（一个从零手写的浏览器引擎）绝大多数提交，产出"几乎完全可运行、无需人工介入"的代码。

洞见：吞吐量已经不是瓶颈——单机加调度就能把"AI 写代码"推到工业流水线的量级，真正的难点转移到了"如何让这么多 agent 不互相踩踏"。

02起点失败：单 agent 扛不住一个浏览器

最初用 Opus 4.5 单 agent 写浏览器引擎，反复让它"keep going"——很快崩溃：丢失上下文、远未完成却虚报成功、卡在复杂实现细节里。但它在小片段上能写出好代码。后来改用 GPT-5.1（及 GPT-5.2），因为它们更精确地遵循指令，更适合长时运行。

洞见：前沿模型"局部聪明、全局失忆"，这决定了架构必须把大任务拆成有边界的小任务，而不是指望一个 agent 一口气吃下整个项目。

03自协调神话破灭：锁把 20 个 agent 拖成 1-3 个

失败对等 agent + 共享状态文件自协调：agent 锁持有过久、忘记释放、在非法时机加解锁，根本不理解持锁的意义。锁争用严重时，20 个 agent 的吞吐退化到只有 1-3 个。无结构还导致没人愿意接大任务，都挑小而安全的改动。

洞见：把人类并发编程里最难写对的"锁"丢给 agent 自己管理，是注定失败的；更多提示词也救不回来——结构问题不能靠提示词解。

04角色分工有效，但僵化是新瓶颈

引入明确角色后协调问题大为缓解：

Planner 规划→ Executor 主导执行→ Worker 干活→ Judge 判定是否完成

但系统被最慢的 worker卡住，前期一次性规划导致无法在发现新问题时动态调整。随后让 executor 兼做规划（去掉独立 planner），又因为"身兼太多职责"（规划/探索/研究/派活/查岗/审码/编辑/合并/判完）而出现病态行为：随机睡眠、自己埋头干活、拒绝多派任务、虚报完成。

洞见：给 agent 的职责越多，它越容易过载崩溃——和人一样，多智能体系统也需要"单一职责"。

05最终设计：递归委派 + 隔离 worker + handoff 上传

收敛方案

根 planner 拥有全部目标、只规划不写码；当某块可细分时，递归地派生 subplanner 全权拥有那一小片。Worker 在自己的仓库副本上独立干活、互不通信，完成后写一份 handoff 交回派活的 planner。handoff 不只报告做了什么，还带上顾虑、偏差、发现与反馈，作为后续消息向上传播。

这让系统"持续运转、自收敛"：信息逐层上传给视野更全局的 owner，而无需全局同步或横向串话。团队还移除了 integrator（数百 worker 都要过一道合并闸门，成了明显瓶颈）。

洞见：这个结构竟与人类软件团队的递归委派惊人相似，且是涌现而非训练所得——暗示这可能本就是组织软件项目的"正确"方式。

06为吞吐量显式设计：接受小而稳定的错误率

要求"每次提交前 100% 正确"会让系统严重串行化：一个小错（API 变更或拼写）就能让整个系统停摆，agent 们越界乱修、扎堆抢修同一问题。改为允许一定松弛后，错误"出现即被同伴快速修复"，错误率保持小而恒定、不爆炸也不恶化。多 agent 改同一文件时也不强行消除，接受短暂"湍流"让系统自然收敛。

"理想的高效系统接受一定错误率，但需要一个最终的'绿色分支'，由 agent 定期快照并在发布前做一次快速修复。"

洞见：把"永远绿"的执念换成"整体快速收敛 + 末端统一修复"，是用工程取舍换吞吐量的关键认知转变。

07意外瓶颈与提示词的量级效应

限制内存后，磁盘成了热点：数百 agent 同时编译会产生每秒数 GB 的构建产物读写，项目结构和编译耗时直接拖累 token/提交吞吐。作者由此提出：Git、Cargo 的共享锁、仓库副本的写时复制与去重，都有低垂的效率提升空间。

提示词上有个反直觉发现：写"生成很多任务"反而保守地只生成少量；写"生成 20-100 个任务"能引出明显更大、更进取的行为。结论是——把模型当"懂工程但不懂你代码库的聪明新人"，只告诉它不知道的东西，用约束（"禁止 TODO、禁止半成品"）而非清单式指令。

洞见：在数千 agent、海量算力的放大下，模糊或欠规格的指令会被指数级放大——"说清意图"成了比 harness 本身更决定产出质量的因素。

最值得带走的 4 条洞见

  1. 结构 > 提示词。锁争用、虚报完成这类问题靠加提示词救不回来，必须用角色分工和隔离的架构来解——20 个 agent 因锁退化到 1-3 个就是铁证。
  2. 单一职责 + 递归委派是涌现的"正解"。给 agent 太多角色会让它过载崩溃；最终收敛出的 planner/subplanner/worker 结构竟与人类团队一致，且未经专门训练。
  3. 为吞吐量而非完美正确性设计。峰值 1000 commits/小时、一周 1000 万次工具调用的代价，是接受小而稳定的错误率，靠"末端绿色分支"统一收尾，而不是每次提交都强求 100% 正确。
  4. 意图与具体数字会被放大。"生成 20-100 个任务"远胜"生成很多"；欠规格的指令在数千 agent 规模下被指数放大——可操控性与可观测性是下一步的关键研究方向。

注：文中提及的"AI 开发 AI 的良性循环"——更好的模型/agent/harness 互相喂养、加速迭代——是作者对这类长时运行 agent 研究方向的核心展望。
