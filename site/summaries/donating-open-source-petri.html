Petri 解决的是“模型在测试中会不会演戏”的问题

对齐评估常常不够真实。模型可能从测试环境的人工痕迹中判断自己正在被评估。

一旦模型知道自己在考试，研究者看到的就不一定是部署时的行为。

为什么这改变理解

Petri 用 auditor model 模拟对齐相关场景，再由 judge model 给 transcript 打分。

这种流程把“可疑行为”变成更可重复的证据。它关注欺骗、谄媚和配合有害请求等风险，而不是只看模型在标准问答中的表现。

Petri 3.0 的三步升级

适配性

新版把 auditor model 和 target model 拆成独立组件。研究者可以分别调整测试者和被测模型。

真实性

Dish 插件使用真实 system prompt 和真实部署 scaffold。测试更接近模型上线后的工作环境。

深度

Petri 与 Bloom 集成。Petri 负责广覆盖筛查，Bloom 负责对指定行为做更深入评估。

从实验室工具到公共评估基础设施

核心洞见

当模型能力越强，评估工具的独立性就越重要。可信的结果不只来自测试方法，也来自谁来维护测试。

最值得带走的洞见

独立性本身是安全能力。

Petri 转交给 Meridian Labs，目标是让结果被更广泛地视为中立和可信。

真实环境会暴露不同风险。

Dish 让测试使用真实 prompt 和 scaffold，减少模型识别测试场景的机会。

广覆盖和深挖需要配合。

Petri 负责扫描多类行为，Bloom 负责深入分析特定行为。
