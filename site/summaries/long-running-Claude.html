面向科学计算的长时间运行 Claude

Long-running Claude for scientific computing

一句话总括：这项实验说明，配备明确验收标准、持久记忆和可靠测试后，AI 代理可以连续工作数天，把复杂科学软件从零推进到接近参考实现的精度。

真正的贡献

关键不是让代理“写代码”，而是让它知道怎样持续纠错

项目把 CLAUDE.md、CHANGELOG.md、测试预言机和 Git 组合成一套闭环。代理能记住失败路线，也能衡量每次修改是否更接近 CLASS。

这改变了人们对 AI 科研能力的判断标准。决定效果的不只是模型知识量，还有任务能否提供稳定、可验证的反馈。

必须保留的边界

亚百分比精度不等于可用于生产

求解器仍未在所有计算区间达到 CLASS 所要求的精度。测试一度只覆盖一个基准参数点，也曾因规范选择等基础问题浪费数小时。

结论：长时间运行可以放大持续进步，也会放大测试盲区。科学判断和覆盖充分的验证仍不可替代。

工作闭环

四个环节让跨会话研究保持方向

1. 写清目标在 CLAUDE.md 中规定交付物、上下文、精度标准和工作规则。
2. 保留记忆用 CHANGELOG.md 记录进度、限制、精度表和已经失败的方法。
3. 持续验证以 CLASS 源码和单元测试作为测试预言机，及时发现回归。
4. 可恢复执行频繁提交并推送。借助 tmux、SLURM 和循环编排继续未完成的工作。
任务结构适合的代理模式原因与启示
C 编译器项目大量代理并行拆分工作较容易分割。Anthropic 的示例跨越约 2,000 个会话。
Boltzmann 求解器单个代理顺序推进，按需派生子代理流水线高度耦合。早期宇宙复合过程中的微小误差会影响所有下游结果。
开放式科学发现目前仍需更强的人类监督如果缺少量化目标、参考实现或测试套件，代理很难可靠判断自己是否取得进展。
真正发生变化的是科研时间的机会成本：当目标明确且算力可用时，夜间没有运行代理，也可能意味着放弃了一段本可获得的进展。

从这次实验中应当记住什么

可验证性决定自治上限。
参考实现、量化目标和回归测试让代理能够自行判断方向。
测试缺口会悄悄积累风险。
单一参数点上的成功，不能证明科学代码在整个参数空间都可靠。
记忆必须包含失败。
如果不记录死路，后续会话很可能重复同样的尝试。
复杂耦合任务不一定适合大规模并行。

完整译文

在本文中，Discovery 团队研究员 Siddharth Mishra-Sharma 解释了如何把持续数天的代理式编程工作流用于科学计算任务。即使任务不属于自己的专业领域，也可以采用测试预言机、持久记忆和编排模式来推进工作。

基本设想

目前，大多数使用 AI 代理的科学家仍在采用对话式循环。他们紧盯过程，并亲自管理每一个步骤。

过去一年左右，模型处理长周期任务的能力显著增强，一种新的工作方式也随之出现。我们不必再介入每个细节，而是可以指定高层目标，让一组代理自主工作。

这种方式让我们有可能在几小时内完成原本需要几天、几周甚至几个月的项目。某些科学任务尤其适合这种模式。例如，重新实现数值求解器，把旧版 Fortran 方言编写的遗留科学软件迁移到现代语言，或者对照参考实现调试大型代码库。

这些任务的范围明确，成功标准清晰。人类只需偶尔监督，不必持续盯守。

Anthropic 的 C 编译器项目展示过这种工作方式。Claude 跨越约 2,000 个会话，构建出一个能够编译 Linux 内核的 C 编译器。

本文介绍如何使用 Claude Code，为科学计算任务搭建类似的工作模式。这里主要考虑的是普通学术实验室的使用环境。

作为具体示例，我会介绍如何使用 Claude Opus 4.6，实现一个可微分的宇宙学 Boltzmann 求解器。这类数值代码用于预测大爆炸余辉，也就是宇宙微波背景（Cosmic Microwave Background，CMB）的统计性质。它通过演化早期宇宙中光子、重子、中微子和暗物质的耦合方程来完成计算。

CLASS 和 CAMB 等 Boltzmann 求解器是宇宙学领域的核心科学基础设施。研究人员依靠它们，使用 Planck、Simons Observatory 等巡天项目的数据约束宇宙学模型。

可微分版本能够让梯度穿过整个求解器传播，从而使用基于梯度的推断方法。这可以大幅加快参数估计。

使用 JAX 编写求解器非常自然。它几乎无需额外工作，就能提供自动微分能力，并兼容 GPU 等加速器。

值得注意的是，这项任务并不属于我的核心研究领域。我对相关工具和科学背景有宏观了解，但并不具备在合理时间内独立完成它所需的专业知识。

真正拥有相关专长的团队已经使用 JAX 构建了可微分求解器，其中包含 CLASS 的一部分功能。这些项目通常凝聚了研究人员数月乃至数年的工作。

这次实验想要验证的是：在非领域专家只提供少量指导的情况下，代理能否走得更远。

这种任务的结构与 C 编译器项目不同。后者可以分派给大量并行工作的代理。

Boltzmann 求解器则是一条高度耦合的流水线。在描述早期宇宙如何发生复合时，一个很小的数值误差或不理想的近似，都可能悄悄改变所有下游结果。

因此，它需要另一套代理能力。调试人员必须沿整条计算链追踪因果关系，并运用领域知识。

这类任务可能更适合由一个代理顺序推进。它可以按需派生子代理，并使用参考实现对差异进行二分定位。

我们会使用运行 SLURM 作业调度器的高性能计算（HPC）集群。不过，无论在哪里运行 Claude Code，核心思路都相同：准备进度文件、测试预言机，以及规则清晰的代理提示。

起草计划，并在本地迭代

当工作方式转向管理一支自主的代理研究团队时，你的大部分时间应当用于编写指令。这个过程可以与 Claude 协作完成。

这些指令需要明确说明项目的交付物和相关上下文，并保存在根目录下的 CLAUDE.md 文件中。

Claude 会特殊处理这个文件。它会让文件内容始终处于上下文中，并在制定整体计划时加以参考。

更重要的是，Claude 可以在工作过程中编辑这些指令。它能够根据遇到的问题更新文件，为后续工作留下新的指导。

在这个宇宙学 Boltzmann 求解器项目中，我准备了一份早期版本的 CLAUDE.md。它展示了首次尝试编写求解器后确定的整体计划和设计决策。

为了得到这份文件，我先指定了项目的高层目标：完整覆盖参考实现 CLASS 的功能，同时保持完全可微分；主要科学交付结果与 CLASS 相比，误差目标为 0.1%。随后，我与 Claude 反复修改计划，直到它看起来足够合理。

CLASS 和 CAMB 这两套经典 Boltzmann 代码通常能达到 0.1% 左右的一致水平。因此，这似乎是一个合适的科学目标。

跨会话记忆

进度文件是代理可携带的长期记忆，作用类似实验室笔记。按照本文使用的约定，我们把它命名为 CHANGELOG.md。在 CLAUDE.md 中，Claude 会被要求持续把进度记录到这个文件里。

一份良好的进度文件可以记录当前状态、已经完成的任务、失败的方法及其失败原因、关键检查点的精度表，以及已知限制。

记录失败的方法尤其重要。缺少这些记录时，后续会话会再次尝试同样的死路。

一条记录可能写成：“尝试用 Tsit5 求解扰动常微分方程，但系统过于刚性。已改用 Kvaerno5。”本示例的变更日志展示了这些要素。

测试预言机

使用代理开展更加开放的科学发现，当然已经不再遥远。不过，在现阶段，长时间运行的自主科学工作仍然极度依赖一种机制：代理必须知道自己是否正在取得进展。

对于科学代码，这种机制可以是参考实现、能够明确量化的目标，或者已有的测试套件。

要求代理在工作过程中扩充测试套件并持续运行测试，也很有帮助。这样可以防止代码发生回归。

在我的示例任务中，Claude 被要求以 CLASS 的 C 源码作为参考实现，构建并持续运行单元测试。

用 Git 协调工作

Git 很适合在减少人工干预的同时监控和协调代理的进展。代理应当在完成每个有意义的工作单元后提交并推送。

这样做可以在出现问题时提供可恢复的历史记录，也能让进展在本地清晰可见。如果计算资源在会话中途耗尽，工作也不至于丢失。

实际使用时，可以在 CLAUDE.md 中加入这样的指令：“完成每个有意义的工作单元后提交并推送。每次提交前运行 pytest tests/ -x -q。绝不要提交会破坏现有已通过测试的代码。”

如果需要引导代理，你随时可以通过 SSH 登录集群，手动再次提示代理，或者更新它的指令。

不过，更方便的方式通常是让本地 Claude Code 实例通过 SSH 登录并替你执行命令。下文介绍的其他操作也可以采用这种方式。

执行循环

如前所述，通常最好先在本地反复修改计划。确认计划合理，并已经写入 CLAUDE.md 后，再进入下一阶段。

随后，在计算节点的 tmux 等终端多路复用器中启动 Claude Code 会话。告诉代理代码库的位置，然后让它自行工作。

由于会话运行在 tmux 内，你可以分离会话并合上笔记本电脑，只需偶尔检查进度。

在 Boltzmann 求解器项目中，我甚至会在排队买咖啡时，用手机查看 GitHub 上的进展。

在 HPC 集群上，可以通过 SLURM 调度器申请一个节点。用于在 tmux 会话中启动 Claude Code 的示例作业脚本，可以采用相应的形式。

作业启动后，连接到 tmux 会话并向 Claude Code 发出指示。例如：“阅读 CHANGELOG.md，然后继续处理下一个任务。”确认它方向正确后，就可以分离会话。

你可以随时重新连接，以便检查进度、调整方向或启动新任务。

Ralph loop：随着模型能力提升，它们对提示工程、检索增强生成（RAG）或上下文填充等定制编排方式的依赖会逐渐减少。

不过，在特定的能力阶段，适量的脚手架仍能提升模型表现。例如，当前模型可能出现“代理式懒惰”（agentic laziness）。面对复杂的多部分任务时，它们有时会找借口在全部工作完成之前停下，比如说：“时间不早了，我们明天再继续？”

一种规避方式是使用 Ralph loop。这种编排模式本质上是一个 for 循环。当代理声称已经完成时，它会把代理重新带回上下文，并询问它是否真的做完了。

这种模式对长时间运行的任务很有帮助。代理可能会承认工作尚未达到要求，然后继续推进，直到真正完成。

其他类似模式包括 GSD、面向特定领域的 GSD 变体，以及 Claude Code 原生的 /loop 命令。

Ralph 可以通过 /plugin 安装。在 Claude Code 中，可以使用相应的典型调用提示。

在这个示例里，Claude 最多会迭代 20 次，直到它用“DONE”这一固定口令确认任务已经完成。

结果

Claude 从零开始，在几天内持续开发了这个项目。它的多个输出与参考实现 CLASS 达到了亚百分比级的一致性。

我让 Claude 重建代码主要输出在整个项目开发过程中的精度变化。这些输出包括不同类型的 CMB 角功率谱。Claude 还标注了开发期间的重要里程碑。

它生成了下图，展示项目逐步达到亚百分比精度的过程。

代理的开发轨迹有些磕绊。例如，它的测试覆盖存在明显缺口。有一段时间，它只在一个基准参数点上测试代码，导致测试发现错误的覆盖面大幅缩小。

它也会犯一些基础错误。例如，它可能在规范约定上出错，或者花费数小时追查宇宙学家一眼就能发现的问题。

尽管如此，它仍然持续朝既定目标前进，并最终实现了亚百分比精度。

这个项目还带来了一个副作用：通过观察 Git 提交历史，我对 Boltzmann 求解器及其描述的物理过程有了出乎意料的深入了解。

这个项目不属于我的核心研究领域。不过，跟随 Claude 的渐进式进展，再去查阅那些我不认识的内容，最终成了一种有效吸收科学知识的方式。

提交日志读起来就像一位速度极快、严格按字面执行任务的博士后留下的实验笔记。

最终得到的求解器还没有达到生产级水平。例如，它并未在所有计算区间内都以可接受的精度匹配参考实现 CLASS。

不过，这次实验说明，由代理驱动的开发可以把研究人员数月甚至数年的工作压缩到几天。

这种压缩改变了“空闲时间”的含义。AI 研究中有一种普遍体验：晚上启动一个实验，例如一次训练任务，然后在早晨看到结果时获得满足感。

不运行实验会带来机会成本。如今，不运行代理似乎也开始具有类似的成本。

如果你拥有可用的算力，也有成功标准明确的项目，那么每一个没有让代理替你工作的夜晚，都可能意味着一段未能获得的进展。
