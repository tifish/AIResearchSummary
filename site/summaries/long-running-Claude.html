把科研编程任务交给长期运行的 AI 代理

Anthropic Research · 2026-03-23 · Science

一句话总括：这篇文章的真正贡献不是证明 Claude 会写科学代码，而是提出了一套让代理在多日尺度上持续推进、可验证、可恢复的科研软件工作流。

适合长期代理的科研任务，必须有清晰目标和可检验终点

作者认为，AI 代理正在从“逐步聊天辅助”转向“给定目标后自主推进”。适合这种模式的科学计算任务包括重写数值求解器、迁移老旧 Fortran 科研软件、用参考实现调试大型代码库。

这些任务的共同点是边界清楚、成功标准明确、人工监督可以低频进行。结论是：代理最先改变的不是开放式科学发现，而是那些已有明确验证标准的科研工程劳动。

核心案例是用 Claude 从零实现可微 Boltzmann 求解器

示例任务是让 Claude Opus 4.6 用 JAX 实现一个可微宇宙学 Boltzmann solver，用来预测 CMB 等早期宇宙观测量，并支持梯度传播。类似 CLASS 和 CAMB 的求解器是宇宙学参数估计的基础设施，而可微版本能让梯度推断方法更快。

作者并非该细分领域专家，因此这个案例测试的是：非领域专家能否用代理推动一个高度专业的科研代码项目。结论是：AI 代理可以显著降低跨领域科研软件原型的启动门槛，但不能消除对验证和专业判断的需求。

工作流的关键不是提示词，而是长期记忆、测试 oracle 和协调机制

项目根目录中的 CLAUDE.md 保存目标、规则和设计取舍，CHANGELOG.md 记录进度、失败尝试、精度表和已知限制。对科研代码来说，参考实现、量化目标或现有测试套件构成 test oracle，让代理知道自己是否真的变好。

Git 则用于低成本监督：每个有意义的工作单元提交和推送，提交前运行测试，便于恢复和查看进展。结论是：长期代理需要的是实验室制度化流程，而不是一次性的“聪明 prompt”。

数字显示它很有用，但还没有达到生产级可靠性

文章给出的目标是主要科学输出相对 CLASS 达到 0.1% 精度，因为 CLASS 与 CAMB 的典型一致性约为 0.1%；Claude 在几天内把多个输出推进到 sub-percent agreement。Anthropic 此前 C 编译器项目曾使用约 2,000 个会话，说明这类长期代理工作已经从演示走向复杂工程。

但代理也暴露出测试覆盖不足、误解规范、长时间追逐领域专家一眼能看出的错误等问题。结论是：它能压缩研发时间，却不能替代科学软件中的精度审查和领域验收。

Ralph loop 解决的是代理过早停工，而不是科学判断

文章提到当前模型在复杂任务中可能出现“agentic laziness”：没有真正达标却声称完成。Ralph loop 的做法是当代理说完成时再次把它拉回上下文，追问是否真的符合规格，直到它继续修复或明确给出完成信号。

这类编排能提高长期任务的完成度，但它依赖清晰规格和可运行测试。结论是：循环督促可以减少半成品，但不能让模糊目标自动变成正确科研成果。

真正的变化是科研空闲时间也变成了可计算资源

作者把长期运行代理类比为夜间训练实验：如果有算力、有清晰目标、有可验证终点，那么不让代理在后台推进也会产生机会成本。对实验室来说，这意味着“等待时间”可以被转化为代码迁移、测试补齐、参考实现对齐和原型探索。

结论是：AI 对科学计算的影响不只是单次加速，而是改变了研究团队安排后台工作的方式。

最值得带走的洞见

  • 长期科研代理最适合有参考实现、测试套件或量化指标的任务。
  • CLAUDE.md、CHANGELOG.md、Git 和测试 oracle 共同构成代理的“实验室操作系统”。
  • 几天达到亚百分比精度很有意义，但距离生产级科研软件仍有明确差距。
  • AI 代理会放大非专家的执行能力，同时也放大验证体系的重要性。
  • 未来实验室的机会成本可能包括：夜里没有让代理继续推进那些可验证的工程任务。
