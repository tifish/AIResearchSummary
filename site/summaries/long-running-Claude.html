Anthropic · Science

让 Claude 长时间自主跑科学计算

来源 Anthropic ｜ 作者 Siddharth Mishra-Sharma（Discovery 团队）｜ 2026-03-23

一句话总括：本文的真正贡献不是"AI 能做科研"这句口号，而是把多日自主 Agent 编程的整套工程实践——计划文件、跨会话记忆、测试预言机、Git 协调、防偷懒循环——搬进科学计算，并用一个亲身例子证明：非本领域专家也能让 Agent 把通常耗时数月到数年的代码工作，压缩到几天。

范式转变从"牵着走"到"放出去自己干"

多数用 AI 的科学家还停在对话式循环里，每一步都盯着、牵着模型走。随着模型在长跨度任务上明显变强，出现了新工作方式：只给高层目标，把一队 Agent 放出去自主干活。适合这种模式的是范围清晰、成功标准明确、只需偶尔人工监督的任务——比如重写数值求解器、把老 Fortran 翻译成现代语言、对照参考实现调试大型代码库。

结论：当任务"对错可判定"时，监督就能从"持续盯梢"降级为"偶尔抽查"，这正是自主化的前提。

两类任务编译器靠并行，求解器靠"一个 Agent 顺着查"

Anthropic 的 C 编译器项目用约 2000 个会话造出能编译 Linux 内核的编译器——那是能拆给大量并行 Agent 的活。而宇宙学 Boltzmann 求解器是深度耦合的流水线：早期宇宙复合过程里一个微小数值误差或粗糙近似，会沿链条悄悄改变下游一切。所以它更适合单个 Agent 顺序作业、按需派生子 Agent、并用参考实现来二分定位差异。

结论：任务的"耦合度"决定编排形态——可切分的并行铺开，强耦合的串行深挖。

实验对象用 Opus 4.6 造一个可微分的宇宙学求解器

作者用 Claude Opus 4.6 从零实现一个可微分的 Boltzmann 求解器（预测大爆炸余晖 CMB 的统计性质）。这类代码（CLASS、CAMB）是宇宙学的核心基础设施；做成可微分版（用 JAX，免费获得自动微分与 GPU 加速）能让基于梯度的推断大幅加速参数估计。关键设定：这不是作者的本行，而真正懂行的团队做出功能子集的可微分版，往往要花数月到数年。

结论：可验证的硬指标（0.1%）是把"开放式科研"变成"可托管工程任务"的关键一步。

工程配方计划、记忆、预言机、协调——缺一不可

这套工作流的四根支柱：

CLAUDE.md（计划） · 写清交付物与上下文，常驻上下文；Claude 可边干边改它，为后续工作沉淀决策。
CHANGELOG.md（记忆） · 跨会话的长期记忆/实验笔记：记状态、已完成、失败的尝试及原因、关键检查点的精度表、已知局限。
测试预言机 · 以 CLASS C 源码为参考实现持续跑单元测试，并让 Agent 自己扩充测试防回归。
Git 协调 · 每完成一个有意义单元就 commit+push；规则如"提交前跑 pytest tests/ -x -q，绝不提交破坏已通过测试的代码"。
为什么"失败记录"如此重要：没有它，后续会话会一次次重蹈覆辙。一条典型记录是——"试过用 Tsit5 解扰动 ODE，系统太僵硬，改用 Kvaerno5。"

结论：让 Agent 长跑的核心不是更强的提示词，而是给它一套"能记住教训、能自判对错、能恢复现场"的脚手架。

防偷懒Ralph loop：你真做完了吗？

当前模型会"偷懒"——面对复杂多步任务，有时会找借口提前收工（"有点晚了，明天接着弄？"）。Ralph loop 本质是个 for 循环：当 Agent 声称完成时，把它踢回上下文反问"真做完了吗"，它常会承认没达标并继续干，直到喊出 "DONE"。文中典型设置为最多循环 20 次。类似机制还有 GSD 与 Claude Code 原生的 /loop。运行环境是 SLURM 调度的 HPC 集群 + tmux：起好会话后可以合上笔记本，偶尔（作者说"排队买咖啡时刷手机看 GitHub"）查看进度。

结论：在模型还会偷懒的当下，少量脚手架就是实打实的能力提升；模型越强，这类编排就越可省。

结果与短板几天达到亚百分比一致，但过程笨拙

Claude 从零干了几天，对 CMB 各角功率谱达到亚百分比（sub-percent）一致。但轨迹"相当笨拙"：测试覆盖一度只测单个基准参数点（大幅缩小抓 bug 面），会犯规范约定（gauge）之类低级错误，也会花数小时追一个宇宙学家一眼能看穿的 bug——但它持续朝目标推进。结果非生产级（并非每个区间都达标），却证明了 Agent 驱动开发能把数月乃至数年的工作压成几天。

结论：Agent 像"一个飞快、极其照字面办事的博士后"——会犯傻，但只要目标可判定，就能稳步逼近。

意外收获看 commit 历史，把一门陌生科学"渗透"进脑子

一个副产品：作者通过追看 Git 提交历史、遇到不懂的就查，竟系统地学到了 Boltzmann 求解器与背后的物理。提交日志"读起来像一位飞快、极度字面化的博士后写的实验笔记"。

结论：Agent 的可观测工作流不只是产出代码，还能成为非专家学习一个领域的高效路径。

最值得带走的洞见

  1. 关键不是"AI 会科研"，而是一套可照搬的工程配方。CLAUDE.md 存计划、CHANGELOG.md 当跨会话记忆（尤其记失败）、参考实现当测试预言机、Git 做协调——四件套缺一不可。
  2. 可验证的硬指标把科研变成可托管任务。把目标钉死在"对标 CLASS 的 0.1%"，Agent 才有办法自判进度；它几天内做到了亚百分比一致。
  3. 耦合度决定编排形态。C 编译器能拆给约 2000 个并行会话；深度耦合的求解器则要单 Agent 顺序深挖、用参考实现二分定位。
  4. 当下还需少量脚手架对抗"偷懒"。Ralph loop（最多 20 次反问到"DONE"）能把声称完成却没达标的 Agent 拽回来继续干；模型越强，这类编排越可省。
  5. "不让 Agent 过夜干活"开始变成一种机会成本。正如跑实验过夜、早上看结果一样，只要有算力和成功标准明确的项目，每个不让 Agent 干活的夜晚都是被白白浪费的进度。

注：本文为方法论示范，结果非生产级——求解器并非在每个区间都达到可接受精度；示例模型为原文所述 Claude Opus 4.6。同行评审 Eric Kauderer-Abrams。
