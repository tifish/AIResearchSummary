一句话总括

用于科学计算的长时运行 Claude

Long-running Claude for scientific computing

文章真正的贡献，是把多日自主代理从“能写代码”推进到“能被科研工作流约束地推进复杂数值项目”。关键不只是模型能力，而是目标、测试、记忆和调度方式。

0.1%主要科学输出对 CLASS 的目标精度。
数天Claude 从零推进到多个输出的亚百分比一致性。
约 2,000C 编译器项目中的 Claude 会话数量。
20 轮Ralph loop 示例中的最大反复验收次数。

科研代理需要可判定的成功标准

作者选择了一个可微分宇宙学 Boltzmann 求解器作为案例。它要预测 CMB 等早期宇宙观测量，并与 CLASS 这样的参考实现对齐。

0.1% 不是随意数字。CLASS 与 CAMB 这两个经典代码之间的典型一致性也在这个量级。

这改变了读者对 AI 科研影响的理解：代理最适合先进入边界清晰、指标可测、参考实现存在的科学工程问题。

核心洞见

长时运行不是让模型自由发挥。它是把模型放进一个有实验记录、有回归测试、有版本历史的科研机器里。

工作流的关键不是提示词，而是闭环

文章给出的流程更像实验室制度，而不是单次聊天技巧。每一步都减少代理跑偏后的恢复成本。

1
CLAUDE.md 写清目标。
它保存交付物、约束和设计选择。Claude 会持续参考，也可以更新它。
2
CHANGELOG.md 充当长期记忆。
它记录成功、失败、精度表和限制。失败路径尤其重要，可以避免新会话重复走死路。
3
测试 oracle 判断进展。
在科学代码里，oracle 可以是参考实现、量化目标或现有测试套件。
4
Git 协调和恢复。
每个有意义单元提交并推送，既能观察进度，也能避免计算资源中断造成损失。
代理压缩的不是科学判断本身，而是大量可验证的工程迭代。人类价值从逐行控制，转向定义问题、设计验收和识别异常。

案例显示了机会，也暴露了边界

观察关键数据和趋势结论
速度显著提高 类似 JAX 可微分 Boltzmann 求解器通常代表数月到数年研究时间。Claude 在数天内达到多个输出的亚百分比一致性。 AI 代理能显著压缩科研软件原型期。
质量仍需审查 结果不是生产级。它并未在所有 regime 中达到可接受精度。 快并不等于可靠。科研代码仍需要专家验算和更宽测试覆盖。
长链错误更难调试 Boltzmann 求解器是深度耦合管线。早期重组模型的小误差会影响下游所有结果。 这类任务更适合单个顺序代理配合子代理，而不是完全并行拆分。

风险：代理会显得忙碌，但测试面可能很窄

作者指出，Claude 一度只在单个 fiducial 参数点测试代码。这会大幅降低发现 bug 的机会。

它还会犯基础错误，例如混淆 gauge convention，或在领域专家能快速定位的问题上花费数小时。结论是：长期代理需要持续扩大测试覆盖，而不是只追求当前指标变好。

最值得带走的洞见 1

可量化 oracle 是当前自主科研代理的核心前提。没有它，模型很难知道自己是否真的进步。

最值得带走的洞见 2

CHANGELOG 不是文档负担。它是跨会话延续实验记忆的最小基础设施。

最值得带走的洞见 3

亚百分比结果证明了潜力，但生产级科学软件仍需要专家审查和更全面验证。

最值得带走的洞见 4

Ralph loop 这类反复验收机制，能对抗代理过早宣布完成的倾向。

最值得带走的洞见 5

当项目有清晰成功标准和可用计算资源时，不运行代理本身也开始形成机会成本。
