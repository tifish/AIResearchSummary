为本地智能体构建安全沙箱

Implementing a secure sandbox for local agents

一句话总括：Cursor 用因平台而异的底层机制构建统一沙箱，让编码智能体获得更大的自主执行空间，同时把文件、网络和系统权限限制在明确边界内。

审批本身也会失效

自动批准命令会放大智能体能力，但也可能导致数据库被删除、故障代码上线或密钥泄露。

逐条审批并非稳固的长期防线。提示积累后，用户会越来越少认真检查。并行运行多个智能体时，频繁切换上下文会进一步加剧审批疲劳。

洞见：安全不能只依赖人在每一次操作前保持高度警觉。

把自由放在边界之内

智能体可在受控环境中自由运行。只有需要越过边界时才申请批准，最常见的情形是访问互联网。

生产数据表明，沙箱化智能体的中断次数减少了 40%。这说明执行边界不仅提升安全性，也能显著改善工作流。

洞见：恰当的默认权限比不断弹出确认框更可靠，也更高效。

同一个 API，三套平台实现

macOS · Seatbelt

Cursor 排除了 App Sandbox、容器和虚拟机。最终选择通过 sandbox-exec 使用 Seatbelt。

策略会结合工作区设置、管理员设置和 .cursorignore 在运行时生成。它可约束整个子进程树。

结论：成熟但已弃用的系统能力，在现实约束下仍可能是最实用的选择。

Linux · Landlock + seccomp

seccomp 阻止不安全的系统调用。Landlock 负责文件系统限制，并让被忽略文件对沙箱进程完全不可访问。

工作区会映射到覆盖文件系统。查找并重新挂载相关文件是 Linux 实现中最慢的环节。

结论：Linux 提供了强大的原语，但把它们组合成好用的产品仍需大量工程工作。

Windows · WSL2

Windows 当前通过 WSL2 运行 Linux 沙箱。原生实现更困难，因为现有原语大多面向浏览器，而不是通用开发工具。

Cursor 正与 Microsoft 合作，推动所需的底层能力落地。

结论：跨平台一致性不等于底层实现一致，过渡方案也可以形成有效边界。

编码智能体执行终端命令的能力正在迅速提高。它们可以借此探索环境并进行修改。允许自动批准这些命令的用户，能够释放智能体更强大的能力，但同时也承担了更大的风险。智能体一旦出错，就可能删除数据库、发布存在故障的代码，或泄露敏感信息。

要求每条命令都由人工批准，可以降低这种风险，但这种效果往往无法长久维持。随着审批不断积累，用户会逐渐不再仔细查看。工程师同时运行多个智能体时，情况会更加严重。他们必须在不同的审批提示之间频繁切换上下文，最终产生审批疲劳。这反而削弱了审批机制原本的意义。

过去三个月，我们通过在 macOS、Linux 和 Windows 上逐步推出智能体沙箱来解决这个问题。沙箱中的智能体可以在受控环境内自由运行。只有需要越过边界时，它们才会申请批准。最常见的情况是访问互联网。

这种方式大幅减少了中断。与未使用沙箱的智能体相比，沙箱化智能体的停止次数减少了 40%，为用户省下了数小时的人工审查和审批时间。

我们的沙箱目标

我们启动沙箱项目时，目标是在提升安全性的同时消除中断。我们希望赋予智能体足够的行动空间，使其能够有效工作，同时拒绝那些会带来风险的权限。

要取得这种平衡，比看起来更困难。许多终端命令需要意料之外的权限。即使是基本的测试或构建步骤，也可能如此。简单粗暴的沙箱会阻止这些命令，导致智能体的工作流中断。设计一个真正可用的沙箱，需要在安全性和可用性之间不断权衡，同时还要受各个操作系统所提供能力的限制。

实现方式

我们对外提供统一的沙箱 API，但在每个平台上采用不同的实现。macOS、Linux 和 Windows 提供的沙箱原语各不相同，这决定了各平台底层设计的选择。

macOS

我们在 macOS 上评估了四种沙箱方案：App Sandbox、容器、虚拟机和 Seatbelt。App Sandbox 是为 Mac App Store 设计的。如果采用它，Cursor 必须为智能体可能执行的每个二进制文件签名。这不仅会显著增加复杂度，还会产生新的滥用途径。智能体生成或修改的二进制文件可能借此继承 Cursor 的信任。容器会把我们限制在 Linux 二进制文件上，而虚拟机的启动延迟和内存开销又无法接受。

因此，只剩下通过 sandbox-exec 使用的 Seatbelt。它于 2007 年引入，并在 2016 年被弃用，但 Chrome 等重要的第三方应用至今仍在使用。它允许命令在沙箱策略下运行，并对完整的子进程树实施行为限制。

这种策略能够细粒度定义权限。它使用一种颇为独特的策略语言，限制系统调用，并限制对特定文件和目录的读取或写入。我们会结合工作区级设置、管理员级设置以及用户的 .cursorignore，在运行时动态生成策略。

Linux

与 macOS 相比，Linux 既更容易，也更困难。内核通过 Landlock 和 seccomp 提供了所需的原语，但用户空间必须负责把它们组合成真正可用的沙箱。多个开源项目已经有效结合了这些机制，但没有一个支持 .cursorignore 等功能。

我们决定直接使用 Landlock 和 seccomp。Seccomp 用于阻止不安全的系统调用，Landlock 则实施文件系统限制。这使我们可以让被忽略的文件对沙箱进程完全不可访问。我们把用户工作区映射到覆盖文件系统中，并用受 Landlock 保护、无法读取或修改的副本覆盖被忽略文件。

查找并重新挂载这些文件，是 Linux 沙箱中最慢的环节。如果能像 macOS 一样延迟过滤文件系统操作，实现起来会更容易。但在 seccomp-bpf 上下文中，Linux 无法让我们轻松取得文件路径。

Windows

在 Windows 上，我们通过 WSL2 运行 Linux 沙箱。构建具有同等能力的原生 Windows 沙箱要困难得多，因为大多数现有沙箱原语都是为浏览器设计的，并不支持通用开发工具。我们正在与 Microsoft 合作，推动所需原语的提供。

教会智能体使用沙箱

只有当智能体能够预判哪些命令可以在沙箱内成功运行，并在需要时识别出必须升级权限，沙箱才能真正发挥作用。要让模型理解沙箱，就必须修改智能体运行框架。

我们先更新了 Shell 工具的说明，用来解释沙箱约束。其中包括：根据用户设置，命令是否拥有文件系统、Git 或网络访问权限，以及智能体如何在需要时申请更高权限。为了得到一个令我们满意的基础运行框架改动，我们进行了大量人工测试。我们会运行几组常见任务，观察实际行为与预期的差异，调整提示，然后重新运行这些任务。

随后，我们使用内部 benchmark Cursor Bench 评估这些变化的影响，对比启用和未启用沙箱的智能体。我们很快发现一种常见的失败模式：智能体会不断重试同一条终端命令，却不改变所申请的权限。

为了解决这个问题，我们调整了 Shell 工具结果的呈现方式。现在，结果会明确指出造成失败的具体沙箱限制。在某些情况下，结果还会建议智能体升级权限。这些提醒发布后，智能体能够更从容地从沙箱相关失败中恢复，离线评测表现也得到显著改善。

不过，离线评测只能呈现全貌的一小部分。为了进一步确认沙箱不会降低用户体验，我们在生产环境中逐步放量。内部和外部收到的反馈，让我们有信心正式推出这项功能。目前，在受支持的平台上，已有三分之一的请求运行在沙箱中。我们还帮助 NVIDIA 等众多企业客户完成了接入。

随着智能体从生成代码走向操作生产系统，提供执行边界变得至关重要。我们当前的实现，是朝这个方向迈出的一步。展望未来，我们尤其期待针对环境约束进行训练的沙箱原生智能体。这些智能体可以获得直接编写并运行脚本和程序的自由，而不再局限于调用工具。
