给本地编码代理装上"安全护栏"

来源：Cursor · 2026-02-18 · 作者：Ani Betts, Yash Gaitonde, Alex Haugland
一句话总括：Cursor 把"代理沙箱化"做成了一个跨三大操作系统的生产级方案—— 代理在受控环境里自由运行，只在需要越界（多为联网）时才请求批准，从而把"批准疲劳"换成了可量化的效率提升：停顿减少 40%。

问题不是"风险"，而是"批准疲劳"

人工逐条批准看似安全，实则会自我瓦解

自动批准让代理更强大，但一个出错的代理可能删库、发布坏代码或泄露密钥。逐条人工批准能缓解风险——但只是暂时的：批准越积越多，用户就不再仔细看；并行跑多个代理、在批准弹窗间反复切换时尤其严重。

洞见：真正的失败模式不是技术漏洞，而是人类注意力的耗尽——沙箱的价值在于把"每条都问"变成"只在越界时问"。

同一套 API，三套截然不同的底层实现

操作系统提供的原语，决定了设计取舍

Cursor 对外暴露统一的沙箱 API，但每个平台的实现完全不同。难点在于：很多普通命令（连测试、构建）都需要意料之外的权限，过于天真的沙箱会直接卡死代理的工作流。

平台方案关键取舍
macOSSeatbelt（sandbox-exec）在 App Sandbox/容器/虚拟机/Seatbelt 四选一；前三者分别带来签名复杂度、仅限 Linux 二进制、启动与内存开销。Seatbelt 2007 推出、2016 弃用，但 Chrome 等仍在用；按 .cursorignore 运行时动态生成策略。
LinuxLandlock + seccompseccomp 拦截危险系统调用，Landlock 限制文件系统；用 overlay 把忽略文件替换为不可读写副本。查找与重挂载这些文件是最慢环节，因为 seccomp-bpf 拿不到文件路径。
WindowsWSL2 里跑 Linux 沙箱原生 Windows 原语多为浏览器设计，不适合通用开发工具；正与微软合作补齐。

洞见：没有"通用沙箱"，每个平台都是安全与可用性之间的一次重新谈判。

光有沙箱不够，还得"教会"模型用它

最典型的失败：反复重试同一命令，却从不升级权限

沙箱只有在代理能预判"哪些命令会成功、何时需要升级权限"时才有效。Cursor 先更新 Shell 工具描述说明约束，再用内部基准 Cursor Bench 对比有无沙箱的表现，发现代理常陷入"无脑重试"。解法是改进工具结果的渲染：显式指出是哪条沙箱约束导致失败，并在适当时建议升级权限——之后代理恢复得更优雅，离线评测显著提升。

洞见：沙箱是"人机协议"的一部分，错误信息的措辞和提示，本身就是让代理可用的工程内容。

从"写代码"走向"操作生产系统"

已落地，并指向"沙箱原生代理"

离线评测只是局部，Cursor 通过逐步灰度放量、收集内外反馈才最终发货；如今约三分之一请求跑在沙箱中，并已上线 NVIDIA 等企业客户。下一步是"沙箱原生代理"——直接基于环境约束训练，让代理能自由编写脚本和程序，而不只是受限于工具调用。

洞见：当代理从"生成代码"跨入"运行生产系统"，执行边界从锦上添花变成了前提条件。

最值得带走的洞见

  1. 可量化收益：沙箱代理停顿减少 40%，约 1/3 请求已启用——这是把"安全 vs 效率"从口号变成了数字。
  2. 真正的敌人是注意力：逐条人工批准会因"批准疲劳"自我失效，沙箱通过"只在越界时打断"才解决根因。
  3. 没有银弹：macOS 用 Seatbelt、Linux 用 Landlock+seccomp、Windows 借 WSL2——统一 API 之下是三套受 OS 原语约束的妥协。
  4. 模型必须"懂沙箱"：清晰的失败信息与升级提示，比沙箱本身的隔离机制同样关键，否则代理会无脑重试。
  5. 方向是沙箱原生代理：未来代理将基于环境约束训练、直接写程序，执行边界会成为代理能力的一部分而非外挂。
