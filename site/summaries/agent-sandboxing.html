给本地代理造一个安全沙箱

来源 Cursor ｜ 作者 Ani Betts, Yash Gaitonde & Alex Haugland ｜ 2026-02-18

一句话总括：Cursor 把"代理跑命令该不该批准"从逐条人工审批换成沙箱默认放行、越界才申请——让代理在一个受控环境里自由探索和改动，只有要踏出环境（最常见是上网）时才请求批准。真正的贡献不是"加了一层限制"，而是用各操作系统原生的沙箱原语，把安全边界做到既不破坏构建/测试等正常工作流、又能消灭审批疲劳，并在三大平台上拿出了可验证的落地数据。

为什么需要"每条命令都批"会退化成"闭眼点同意"

能自己跑终端命令的代理越来越强，自动放行能解锁更强的能力，但代价是风险：跑偏的代理可能删库、发坏代码、泄露密钥。直觉解法是"每条命令都让人审"——可批准一多，人就不再认真看；当工程师并行跑多个代理、在多个审批弹窗间来回切换时更糟。结果是审批疲劳，让审批本身失去意义。

结论：安全不能靠"多按几次同意"，否则审批沦为形式。

核心做法沙箱内自由跑，越界才打断

过去三个月，Cursor 在 macOS、Linux、Windows 上陆续上线代理沙箱：代理在受控环境里自由运行，只在需要走出环境时才请求批准。目标是给代理足够的施展空间却拒绝高风险权限。难点在于平衡——很多终端命令连基本的测试/构建都会用到意料之外的权限，过于天真的沙箱会把这些挡掉、直接搞坏工作流。效果很直接：沙箱化代理的中断次数比未沙箱化时减少 40%，为用户省下大量人工审查时间。

40%
沙箱化代理"停下来等批准"的次数减少幅度
1/3

结论：把默认从"问"改成"放行 + 越界才问"，是同时改善安全与体验的关键转向。

实现同一套 API，三平台各用各的原语

Cursor 对外暴露统一的沙箱 API，但底层因操作系统各异而实现不同——平台提供的沙箱原语，直接决定了设计取舍。

macOS · 选 Seatbelt
在 App Sandbox、容器、虚拟机、Seatbelt 四选一。App Sandbox 要给代理执行的每个二进制签名，复杂且会让代理改过的二进制继承 Cursor 的信任，反成新攻击面；容器只能跑 Linux 二进制；虚拟机启动慢、内存开销大。最终选 Seatbelt（经 sandbox-exec）：2007 年推出、2016 年弃用，但 Chrome 等关键应用仍在用，能用一套策略约束整棵子进程树的系统调用与文件读写，并按 .cursorignore 等设置在运行时动态生成。
Linux · Landlock + seccomp
内核给了原语，但要靠用户态自己拼成可用沙箱，现成开源项目都不支持 .cursorignore 这类特性。于是直接用 seccomp 拦危险系统调用 + Landlock 管文件：把工作区映射到 overlay 文件系统，再用 Landlock 把被忽略的文件覆盖成不可读写的副本。最慢的一步是找到并重挂这些文件——因为 seccomp-bpf 上下文里拿不到文件路径，没法像 macOS 那样"惰性过滤"。
Windows · 跑在 WSL2 里
直接把 Linux 沙箱放进 WSL2 运行。造一个等效的原生 Windows 沙箱要难得多——现有原语多是为浏览器设计的，不支持通用开发工具。Cursor 正与微软合作推动这些原语落地。

结论：没有"一招通吃"的沙箱，可用性来自对每个系统原语脾气的精细迁就。

让模型会用不只是关进去，还要教它认识围栏

沙箱要有效，代理得能预判哪些命令在里面会成功、并认出何时需要提权。Cursor 先改写 Shell 工具说明，讲清当前是否有文件、git、网络权限以及如何申请提权——这一步靠大量手工测试反复调提示词。随后用内部基准 Cursor Bench 对比开/关沙箱，发现一个典型失败：代理会反复重试同一条命令却不改权限。对策是改写 Shell 结果的回显，把"导致失败的具体沙箱限制"明确摆出来、必要时建议提权。上线这些提醒后，代理从沙箱相关失败中恢复得优雅得多，离线评测显著提升。

命令失败→ 回显点明"被哪条限制挡住"→ 建议提权 / 改路径→ 代理优雅恢复，不再死循环重试

结论：离线评测只是一角，真正给出信心的是逐步灰度放量后的真实反馈。

落地与展望已接入 NVIDIA，下一步是"沙箱原生"代理

在内外部真实反馈支撑下，沙箱正式发布：目前受支持平台上约三分之一的请求在沙箱内运行，并已接入 NVIDIA 等多家企业客户。文章把这件事放进更大的趋势里：当代理从"写代码"走向"操作生产系统"，提供执行边界至关重要。Cursor 接下来最期待沙箱原生（sandbox-native）的代理——直接在环境约束上训练，从而被允许自己写脚本和程序，而不再局限于工具调用。

结论：沙箱不只是安全补丁，而是让代理获得更大自由度的前提条件。

最值得带走的洞见

  1. 逐条审批是伪安全。批准一多、并行代理一多，人就不再细看，审批退化成闭眼点同意——沙箱把默认改成"放行 + 越界才问"，正面破解了审批疲劳。
  2. 40% 与 1/3 是硬指标。沙箱化代理中断减少 40%，受支持平台上已有三分之一请求在沙箱内跑，并接入 NVIDIA——说明这不是实验性开关而是规模化能力。
  3. 没有通用沙箱。macOS 用弃用但仍可靠的 Seatbelt、Linux 拼 Landlock+seccomp、Windows 借 WSL2——每个平台的原语脾气直接塑造了实现。
  4. 安全件必须"懂工作流"。天真沙箱会挡掉连测试/构建都需要的权限；可用性来自对意外权限需求的精细迁就。
  5. 光关进去不够，得教它认围栏。把"被哪条限制挡住"明确回显给模型，比单纯加限制更能让代理优雅恢复、不再死循环重试。
