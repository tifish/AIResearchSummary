一句话总括

为本地智能体实现安全沙箱

Implementing a secure sandbox for local agents

Cursor 的贡献不是单纯加一道权限弹窗，而是把本地编码智能体放进可执行、可解释、跨平台的边界里，让自动化能力增长时，系统风险不随之失控。

40%启用沙箱后，智能体中断次数比未启用时更少。
3 个月Cursor 在 macOS、Linux、Windows 上逐步推出沙箱。
1/3受支持平台上约三分之一请求已在沙箱内运行。
3 类系统同一 API 背后分别使用不同操作系统原语。

问题不是审批太少，而是审批会失效

本地智能体能运行终端命令，就能探索环境、修改代码和执行构建。自动批准会提升效率，但错误命令可能删除数据库、发布损坏代码，或泄露密钥。

逐条人工审批只能短期缓解风险。审批越来越多时，用户会停止认真检查。多智能体并行工作会让这种疲劳更明显。

沙箱把自由放在边界内

Cursor 的设计目标是减少打断，同时提高安全性。智能体可以在受控环境中自由工作，只有需要越界时才请求授权。

最常见的越界场景是访问互联网。这说明沙箱改变的是默认权限模型，而不只是增加提示文案。

跨平台实现各不相同

平台 采用方式 关键取舍 带来的理解变化
macOS 使用 Seatbelt，通过 sandbox-exec 运行子进程树。 App Sandbox、容器和虚拟机各有成本。Seatbelt 虽已废弃，但仍能细粒度限制系统调用和文件访问。 本地 AI 工具的安全工程常常依赖现实可用的系统能力，而不是最理想的抽象。
Linux 直接组合 Landlock 和 seccomp。 seccomp 阻止危险系统调用，Landlock 约束文件系统。Cursor 还用 overlay filesystem 处理被忽略文件。 安全边界不是一个开关。它需要把内核能力、工作区映射和忽略规则一起编排。
Windows 先在 WSL2 中运行 Linux 沙箱。 原生 Windows 沙箱更难，因为现有原语多面向浏览器，不适合通用开发工具。 智能体安全会推动操作系统暴露新的开发者级隔离能力。

让智能体理解沙箱，和实现沙箱一样重要

1

Cursor 先更新 Shell 工具说明，让模型知道文件系统、Git、网络权限会受用户设置影响。这样智能体能预判哪些命令会失败。

2

团队用常见 rollout 手工测试提示词。发现异常后调整 harness，再重复执行。这个过程把安全规则变成模型能操作的上下文。

3

在 Cursor Bench 中，他们发现智能体会反复重试同一命令，却不改变权限。失败提示随后被改成明确指出是哪条沙箱约束造成了失败。

4

上线后，智能体从沙箱失败中恢复得更好，离线评测表现明显改善。生产环境的渐进发布又验证了用户体验没有明显退化。

真正的安全自主权，不是让模型少做事，而是让模型知道自己在哪些边界内可以大胆做事。

最值得带走的洞见

审批不是长期扩展方案

当智能体数量和命令数量上升时，人类审批会变成噪声。沙箱把注意力留给真正越界的操作。

安全边界要贴近工作流

如果沙箱阻断测试、构建和常见命令，它会被绕开。可用性本身就是安全设计的一部分。

平台差异无法被抹平

统一 API 之下仍要分别处理 Seatbelt、Landlock、seccomp 和 WSL2。

模型也需要权限素养

工具提示和错误渲染会影响智能体是否能正确升级权限，而不是原地重试。

下一步是沙箱原生智能体

如果训练时就理解环境约束，智能体可以更自由地写脚本和程序，而不只依赖工具调用。
