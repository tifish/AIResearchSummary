本地 Agent 沙箱：让自动执行更安全

一句话总括：这篇文章的真正贡献，是把本地 coding agent 的安全重点从“人类反复点批准”转向“默认受限执行、必要时清晰提权”的系统设计。

问题不是权限太多，而是审批会失效

自动批准命令能显著提升 coding agent 的能力，但错误命令可能删除数据库、发布坏代码或泄露密钥；逐条人工批准虽然看似安全，却会在频繁提示中制造 approval fatigue。这改变了对 AI 影响的理解：风险不只来自模型犯错，也来自人机协作界面让人停止认真审查。

沙箱的目标是减少打断，同时收紧边界

Cursor 在过去 三个月把 agent sandboxing 推到 macOS、Linux 和 Windows：agent 可在受控环境内自由运行，只有需要越界时才请求批准，最常见的是访问互联网。结果是沙箱 agent 比非沙箱 少停止 40%，说明安全边界如果设计得好，不必以明显牺牲效率为代价。

统一 API 背后，是三套不同操作系统实现

macOS 评估了 4 种方案：App Sandbox、容器、虚拟机和 Seatbelt，最后选择通过 sandbox-exec 使用 Seatbelt，因为它能约束整个子进程树并按文件、目录和 syscall 细粒度授权。这个选择说明，agent 安全不是抽象策略就能解决，必须贴近操作系统原语和开发工具的真实行为。

Linux 更开放，但组合成本更高

Linux 直接使用 seccomp 和 Landlock：seccomp 阻止危险 syscall，Landlock 限制文件系统访问，并让 .cursorignore 中的文件对沙箱进程不可读、不可改。难点在于查找和重新挂载这些被忽略文件是最慢步骤，这提醒读者：更强的权限控制往往会把复杂性转移到文件系统映射、性能和兼容性上。

Windows 目前借助 WSL2，而不是原生沙箱

在 Windows 上，Cursor 把 Linux 沙箱运行在 WSL2 内，因为现有 Windows 沙箱原语更多面向浏览器，难以覆盖通用开发工具。这个结论很实际：跨平台 agent 产品不能假设每个系统都有同等成熟的安全基础设施，短期可能需要折中实现。

模型也要学会“沙箱意识”

Cursor 不只改底层隔离，还改了 Shell 工具说明和失败结果展示，让 agent 知道当前是否有文件系统、git 或网络权限，以及何时应请求提权。在 Cursor Bench 中，一个常见失败是 agent 反复重试同一命令却不改变权限；显式提示失败约束后，agent 对沙箱失败的恢复更好。这说明 AI 工具安全不仅是权限系统，也是反馈设计和模型行为训练问题。

生产采用说明沙箱已进入主流程

Cursor 通过离线评测和生产灰度验证体验影响，目前支持平台上约 三分之一请求运行在沙箱中，并已服务 NVIDIA 等企业客户。结论是：当 agent 从生成代码走向操作真实系统，执行边界会成为基础设施，而不是高级可选项。

最值得带走的洞见

  • 审批不是长久安全机制；高频审批会把安全判断变成机械动作。
  • 有效沙箱要给 agent 足够自由，否则会破坏构建、测试和调试流程。
  • 跨平台安全实现必须尊重系统差异：macOS、Linux、Windows 的最优路径并不相同。
  • 工具返回的失败原因会直接影响 agent 行为，好的错误提示本身就是安全能力。
  • 未来更强的本地 agent 可能不是被更多工具约束，而是被训练成理解并利用自己的执行边界。
