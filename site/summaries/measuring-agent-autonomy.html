能力评测告诉我们模型最多能做什么。这项研究关心另一个问题：在真实世界里，人们实际上让智能体独立做到什么程度。

自主性在增长，但不是模型升级单独造成的

Claude Code 的中位单轮时长仍约为 45 秒。真正发生变化的是分布尾部。最久的一批任务持续时间明显拉长，而且增长曲线跨越多个模型版本，过程相当平滑。

这说明现有模型可能早已具备更大的自主工作空间。用户建立信任、任务变得更复杂，以及产品持续改进，都在推动实际自主性上升。

能力与部署之间仍有余量

METR 估计 Claude Opus 4.5 能以 50% 成功率完成相当于人类近 5 小时工作量的任务。但 Claude Code 的 99.9 分位实际单轮时长约为 42 分钟。

两项指标不能直接比较。不过它们共同指向一个结论：现实中授予模型的自主空间，仍落后于模型能够处理的范围。

经验改变的不是监督强度，而是监督方式

约 10 次会话

新用户常在每一步操作前审批。只有约 5% 的单轮任务会被中途打断。

少于 50 次会话

约 20% 的会话使用完全自动批准。用户仍主要依赖事前许可。

约 750 次会话

完全自动批准超过 40%，中断率约为 9%。熟练用户更愿意让 Claude 连续工作，同时更主动地纠偏。

核心洞见

有效监督不必等于批准每一个动作。关键是让人看得见过程，并能在真正重要的时刻迅速介入。

当前总体风险较低，但平均值会遮住前沿风险

令人安心的总体数据

80% 的工具调用似乎至少有一种保护措施。73% 有某种人类参与。只有 0.8% 看起来不可逆。

不可忽视的尾部

高风险、高自主性的集群数量不多，但并非不存在。它们涉及敏感安全操作、金融交易和医疗信息。一次错误仍可能带来严重后果。

行业分布尚不均衡

软件工程占近一半工具调用。商业智能、客服、销售、金融和电商各自只占几个百分点。

风险边界会继续外扩

随着智能体进入医疗、法律和金融等难以验证的领域，建立信任和发现错误都会更困难。今天的低风险均值不能代表未来。

模型也参与监督。任务越复杂，Claude 越常主动停下来询问澄清。在最复杂的任务中，它主动提问的频率超过人类中断频率的两倍。监督因此不只是人约束模型，也包括模型识别自身的不确定性。

研究能说明什么

它覆盖数百万次 Claude Code 与公共 API 交互。Claude Code 提供完整会话，适合研究自主时长和中断。公共 API 覆盖数千种部署，适合观察行业、风险与保护措施。

研究不能说明什么

数据只来自 Anthropic。公共 API 只能按单次工具调用分析，无法还原完整会话。分类由 Claude 生成，也无法可靠区分生产操作、评测和红队测试。

样本还集中在 2025 年末至 2026 年初。智能体生态变化很快，因此这些数字是阶段性测量，不是永久规律。

最值得带走的洞见

01

自主性是共同建构的。模型行为、用户的监督策略和产品设计缺一不可。单靠发布前 benchmark 无法解释真实部署中的自主程度。

02

熟练用户并非放弃监督，而是在从逐步批准转向观察和纠偏。

03

训练模型主动暴露不确定性，是对权限控制和人类监督的重要补充。

04

下一阶段的重点应是隐私保护下的部署后监测、可信的过程可见性，以及低摩擦的实时干预机制，而不是强制每一步都由人批准。

完整译文

AI 智能体已经到来。它们也已被部署到后果差异极大的场景中，从邮件分类到网络间谍活动都有。要安全部署 AI，理解这一完整的应用谱系至关重要。然而，对于人们在现实世界中究竟如何使用智能体，我们知道的却少得惊人。

我们使用保护隐私的工具，分析了 Claude Code 和公共 API 中数百万次人类与智能体的交互。我们希望回答几个问题：人们会给智能体多少自主权？随着经验增加，这种授权会如何变化？智能体正在哪些领域运行？它们采取的行动是否存在风险？

我们发现：

Claude Code 自主工作的时间正在延长。在持续时间最长的一批会话中，Claude Code 停下来之前的工作时间在三个月内接近翻倍，从不足 25 分钟增至超过 45 分钟。这一增长平滑地跨越了多个模型版本，说明它不完全由能力提升造成。现有模型实际能够承担的自主工作，可能比现实中已经发挥出来的更多。

Claude Code 的熟练用户更常使用自动批准，也更常中断它。随着使用经验增加，用户往往不再逐项审查每个操作，而是让 Claude 自主运行，只在必要时介入。新用户约有 20% 的会话使用完全自动批准。随着经验增加，这一比例会超过 40%。

Claude Code 主动停下来澄清问题的频率，高于人类中断它的频率。除了人类发起的停止，智能体主动停止也是已部署系统中的一种重要监督方式。在最复杂的任务上，Claude Code 停下来询问澄清的频率，是人类中断它的两倍以上。

智能体已被用于高风险领域，但规模还不大。公共 API 上的大多数智能体操作风险较低，而且可以撤销。软件工程占智能体活动的近 50%。与此同时，我们也观察到了医疗、金融和网络安全领域的新兴应用。

下面，我们会更详细地介绍研究方法和发现。最后，我们将向模型开发者、产品开发者和政策制定者提出建议。我们的核心结论是，要有效监督智能体，需要建立新的部署后监测基础设施。还需要形成新的人机交互范式，让人类与 AI 能够共同管理自主性和风险。

我们认为，这项研究只是一个规模不大但很重要的起点。它帮助我们通过实证方式理解人们如何部署和使用智能体。随着智能体得到更广泛采用，我们会继续改进研究方法，并公布新的发现。

研究真实环境中的智能体

对智能体开展实证研究并不容易。首先，人们还没有就“什么是智能体”达成一致定义。其次，智能体发展得非常快。去年，许多最先进的智能体，包括 Claude Code，仍以单一对话线程为基础。如今，多智能体系统已经可以连续自主运行数小时。最后，模型提供商对客户智能体的整体架构了解有限。例如，我们没有可靠方法把客户向 API 发出的独立请求关联成智能体活动的“会话”。我们会在本文末尾更详细地讨论这一难题。

面对这些挑战，我们该如何用实证方法研究智能体？

作为起点，本研究采用了一个既有概念基础、又能实际操作的定义：智能体是一种配备工具的 AI 系统。这些工具让它能够采取行动，例如运行代码、调用外部 API，以及向其他智能体发送消息。1 研究智能体使用的工具，可以让我们深入了解它们正在现实世界中做什么。

接下来，我们制定了一组指标。它们同时使用公共 API 中的智能体应用数据，以及我们自有编程智能体 Claude Code 的数据。这两个来源在广度和深度之间各有取舍：

公共 API 让我们能够广泛观察数千名不同客户的智能体部署。我们没有尝试推断客户的智能体架构，而是把分析单位设为单次工具调用。2 这一简化假设让我们能够对现实中的智能体作出有依据且一致的观察，即使这些智能体的部署环境差异很大。它的局限是，我们只能孤立地分析每个操作，无法重建多个独立操作如何随时间组成更长的行为序列。

Claude Code 提供了相反的取舍。因为 Claude Code 是我们自己的产品，所以我们可以把会话中的不同请求关联起来，从头到尾了解完整的智能体工作流。这使 Claude Code 特别适合研究自主性。例如，我们可以分析智能体在没有人类介入时运行多久、什么情况会触发中断，以及用户在积累经验后如何继续监督 Claude。不过，Claude Code 只是一个产品，因此无法像 API 流量一样反映智能体应用的多样性。

通过保护隐私的基础设施结合这两类数据，我们可以回答任何单一数据来源都无法独立回答的问题。

Claude Code 的自主工作时间正在延长

智能体在没有人类参与时，实际能连续运行多久？在 Claude Code 中，我们可以直接测量这一点。我们按每一轮任务追踪 Claude 从开始工作到停止之间的时间。停止可能是因为任务已经完成、Claude 提出了问题，或用户中断了它。3

单轮持续时间只能近似反映自主性，并不完美。4 例如，能力更强的模型可能更快完成相同的工作。子智能体也能让更多工作并行发生。这两个因素都会缩短单轮时间。5 与此同时，用户可能在尝试越来越有挑战性的任务，这又会延长单轮时间。此外，Claude Code 的用户群正在快速增长，因此用户构成也在变化。我们无法孤立测量这些变化。最终测到的是多种因素共同作用的净结果，包括用户允许 Claude 独立工作多久、交给它的任务有多难，以及产品本身的效率。产品效率每天都在提高。

Claude Code 的多数单轮任务都很短。中位持续时间约为 45 秒，过去几个月只出现小幅波动，范围在 40 至 55 秒之间。事实上，99 分位以下的几乎所有分位数都相对稳定。6 对于一个正在快速增长的产品，这种稳定性符合预期。新用户开始使用 Claude Code 时经验较少。正如下一节所示，他们也不太愿意给予 Claude 完全自由的行动空间。

更能说明问题的信号来自分布尾部。持续时间最长的单轮任务，最能反映 Claude Code 最有挑战性的应用，也能指示自主性的发展方向。2025 年 10 月至 2026 年 1 月，99.9 分位的单轮持续时间接近翻倍，从不足 25 分钟增至超过 45 分钟，见图 1。

值得注意的是，这一增长平滑地跨越了多个模型版本。如果自主性完全由模型能力决定，那么每次推出新模型时，曲线应该会出现明显跳升。但实际趋势相对平稳。这说明可能有多个因素共同发挥作用，包括高级用户逐渐建立对工具的信任、把 Claude 用于越来越有挑战性的任务，以及产品本身不断改进。

自 1 月中旬以来，极端单轮持续时间有所下降。我们推测有几个原因。第一，从 1 月到 2 月中旬，Claude Code 的用户群翻了一倍。规模更大、构成更多样的会话群体可能会改变分布。第二，用户结束假期返工后，带给 Claude Code 的项目可能从兴趣项目变成范围界定更严格的工作任务。最可能的情况是，这些因素与我们尚未发现的其他因素共同造成了下降。

我们还分析了 Anthropic 内部的 Claude Code 使用情况，以了解独立性和实用性如何共同演变。从 8 月到 12 月，Claude Code 在内部用户最具挑战性任务上的成功率翻了一倍。与此同时，每个会话中人类介入的平均次数从 5.4 次降至 3.3 次。7 用户正在给予 Claude 更多自主权。至少在内部使用中，他们取得了更好的结果，同时需要介入的次数更少。

这两项测量都显示出显著的“部署余量”：模型能够处理的自主工作，超过了它在现实中实际承担的范围。

把这些发现与外部能力评测对照起来，会很有帮助。最常被引用的能力评测之一是 METR 的“衡量 AI 完成长任务的能力”。该评测估计，Claude Opus 4.5 能以 50% 的成功率完成一项任务，而同一任务需要人类花费近 5 小时。相比之下，Claude Code 的 99.9 分位单轮持续时间约为 42 分钟，中位数则短得多。不过，这两个指标不能直接比较。METR 评测测量的是模型在理想环境中的能力。环境中没有人类互动，也没有现实后果。我们的测量反映现实中实际发生的情况。在这种环境下，Claude 会停下来请求反馈，用户也会中断它。8 此外，METR 所说的 5 小时衡量的是任务难度，也就是人类完成任务所需的时间，而不是模型实际运行的时间。

单看能力评测或我们的实际测量，都无法完整描述智能体自主性。但把两者结合起来，可以看出一个趋势：现实中授予模型的行动空间，落后于模型能够处理的范围。

Claude Code 熟练用户更常自动批准，也更常中断

随着时间推移，人类会如何调整与智能体协作的方式？我们发现，使用 Claude Code 的经验越多，人们给予它的自主权就越大，见图 2。较新的用户，也就是会话数少于 50 次的用户，约有 20% 的会话使用完全自动批准。会话数达到 750 次时，这一比例会超过 40%。

这种变化是渐进的，说明信任在稳定积累。还需要注意，Claude Code 的默认设置要求用户手动批准每个操作。因此，这种变化可能部分来自用户对产品配置的调整。随着他们熟悉 Claude 的能力，会把产品设为更符合自己偏好的模式，并给予 Claude 更大的独立空间。

批准操作只是监督 Claude Code 的一种方法。用户也可以在 Claude 工作时中断它并提供反馈。我们发现，中断率会随使用经验增加。会话数约为 10 次的新用户，会在 5% 的单轮任务中中断 Claude。经验更丰富的用户则会在约 9% 的单轮任务中中断它，见图 3。

中断和自动批准都会随经验增加。这个看似矛盾的现象，反映出用户监督策略的转变。新用户更可能在操作执行前逐项批准，因此很少需要在执行过程中中断 Claude。经验丰富的用户更愿意让 Claude 自主工作，并在出现问题或需要调整方向时介入。较高的中断率也可能意味着熟练用户正在主动监控。他们更清楚什么时候需要自己介入。我们预计，随着用户形成稳定的监督方式，单轮中断率最终会趋于平稳。经验最丰富用户的曲线可能已经开始变平。不过，会话数越高，置信区间越宽，因此很难确认这一点。9

我们在公共 API 上也看到了类似规律。在复杂度最低的任务中，例如修改一行代码，87% 的工具调用存在某种形式的人类参与。对于高复杂度任务，例如自主寻找零日漏洞或编写编译器，这一比例只有 67%。10 这可能有悖直觉，但存在两种合理解释。第一，任务步骤越多，逐步审批就越不现实。因此，复杂任务在结构上更难做到每一步都受监督。第二，Claude Code 数据显示，熟练用户往往给予工具更多独立空间，而复杂任务可能更多来自熟练用户。我们无法直接测量公共 API 用户的使用年限，但整体规律与 Claude Code 中的观察一致。

综合来看，这些发现说明，熟练用户未必是在放弃监督。自动批准率上升的同时，中断率也在上升，这表明用户仍在进行某种主动监控。它进一步印证了我们此前提出的观点：有效监督不要求批准每个操作，而是要求人类能够在真正重要的时刻介入。

Claude Code 主动询问澄清的频率高于人类中断它的频率

当然，人类并不是唯一影响实际自主程度的参与者。Claude 也会主动参与这一过程。当它不确定该如何继续时，会停下来请求澄清。我们发现，随着任务复杂度上升，Claude Code 会更频繁地要求澄清，而且频率高于人类主动中断它的频率，见图 4。

在最复杂的任务上，Claude Code 请求澄清的频率，是最低复杂度任务的两倍以上。这说明 Claude 对自身的不确定性具备一定程度的校准。不过，不应过度解读这一发现。Claude 未必总能在正确的时机停止，也可能提出不必要的问题。它的行为还可能受到 Plan Mode 等产品功能影响。无论如何，随着任务变难，Claude 越来越常主动停下来咨询人类，从而限制自己的自主性，而不是等人类介入。11

什么会让 Claude Code 停下来？

这些发现表明，智能体主动停止是已部署系统中的一种重要监督机制。训练模型识别自身的不确定性并据此采取行动，是一种重要的安全特性。它可以补充权限系统和人类监督等外部保护措施。在 Anthropic，我们会训练 Claude 在面对含糊任务时提出澄清问题，也鼓励其他模型开发者这样做。

智能体已进入高风险领域，但尚未形成规模

人们正在用智能体做什么？这些部署的风险有多大？智能体的自主程度有多高？风险与自主性之间是否存在取舍？

为了回答这些问题，我们让 Claude 按 1 到 10 分评估公共 API 中单次工具调用的相对风险和自主程度。简单来说，风险得分为 1，表示操作出错也不会造成后果。风险得分为 10，则表示操作可能造成重大伤害。自主性也使用相同量表。低自主性表示智能体似乎在遵循人类的明确指令。高自主性表示它在独立运行。13 随后，我们把相似操作归入不同集群，并计算每个集群的平均风险和自主性得分。

高风险或高自主性的工具使用集群

由于我们对这些智能体的了解有限，这些操作也可能只是模拟。例如，我们无法判断 Claude 下达的金融交易指令是否真的得到执行。我们也认为，许多“数据外传”属于评测活动。尽管高风险操作在整体流量中的占比很低，但单次错误仍可能造成严重后果。

我们发现，80% 的工具调用来自似乎至少具备一种保护措施的智能体，例如受限权限或人工批准要求。73% 的调用似乎以某种方式让人类参与其中。只有 0.8% 的操作看起来不可逆，例如向客户发送邮件。14

为了展示所有集群中风险与自主性的联合分布，我们根据两个维度的平均得分绘制每个集群。图 5 中的每个点都代表一组相关操作。点的位置由该集群的平均风险与平均自主性决定。

公共 API 上的绝大多数操作风险较低。虽然多数智能体部署相对无害，但我们也在风险与自主性的前沿观察到了一些新用途。15 风险最高的集群往往涉及敏感的安全操作、金融交易和医疗信息。不过，我们再次强调，其中许多可能属于评测。风险主要集中在量表低端，而自主性分布更广。在低自主性一端，也就是得分为 3 至 4 时，智能体为人类完成规模小、范围明确的任务，例如预订餐厅或对代码做小幅修改。在高自主性一端，也就是得分高于 6 时，智能体会向数据科学竞赛提交机器学习模型，或对客户服务请求进行分类处理。

我们预计，处在高风险和高自主性极端位置的智能体会越来越常见。如今，智能体主要集中在单一行业：软件工程占公共 API 工具调用的近 50%，见图 6。除编程外，我们还观察到商业智能、客户服务、销售、金融和电子商务领域的一些小规模应用，但没有任何一个领域的流量占比超过几个百分点。随着智能体扩展到这些领域，风险与自主性的前沿预计会继续外扩。这些领域中许多任务的风险，都高于修复一个软件缺陷。

这些规律表明，我们仍处在智能体应用的早期阶段。软件工程师最先大规模构建和使用智能体工具。图 6 显示，其他行业也开始试验智能体。16 我们的方法让我们可以监测这些规律如何随时间演变。特别是，我们能够观察应用是否正转向更自主、风险更高的任务。

从总体数据看，情况令人安心。多数智能体操作风险较低且可以撤销，人类通常也在流程中。但平均值可能掩盖前沿部署。智能体应用集中在软件工程，同时又开始进入新领域。这意味着风险与自主性的前沿将继续外扩。在本文末尾的建议部分，我们会讨论这对模型开发者、产品开发者和政策制定者意味着什么。

局限

这项研究只是一个起点。我们只能看到智能体活动的一部分，也希望坦率说明数据能够和不能告诉我们什么：

  • 我们只能分析单一模型提供商 Anthropic 的流量。基于其他模型构建的智能体，可能呈现不同的采用规律、风险特征和互动模式。
  • 两个数据来源可以互相补充，但都不完整。公共 API 流量覆盖数千种部署，具有广度，但我们只能孤立分析单次工具调用，无法分析完整的智能体会话。Claude Code 提供完整会话，但它只是一个产品，而且绝大多数用途都与软件工程有关。许多最有力的发现来自 Claude Code 数据，因此未必能推广到其他领域或产品。
  • 分类结果由 Claude 生成。我们为每个维度设置了可退出类别，例如“无法推断”和“其他”。在可能的情况下，我们也会用内部数据进行验证，详情见附录。不过，由于隐私限制，我们无法人工检查底层数据。有些保护措施或监督机制也可能存在于我们无法观察的上下文之外。
  • 本次分析只反映一个特定时间窗口，也就是 2025 年末至 2026 年初。智能体生态正在快速变化。随着能力提升和应用发展，这些规律可能改变。我们计划持续扩展这项分析。
  • 公共 API 样本以单次工具调用为抽样单位。这意味着包含许多连续工具调用的部署会被过度代表，例如反复编辑文件的软件工程工作流。相比之下，用较少操作就能完成目标的部署所占权重会更低。这种抽样方式反映智能体活动量，但不一定反映智能体部署或用途的实际分布。
  • 我们研究 Claude 在公共 API 上使用的工具，以及这些操作周围的上下文。但对于客户基于公共 API 构建的更大系统，我们了解有限。一个在 API 层面看似自主运行的智能体，可能会在后续流程中接受我们无法观察的人类审查。尤其需要注意的是，我们对风险、自主性和人类参与的分类，取决于 Claude 能从单次工具调用上下文中推断出什么。分类无法区分生产环境中的操作与评测或红队测试中的操作。风险最高的若干集群似乎属于安全评测。这也凸显了我们无法了解每个操作完整背景的局限。

展望未来

智能体应用仍处在早期阶段，但自主性正在提高，高风险部署也开始出现。Cowork 等产品让智能体更容易使用，这一趋势尤其明显。下面，我们向模型开发者、产品开发者和政策制定者提出一些建议。由于我们才刚刚开始测量真实环境中的智能体行为，因此会避免作出强硬规定，而是重点指出值得继续研究的方向。

模型和产品开发者应投资部署后监测

部署后监测对于理解智能体的实际用途至关重要。部署前评测会在受控环境中测试智能体能够做什么，但我们发现的许多现象无法仅靠部署前测试观察到。除了了解模型的能力，还必须理解人们在实践中如何与智能体互动。我们之所以能够报告这些数据，是因为我们选择建设收集数据所需的基础设施。但仍有很多工作要做。我们目前无法可靠地把公共 API 的独立请求关联成连贯的智能体会话。这限制了我们对 Claude Code 等第一方产品之外智能体行为的了解。如何在保护隐私的前提下开发这些方法，是一个需要全行业共同研究与合作的重要领域。

模型开发者应考虑训练模型识别自身的不确定性

训练模型识别自身的不确定性，并主动向人类提出问题，是一种重要的安全特性。它能补充人工审批流程和访问限制等外部保护措施。我们会这样训练 Claude。本次分析也显示，Claude Code 主动提问的频率高于人类中断它的频率。我们鼓励其他模型开发者采取同样做法。

产品开发者应围绕用户监督进行设计

有效监督智能体，不只是把人放进审批链条。我们发现，随着用户积累使用经验，他们往往不再逐个批准智能体的操作，而是转向监控智能体的行为，并在必要时介入。例如，Claude Code 熟练用户更多使用自动批准，但也更常中断。公共 API 上也有相关规律：随着目标复杂度提高，人类参与似乎会减少。产品开发者应投资开发工具，为用户提供可信的智能体行为可见性。产品还应提供简单的干预机制，让用户能够在出现问题时重新引导智能体。我们会继续在 Claude Code 中投入这方面的建设，例如实时引导和 OpenTelemetry，也鼓励其他产品开发者这样做。

现在要求特定的交互模式还为时过早

有一个方面，我们很有把握给出建议，那就是什么不应被强制要求。我们的发现显示，熟练用户会减少对智能体操作的逐项批准，转而采用监控并在必要时介入的方式。如果监督要求强制规定具体交互模式，例如要求人类批准每个操作，就会增加摩擦，却未必带来安全收益。随着智能体和智能体测量科学逐渐成熟，关注重点应是人类能否有效监控和介入，而不是要求人类必须采用某一种参与形式。

这项研究带来的一个核心启示是：智能体在实践中展现的自主性，由模型、用户和产品共同建构。Claude 会在不确定时停下来提问，从而限制自己的独立性。用户在与模型协作的过程中建立信任，并相应调整监督策略。我们在任何一次部署中观察到的行为，都是这三股力量共同作用的结果。因此，只靠部署前评测无法完整描述它。要理解智能体实际如何行动，就必须在现实世界中测量它们，而开展这项工作所需的基础设施仍处于起步阶段。
