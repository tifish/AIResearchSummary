通过 API 中的新模型推进语音智能

Advancing voice intelligence with new models in the API

一句话总括：OpenAI 的贡献不是只发布更自然的声音，而是把实时语音变成能推理、调用工具、翻译和转写的产品运行层。

3个新音频模型进入 Realtime API，覆盖语音代理、实时翻译和流式转写。

核心变化

语音接口正在从“轮流说话”变成“边聊边办事”

文章把语音 AI 的门槛重新定义为任务能力。好用的语音代理需要听懂意图，记住上下文，处理用户改口，还要在不中断对话的情况下调用工具。

这改变了读者对 AI 影响的理解。语音不再只是输入方式，而是软件行动的前台界面。

三种产品模式

Voice-to-action：用户说出目标，系统推理、查工具并完成任务。

Systems-to-voice：软件把后台状态转成即时语音提示，例如航班延误后的路线建议。

Voice-to-voice：跨语言、跨任务、跨上下文的实时对话继续进行。

结论是，语音 AI 的商业价值来自任务闭环，而不是单点识别准确率。

GPT‑Realtime‑2 把代理行为做进了实时对话

它支持 preambles、并行工具调用、工具透明提示、更强恢复、更长上下文、专业术语保持和可调 reasoning effort。每个能力都指向同一个趋势：用户不该等待一个沉默的系统。

1先回应

短句提示“我来查一下”，让用户知道请求已被接住。

2同时查

模型可并行调用多个工具，减少实时任务中的停顿感。

3说清动作

把“正在查日历”等动作说出来，提升可理解性。

4恢复失败

出错时给出自然解释，而不是静默或断线式失败。

5继续推进

128K 上下文让长会话和复杂任务更连贯。

风险与信任
实时语音越像人类协作，安全边界就越要进入会话本身，而不是只停留在事后审核。

OpenAI 描述了多层防护，包括 Realtime API 会话中的主动分类器。某些违反有害内容规则的对话可以被中止。

开发者还可以用 Agents SDK 增加自己的 guardrails。政策要求也很明确：不能把输出用于垃圾信息、欺骗或其他有害目的，并且通常要让终端用户知道自己在和 AI 互动。

这让读者看到一个关键点：语音代理进入生产环境后，低延迟和合规同样是基础设施问题。

价格与可用性决定了落地节奏

GPT‑Realtime‑2

音频输入 $32 / 1M tokens，缓存输入 $0.40 / 1M tokens，音频输出 $64 / 1M tokens。

GPT‑Realtime‑Translate

$0.034 / 分钟，适合实时客服、教育、活动、媒体和创作者平台。

GPT‑Realtime‑Whisper

$0.017 / 分钟，面向低延迟字幕、会议记录和持续语音理解。

结论是，OpenAI 把语音能力拆成不同成本结构。开发者可以按任务选择推理、翻译或转写，而不是用一个模型承担所有工作。

最值得带走的洞见

语音正在成为操作系统级入口

文章最重要的信号是，用户可以通过自然语言直接触发软件动作。未来的语音体验会更像协作界面，而不是语音搜索框。

实时性不是只看延迟

真正的实时包括边说边理解、边查工具、边恢复错误。响应方式会影响用户是否信任代理。

多语言体验进入现场生产

70+ 输入语言和 13 种输出语言让实时翻译适合客服、活动和教育场景。跨语言沟通不必等待后期制作。

安全会成为语音代理的产品功能

实时会话中的主动分类器、开发者 guardrails 和 AI 身份披露，会直接影响语音产品能否进入敏感行业。
