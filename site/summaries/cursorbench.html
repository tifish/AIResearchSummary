我们如何在 Cursor 中比较模型质量

How we compare model quality in Cursor

Cursor 的真正贡献不是又发布一个榜单，而是把离线真实任务、线上产品实验和持续刷新机制连成一个闭环，用来衡量编码代理对开发者是否真的有用。

数月CursorBench 每隔几个月刷新一次，以跟上开发者使用代理的变化。

公开基准的问题，不只是分数饱和

文章指出，公共 benchmark 越来越难反映开发者真实体验。许多任务来自公开仓库，问题描述也更像已整理好的工单。真实用户给代理的请求更短、更模糊，也更依赖上下文。

这会改变我们理解 AI 能力的方式。一个模型在标准题上拿高分，不等于它能处理多文件、多工具和多步骤的真实开发任务。

对齐风险：许多 SWE 基准仍偏向修 bug，Terminal-Bench 又包含偏谜题式任务。这些并不代表日常编码代理请求。
评分风险：真实需求常常欠规格化。固定答案会惩罚合理替代方案，额外添加要求又会扭曲原问题。
污染风险：公开仓库任务可能进入训练数据。模型可能记住补丁，而不是现场解决问题。

CursorBench 的构造方式

Cursor 用 Cursor Blame 把提交代码追溯到产生它的代理请求。这样可以自然得到“开发者请求”和“真实解决方案”的配对。

从真实会话取样。
任务来自 Cursor 工程团队的真实使用，而不是公开仓库题目。
保留短请求。
任务描述故意保持简短，贴近开发者和代理对话时的模糊表达。
使用 agentic graders。
评分器需要判断多种可能的正确解，而不是只匹配单一补丁。
持续刷新。
套件每几个月更新，避免评价停留在旧工作流上。

它比公开 benchmark 更能拉开前沿模型差距

比较维度公开离线基准CursorBench
任务来源 多来自公开仓库或标准化问题集，存在训练污染风险。 来自真实 Cursor 会话、内部代码库和受控来源，污染风险更低。
任务形态 常以详细 GitHub issue 或固定题面出现。 更接近开发者对代理说话的方式，短、模糊、依赖上下文。
模型区分度 前沿模型上容易饱和，甚至出现较小模型追平高端模型的现象。 在前沿水平上显示更大分离度，更接近开发者实际感受。

离线分数必须接受线上体验校验。 一个答案看起来能通过评分器，不代表它在产品里让开发者更高效。

线上评估补上离线盲区

Cursor 通过受控线上实验观察代理改动是否真的改善开发者体验。它不只优化单一指标，而是同时看交互质量和输出质量的代理信号。

语义搜索实验。
Cursor 曾在迭代检索能力时移除语义搜索工具做消融实验。结果帮助定位语义搜索最重要的场景，例如大代码库中的仓库问答。
产品体验。
线上指标能捕捉“评分不错但用起来变差”的回归。
排序可信度。
CursorBench 排名更接近开发者在 Cursor 中感受到的模型质量。
评估目标。
真正要测的是代理能否完成实际工作，而不是能否适应某个榜单。
方法结论。
离线和线上结合，才更能跟上工作流变化。

评估会走向长时间运行的代理

CursorBench-3 的任务已经比公开 benchmark 更长，但仍在单个会话内解决。文章预计未来一年，大多数开发工作会转向在自己电脑上长时间运行的代理。

这会带来新的评估难题。评分要更便宜，外部服务交互要可复现，离线评估还要继续缩小与开发者体验之间的距离。

最值得带走的洞见

  • 编码代理评估不能只看公开题库，因为真实开发请求更长、更模糊，也更依赖上下文。
  • 训练污染会让公开基准高估模型能力，尤其是在前沿模型已经能记住部分补丁时。
  • CursorBench 的价值在于用真实会话构造任务，并且定期刷新任务分布。
  • 线上实验是必要校验。它能发现离线评分器看不出的产品体验回归。
  • 下一代评估会面向长时间运行的代理，核心难点将是低成本评分和可复现环境。
