我们如何在 Cursor 中比较模型质量

How we compare model quality in Cursor

一句话总括：Cursor 把真实开发会话制成离线评测集，再用线上受控实验校准结果，从而让模型排名更接近开发者真正感受到的质量。

核心判断

排行榜高分，不一定代表真实开发体验更好

前沿模型正在让公开 benchmark 趋于饱和。某些公开榜单甚至会显示 Haiku 追平或超过 GPT-5，但开发者在 Cursor 中感受到的能力差异依然明显。

CursorBench 能拉开这些模型的差距。其排序也更贴近线上评估所反映的实际体验。

公开评测的三类失真

  1. 任务错位：许多 SWE benchmark 仍以修复缺陷为主，Terminal-Bench 还包含棋局等解谜任务。这与真实编码工作的范围不一致。
  2. 评分过窄：真实请求往往没有唯一答案。固定答案会错罚合理方案，附加人为约束又会扭曲原任务。
  3. 数据污染：公开仓库可能进入训练数据。模型因此可能凭记忆复现标准补丁，而不是真正解决问题。

Cursor 的线上—离线闭环

01 · 从真实会话取样Cursor Blame 把已提交代码追溯到产生它的智能体请求，形成开发者问题与真实解法的自然配对。
02 · 多维离线评估CursorBench 不只衡量答案是否正确，也检查代码质量、效率和交互行为。
03 · 用线上实验校准真实流量中的受控实验可发现离线评分遗漏的退化，并验证改进是否真的帮助了开发者。
评测的价值不在于制造一个绝对分数，而在于持续回答：模型在真实工作中是否更有用？

更难，也更像开发者真正提出的问题

CursorBench 的任务描述刻意保持简短。它保留真实请求中常见的模糊性，再由智能体评分器判断不同的有效解法。

CursorBench-3 纳入了多工作区与 monorepo、生产日志调查以及长时间实验等任务。其代码规模明显超过 SWE-bench Verified、Pro 和 Multilingual。

代码行数并非完美的难度指标。但规模增长表明，评测对象已经从局部修补走向跨文件、跨工具、跨步骤的完整工作流。

最值得带走的洞见

真实任务来源比题目数量更重要。从实际开发会话抽取任务，可以同时提高评测的相关性，并降低公开数据污染风险。
不能跨版本硬比分数。CursorBench 3.1 改变了问题分布，因此只能在同一评测版本内比较结果。
离线正确不等于产品好用。开发者可能觉得一个被评分器判为正确的答案仍然难用，线上信号正是为捕捉这种差距而存在。
下一代评测要面向长期运行的智能体。它需要降低评分成本，解决外部服务交互的可复现性，并进一步缩小离线结论与真实体验之间的距离。

说明：随着智能体能力演进，CursorBench 也在持续更新。目前的生产版本是 CursorBench 3.1；最新排行榜请参阅该版本的页面。

开发者正让编码智能体承担持续时间更长、复杂度更高的任务。这些任务会跨越多个文件，使用多种工具，并经过多个步骤。随着请求范围扩大，用来衡量智能体表现的评测也必须随之演进。

在 Cursor，我们采用线上与离线相结合的评测流程。这样可以让我们对模型质量的理解，始终贴近开发者真正开展的工作。

离线部分使用 CursorBench。它是我们的内部评测套件，基于工程团队真实的 Cursor 会话构建。任务来自 Cursor 的实际使用，而不是公开仓库。因此，相比公开 benchmark，CursorBench 更能区分不同模型，也更符合开发者的真实结果。

我们构建 CursorBench，是为了衡量智能体表现的多个维度。其中包括方案正确性、代码质量、效率和交互行为。本文重点介绍方案正确性的结果，但在实际工作中，我们会从所有这些维度评估智能体。

2026 年 5 月更新：我们后来把 CursorBench 升级到了 3.1，并加入了难度更高的问题。由于问题分布已经改变，CursorBench 3.1 的分数可能与本文中的数字和图表不同。分数应当在相同的评测版本内比较。

我们还会使用真实流量上的受控分析来补充 CursorBench。这些线上评估能够发现离线套件遗漏的退化。例如，智能体的输出在评分器看来可能正确，但开发者在产品中使用时却觉得体验更差。

线上与离线组成的闭环，让我们对模型质量的判断始终立足于生产环境。即使工作流发生变化，我们也能据此在 Cursor 中打造尽可能好的智能体体验。

公开 benchmark 的局限

一个好的 benchmark 需要区分那些在实践中表现不同的模型。同时，它还要符合开发者对这些模型的真实体验。公开的离线评测在这两方面都遇到了困难。

第一个问题是对齐。开发者正在借助智能体处理越来越复杂、越来越多样的工作。静态或错位的 benchmark 最终可能完全测错了对象。例如，多数 SWE benchmark 仍然聚焦于缺陷修复任务。Terminal-Bench 同样强调范围很广的解谜式任务，例如根据棋盘局面寻找最佳落子。我们发现，这些任务与开发者交给智能体的编码工作并不契合。

第二个问题是评分。许多公开 benchmark 任务假定只有一组范围很窄的正确答案。然而，大多数开发者请求都没有给出完整限定，因此可能存在多种有效做法。结果是，benchmark 要么惩罚其他同样正确的方法，要么加入人为编造的要求，以消除问题中的不确定性。这两种办法都无法准确衡量真实表现。

第三个问题是污染。SWE-bench Verified、Pro 和 Multilingual 的任务都来自公开仓库，而这些内容最终会进入模型训练数据，从而抬高分数。OpenAI 最近已经完全停止报告 SWE-bench Verified 的结果。其调查发现，前沿模型可以凭记忆复现标准补丁，而尚未解决的问题中有近 60% 存在测试缺陷。

因此，在前沿能力水平上，这些 benchmark 已无法区分那些对开发者实际价值差异很大的模型。

构建 CursorBench

我们通过 Cursor Blame 为 CursorBench 收集任务。Cursor Blame 可以从已提交的代码追溯到生成这段代码的智能体请求。这样，我们便能自然地获得开发者查询与真实解决方案的配对。许多任务来自我们的内部代码库和受控来源，可以降低模型在训练期间见过这些任务的风险。我们每隔几个月刷新一次套件，以跟上开发者使用智能体方式的变化。

从最初版本到当前的 CursorBench-3，我们正确性评测中的问题规模约扩大了一倍。代码行数和平均文件数都呈现了这一变化。CursorBench-3 任务涉及的代码行数明显多于 SWE-bench Verified、Pro 或 Multilingual。代码行数并不是衡量难度的完美指标，但这一指标的增长反映出我们已经纳入更具挑战性的任务。例如处理包含 monorepo 的多工作区环境、调查生产日志，以及执行长时间运行的实验。

CursorBench 任务也保留了开发者与智能体交流时那种限定不足、经常带有歧义的表达方式。与公开 benchmark 中详细的 GitHub issue 相比，我们刻意让任务描述保持简短，并使用智能体评分器进行可靠评分。

CursorBench 能更明显地区分模型

任务复杂度和说明方式上的这些差异，会直接影响 benchmark 的实用价值。在公开 benchmark 日益饱和的前沿水平上，CursorBench 可以更明显地拉开模型之间的差距。在某些情况下，公开 benchmark 甚至会显示 Haiku 能够追平或超过 GPT-5。对于开发者在使用中认为存在显著差异的模型，CursorBench 可以可靠地区分它们。

CursorBench 分数与线上评估一致

线上评估用于衡量智能体的改进是否真的能在实践中帮助开发者。我们跟踪一组反映智能体结果的高层指标，其中既有交互质量信号，也有输出质量信号。我们关注这些指标是否出现一致变化，而不是只针对某一个指标进行优化。把这些信号汇总起来，可以帮助我们发现一类退化：智能体的输出在离线评分器下表现很好，却无法真正为开发者带来良好效果。

我们使用受控的线上实验来判断影响来自哪里。例如，在迭代语义搜索和检索功能时，我们做了一次消融实验，完全移除了语义搜索工具。这使我们能够准确找出语义搜索最重要的场景，例如在大型代码库中，依据仓库内容回答问题。

根据我们的线上评估指标，CursorBench 的排名也更接近开发者在 Cursor 中实际感受到的模型质量。

下一套评测体系

CursorBench-3 的任务虽然比公开 benchmark 更长，但仍能在单次会话中完成。我们预计，未来一年中，绝大多数开发工作会转向长期运行的智能体。这些智能体将在自己的计算机上工作。我们正计划据此调整 CursorBench。

要做到这一点，我们需要找到降低评分成本的方法。我们还要解决涉及外部服务的任务如何复现，并缩小离线评估与开发者体验之间的差距。

线上与离线组成的闭环，提供了我们认为正确的基础。随着后续工作的推进，我们计划分享更多信息。
