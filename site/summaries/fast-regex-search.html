快速正则搜索：为智能体工具建立文本索引

Fast regex search: indexing text for agent tools

一句话总括：Cursor 把正则搜索从“扫描整个代码库”改造成“先用本地索引筛选候选文件，再做确定性匹配”，让智能体在大型仓库中更快获得准确且足够新鲜的文本上下文。

来源：Cursor　·　日期：2026 年 3 月 23 日　·　作者：Vicent Marti

先看真正的问题：快模型仍会被慢搜索卡住

ripgrep 已经把单个文件内的正则匹配做得很快。但它仍要读取所有文件。仓库越大，搜索延迟就越明显。对于频繁调用搜索工具的智能体，这不是一次性的等待，而是反复进入推理链路的系统性瓶颈。

风险：扫描成本会随仓库扩大

grep 是少数会直接受代码库规模和复杂度影响的智能体操作。模型越快，这段等待越突出。缺陷调查尤其依赖多轮搜索，因此更容易被延迟拖慢。

结论：只优化正则引擎并不够。必须减少真正需要扫描的文件数量。

机会：把文本检索也变成本地索引操作

索引不直接给出最终答案。它先筛出可能匹配的文件，再由正则引擎读取原文并确认结果。

结论：索引允许出现假阳性，却不能漏掉真匹配。这样既能提速，也不牺牲正确性。

技术路线不是凭空出现的

  1. 三元组倒排索引：用重叠的 3 字符片段建立 posting list。它在键数量和列表长度之间取得折中。
  2. 后缀数组：通过二分搜索定位匹配位置。查询能力强，但动态更新和多文件映射成本很高。
  3. 概率掩码：为三元组附加后继字符和位置掩码。它能像四元组一样收窄候选，但 Bloom filter 会饱和。
  4. 稀疏 N-gram：索引阶段多做工作，查询阶段只查覆盖匹配所需的少量片段。Cursor 选择它作为本地方案的基础。

洞见：这些方案的差异不只是查询速度。索引体积、更新代价和候选集精度同样决定它能否服务持续改写代码的智能体。

性能

智能体的实际速度取决于工具链，而不只取决于模型 TPS。一次超过 15 秒的搜索足以抵消快速生成带来的体验优势。

方法

索引的职责是安全地排除不可能匹配的文件。最终结果仍由原文上的正则匹配确定。

更新

文本索引必须比语义索引更新得更及时。智能体若搜不到自己刚写入的精确文本，可能浪费更多搜索和推理步骤。

架构

本地索引同时减少网络延迟、文件同步成本和代码托管风险。对于大型私有仓库，这些收益相互强化。

时间仿佛是一个循环。第一版 grep 于 1973 年发布。它当时只是一个基础工具，用来在文件系统的文本文件中匹配正则表达式。

随着开发工具日益先进，grep 逐渐被更专业的工具取代。最先出现的是 ctags 这类粗略的语法索引。后来，许多开发者转向针对特定编程语言打造的 IDE。这些 IDE 会解析代码并建立语法索引，通常还会补充类型层面的信息，因此能让开发者高效浏览代码库。

最终，这套能力通过语言服务器协议（Language Server Protocol，LSP）实现了标准化。新旧文本编辑器都能使用这些索引。可就在 LSP 即将成为行业标准时，智能体编程出现了。结果颇为有趣：智能体非常喜欢使用 grep。

当然，还有其他先进技术可以为智能体收集上下文。我们过去讨论过，语义索引能在许多任务中大幅改善智能体表现。但有些查询只能由模型借助正则表达式搜索来解决。这意味着我们又回到了 1973 年，尽管这个领域此后已经进步了不少。

包括 Cursor 在内，大多数智能体运行框架都会默认使用 ripgrep 提供搜索工具。它是 Andrew Gallant 开发的独立可执行程序，可以替代经典的 grep。它提供了更合理的默认行为，例如忽略文件的方式，而且性能高得多。

ripgrep 以速度快而闻名。这是因为 Andrew 花了大量时间研究如何提升正则表达式匹配速度。

然而，无论 ripgrep 匹配单个文件内容的速度有多快，它都有一个严重限制：它必须检查所有文件的内容。

在小型项目中，这不成问题。但 Cursor 的许多用户，尤其是大型企业客户，都在规模极大的单体仓库中工作。那些仓库大得令人痛苦。

我们经常看到一次 rg 调用耗时超过 15 秒。对于正在主动指导智能体编写代码的人来说，这会严重打断工作流程。

正则表达式匹配已经成为智能体开发中的关键环节。我们认为，必须专门优化这项操作。传统 IDE 会在本地建立语法索引，以支持“转到定义”等操作。与此类似，我们正在为现代智能体查找文本时最核心的操作建立索引。

经典算法

通过索引文本数据来加快正则表达式匹配，并不是新想法。Zobel、Moffat 和 Sacks-Davis 在 1993 年发表的论文《使用压缩倒排文件搜索部分指定的词项》（Searching Large Lexicons for Partially Specified Terms using Compressed Inverted Files）中首次提出了这种方法。

他们使用 N-gram，也就是宽度为 N 个字符的字符串片段，来建立倒排索引。他们还提出了一些启发式方法，把正则表达式分解为一棵 N-gram 树，再通过索引查询这些片段。

如果你以前听说过这个概念，更可能是从 Russ Cox 于 2012 年发表的一篇博客文章中了解到的。那篇文章发布于 Google Code Search 关闭后不久。

下面快速回顾一下这类索引的基本组成。此后开发的其他索引方法，基本都沿用了这些构件。

倒排索引

倒排索引是搜索引擎背后的基础数据结构。

建立倒排索引时，首先要把每份待索引文档拆分成 token。这个过程称为分词（tokenization），实现方式有很多。为了便于说明，这里采用最简单的方式：把每个单词视为一个 token。

这些 token 会成为类似字典的数据结构中的键。每个键对应的值，是所有包含该 token 的文档列表。

这份列表通常称为 posting list。每份文档都会由一个唯一数字标识，这个数字称为 posting。

搜索一个或多个 token 时，我们会加载相应的 posting list。如果列表不止一个，就计算它们的交集，从而找出同时出现在所有列表中的文档。

编辑文档，观察索引如何更新。单击某个词项即可选中并查看它。

将鼠标悬停在词项上，或轻触词项，即可追踪同一个索引条目。

每个词项都会映射到包含它的文档。悬停或轻触即可逐一查看条目。

搜索词项，查找匹配的文档。

可以尝试搜索 cat、the 或 ran。

如今大多数搜索引擎都以这种设计为基础，只是在上面增加了大量复杂机制。

然而，这些搜索引擎面向的是自然语言。我们要搜索的却是正则表达式，而且匹配对象是源代码。这套方法并不完全适用。

一种可能的办法，是在分词上投入大量精力。系统可以理解每种编程语言的语法，也可以拆分源代码中的标识符。但这很难做好。

GitHub 早期的 Code Search 就采用过这条路线。它使用非常复杂的编程语言分词器，并依赖规模庞大的 ElasticSearch 集群。

搜索效果并不好，用户对这个功能的评价也很差。它勉强能搜索标识符，却无法匹配正则表达式。要解决这个问题，必须采用更好的分词方式。

三元组分解

对源代码进行朴素分词，不适合正则表达式匹配。我们需要把文档拆成更基础的片段。

经典算法选择三元组（trigram）。输入字符串中每个相互重叠的三字符片段，都会成为一个 token。

为什么是三个字符？因为这些三元组将作为倒排索引的键。

如果选择二元组，也就是长度为 2 的片段，索引中的键会很少，最多约为 64K。但每个键对应的 posting list 会非常庞大，难以高效处理。

如果选择四元组，也就是长度为 4 的片段，posting list 会很小。这当然很好，但倒排索引可能出现数十亿个键，同样难以处理。

因此，三元组是一个不错的折中。建立文档索引时，分词过程也很简单：提取文档中所有相互重叠的三字符序列，并把它们作为倒排索引中的 token。

真正复杂的部分，是如何对正则表达式分词，使它能够通过索引进行匹配。

正则表达式包含语法。系统必须先解析表达式，再用启发式方法判断哪些实际表示文本的片段能够提取出三元组。

把字面字符串分解成三元组很直接，算法与建立文档索引时相同。只需提取字符串中的所有重叠三元组。

包含全部这些三元组的文档，很可能包含这个字面字符串，但并非一定如此。

选择分支需要分别分解。分解后会产生两个分支，只要文档包含其中一个分支，就可能匹配。查询倒排索引时，我们会合并 posting list，而不是计算交集。

字符类可以分解成多个三元组。像 [rbc]at 这样的小型字符类，会为类中的每个元素生成一个三元组。

对于范围更广的字符类，我们会直接跳过跨越这些边界的三元组提取。

组合成完整方案

现在，我们知道三元组适合用来切分文档，也知道如何在建立索引时切分文档，以及如何在查询时切分表达式。把这些部分组合起来，就能得到一个高效匹配正则表达式的搜索索引。

系统先把正则表达式分解成一组三元组，再从倒排索引中加载相关的 posting list。这样会得到一份可能匹配该正则表达式的文档列表。

这里必须强调“可能匹配”。最终结果仍然需要加载所有候选文档，并用传统方式执行正则表达式匹配。

不过，扫描候选文档子集始终比逐个扫描并匹配整个代码库更快。

编辑文档，观察索引如何更新。单击三元组即可选中并查看它。

将鼠标悬停在三元组上，或轻触三元组，即可追踪同一个索引条目。

每个三元组都会映射到包含它的文档。悬停或轻触即可逐一查看条目。

使用字面字符串或正则表达式搜索，观察三元组如何分解。

可以尝试搜索 cat、the rat，或正则表达式 c[au]t。

无论从哪个角度看，这套设计都完全可用。google/codesearch 和 sourcegraph/zoekt 等项目，都使用三元组倒排索引为大型索引提供了良好性能。与所有搜索引擎一样，它们还在基础设计之上增加了许多复杂机制。

但这套方案也有明显缺点。索引体积并不小，而且查询时的分解策略必须做出取舍。

如果采用简单的启发式方法，查询只会被分解成少量三元组。这样会产生大量候选文档。

如果采用复杂的启发式方法，查询可能被分解成几十个，甚至数百个三元组。从倒排索引中加载所有这些条目，可能和从头搜索全部内容一样慢。

我们还可以做得更好。

后缀数组：一段题外话

既然正在回顾正则表达式文本索引的发展历史，我想稍微离开主线，谈谈 Nelson Elhage 在 2015 年为其 livegrep Web 服务开发的实现。

与其他大型产业项目相比，livegrep 的规模很小。它只索引 Linux Kernel 的最新版本。

但正因为范围有限，它的实现与其他方案很不一样。这也让它格外有趣，值得讨论。

Nelson 从第一性原理出发解决问题。这个搜索引擎背后没有倒排索引。相反，所有源代码都被索引进一个后缀数组。

后缀数组的概念从名称就能看懂：它是一个字符串所有后缀按顺序排列而成的数组。

如果尝试为较长的字符串建立后缀数组，就会发现这个数据结构增长得很快。它看起来是一种成本很高的索引，很多方面也确实如此。

不过，如果能够访问原始字符串，它的存储可以得到很好的压缩。只需保存每个后缀起始位置的偏移量即可。

为待搜索的语料建立后缀数组后，就可以把正则表达式分解成字面量，从而高效执行搜索。

随后，可以在后缀数组上执行二分搜索，找出正则表达式所有可能的匹配位置。

尝试搜索 th 这样的短字符串，观察二分搜索如何圈定文档中的所有匹配位置。

利用后缀数组的相同性质，还能匹配正则表达式中的复杂结构。

例如，要匹配 [a-z] 这样的字符范围，可以分别对范围起点和终点执行二分搜索。两个端点之间的内容必然符合这个范围。

输入一个字符串来建立后缀数组。字符串中的每个位置都会定义一个后缀。

将鼠标悬停在某个后缀行上，或轻触该行，即可查看该后缀在字符串中的起始位置。

所有后缀按字典序排列。数组只保存索引，后缀内容由原始字符串推导得出。

这套方案有什么缺点？后缀数组必须根据一个输入字符串建立。这是很大的限制。

如果要索引大型代码库，甚至多个不同代码库，首先需要把全部内容连接成一个字符串，再基于它建立后缀数组。

在后缀数组中匹配时，还需要辅助数据结构，把匹配位置映射回包含该位置的原始文件。

这些复杂问题并非无法解决，但会让索引的动态更新变得非常昂贵。这是一种很难扩展的方案。

使用概率掩码的三元组查询

现在回到更传统的设计。下面介绍一种最初由 GitHub 为 Project Blackbird 开发的方法。

这是一个旨在替代旧版 Code Search 的研究项目。前面提到，旧版搜索通过切分源代码来实现，无法匹配正则表达式。新实现的目标，就是加入这种能力。

早期版本尝试采用经典的三元组倒排索引，但很快遇到了容量问题。

GitHub 上有海量代码。使用三元组建立索引后，posting list 大得难以搜索。

既然三元组不够理想，下一步就是为索引寻找更合适的 N-gram 长度。

我们已经看到，二元组覆盖范围太宽，posting list 会大到难以管理。四元组又过于具体，会让索引产生太多键。

三元组处在两者之间。但在实际应用中，理想长度更像是“三点五元组”。可是，我们总不能把一个字符切成两半吧？

事实上，我们可以实现非常接近的效果。

这套设计仍用三元组作为倒排索引的键，但会在 posting list 中加入额外信息，记录特定文档中跟在该三元组之后的“第四个字符”。

我们可以直接用一个额外字节保存第四个字符。但这样会让索引变成四元组索引，而前面已经说明，四元组索引太大。

因此，实际保存的是一个 Bloom filter。它包含跟在特定三元组之后的所有字符。

每个三元组都带有一个 8-bit locMask（位置对 8 取模）和一个 nextMask（每个后继字符的哈希）。

将鼠标悬停在三元组上，或轻触三元组，即可追踪索引中的同一个条目。

每个（三元组，文档）条目保存两个 8-bit 掩码。悬停或轻触即可逐一查看三元组。

你可能觉得 Bloom filter 是一种庞大而复杂的数据结构，但它不一定如此。Bloom filter 可以压缩到很少的几个 bit。

如果编码足够仔细，8 bit 也能容纳很多信息。每个 posting 只需增加两个字节，就能绕开经典三元组索引的两个最大问题。

有了保存三元组后继字符的掩码，倒排索引仍可使用三元组作为键，却能使用四元组进行查询。

与简单的三元组索引相比，这已经可以大幅缩小候选文档范围。

第二个增强掩码保存三元组在文档中的位置偏移。它可以解决三元组的歧义问题。

文档包含两个三元组，不代表它们在文档中彼此相邻。但要匹配查询，它们必须相邻。

我们把第二个三元组的位置掩码左移一位，再与第一个三元组的掩码比较。这样就能确认两者确实相邻。

对于非常常见的三元组，这种能力尤其重要。它可以进一步缩小候选文档列表。

这些信息当然都是概率性的。与 Bloom filter 中保存的任何信息一样，它可能产生假阳性。

但在这里，假阳性完全可以接受。最终匹配仍会在文本本身上以确定性方式执行。索引的目标，只是尽量减少需要扫描的候选文档数量。

最终得到的索引效率极高，但存在一个重大缺点：Bloom filter 可能饱和。

这是 Bloom filter 令人遗憾的特性。它可以更新，但如果加入过多数据，最终过滤器中的所有 bit 都会被置位。

Bloom filter 一旦饱和，就会匹配所有内容。这样又会退回到最初那种索引的性能水平。

这套索引最大限度地减少了存储空间，却很难高效地进行原地更新。

稀疏 N-gram：更聪明地选择三元组

下面是另一个非常巧妙的想法。你可能在 ClickHouse 的正则表达式运算符中见过它。GitHub 几年前推出的新版 Code Search 也采用了它，并且支持正则表达式匹配。

它叫作稀疏 N-gram（Sparse N-grams），是各种折中方案中最出色的一种。

传统三元组索引会提取每一个连续的三字符序列。这样会产生大量冗余，因为每个三元组中的字符都会在相邻三元组中重复出现。

这套算法会提取数量不定的 N-gram，而且每个 N-gram 的长度也不固定。

这里的“随机”当然不能是真正的随机，否则索引就无法查询。

我们会为文档中的每一对字符分配一个“权重”。只要结果具有确定性，权重可以用任何方式计算。ClickHouse 使用两个字符的 crc32 哈希。

随后，我们选择这样的子字符串作为稀疏 N-gram：子字符串两端的权重，都严格大于内部的所有权重。

关键在于，稀疏 N-gram 可以具有任意长度。它们并不统一。

这也意味着，算法可能生成很多 N-gram，甚至比单纯提取三元组还多。

不过，N-gram 是以确定性方式生成的，因此查询时可以进行一些非常重要的优化。下面看看它如何工作。

这套算法并不容易理解，所以需要通过交互来观察。你可以使用可视化界面中的前进和后退箭头，逐步查看运行过程。

输入内容的字符分解上方，会显示分配给每对字符的随机权重。这些权重决定了哪些片段会被提取为 N-gram。

最下方区域会显示输入字符串提取出的稀疏 N-gram 数量，以及采用二元组、三元组或四元组时会提取的数量。差异非常明显：算法确实提取了大量稀疏 N-gram。

那么，这到底有什么意义？我们只是在做一件愚蠢的事吗？并不是。

我们在建立索引时承担较高的前期成本，以换取查询阶段的极快速度。

当前展示的 build_all 算法用于索引文档。它会从输入中提取所有可能的稀疏 N-gram。

但查询时不必这样做。因为权重虽然像随机值，却具有确定性，所以查询时可以使用覆盖算法，只生成在索引中完成匹配所需的最少 N-gram。

我们知道这些 N-gram 已经是最少集合。原因在于，建立索引时，只有片段内部的所有权重都低于两端权重，算法才会生成该 N-gram。

因此，查询时只需提取边缘位置的稀疏 N-gram。它们的数量远少于全部三元组，同时仍能以很高的精度筛选候选文档。

还能做得更好吗？可以，而且能好很多。

前面的示例使用 crc32 作为权重函数。不过，只要结果具有确定性，任何哈希函数都可以使用。

我们可以选择一种更聪明的函数：实际非常罕见的字符对获得高权重，非常常见的字符对获得低权重。

这种函数很容易计算。因为索引对象是源代码，我们可以从互联网上收集数 TB 的开源代码，再统计其中所有字符对的频率表。

这张频率表就是我们的哈希函数。

把它应用到算法后，最高权重会出现在最不常见的字符对下方。因此，覆盖模式需要查询的 N-gram 会更少，可能匹配的文档也会更少。

这种方案能最大限度地减少 posting 查询次数。它非常适合作为起点，用来构建可在用户电脑上高效查询的索引。

把这一切放进你的电脑

用于加快正则表达式搜索的索引总要存放在某个地方。

此前看到的所有设计都部署在服务端。我们讨论过的语义索引，也由服务端负责管理和查询。

但这一次，我们选择了不同方向：在用户的电脑上建立并查询索引。

把索引留在本地有几个合理的原因。

首先，索引只是正则表达式匹配所需的一部分。它能筛出可能匹配的文档子集，但系统仍要逐一扫描其中的文件。

如果在服务端完成这项工作，就必须同步所有文件，或者在服务端与客户端之间进行成本高昂的多次往返。

在客户端执行则很简单。它还能避开数据存储相关的许多安全和隐私问题。

延迟对这项功能也十分重要。

我们的 Composer 模型拥有业界领先的每秒 token 数（TPS）表现之一。我们也一直在努力让它变得更聪明、更快。

模型会持续调用搜索这项关键操作，而且经常并行调用。为它加入网络往返只会增加阻力和停顿，与我们改善智能体交互体验的目标背道而驰。

与语义索引不同，正则表达式搜索索引还必须非常新鲜。模型能否读到自己刚写入的内容，尤其重要。

语义索引不必持续更新。文件修改后，即使重新计算 embedding，新向量通常也不会在多维空间中发生显著位移。最近邻搜索仍会把智能体引向正确方向。

但如果智能体搜索某段具体文本却没有找到，它往往会开始无效追查。这会浪费 token，也会从根本上抵消性能优化的意义。

把索引带到客户端，也会产生新的挑战。同步磁盘数据既复杂又昂贵，但我们在实践中把它做得很高效。

我们以底层 Git 仓库中的一个 commit 为基准，控制索引状态。用户和智能体的改动会作为叠加层存储在它之上。

因此，索引可以快速更新，也能在启动时快速加载和同步。

为了尽量降低编辑器中的内存占用，我们把索引存储在两个独立文件中。

第一个文件依次保存索引的所有 posting list。构建索引时，我们会把它们直接刷新到磁盘。

另一个文件保存一张排序表，其中包含所有 N-gram 的哈希，以及对应 posting list 在 posting 文件中的偏移量。

这里只保存哈希，不保存完整 N-gram，而且始终是安全的。

如果两个哈希发生碰撞，posting list 的覆盖范围可能变大。这在实践中极少发生，但即使发生，也不会产生错误结果。

这种方式还能让查找表的布局非常紧凑。

随后，我们在编辑器进程中只对这张表执行 mmap，并使用二分搜索处理查询。

搜索会返回一个偏移量。系统再直接读取 posting 文件中对应偏移位置的数据。

将鼠标悬停在查找表中的某一行，或轻触该行，即可追踪对应 posting 在磁盘上的位置。

结论

我们发现，为 Composer 2 这类快速模型提供文本搜索索引，会给智能体工作流带来质的变化。

这种影响在大型企业仓库中更加明显。grep 是少数延迟会随代码规模和复杂度增长的智能体操作。

可以看看这些由 Composer 2 运行的示例工作流。彻底消除代码库搜索耗时，能节省可观的时间。

智能体调查缺陷时，收益尤其明显。搜索加快后，迭代也会有效得多。

切换模式，然后将鼠标悬停在某个片段上，或轻触该片段，即可查看它的持续时间。

下一步会是什么？谁也不知道。

我们会继续优化现有方法的性能，其中也包括语义索引。

我们还希望推出全新的方法，进一步提升智能体表现。同时，我们会始终确保这些方法能在最重要的环境中正常运行。

这里所说的环境，就是世界上规模最大的代码仓库。智能体开发的未来，正在这些仓库中真正形成势头。

时间仿佛是一个循环。第一版 grep 于 1973 年发布。它当时只是一个基础工具，用来在文件系统的文本文件中匹配正则表达式。

随着开发工具日益先进，grep 逐渐被更专业的工具取代。最先出现的是 ctags 这类粗略的语法索引。后来，许多开发者转向针对特定编程语言打造的 IDE。这些 IDE 会解析代码并建立语法索引，通常还会补充类型层面的信息，因此能让开发者高效浏览代码库。

最终，这套能力通过语言服务器协议（Language Server Protocol，LSP）实现了标准化。新旧文本编辑器都能使用这些索引。可就在 LSP 即将成为行业标准时，智能体编程出现了。结果颇为有趣：智能体非常喜欢使用 grep。

当然，还有其他先进技术可以为智能体收集上下文。我们过去讨论过，语义索引能在许多任务中大幅改善智能体表现。但有些查询只能由模型借助正则表达式搜索来解决。这意味着我们又回到了 1973 年，尽管这个领域此后已经进步了不少。

包括 Cursor 在内，大多数智能体运行框架都会默认使用 ripgrep 提供搜索工具。它是 Andrew Gallant 开发的独立可执行程序，可以替代经典的 grep。它提供了更合理的默认行为，例如忽略文件的方式，而且性能高得多。

ripgrep 以速度快而闻名。这是因为 Andrew 花了大量时间研究如何提升正则表达式匹配速度。

然而，无论 ripgrep 匹配单个文件内容的速度有多快，它都有一个严重限制：它必须检查所有文件的内容。

在小型项目中，这不成问题。但 Cursor 的许多用户，尤其是大型企业客户，都在规模极大的单体仓库中工作。那些仓库大得令人痛苦。

我们经常看到一次 rg 调用耗时超过 15 秒。对于正在主动指导智能体编写代码的人来说，这会严重打断工作流程。

正则表达式匹配已经成为智能体开发中的关键环节。我们认为，必须专门优化这项操作。传统 IDE 会在本地建立语法索引，以支持“转到定义”等操作。与此类似，我们正在为现代智能体查找文本时最核心的操作建立索引。

经典算法

通过索引文本数据来加快正则表达式匹配，并不是新想法。Zobel、Moffat 和 Sacks-Davis 在 1993 年发表的论文《使用压缩倒排文件搜索部分指定的词项》（Searching Large Lexicons for Partially Specified Terms using Compressed Inverted Files）中首次提出了这种方法。

他们使用 N-gram，也就是宽度为 N 个字符的字符串片段，来建立倒排索引。他们还提出了一些启发式方法，把正则表达式分解为一棵 N-gram 树，再通过索引查询这些片段。

如果你以前听说过这个概念，更可能是从 Russ Cox 于 2012 年发表的一篇博客文章中了解到的。那篇文章发布于 Google Code Search 关闭后不久。

下面快速回顾一下这类索引的基本组成。此后开发的其他索引方法，基本都沿用了这些构件。

倒排索引

倒排索引是搜索引擎背后的基础数据结构。

建立倒排索引时，首先要把每份待索引文档拆分成 token。这个过程称为分词（tokenization），实现方式有很多。为了便于说明，这里采用最简单的方式：把每个单词视为一个 token。

这些 token 会成为类似字典的数据结构中的键。每个键对应的值，是所有包含该 token 的文档列表。

这份列表通常称为 posting list。每份文档都会由一个唯一数字标识，这个数字称为 posting。

搜索一个或多个 token 时，我们会加载相应的 posting list。如果列表不止一个，就计算它们的交集，从而找出同时出现在所有列表中的文档。

编辑文档，观察索引如何更新。单击某个词项即可选中并查看它。

将鼠标悬停在词项上，或轻触词项，即可追踪同一个索引条目。

每个词项都会映射到包含它的文档。悬停或轻触即可逐一查看条目。

搜索词项，查找匹配的文档。

可以尝试搜索 cat、the 或 ran。

如今大多数搜索引擎都以这种设计为基础，只是在上面增加了大量复杂机制。

然而，这些搜索引擎面向的是自然语言。我们要搜索的却是正则表达式，而且匹配对象是源代码。这套方法并不完全适用。

一种可能的办法，是在分词上投入大量精力。系统可以理解每种编程语言的语法，也可以拆分源代码中的标识符。但这很难做好。

GitHub 早期的 Code Search 就采用过这条路线。它使用非常复杂的编程语言分词器，并依赖规模庞大的 ElasticSearch 集群。

搜索效果并不好，用户对这个功能的评价也很差。它勉强能搜索标识符，却无法匹配正则表达式。要解决这个问题，必须采用更好的分词方式。

三元组分解

对源代码进行朴素分词，不适合正则表达式匹配。我们需要把文档拆成更基础的片段。

经典算法选择三元组（trigram）。输入字符串中每个相互重叠的三字符片段，都会成为一个 token。

为什么是三个字符？因为这些三元组将作为倒排索引的键。

如果选择二元组，也就是长度为 2 的片段，索引中的键会很少，最多约为 64K。但每个键对应的 posting list 会非常庞大，难以高效处理。

如果选择四元组，也就是长度为 4 的片段，posting list 会很小。这当然很好，但倒排索引可能出现数十亿个键，同样难以处理。

因此，三元组是一个不错的折中。建立文档索引时，分词过程也很简单：提取文档中所有相互重叠的三字符序列，并把它们作为倒排索引中的 token。

真正复杂的部分，是如何对正则表达式分词，使它能够通过索引进行匹配。

正则表达式包含语法。系统必须先解析表达式，再用启发式方法判断哪些实际表示文本的片段能够提取出三元组。

把字面字符串分解成三元组很直接，算法与建立文档索引时相同。只需提取字符串中的所有重叠三元组。

包含全部这些三元组的文档，很可能包含这个字面字符串，但并非一定如此。

选择分支需要分别分解。分解后会产生两个分支，只要文档包含其中一个分支，就可能匹配。查询倒排索引时，我们会合并 posting list，而不是计算交集。

字符类可以分解成多个三元组。像 [rbc]at 这样的小型字符类，会为类中的每个元素生成一个三元组。

对于范围更广的字符类，我们会直接跳过跨越这些边界的三元组提取。

组合成完整方案

现在，我们知道三元组适合用来切分文档，也知道如何在建立索引时切分文档，以及如何在查询时切分表达式。把这些部分组合起来，就能得到一个高效匹配正则表达式的搜索索引。

系统先把正则表达式分解成一组三元组，再从倒排索引中加载相关的 posting list。这样会得到一份可能匹配该正则表达式的文档列表。

这里必须强调“可能匹配”。最终结果仍然需要加载所有候选文档，并用传统方式执行正则表达式匹配。

不过，扫描候选文档子集始终比逐个扫描并匹配整个代码库更快。

编辑文档，观察索引如何更新。单击三元组即可选中并查看它。

将鼠标悬停在三元组上，或轻触三元组，即可追踪同一个索引条目。

每个三元组都会映射到包含它的文档。悬停或轻触即可逐一查看条目。

使用字面字符串或正则表达式搜索，观察三元组如何分解。

可以尝试搜索 cat、the rat，或正则表达式 c[au]t。

无论从哪个角度看，这套设计都完全可用。google/codesearch 和 sourcegraph/zoekt 等项目，都使用三元组倒排索引为大型索引提供了良好性能。与所有搜索引擎一样，它们还在基础设计之上增加了许多复杂机制。

但这套方案也有明显缺点。索引体积并不小，而且查询时的分解策略必须做出取舍。

如果采用简单的启发式方法，查询只会被分解成少量三元组。这样会产生大量候选文档。

如果采用复杂的启发式方法，查询可能被分解成几十个，甚至数百个三元组。从倒排索引中加载所有这些条目，可能和从头搜索全部内容一样慢。

我们还可以做得更好。

后缀数组：一段题外话

既然正在回顾正则表达式文本索引的发展历史，我想稍微离开主线，谈谈 Nelson Elhage 在 2015 年为其 livegrep Web 服务开发的实现。

与其他大型产业项目相比，livegrep 的规模很小。它只索引 Linux Kernel 的最新版本。

但正因为范围有限，它的实现与其他方案很不一样。这也让它格外有趣，值得讨论。

Nelson 从第一性原理出发解决问题。这个搜索引擎背后没有倒排索引。相反，所有源代码都被索引进一个后缀数组。

后缀数组的概念从名称就能看懂：它是一个字符串所有后缀按顺序排列而成的数组。

如果尝试为较长的字符串建立后缀数组，就会发现这个数据结构增长得很快。它看起来是一种成本很高的索引，很多方面也确实如此。

不过，如果能够访问原始字符串，它的存储可以得到很好的压缩。只需保存每个后缀起始位置的偏移量即可。

为待搜索的语料建立后缀数组后，就可以把正则表达式分解成字面量，从而高效执行搜索。

随后，可以在后缀数组上执行二分搜索，找出正则表达式所有可能的匹配位置。

尝试搜索 th 这样的短字符串，观察二分搜索如何圈定文档中的所有匹配位置。

利用后缀数组的相同性质，还能匹配正则表达式中的复杂结构。

例如，要匹配 [a-z] 这样的字符范围，可以分别对范围起点和终点执行二分搜索。两个端点之间的内容必然符合这个范围。

输入一个字符串来建立后缀数组。字符串中的每个位置都会定义一个后缀。

将鼠标悬停在某个后缀行上，或轻触该行，即可查看该后缀在字符串中的起始位置。

所有后缀按字典序排列。数组只保存索引，后缀内容由原始字符串推导得出。

这套方案有什么缺点？后缀数组必须根据一个输入字符串建立。这是很大的限制。

如果要索引大型代码库，甚至多个不同代码库，首先需要把全部内容连接成一个字符串，再基于它建立后缀数组。

在后缀数组中匹配时，还需要辅助数据结构，把匹配位置映射回包含该位置的原始文件。

这些复杂问题并非无法解决，但会让索引的动态更新变得非常昂贵。这是一种很难扩展的方案。

使用概率掩码的三元组查询

现在回到更传统的设计。下面介绍一种最初由 GitHub 为 Project Blackbird 开发的方法。

这是一个旨在替代旧版 Code Search 的研究项目。前面提到，旧版搜索通过切分源代码来实现，无法匹配正则表达式。新实现的目标，就是加入这种能力。

早期版本尝试采用经典的三元组倒排索引，但很快遇到了容量问题。

GitHub 上有海量代码。使用三元组建立索引后，posting list 大得难以搜索。

既然三元组不够理想，下一步就是为索引寻找更合适的 N-gram 长度。

我们已经看到，二元组覆盖范围太宽，posting list 会大到难以管理。四元组又过于具体，会让索引产生太多键。

三元组处在两者之间。但在实际应用中，理想长度更像是“三点五元组”。可是，我们总不能把一个字符切成两半吧？

事实上，我们可以实现非常接近的效果。

这套设计仍用三元组作为倒排索引的键，但会在 posting list 中加入额外信息，记录特定文档中跟在该三元组之后的“第四个字符”。

我们可以直接用一个额外字节保存第四个字符。但这样会让索引变成四元组索引，而前面已经说明，四元组索引太大。

因此，实际保存的是一个 Bloom filter。它包含跟在特定三元组之后的所有字符。

每个三元组都带有一个 8-bit locMask（位置对 8 取模）和一个 nextMask（每个后继字符的哈希）。

将鼠标悬停在三元组上，或轻触三元组，即可追踪索引中的同一个条目。

每个（三元组，文档）条目保存两个 8-bit 掩码。悬停或轻触即可逐一查看三元组。

你可能觉得 Bloom filter 是一种庞大而复杂的数据结构，但它不一定如此。Bloom filter 可以压缩到很少的几个 bit。

如果编码足够仔细，8 bit 也能容纳很多信息。每个 posting 只需增加两个字节，就能绕开经典三元组索引的两个最大问题。

有了保存三元组后继字符的掩码，倒排索引仍可使用三元组作为键，却能使用四元组进行查询。

与简单的三元组索引相比，这已经可以大幅缩小候选文档范围。

第二个增强掩码保存三元组在文档中的位置偏移。它可以解决三元组的歧义问题。

文档包含两个三元组，不代表它们在文档中彼此相邻。但要匹配查询，它们必须相邻。

我们把第二个三元组的位置掩码左移一位，再与第一个三元组的掩码比较。这样就能确认两者确实相邻。

对于非常常见的三元组，这种能力尤其重要。它可以进一步缩小候选文档列表。

这些信息当然都是概率性的。与 Bloom filter 中保存的任何信息一样，它可能产生假阳性。

但在这里，假阳性完全可以接受。最终匹配仍会在文本本身上以确定性方式执行。索引的目标，只是尽量减少需要扫描的候选文档数量。

最终得到的索引效率极高，但存在一个重大缺点：Bloom filter 可能饱和。

这是 Bloom filter 令人遗憾的特性。它可以更新，但如果加入过多数据，最终过滤器中的所有 bit 都会被置位。

Bloom filter 一旦饱和，就会匹配所有内容。这样又会退回到最初那种索引的性能水平。

这套索引最大限度地减少了存储空间，却很难高效地进行原地更新。

稀疏 N-gram：更聪明地选择三元组

下面是另一个非常巧妙的想法。你可能在 ClickHouse 的正则表达式运算符中见过它。GitHub 几年前推出的新版 Code Search 也采用了它，并且支持正则表达式匹配。

它叫作稀疏 N-gram（Sparse N-grams），是各种折中方案中最出色的一种。

传统三元组索引会提取每一个连续的三字符序列。这样会产生大量冗余，因为每个三元组中的字符都会在相邻三元组中重复出现。

这套算法会提取数量不定的 N-gram，而且每个 N-gram 的长度也不固定。

这里的“随机”当然不能是真正的随机，否则索引就无法查询。

我们会为文档中的每一对字符分配一个“权重”。只要结果具有确定性，权重可以用任何方式计算。ClickHouse 使用两个字符的 crc32 哈希。

随后，我们选择这样的子字符串作为稀疏 N-gram：子字符串两端的权重，都严格大于内部的所有权重。

关键在于，稀疏 N-gram 可以具有任意长度。它们并不统一。

这也意味着，算法可能生成很多 N-gram，甚至比单纯提取三元组还多。

不过，N-gram 是以确定性方式生成的，因此查询时可以进行一些非常重要的优化。下面看看它如何工作。

这套算法并不容易理解，所以需要通过交互来观察。你可以使用可视化界面中的前进和后退箭头，逐步查看运行过程。

输入内容的字符分解上方，会显示分配给每对字符的随机权重。这些权重决定了哪些片段会被提取为 N-gram。

最下方区域会显示输入字符串提取出的稀疏 N-gram 数量，以及采用二元组、三元组或四元组时会提取的数量。差异非常明显：算法确实提取了大量稀疏 N-gram。

那么，这到底有什么意义？我们只是在做一件愚蠢的事吗？并不是。

我们在建立索引时承担较高的前期成本，以换取查询阶段的极快速度。

当前展示的 build_all 算法用于索引文档。它会从输入中提取所有可能的稀疏 N-gram。

但查询时不必这样做。因为权重虽然像随机值，却具有确定性，所以查询时可以使用覆盖算法，只生成在索引中完成匹配所需的最少 N-gram。

我们知道这些 N-gram 已经是最少集合。原因在于，建立索引时，只有片段内部的所有权重都低于两端权重，算法才会生成该 N-gram。

因此，查询时只需提取边缘位置的稀疏 N-gram。它们的数量远少于全部三元组，同时仍能以很高的精度筛选候选文档。

还能做得更好吗？可以，而且能好很多。

前面的示例使用 crc32 作为权重函数。不过，只要结果具有确定性，任何哈希函数都可以使用。

我们可以选择一种更聪明的函数：实际非常罕见的字符对获得高权重，非常常见的字符对获得低权重。

这种函数很容易计算。因为索引对象是源代码，我们可以从互联网上收集数 TB 的开源代码，再统计其中所有字符对的频率表。

这张频率表就是我们的哈希函数。

把它应用到算法后，最高权重会出现在最不常见的字符对下方。因此，覆盖模式需要查询的 N-gram 会更少，可能匹配的文档也会更少。

这种方案能最大限度地减少 posting 查询次数。它非常适合作为起点，用来构建可在用户电脑上高效查询的索引。

把这一切放进你的电脑

用于加快正则表达式搜索的索引总要存放在某个地方。

此前看到的所有设计都部署在服务端。我们讨论过的语义索引，也由服务端负责管理和查询。

但这一次，我们选择了不同方向：在用户的电脑上建立并查询索引。

把索引留在本地有几个合理的原因。

首先，索引只是正则表达式匹配所需的一部分。它能筛出可能匹配的文档子集，但系统仍要逐一扫描其中的文件。

如果在服务端完成这项工作，就必须同步所有文件，或者在服务端与客户端之间进行成本高昂的多次往返。

在客户端执行则很简单。它还能避开数据存储相关的许多安全和隐私问题。

延迟对这项功能也十分重要。

我们的 Composer 模型拥有业界领先的每秒 token 数（TPS）表现之一。我们也一直在努力让它变得更聪明、更快。

模型会持续调用搜索这项关键操作，而且经常并行调用。为它加入网络往返只会增加阻力和停顿，与我们改善智能体交互体验的目标背道而驰。

与语义索引不同，正则表达式搜索索引还必须非常新鲜。模型能否读到自己刚写入的内容，尤其重要。

语义索引不必持续更新。文件修改后，即使重新计算 embedding，新向量通常也不会在多维空间中发生显著位移。最近邻搜索仍会把智能体引向正确方向。

但如果智能体搜索某段具体文本却没有找到，它往往会开始无效追查。这会浪费 token，也会从根本上抵消性能优化的意义。

把索引带到客户端，也会产生新的挑战。同步磁盘数据既复杂又昂贵，但我们在实践中把它做得很高效。

我们以底层 Git 仓库中的一个 commit 为基准，控制索引状态。用户和智能体的改动会作为叠加层存储在它之上。

因此，索引可以快速更新，也能在启动时快速加载和同步。

为了尽量降低编辑器中的内存占用，我们把索引存储在两个独立文件中。

第一个文件依次保存索引的所有 posting list。构建索引时，我们会把它们直接刷新到磁盘。

另一个文件保存一张排序表，其中包含所有 N-gram 的哈希，以及对应 posting list 在 posting 文件中的偏移量。

这里只保存哈希，不保存完整 N-gram，而且始终是安全的。

如果两个哈希发生碰撞，posting list 的覆盖范围可能变大。这在实践中极少发生，但即使发生，也不会产生错误结果。

这种方式还能让查找表的布局非常紧凑。

随后，我们在编辑器进程中只对这张表执行 mmap，并使用二分搜索处理查询。

搜索会返回一个偏移量。系统再直接读取 posting 文件中对应偏移位置的数据。

将鼠标悬停在查找表中的某一行，或轻触该行，即可追踪对应 posting 在磁盘上的位置。

结论

我们发现，为 Composer 2 这类快速模型提供文本搜索索引，会给智能体工作流带来质的变化。

这种影响在大型企业仓库中更加明显。grep 是少数延迟会随代码规模和复杂度增长的智能体操作。

可以看看这些由 Composer 2 运行的示例工作流。彻底消除代码库搜索耗时，能节省可观的时间。

智能体调查缺陷时，收益尤其明显。搜索加快后，迭代也会有效得多。

切换模式，然后将鼠标悬停在某个片段上，或轻触该片段，即可查看它的持续时间。

下一步会是什么？谁也不知道。

围绕如何为智能体提供上下文，正在出现许多令人兴奋的新进展。包括 Cursor 的研究人员在内，很多人都在研究这个方向。

我们会继续优化现有方法的性能，其中也包括语义索引。

我们还希望推出全新的方法，进一步提升智能体表现。同时，我们会始终确保这些方法能在最重要的环境中正常运行。

这里所说的环境，就是世界上规模最大的代码仓库。智能体开发的未来，正在这些仓库中真正形成势头。
