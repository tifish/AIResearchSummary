给 Agent 造一套更快的正则搜索：把 grep 变成本机索引

一句话总括

Agent 离不开 grep，但 ripgrep 必须扫每一个文件——在超大仓库里这是少数延迟随代码规模线性增长的操作。Cursor 的真正贡献是：像 IDE 为"跳转定义"建本地语法索引一样，给"正则搜索"也建一套本机增量索引，把这块等待时间整个抹掉。

01问题：Agent 爱用 grep，但 grep 不会变快，只会被仓库拖慢

工具链从 grep（1973）一路进化到 ctags、IDE 语法索引、再到标准化的 LSP，可 Agentic 编码一来，Agent 偏偏又爱上了 grep。语义索引能解决很多任务，但有些查询模型只能靠正则搜索。ripgrep 已经够快，瓶颈却是结构性的：它必须匹配所有文件的内容。小项目无所谓，但 Cursor 的大型企业客户工作在"大到离谱"的 monorepo 里——一次 rg 调用常常超过 15 秒，正在实时指挥 Agent 写代码的人就这么被卡住。
洞见：grep 是少数延迟随代码规模增长的 Agent 操作，所以值得像 IDE 那样为它单独建索引——这是把"搜索"从瓶颈变成常数级操作的关键判断。

02经典解法：倒排索引 + 三元组分解

正则索引并非新事——1993 年 Zobel/Moffat/Sacks-Davis 已提出用 n-gram 建倒排索引，2012 年 Russ Cox 的博文让它广为人知。倒排索引把文档切成 token，每个 token 映射到一串"出现过它的文档"（posting list），多 token 查询就求交集。但对源码做自然语言式分词（GitHub 早期靠复杂分词器 + 巨型 ElasticSearch 集群）效果很差，根本匹配不了正则。正解是用 trigram（每 3 个字符的重叠片段）当键：bigram（2 字符）只有最多 64k 个键、posting list 大到没法用；quadgram（4 字符）posting 很小却产生数十亿个键。三元组是恰到好处的折中。难点在查询端：正则有语法，要解析后用启发式把字面量、字符类、分支拆成可查的 trigram。
洞见：索引只负责缩小候选集——最终匹配仍要把候选文档逐个加载、按老办法跑一遍正则。但"先筛掉一大半"永远比全仓扫描快，这是所有方案的共同骨架。

03两条岔路：后缀数组的优雅，与它无法扩展的代价

2015 年 Nelson Elhage 的 livegrep 走了一条完全不同的路：不用倒排索引，把全部源码塞进一个后缀数组（所有后缀按字典序排好的数组，只存起始偏移即可压缩）。正则被分解成字面量后，每个可能的匹配位置都能在后缀数组上二分查找定位；字符区间 [a-z] 这类结构还能靠二分边界一次圈定。优雅，但有硬伤：后缀数组必须由单一字符串构造——索引整个代码库得先把所有内容拼成一根长字符串，再额外维护"匹配位置→源文件"的映射。这让动态更新代价极高，几乎无法规模化。
洞见：livegrep 只索引最新版 Linux 内核这一件事，恰恰暴露了第一性原理方案的边界——能否便宜地增量更新，往往比单次查询多快更决定一个索引能不能用在真实工程里。

04把三元组榨到极致：概率掩码与 Sparse N-grams

GitHub 的 Project Blackbird 想要"3.5 元组"——既要 trigram 的小键空间，又要 quadgram 的精确。做法是：键仍是 trigram，但每条 (trigram, 文档) posting 多挂两个 8-bit 掩码——一个 nextMask（用布隆过滤器编码"紧跟其后的第四个字符"，于是能用 quadgram 来查），一个 locMask（位置信息，把第二个 trigram 的掩码左移一位再比对，确认两个 trigram 真的相邻）。只用每 posting 两字节就绕开了三元组的两大顽疾，且布隆过滤器的假阳性无所谓——反正最终在文本上精确匹配。代价：布隆过滤器会饱和，塞太多数据后所有位都置 1、退化成"匹配一切"，原地更新很痛。

更甜的中间地带是 Sparse N-grams（ClickHouse、GitHub 新版 Code Search 都在用）：给每对字符算一个确定性"权重"，只在两端权重都严格大于内部所有权重的子串处切出 n-gram，长度可变。索引时用 build_all 全量抽取（量很大），查询时却只需"覆盖算法"抽边缘那几个 n-gram。再把权重函数换成"基于几 TB 开源代码统计的字符对频率表"——让越罕见的字符对权重越高，查询要查的 posting 就更少、候选文档更精。
洞见：这条线索的精髓是把成本从查询时挪到索引时：索引端多花力气全量抽取，换来查询端只需最少的查找——而"用真实代码频率当权重"让筛选精度再上一个台阶。

05Cursor 的选择：把索引搬进你自己的机器

前面所有方案都跑在服务器上，Cursor 偏要放本机，理由有三。其一本就要在客户端扫文件：索引只给候选子集，逐个文件的最终匹配在客户端做才不必同步全部文件或来回往返，也顺手避开数据存储的安全与隐私问题。其二延迟：Composer 模型 TPS 业界领先且常并行调用搜索，任何网络往返都是逆向摩擦。其三新鲜度：正则索引和语义索引不同——语义索引改个文件嵌入不会明显漂移，方向仍对；但正则搜索若读不到模型刚写下的文本，Agent 就会陷入"瞎找"、白烧 token。

工程实现也很省：以底层 Git 仓库的某个 commit 为基线，用户和 Agent 的改动作为叠加层，启动时加载与同步都很快。索引拆两个文件——一个顺序存放所有 posting list 并直接刷盘；另一个是排好序的查找表，只存 n-gram 的哈希与对应 posting 的偏移（存哈希而非完整 n-gram，最坏只是哈希碰撞让某个 posting 变宽，绝不会出错）。编辑器进程只 mmap 这张紧凑查找表，二分查到偏移后直接去 postings 文件读，内存占用极低。
洞见："放本机 + 以 Git commit 为基线 + 改动叠加层 + 只 mmap 查找表"——这套组合同时解掉了新鲜度、低延迟、低内存、隐私四个约束，正是它能服务于真实大仓库的原因。

06结论：搜索时间归零，给快模型带来质变

Cursor 发现，给 Composer 2 这类快模型配上文本搜索索引，会让 Agentic 工作流产生质的差别，且在大型企业仓库里效果尤其明显——因为 grep 是少数延迟随代码规模与复杂度增长的操作。在示例工作流里，把搜索代码的时间整个抹掉带来实打实的提速，尤其当 Agent 在排查 bug 时，迭代效率显著提升。Cursor 表示会继续优化包括语义索引在内的现有方法，并探索全新路径，目标始终是让 Agent 在世界上最大的仓库里也能跑得动。
洞见：当模型本身越来越快，瓶颈就从"模型想多久"转移到"工具等多久"——把工具延迟降到常数级，是让快模型真正发挥的隐形前提。

最值得带走的 5 条洞见

  1. grep 是 Agent 的隐形瓶颈。它是少数延迟随仓库规模线性增长的操作，大企业仓库里一次调用常超 15 秒——值得像 IDE 建语法索引那样，为正则搜索单独建索引。
  2. 索引只负责"缩小候选集"。从 trigram 倒排到 Sparse N-grams，所有方案最终都得把候选文档加载后跑一遍真正的正则；筛掉一大半，就已经赢了。
  3. 真正的工程难题是"增量更新"。后缀数组查询很优雅却无法便宜地更新，因而难扩展；布隆过滤器会饱和退化。能否便宜地保持新鲜，往往比单次查询多快更决定成败。
  4. 把成本从查询时挪到索引时。Sparse N-grams 索引端全量抽取、查询端只覆盖边缘；再用真实代码的字符对频率当权重，让罕见组合权重更高，查找更少、更精。
  5. 放本机是关键决定。以 Git commit 为基线、改动叠加为增量层、只 mmap 紧凑查找表——一举解决新鲜度、低延迟、低内存与隐私，让快模型的速度优势不被网络往返吃掉。
