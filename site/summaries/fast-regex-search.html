一句话总括

正则搜索正在从“扫完整个仓库”变成 AI 代理的本地索引能力

文章真正的贡献，是把代理编程中的 grep 延迟问题具体化：大型代码库里，正则搜索已经是模型工作流的核心瓶颈，因此需要专门为代理构建低延迟、可更新、隐私友好的本地文本索引。

>15 秒Cursor 在大型 monorepo 中经常观察到的 rg 调用耗时。
2 字节Blackbird 风格方案中每个 posting 用两个 8-bit 掩码增强过滤能力。

ripgrep 很快，但代理的问题是它仍要扫描全部文件

背景grep 从 1973 年开始就是文本匹配工具，ripgrep 已经把正则匹配本身做得很快，但它在大仓库中仍然需要遍历大量文件。Cursor 提到，大型企业 monorepo 里 rg 调用超过 15 秒并不少见。

这改变了对 AI 编程效率的理解：瓶颈不只在模型推理速度，也在模型为了找上下文而反复调用工具的延迟。结论是，正则搜索应被视为代理基础设施，而不是普通命令行工具调用。

三元组倒排索引是可行基线，但查询和存储会拉扯

方法经典方案来自 1993 年的 n-gram 倒排索引思想：把文档切成重叠的 3 字符片段，查询时把正则表达式分解成可查的三元组，先用 posting list 找候选文件，再对候选文件做确定性的正则匹配。

bigram 键数少，最多约 64k，但 posting list 太大；quadgram 候选更精确，却可能产生数十亿个键；trigram 是折中。结论是，索引不替代正则匹配，而是减少必须扫描的文件数量。

后缀数组展示了另一条路，但动态更新成本太高

对比2015 年 livegrep 使用后缀数组，把语料拼成一个大字符串并保存所有后缀的排序偏移，查询时可通过二分搜索定位文字片段或字符范围。

它适合较静态、范围受控的语料，例如 livegrep 只索引最新 Linux Kernel；但对多仓库、大仓库和频繁变更的本地工作区，拼接、映射回文件和增量更新都会复杂化。结论是，理论上优雅的结构未必适合代理不断读写代码的场景。

概率掩码让三元组更接近 3.5-gram，但会遇到饱和问题

优化GitHub Project Blackbird 的思路是在 trigram posting 上增加信息：一个 8-bit nextMask 记录三元组后续字符的哈希，一个 8-bit locMask 记录位置 mod 8。这样只用约 2 字节额外数据，就能用三元组索引模拟更接近 quadgram 的过滤效果。

这些掩码允许误报，因为最终仍会扫描文本做确定性匹配；问题是 Bloom filter 可能饱和，饱和后过滤能力下降。结论是，压缩索引能显著减少候选文件，但原地更新和长期新鲜度仍是硬约束。

Sparse N-grams 用更高建索引成本换更少查询读取

核心Sparse N-grams 不再抽取所有连续三元组，而是用确定性权重选择变长 n-gram：只有边界权重大于内部权重的片段才进入索引。查询时不必生成全部片段，只用覆盖算法取最少的必要 n-gram。

如果权重来自源码字符对频率表，稀有字符对权重更高，查询会优先选择更有区分度的片段。结论是，代理搜索更需要“少读 posting、快缩候选”，因此愿意把更多成本放到建索引阶段。

本地索引比服务器索引更适合代理的低延迟和新鲜度需求

部署Cursor 选择在用户机器上构建和查询索引，而不是把正则搜索索引放到服务器。原因包括：最终匹配仍要读本地文件；服务器方案会带来同步或往返开销；本地处理还能减少代码存储的安全和隐私风险。

更关键的是，正则索引必须非常新鲜，尤其要让模型能搜索到自己刚写入的文本。结论是，语义索引可以稍微陈旧，正则索引不行，因为一次“找不到”就可能让代理浪费大量 token 走错方向。

Cursor 的工程落点是 Git 基线、增量层和 mmap 小表

实现Cursor 以底层 Git commit 控制索引状态，把用户和代理的修改放在上层增量层，从而让启动同步和更新更快。为降低编辑器内存占用，索引拆成两个文件：posting lists 顺序写入一个文件，n-gram hash 与 posting offset 的有序查找表写入另一个文件。

编辑器进程只 mmap 查找表，用二分搜索得到 offset，再从 posting 文件对应位置读取。结论是，这套设计把索引能力做成了轻量本地运行时组件，而不是一个沉重的搜索服务。

最值得带走的洞见

  • AI 代理的工具延迟会被模型反复放大；搜索慢几秒，可能意味着整个交互节奏被打断。
  • 正则搜索索引的目标不是直接给出最终答案，而是把候选文件缩到足够小。
  • 三元组、概率掩码和 Sparse N-grams 的差异，本质上是在存储、查询读取量、误报和更新成本之间取舍。
  • 对代理来说，索引新鲜度和低延迟与模型能力同样重要，尤其是在模型需要读取自己刚写入代码时。
  • 大型仓库中的 AI 编程体验，很大一部分会取决于本地上下文基础设施，而不只是云端模型本身。
