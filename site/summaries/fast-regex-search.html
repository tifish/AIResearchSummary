快速正则搜索：为 Agent 工具索引文本

Fast regex search: indexing text for agent tools

一句话总括：文章的真正贡献，是把 Agent 常用的正则搜索从“每次扫完整个仓库”转向“本地、可更新、面向代码查询的文本索引”。

1973grep 最早发布，正则文件搜索成为基础工具。
15s+Cursor 经常看到大型 monorepo 中的 rg 调用超过 15 秒。
1993n-gram 倒排索引用于部分指定词搜索的论文发表。
2B概率掩码方案每个 posting 只增加两个 8-bit 掩码。
2 files本地索引拆成 posting 文件和 mmap 查找表。

问题不是 ripgrep 慢，而是扫描范围太大

ripgrep 的正则匹配非常快。它的问题在于必须查看所有文件。

在小项目里，这通常可以接受。在大型企业 monorepo 中，搜索延迟会直接阻塞人与 Agent 的交互。

这改变了读者对 AI 工具性能的理解：瓶颈不只在模型推理，也在模型每一步拿上下文的系统路径上。

核心判断

正则搜索已成为 Agentic 开发的基础操作

Agent 经常需要查找具体文本、符号片段、错误信息和代码模式。语义索引适合许多上下文检索任务，但不能替代所有正则查询。

因此，Cursor 把正则搜索视为类似 Go To Definition 的基础能力。它需要专门的本地索引，而不是每次退回全仓库扫描。

技术路线是一条逐步收窄候选文件的历史

这些方案的共同目标很直接：先用索引找到“可能匹配”的文件，再用确定性的正则匹配确认结果。

1973

grep

逐文件匹配正则。简单可靠，但延迟随代码库规模增长。

1993

n-gram 倒排索引

把文本切成固定宽度片段，用 posting list 快速缩小范围。

2012

Code Search 思路普及

Russ Cox 的文章让三元组索引成为很多工程师熟悉的方案。

2015

livegrep 后缀数组

把源码拼成可二分查找的后缀数组，查询快但动态更新困难。

2B

概率掩码

用 nextMask 和 locMask 增强三元组，降低候选文件数量。

Now

Sparse N-grams

索引时多做工作，查询时只取最少且高区分度的 n-gram。

四类索引的取舍

文章最有价值的部分，是没有把某一种结构包装成银弹，而是展示了每种结构为什么会在规模、更新和查询之间付出代价。

方案 优势 代价 对 Agent 的意义
经典三元组倒排索引 实现清晰，可复用搜索引擎结构。 索引不小，复杂正则可能需要加载大量 posting list。 能避免全仓库扫描，但查询分解质量决定收益。
后缀数组 可用二分搜索定位文字片段。 需要把语料拼成一个大字符串，增量更新昂贵。 适合静态语料，不适合频繁变化的工作区。
带概率掩码的三元组 用少量位信息模拟更强的四元组过滤。 Bloom filter 会饱和，原地更新会变痛苦。 说明“允许假阳性”是搜索加速的关键工程自由度。
Sparse N-grams 查询时只查少量高信息量片段。 索引构建更重，需要确定性权重函数。 更接近 Agent 工具需要的低延迟、高选择性检索。
!

关键洞见：索引不需要直接给出最终答案。它只需要以很低的成本排除大多数不可能匹配的文件，然后把剩下的少量文件交给确定性正则匹配。

为什么索引要放在用户机器上

Cursor 的选择不是把所有代码送到服务器，而是在本机建立和查询索引。这一决定同时影响延迟、隐私和新写入代码的可见性。

延迟

Agent 会频繁并行调用搜索。网络往返会放大等待时间，尤其会削弱快速模型的交互优势。

隐私

服务器端搜索要同步文件或反复回传内容。本地扫描避免了额外的数据存储风险。

新鲜度

正则索引必须读到 Agent 刚写下的文本。否则模型会误判不存在，继续浪费 token 搜索。

本地实现的关键设计

Cursor 用 Git commit 作为基础层，用户和 Agent 的修改作为覆盖层。这样启动同步和增量更新都更轻。

1

posting 列表顺序写入磁盘，构建时不需要把全部 posting 留在编辑器内存中。

2

查找表保存 n-gram 哈希和 posting 文件偏移量，并按哈希排序。

3

编辑器只 mmap 查找表。查询时二分找到偏移，再从 posting 文件读取对应列表。

4

哈希碰撞最多让候选集合变宽，不会产生错误结果。最终匹配仍由原始文本上的正则确认。

最值得带走的洞见

搜索延迟会塑造 Agent 行为

当 rg 一次超过 15 秒时，问题已经不是工具小优化，而是交互循环被打断。

语义索引不能替代正则

具体文本、错误字符串和代码模式仍需要精确查找。Agent 工具链必须同时支持两类检索。

假阳性可以接受

索引阶段只负责缩小候选集。只要最后一步是确定性匹配，宽一点不会损害正确性。

本地索引是产品判断

它同时服务于低延迟、隐私和读写一致性。对大型仓库中的 Agentic 开发来说，这比单纯追求服务器端集中索引更贴近工作流。
