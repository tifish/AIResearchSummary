为 Agent 加速正则搜索：把文本索引搬进你的机器

作者 Vicent Marti ·《Fast regex search: indexing text for agent tools》
一句话总括：AI Agent 极度依赖 grep，而在超大代码库里 grep 的延迟会随规模线性爆炸——Cursor 的真正贡献是把"为正则搜索建索引"这件事，从服务器端搬到了用户本地机器，并配合快速模型 Composer 2，让 Agent 的文本查找几乎"零等待"。

问题Agent 又回到了 1973 年的 grep

开发工具从 grep 进化到 ctags、IDE 语法索引、再到标准化的 LSP；可 Agentic 编码一来，Agent 偏偏最爱用 grep。多数 harness（包括 Cursor）默认用 ripgrep——它比 grep 更快、默认更合理。但它有个根本限制：必须扫描所有文件的内容。

1973
grep 诞生年，半个世纪后 Agent 又用上它

grep 是少数"延迟随代码库规模增长"的 Agent 操作，所以它值得像 IDE 的"跳转定义"那样被单独建索引优化。

基础为什么是 Trigram（3 字符）

正则索引并非新事：最早见于 1993 年 Zobel/Moffat/Sacks-Davis 论文，2012 年 Russ Cox 在 Google Code Search 关闭后撰文普及。核心是把文档切成 n-gram 建倒排索引，再把正则拆解成 n-gram 去查。n 的大小是关键权衡：

2-gram
键最多 64k，但每个 posting list 太大，难用
3-gram
甜蜜点：键数与列表大小折中
4-gram
列表很小，但键数高达数十亿，难管理

索引返回的是"可能匹配"的候选文档子集，最终仍要在这些文件上真正跑一遍正则——但扫子集永远快于扫全库。

索引不负责给出正确答案，它只负责把"要逐字扫描的文件"从全库缩到极小一撮。

技术史四种方案，各有死穴

经典 Trigram 倒排

google/codesearch、sourcegraph/zoekt 采用。

死穴：索引偏大；查询拆解要么太粗（候选太多）要么太细（加载几百个 trigram 跟全扫一样慢）。

Suffix Array（后缀数组）

Nelson Elhage 2015 年 livegrep，仅索引 Linux 内核，靠二分查找定位。

死穴：必须把全部代码拼成一个大字符串构建，增量更新极昂贵，难扩展。

概率掩码（GitHub Blackbird）

每个 posting 仅 2 字节、两个 8-bit bloom mask：一个记"第四个字符"（用 trigram 键却能按 quadgram 查），一个记位置以判断相邻。

死穴：bloom filter 会饱和——填满后匹配一切，退化回原点。

Sparse N-grams ★

ClickHouse 与 GitHub 新版 Code Search 采用。按字符对的确定性权重切分，长度可变。索引时 build_all 全量提取，查询时用"覆盖算法"只取最少 n-gram。

升级：用数 TB 开源代码统计的字符对频率表当权重函数，让罕见字符对权重高，查询时 n-gram 更少、候选更精。

从倒排到后缀数组到掩码再到稀疏 n-gram，演进主线只有一条：用更聪明的切分，把"候选文档数"和"查询/存储成本"同时压下去。

关键抉择为什么索引要放在你的本机

以上方案历来都部署在服务器。Cursor 反其道而行，把索引建在并查询于用户本地，理由有三：

① 反正要在本地扫文件

索引只给候选子集，逐文件精确匹配本就在客户端做最省事，还避开数据同步与隐私安全顾虑。

② 延迟致命

Composer 模型 TPS 业界领先且常并行搜索，任何网络往返都是拖累。

③ 索引必须"新鲜"

不同于语义索引，正则搜索要能读到模型自己刚写的内容，否则 Agent 找不到就会瞎折腾、白烧 token。

工程实现：索引以一个 Git commit 为基准，用户/Agent 的改动作为增量层叠加，启动加载快、更新快。索引拆成两个文件——postings 文件（所有列表顺序落盘）与一张 hash→偏移查找表（只存 hash 不存完整 n-gram，碰撞只会让列表变宽、不会出错）。编辑器只 mmap 那张查找表，二分查到偏移后直接去 postings 文件读取，内存占用极低。

"以 Git commit 为基准 + 增量层"是兼顾新鲜度与速度的巧解；只 mmap 索引表而非全部数据，是把内存成本压到最低的关键。

结果对大仓库的质变

给 Composer 2 这类快速模型提供本地文本搜索索引，为 Agentic 工作流带来质的差异，在大型企业仓库尤为明显——因为 grep 是少数延迟随代码规模上升的 Agent 操作。彻底消除搜索耗时，在排查 bug 时节省尤为可观，迭代更高效。

价值不在"快一点"，而在让 latency 不再随仓库变大而恶化——这正是 Agentic 开发最先落地的"世界级大仓库"最需要的。

最值得带走的洞见

  1. grep 是 Agent 的延迟瓶颈，且会随仓库规模线性恶化——企业 monorepo 里 rg 单次可超 15 秒，这是必须专门优化的核心操作，而非边角。
  2. 所有方案的本质都是"先用索引缩小候选、再逐字精确匹配"，索引允许假阳性，因为最终判定由文本本身完成。
  3. n-gram 大小是永恒的权衡：trigram 是经典甜蜜点，而 Sparse N-grams + 频率感知权重函数是当前最优——查询时只取最少 n-gram，精度最高。
  4. Cursor 最大的差异化是"本地化"：为了零网络往返、隐私安全，以及让模型能即时读到自己刚写的内容（索引必须新鲜）。
  5. 以 Git commit 为基准 + 增量层 + 只 mmap 查找表，是同时拿下新鲜度、更新速度与低内存的工程组合拳。
