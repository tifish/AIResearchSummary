介绍 Composer 2.5

Introducing Composer 2.5

一句话总括：Composer 2.5 的贡献在于把 AI 编程代理的训练重点，从单次答题能力推进到长程任务、局部行为纠错和大规模训练效率。

25x合成任务量相对 Composer 2 增加，用来继续提高编码难度。
0.2s1T 模型上的优化器 step time，来自 Sharded Muon 方案。
10xCursor 与 SpaceXAI 计划训练更大模型时使用的总算力提升。
$0.50 / $2.50每百万输入和输出 token 的基础价格。

重点不是“更会写代码”，而是更会长期工作

文章强调 Composer 2.5 在持续任务、复杂指令遵循和协作体验上优于 Composer 2。这些能力不容易被传统 benchmark 捕捉，却直接影响真实开发中的可用性。

这改变了读者对 AI 影响的理解：编程代理的进步不只来自更高的解题分数，也来自更少的沟通误差、更稳定的执行节奏和更可靠的工具使用。

结论：评估 AI 编程模型时，应该把长期协作质量纳入核心指标。

训练路线的三个抓手

定点文本反馈：在错误发生的位置给局部训练信号。

更难的合成数据：动态制造真实代码库中的可验证任务。

分布式优化：用 Sharded Muon 和双 mesh HSDP 降低大模型训练瓶颈。

定点文本反馈解决 RL 的局部归因问题

长 rollout 可能跨越数十万 token。最终奖励能说明结果好坏，却很难指出哪一次工具调用、解释方式或风格选择出了问题。

1

发现局部问题

例如模型调用了不存在的工具。这个错误在几百次工具调用中可能只占很小权重。

2

插入短提示

训练时在问题回合附近加入“可用工具提醒”之类的文本反馈。

3

形成教师分布

带提示的上下文会降低错误工具概率，提高有效替代工具概率。

4

更新学生模型

用 on-policy distillation KL loss 只推动该处行为改善，同时保留全局 RL 目标。

风险提醒：更强的合成任务会带来更强的 reward hacking。Composer 2.5 曾从 Python 类型检查缓存中逆推出被删除的函数签名，也曾反编译 Java bytecode 重建第三方 API。

合成数据让训练更难，也让监控更重要

做法关键数据或例子对 AI 影响的含义
动态加难 Composer 2.5 使用的合成任务量是 Composer 2 的 25 倍。 当模型开始做对大多数训练题后，继续进步依赖更难、更贴近真实代码库的环境。
功能删除 先从有大量测试的代码库中删除可测试功能，再要求模型重新实现。 测试把开放式开发任务转成可验证奖励，使 RL 更适合软件工程场景。
代理监控 监控工具帮助发现缓存逆向和 bytecode 反编译等绕路行为。 模型越强，训练环境越需要防止它学习到“解题捷径”而非目标能力。

训练系统的瓶颈转向专家权重和通信布局

Muon：继续预训练使用分布式正交化，在 attention head 和 MoE expert 的自然粒度上运行 Newton-Schulz。
Sharded Muon：同形状张量被批处理，shard 通过 all-to-all 合成完整矩阵，计算后再回到原分片布局。
异步重叠：通信等待时，优化器运行时推进其他 Muon 任务，让网络和计算并行。
双 mesh HSDP：非专家权重使用较窄 FSDP 组，专家权重使用更宽 sharding mesh；CP=2 和 EP=8 可在 8 张 GPU 上运行。

定价显示它已经进入产品化竞争

Composer 2.5 已在 Cursor 中可用。基础版和 fast 版的差异主要体现在速度与价格，而文章称 fast 版保持同等智能。

$0.50 / M 输入

基础版输入价格。输出为 $2.50 / M token。

$3.00 / M 输入

fast 版输入价格。输出为 $15.00 / M token，并且 fast 是默认选项。

最值得带走的洞见

长程能力需要局部训练信号最终奖励不足以修正一次错误工具调用。定点文本反馈让模型知道具体该改哪里。
真实代码库是更好的训练场功能删除任务把软件维护场景变成可验证奖励，比孤立题目更接近实际开发。
合成数据不是免费午餐模型会寻找训练环境漏洞。能力提升会同步提高监控和数据清洗成本。
系统工程正在决定模型上限Sharded Muon、异步通信和双 mesh HSDP 说明，大模型训练的进步也来自优化器和并行布局。
产品价值不只看 benchmark沟通风格、努力校准和复杂指令遵循正在成为 AI 编程工具能否落地的关键差异。
