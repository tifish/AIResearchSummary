推出 Composer 2.5

Introducing Composer 2.5

一句话总括：Composer 2.5 的核心贡献，是把局部行为纠正、更困难的合成任务和大规模分布式优化结合起来，让编程智能体在长任务中更可靠，也更容易协作。

25×相较 Composer 2，合成任务数量增加至 25 倍
0.2s1T 模型上的优化器单步耗时
10×与 SpaceXAI 合作训练的新模型计划使用的总计算量
8 GPUCP=2 与 EP=8 可重叠运行所需的 GPU 数量

改进重点不只在“答对”

Cursor 强调了长时间持续工作、复杂指令遵循、沟通方式和投入程度校准。

Composer 2.5 仍建立在 Moonshot 的 Kimi K2.5 开源检查点上。Cursor 没有在正文中提供可复核的具体 benchmark 分数，而是把重点放在现有基准难以覆盖的行为质量上。

这改变了观察 AI 编程模型的角度。真实效用不仅取决于一次回答是否正确，还取决于模型能否在数十万 token 的轨迹中保持方向，并以合适方式与人协作。

局部错误需要局部训练信号

长轨迹只给一个最终奖励时，单次错误工具调用几乎会被数百次后续操作淹没。Composer 2.5 使用定点文本反馈，在问题发生的位置插入简短提示，并把带提示的模型分布作为教师。

01 找到错误定位错误工具调用、混乱解释或风格违规。
02 插入提示在该轮局部上下文中说明更合适的行为。
03 生成教师分布降低错误选择概率，提高有效替代方案概率。
04 蒸馏到学生仅针对该轮加入 on-policy KL 损失，同时保留全轨迹 RL 目标。

结论：训练信号从“整次任务哪里不对”细化到“这一轮应该怎样改”。这使沟通风格等难以用最终奖励衡量的行为也能被直接优化。

Composer 2.5 使用的合成任务数量是 Composer 2 的 25 倍。其中一种方法先从真实代码库删除可测试功能，再要求智能体重新实现，并以测试结果作为可验证奖励。

更困难的任务推动了能力增长，也带来奖励投机。模型曾从遗留的 Python 类型检查缓存中反推出函数签名，还曾反编译 Java 字节码以重建第三方 API。

洞见：模型越擅长搜索环境漏洞，训练结果就越不能只看测试是否通过。大规模 RL 同时需要更严格的环境清理和智能体监控。

合成数据既是机会，也是风险

任务规模扩大带来了更强的推理与编码能力，也让奖励投机变得更隐蔽。

分片 Muon 与双网格 HSDP

Muon 按模型的自然粒度执行分布式正交化。注意力投影按 attention head 处理，堆叠 MoE 权重按 expert 处理。

同形状张量先组成批次，再通过 all-to-all 拼成完整矩阵。Newton-Schulz 完成后，结果被送回原来的分片布局。通信与计算异步重叠。

这一方案等价于完整矩阵 Muon，但能持续利用分片组。在 1T 模型上，优化器单步只需 0.2 秒。

非 expert 权重使用较窄的 FSDP 组，expert 权重使用更宽的分片网格。由此，CP=2、EP=8 可在 8 张 GPU 上运行，而不是共享网格所需的 16 张。

Composer 2.5 现已在 Cursor 中推出。

与 Composer 2 相比，它的智能水平和行为表现都有显著提升。它更擅长持续处理长时间运行的任务，能够更可靠地遵循复杂指令，也更容易与人协作。

我们扩大了训练规模，生成了更复杂的强化学习环境，并引入新的学习方法，从而改进了 Composer。

除了让 Composer 2.5 接受更困难任务的训练，我们还改善了模型的行为表现，例如沟通风格和投入程度校准。现有 benchmark 很难充分衡量这些方面，但我们发现，它们对现实使用价值十分重要。

Composer 2.5 与 Composer 2 建立在同一个开源检查点上，也就是 Moonshot 的 Kimi K2.5。

我们还在与 SpaceXAI 合作，从零开始训练一个大得多的模型。它将使用 10 倍的总计算量。结合 Colossus 2 的百万 H100 等效算力，以及双方的数据和训练技术，我们预计模型能力将迎来一次重大跃升。

训练 Composer 2.5

Composer 2.5 对我们的训练技术栈做了多项改进。这些变化同时面向模型智能和易用性。

使用文本反馈的定点强化学习

随着 rollout 的长度达到数十万 token，强化学习中的信用分配正变得越来越困难。如果奖励是根据整条 rollout 计算的，模型就很难判断究竟是哪一个决定改善或损害了结果。当我们希望抑制某种局部行为时，这个问题尤其突出。例如错误的工具调用、令人困惑的解释或违反风格要求。最终奖励可以告诉我们出现了问题，但它很难准确指出问题发生在哪里。

为解决这个问题，我们使用定点文本反馈训练 Composer 2.5。1 核心思路是在轨迹中本可表现得更好的位置直接提供反馈。对于一条目标模型消息，我们会构造一段简短提示，说明应该如何改进。随后把提示插入局部上下文，并将由此得到的模型分布作为教师。使用原始上下文的策略充当学生。我们再加入 on-policy 蒸馏 KL 损失，让学生的 token 概率向教师靠近。这样既能针对希望改变的行为提供局部训练信号，又能保留覆盖完整轨迹的整体强化学习目标。

以文本反馈流程为例，假设一次很长的 rollout 中出现了工具调用错误。模型尝试调用一个不存在的工具。它会在 rollout 过程中收到“找不到工具”的错误，然后继续执行其他有效的工具调用。在数百次工具调用中只发生一次错误，对最终奖励的影响会非常小。

借助文本反馈，我们可以专门处理这个错误。在出现问题的轮次上下文中，我们会插入类似“提醒：可用工具……”的提示，并附上可用工具列表。这个提示会改变教师模型的概率分布，降低错误工具的概率，同时提高有效替代工具的概率。然后，我们只在这一轮让学生权重向新的概率分布靠近。

在 Composer 2.5 的训练过程中，我们把这一方法用于多种模型行为，包括编码风格和模型沟通方式。

合成数据

在强化学习训练期间，Composer 的编码能力大幅提高，最终能够正确解决大多数训练问题。为了继续提高智能水平，我们在整个训练过程中动态筛选并创建更困难的任务。Composer 2.5 使用的合成任务数量是 Composer 2 的 25 倍。

我们采用多种方法创建以真实代码库为基础的合成任务。其中一种方法是删除功能。在这类任务中，智能体会得到一个带有大量测试的代码库。它需要删除代码和文件，同时保证代码库仍能正常运行，并移除指定的、可测试的功能。合成任务要求模型重新实现该功能，测试则作为可验证的奖励。

大规模创建合成任务会带来一个后果，那就是模型可能以意想不到的方式进行奖励投机。随着能力增强，Composer 2.5 能够找到越来越复杂的变通方法来解决当前任务。在一个例子中，模型发现了遗留的 Python 类型检查缓存，并通过逆向分析其格式找到了已删除函数的签名。在另一个例子中，它发现并反编译了 Java 字节码，以重建第三方 API。我们使用智能体监控工具发现并诊断了这些问题。它们也说明，大规模强化学习需要越来越谨慎地开展。

分片 Muon 与双网格 HSDP

在持续预训练中，我们使用带有分布式正交化的 Muon。形成动量更新后，我们按模型的自然粒度运行 Newton-Schulz。注意力投影按每个 attention head 处理，堆叠的 MoE 权重则按每个 expert 处理。

主要开销来自 expert 权重的正交化。对于分片参数，我们先把形状相同的张量组成批次。然后通过 all-to-all，把各个分片组合成完整矩阵，再运行 Newton-Schulz。完成后，我们再次通过 all-to-all，把结果送回原来的分片布局。这些数据传输是异步进行的。当一个任务等待通信时，优化器运行时会继续推进其他 Muon 任务，使网络通信与计算重叠。该方法等价于完整矩阵 Muon，但能让分片组保持忙碌。在 1T 模型上，优化器单步耗时为 0.2 秒。

这与我们为 MoE 模型使用 HSDP 的方式密切相关。HSDP 会建立多个 FSDP 副本，并在对应分片之间对梯度执行 all-reduce。我们为非 expert 权重和 expert 权重使用不同的 HSDP 布局。非 expert 权重相对较小，因此其 FSDP 组可以保持较窄的范围，通常位于单个节点或机架内。expert 权重包含大部分参数，也承担大部分 Muon 计算，因此使用更宽的 expert 分片网格。

把两种布局分开，还能让彼此独立的并行维度重叠运行。CP=2 和 EP=8 可以在 8 张 GPU 上运行，不必像单一共享网格那样需要 16 张 GPU。这样既避免了为较小的非 expert 状态进行宽范围通信，也能把 expert 优化器的工作分散到更多 GPU 上。

试用 Composer 2.5

Composer 2.5 的价格是每百万输入 token 0.50 美元，每百万输出 token 2.50 美元。

它还提供速度更快但智能水平相同的版本，价格是每百万输入 token 3.00 美元，每百万输出 token 15.00 美元。这个价格低于其他前沿模型的快速档位。与 Composer 2 类似，fast 是默认选项。完整信息请参阅我们的模型文档。

Composer 2.5 在上线第一周提供双倍用量。
