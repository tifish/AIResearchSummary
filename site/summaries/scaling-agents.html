扩展长期运行的自主编程能力

Scaling long-running autonomous coding

一句话总括：Cursor 通过数百个智能体连续运行数周的实验发现，大规模自主编程可以取得实质进展，但前提是用清晰的角色分工取代扁平协调，并为不同角色选择合适的模型。

实验规模

这不只是增加智能体数量，而是重新设计协作方式

单个智能体擅长边界清晰的任务。面对需要人类团队数月完成的项目，它的速度和持续专注能力仍然不足。Cursor 因此让数百个智能体并行工作，并观察它们如何处理同一个大型代码库。

实验累计使用了数万亿 token。智能体写出了超过百万行代码。真正的发现是，规模本身并不能解决协作问题。系统结构决定了新增算力能否转化为有效进展。

100 万+浏览器实验生成的代码行数
1,000浏览器项目涉及的文件数
3 周+Solid 到 React 的迁移时间
25×Rust 版本带来的视频渲染提速

失败路线

平等自治容易变成等待和回避

第一套方案让所有智能体通过共享文件自行认领任务。锁用于避免重复领取，但很快成为瓶颈。20 个智能体最终只发挥出约 2 到 3 个智能体的吞吐量，大部分时间都耗在等待上。

改用乐观并发控制后，状态写入更可靠，但更深层的问题没有消失。没有层级时，智能体会倾向于选择小而安全的修改。困难任务和端到端责任无人承担。

协调机制暴露的风险

  • 智能体长时间持有锁，甚至忘记释放。
  • 执行失败时，锁和共享状态可能一起失效。
  • 所有智能体都能选择任务，却没有谁对整体结果负责。

结论：减少冲突并不等于推动项目。系统还需要明确的决策权和责任边界。

有效结构

规划者、执行者和裁判组成循环

规划者

持续探索代码库并创建任务。它们还能派生子规划者，让规划本身并行展开。

→

执行者

只关注分配到的任务。完成后直接推送修改，不承担全局协调工作。

→

裁判

在每轮结束时判断是否继续。下一轮重新开始，以减轻漂移和视野变窄。

这种流水线解决了大部分协调问题。单个智能体不再需要同时理解全局、争抢任务和完成实现。角色分离让数百名执行者可以并行工作，同时保持项目方向。

真实项目

长时间运行已经产生可测量的工程成果

+26.6 万/-19.3 万

Cursor 代码库中的 Solid 到 React 原位迁移规模。实验持续超过三周，并已通过 CI 和早期检查，但仍需仔细审查。

7,400 次提交

Java LSP 实验达到约 55 万行代码。它说明长时间自治不仅能生成一次性原型，也能持续积累大型工程修改。

160 万行

Excel 实验在约 12,000 次提交后达到的规模。另一个 Windows 7 模拟器实验已有 14,600 次提交和 120 万行代码。

这些数字不能直接证明代码已经达到生产质量。它们证明的是另一件事：新加入的智能体仍能理解不断扩大的代码库，并在冲突较少的情况下继续推进工作。

模型与提示词

长期自治不是通用模型之间的简单排名

Cursor 发现，GPT-5.2 在长时间执行中更能遵循指令、保持专注、避免漂移，并完整落实要求。Opus 4.5 更容易提前停止，或在方便时采用捷径。

角色差异也很重要。GPT-5.2 的规划表现优于 GPT-5.1-Codex，尽管后者专门面向编程训练。因此，Cursor 开始按角色选模型，而不是让一个模型承担全部职责。

框架和模型都会影响结果，但提示词对协调、专注和异常行为的影响更大。

最值得带走的洞见

规模有效，但只有适度结构才能把规模变成进展

角色分工优于扁平协调规划者负责方向，执行者负责交付，裁判负责决定是否进入下一轮。

简单结构往往更有效整合者角色增加了瓶颈，而执行者已经能自行解决多数冲突。
模型应按角色选择编程专用模型不一定是更好的规划者，长期任务尤其需要持续专注能力。
定期刷新仍不可少智能体会运行过久，也会出现漂移和视野变窄。当前系统离最优解仍有距离。

我们一直在尝试让编程智能体自主运行数周。

我们的目标，是了解智能体编程的边界究竟能推进多远。实验针对的项目，通常需要人类团队花费数月才能完成。

本文介绍了我们从实验中得到的经验。我们让数百个智能体在同一个项目上并发工作，协调它们的任务，并观察它们写出超过 100 万行代码、消耗数万亿 token。

单个智能体的局限

如今的智能体能够很好地完成目标明确的任务，但处理复杂项目时速度较慢。自然的下一步，是并行运行多个智能体。然而，如何协调它们并不容易。

我们最初认为，预先规划会显得过于僵化。大型项目的推进路径并不明确，一开始也很难确定合理的任务分工。因此，我们先尝试了动态协调。每个智能体根据其他智能体正在进行的工作，自行决定下一步做什么。

学习如何协调

最初的方案让所有智能体地位平等，并通过一个共享文件自行协调。每个智能体会查看其他智能体正在做什么，认领一项任务，再更新任务状态。为了避免两个智能体拿到同一任务，我们加入了锁机制。

这套方案以一些很有意思的方式失败了：

智能体会长时间持有锁，有时甚至完全忘记释放。即使锁机制正常工作，它也会成为瓶颈。20 个智能体的有效吞吐量会下降到只相当于 2 到 3 个智能体。大部分时间都浪费在等待上。

系统也很脆弱。智能体可能在持有锁时失败，尝试获取自己已经持有的锁，或者根本没有先获取锁就更新协调文件。

我们尝试用乐观并发控制取代锁。智能体可以自由读取状态。如果状态在上次读取后发生变化，写入就会失败。这套机制更简单，也更可靠，但更深层的问题仍然存在。

由于缺少层级，智能体会变得厌恶风险。它们避开困难任务，转而进行小而安全的修改。没有智能体愿意负责棘手问题，也没有谁承担端到端实现的责任。结果是系统长时间反复忙碌，却没有真正取得进展。

规划者与执行者

下一套方案拆分了不同角色。我们不再使用每个智能体什么都做的扁平结构，而是建立了一条职责明确的流水线。

规划者持续探索代码库并创建任务。它们可以针对特定领域派生子规划者，让规划工作本身也能并行和递归展开。

执行者领取任务后，只专注于完成任务。它们不与其他执行者协调，也不需要操心全局。它们会持续处理分配到的任务，直到任务完成，然后推送修改。

每轮结束时，一个裁判智能体会判断是否继续。随后，下一轮会从全新的状态开始。这解决了大部分协调问题，也让系统可以扩展到非常大的项目，同时避免任何单个智能体陷入狭窄视角。

连续运行数周

为了测试这套系统，我们给它设定了一个雄心勃勃的目标：从零构建一个 Web 浏览器。智能体运行了接近一周，在 1,000 个文件中写出了超过 100 万行代码。你可以在 GitHub 上查看源代码。

尽管代码库已经很大，新加入的智能体仍能理解它，并继续完成有意义的工作。数百名执行者同时运行，将修改推送到同一个分支，而冲突很少。

从截图上看，它或许显得很简单，但从零构建浏览器是一项极其困难的工作。

另一个实验，是在 Cursor 代码库中将 Solid 原位迁移到 React。这个过程持续了三周以上，产生了增加 26.6 万行、删除 19.3 万行的修改。它仍然需要仔细审查，但当时已经通过了我们的 CI 和早期检查。

我们还尝试改进一款即将推出的产品。一个长期运行的智能体用高效的 Rust 版本，将视频渲染速度提高了 25 倍。它还加入了平滑缩放和平移功能。画面会跟随光标，并使用自然的弹簧过渡和运动模糊。这些代码已经合并，很快会进入生产环境。

我们还有几个正在运行的有趣实验：

  • Java LSP：7,400 次提交，55 万行代码
  • Windows 7 模拟器：14,600 次提交，120 万行代码
  • Excel：12,000 次提交，160 万行代码

我们的发现

为了实现同一个目标，我们已让这些智能体消耗了数万亿 token。系统的效率并不完美，但效果远远超出我们的预期。

对于运行时间极长的任务，模型选择非常重要。我们发现，GPT-5.2 系列更擅长长时间自主工作。它们更能遵循指令、保持专注、避免漂移，并精确而完整地落实要求。

Opus 4.5 往往更早停止，也更容易在方便时采用捷径，并迅速交回控制权。我们还发现，不同模型擅长不同角色。GPT-5.2 比 GPT-5.1-Codex 更适合作为规划者，尽管后者是专门针对编程训练的模型。现在，我们会为每个角色选择最合适的模型，而不是使用一个通用模型承担所有工作。

我们的许多改进来自减少复杂度，而不是增加复杂度。最初，我们设置了一个整合者角色，负责质量控制和冲突解决。后来我们发现，它造成的瓶颈比解决的问题更多。执行者本身已经能够处理冲突。

最好的系统往往比想象中更简单。起初，我们尝试借鉴分布式计算和组织设计中的系统。但并不是所有这类系统都适用于智能体。

合适的结构程度位于两个极端之间。结构太少，智能体就会彼此冲突、重复工作并偏离目标。结构太多，系统又会变得脆弱。

令人意外的是，系统的许多行为最终都取决于我们如何提示智能体。要让它们顺利协调、避免病态行为，并在长时间运行中保持专注，需要进行大量实验。运行框架和模型都很重要，但提示词更重要。

下一步

多智能体协调仍是一个困难的问题。我们目前的系统可以工作，但离最优状态还很远。规划者应该在任务完成时自动唤醒，并规划下一步。有些智能体偶尔会运行得过久。为了对抗漂移和视野变窄，我们仍然需要定期重新开始。

不过，对于那个核心问题，也就是能否通过投入更多智能体来扩展自主编程能力，我们得到的答案比预期更乐观。数百个智能体可以在同一个代码库中协作数周，并在雄心勃勃的项目上取得真正的进展。
