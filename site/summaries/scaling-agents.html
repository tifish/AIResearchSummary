百个 agent、百万行代码：自主编码能不能靠"堆数量"扩展？

来源 Cursor · 2026-01-14 · 作者 Wilson Lin · 原文
一句话总括：这篇文章真正的贡献不是"多 agent 很厉害"，而是用百万行级的真实实验，记录了把上百个编码 agent 协调到一起所必经的失败路径——并得出一个反直觉结论：让 agent 协作好的关键既不是模仿分布式系统、也不是堆更多角色，而是恰到好处的中等结构加上比模型和工具更重要的 prompt 工程。

01单个 agent 不是不行，是太慢——而并行的真正难题是"协调"

当下的 agent 擅长聚焦的小任务，但面对人类团队要花数月的复杂项目就显得很慢。直觉的下一步是并行跑多个 agent，可一旦并行，怎么协调它们就成了核心障碍。作者一开始判断"提前规划太僵硬"，因为大项目的路径本就模糊、分工也不明显，于是先选了动态协调。

把多个 agent 凑在一起容易，让它们不互相踩脚才是真问题——这决定了后面所有的架构演进。

02"平权 + 加锁"失败了：20 个 agent 跑出 2-3 个的吞吐

最初让所有 agent 地位平等、通过一个共享文件自协调：各自查看别人在做什么、认领任务、更新状态，再用锁防止两个 agent 抢同一任务。结果锁成了瓶颈——agent 长期持锁甚至忘记释放，系统脆弱（持锁时崩溃、重复获取已持有的锁、不加锁就改协调文件）。

20 → 2~3
加锁后有效吞吐塌缩：大部分时间都在等待

换成乐观并发控制（自由读、若状态已变则写入失败）后更简单稳健，但更深的问题暴露出来：没有层级，agent 变得规避风险——专挑安全的小改动，没人对硬问题或端到端实现负责，工作长时间空转。

协调机制的 bug 只是表象，真正的病根是"扁平结构让没人愿意扛难题"。

03解法是分层流水线：规划者出活、工作者埋头、裁判把关

放弃"人人都做所有事"，改成职责分明的管线：

  • 规划者（Planners）：持续探索代码库、生成任务，并可针对特定领域递归派生子规划者，让规划本身也并行。
  • 工作者（Workers）：只认领并死磕单个任务，不和别人协调、不操心全局，完成即推送。
  • 裁判（Judge）：每轮结束判断是否继续，然后下一轮全新开始，避免 agent 钻牛角尖（tunnel vision）。

用"分工 + 周期性重启"替代"加锁 + 共识"，绕开了几乎全部协调难题，才得以扩展到超大项目。

04真实战绩：从零造浏览器、三周迁移框架、渲染提速 25 倍

05模型不是越"会写代码"越好：GPT-5.2 守得住，Opus 4.5 易走捷径

在极长任务上模型差异显著：GPT-5.2 更擅长守指令、保持专注、避免漂移、把事做完整；而 Opus 4.5 倾向于提前停下、在方便时走捷径、快速交还控制权。更反直觉的是，GPT-5.2 作为"规划者"竟优于专为编码训练的 GPT-5.1-Codex。因此团队改为"每个角色用最适合的模型"，而非一个万能模型。

"训练目标是编码"不等于"擅长长程自主工作"；角色化选模比追求单一最强模型更实际。

06最大的杠杆是 prompt，最好的优化常是"做减法"

很多改进来自移除复杂性而非增加：他们曾建一个专门的 integrator 角色做质检与冲突解决，结果它制造的瓶颈比解决的还多——工作者自己就能处理冲突。试图照搬分布式计算和组织设计的模型，也不是都适用于 agent。结构要"中等"：太少则冲突、重复、漂移；太多则脆弱。而系统行为很大程度取决于 prompt——作者直言"harness 和模型重要，但 prompt 更重要"。

在 agent 系统里，prompt 工程往往是比换模型、加角色更高的杠杆。

07仍未最优：漂移、跑过头与"定期重启"的取舍

多 agent 协调仍是难题：规划者应在任务完成时被唤醒去规划下一步，agent 偶尔会跑太久，仍需周期性的全新开始来对抗漂移与钻牛角尖。但对于核心问题——"靠堆更多 agent 能否扩展自主编码"——答案比预期更乐观：上百个 agent 可以在单一代码库上协作数周，在雄心勃勃的项目上取得真实进展。

现状"能用但远非最优"，恰恰说明这是一个仍有巨大上升空间的工程方向。

最值得带走的 5 条洞见

  1. 协调比智能更稀缺：百万行实验里，决定成败的不是单个 agent 多聪明，而是上百个 agent 如何分工不打架——加锁会把 20 个 agent 的吞吐压成 2-3 个。
  2. 层级化（规划者/工作者/裁判）打败扁平自协调：扁平结构让 agent 集体规避难题，分层 + 周期性重启才让规模化成为可能。
  3. "会写代码"≠"能长跑"：GPT-5.2 在长程自主任务（守指令、不漂移、做完整）上胜过更早停手的 Opus 4.5，甚至当规划者也优于专训编码的 GPT-5.1-Codex，应按角色选模。
  4. 做减法常优于做加法：删掉专门的 integrator 角色反而更顺；照搬分布式/组织理论不一定适用，结构取"中等"为佳。
  5. prompt 是最高杠杆：团队的明确判断是"harness 和模型重要，但 prompt 更重要"——多 agent 系统的可靠性主要靠 prompt 调出来。
