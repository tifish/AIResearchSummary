一句话总括

多 agent 编程能长期推进复杂项目，但前提是把“自由协作”改造成“适度分工”。

Cursor 的核心贡献不是展示 agent 会写很多代码，而是说明在周级自主运行中，真正决定成败的是协作结构、角色边界、模型选择、提示设计和周期性重启。

数百个 并发 agent 在同一项目上工作
100 万+ 浏览器实验生成代码行数
25x 某产品视频渲染优化幅度

单个 agent 能做聚焦任务，但大型项目的瓶颈很快变成协作

文章先指出，今天的 agent 已经适合完成范围清楚的编码任务，但面对需要人类团队数月完成的工程，单体 agent 会太慢。把多个 agent 并行运行是自然方向，但并行本身不会自动带来效率。

为什么这改变理解：AI 编程能力的上限不只取决于单次补全或单个任务成功率，还取决于多个智能体能否在同一代码库中避免重复、冲突和方向漂移。

扁平自协调失败在锁、等待和无人负责难题

Cursor 最初让所有 agent 地位相同，通过共享文件认领任务，并用锁避免重复领取。结果是锁经常被长期持有或遗忘释放，即便机制正确，20 个 agent 的吞吐也会退化到大约 2-3 个 agent，大量时间耗在等待上。

后来他们尝试乐观并发控制，写入冲突时失败重试，机制更简单也更稳，但仍没解决深层问题：没有层级时，agent 会变得保守，只做小而安全的改动，难题和端到端实现没人真正承担。

结论：多 agent 系统不能只复制分布式锁或共享状态表；如果没有明确责任分配，系统会忙碌很久却缺少实质进展。

Planner、worker、judge 的分工让系统获得可扩展性

更有效的方案是角色分离：planner 持续探索代码库、拆分任务，并能生成子 planner；worker 只专注完成分配任务，不再关心全局协调；每轮结束后由 judge 判断是否继续，并开启新的迭代。

这种结构减少了 worker 之间的协商成本，也避免单个 agent 陷入局部视野。它说明长期自主编码需要的不是完全自由的群体智能，而是足够简单、能反复刷新上下文的流水线。

周级实验显示规模化可行，但仍需要人工级审查

Cursor 用这个系统从零构建浏览器，运行接近一周，写出超过 100 万行代码，覆盖 1,000 个文件；新 agent 仍能理解代码库并继续推进，数百个 worker 能并发推到同一分支且冲突较少。

另一个实验是把 Cursor 代码库中的 Solid 原地迁移到 React，耗时超过三周，产生 +266K/-193K 改动，CI 和早期检查通过，但仍需要细致 review。还有一个产品优化中，长期 agent 用 Rust 版本把视频渲染提升 25x，并加入平滑缩放、平移、弹簧过渡和运动模糊，相关代码已合并。

为什么重要：这些例子把 agent 从“完成 issue”推进到“承担大型工程改造”，但通过 CI 不等于生产级可信，审查和质量门禁仍是关键环节。

长期运行中，模型角色匹配和提示设计比想象中更关键

文章称他们在单一目标上消耗了 trillions of tokens，并发现 GPT-5.2 更适合长时间自主工作：更能遵循指令、保持专注、避免漂移并完整实现。Opus 4.5 更倾向较早停止或走捷径；GPT-5.2 作为 planner 也优于专门面向编码训练的 GPT-5.1-Codex。

他们还发现，减少复杂度常比增加角色更有效。曾经加入的 integrator 角色本想负责质量控制和冲突解决，但实际制造了瓶颈，因为 worker 已能自行处理不少冲突。

洞见：长期 agent 系统的工程重点会从“选最强模型”转向“把不同模型放在合适角色中，并用提示约束其行为”。

仍未解决的问题是漂移、过长运行和下一步规划时机

Cursor 承认当前系统远未最优：planner 应在任务完成时被唤醒以规划下一步，agent 偶尔运行太久，系统仍需要周期性 fresh start 来对抗漂移和隧道视野。

这意味着多 agent 编程的未来不是完全无人监管，而是逐步形成更可靠的自治周期：计划、执行、评估、刷新，再进入下一轮。

最值得带走的洞见

  • 多 agent 编程的主要难题不是“让更多 agent 同时写代码”，而是让它们在同一目标下有清晰责任和低摩擦交接。
  • 扁平协作会放大等待、冲突和风险规避；planner-worker-judge 这种轻量层级更适合周级任务。
  • 大型 agent 工程需要周期性刷新上下文，否则长期运行会出现漂移、局部最优和无人承担难题。
  • 模型选择要按角色匹配：规划、执行、审查可能需要不同模型，而不是一个通用最强模型包办。
  • 真实影响会先出现在代码迁移、性能重写、原型扩展等工程场景，但生产落地仍离不开 CI、review 和质量控制。
