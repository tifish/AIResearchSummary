规模化长时自主编码：让数百个 agent 连写数周代码

来源：Cursor 博客 · 作者 Wilson Lin · 2026 年 1 月 14 日
真正的贡献不是"agent 能写代码"，而是 Cursor 用真实大型项目证明了：把数百个 agent 编排到同一个代码库上连续协作数周是可行的——但成败的决定因素既不是更聪明的分布式锁，也不是更复杂的组织结构，而是"刚刚好"的分层结构 + 精心打磨的 prompt。

01单个 agent 的天花板，逼出了多 agent 协调

为什么要做这件事

今天的 agent 擅长聚焦的小任务，但面对要人类团队数月完成的大项目就太慢。自然的下一步是并行跑多个 agent——难点全在"如何协调"。Cursor 一开始假设"提前规划太僵硬"，于是选了动态协调：让每个 agent 看别人在做什么、再决定自己做什么。

大项目的路径本身是模糊的，起点上"如何分工"并不明显——这正是协调成为核心难题的根源。

02平权自协调失败了：锁是瓶颈，无层级让 agent 变胆小

两次踩坑，定位出真正的问题

第一版让所有 agent 平权，通过一个共享文件认领任务，用加锁防止两个 agent 抢同一任务。结果锁被长期持有甚至忘记释放；即使锁正常工作也成了瓶颈。

20 → 2-3
20 个 agent 因锁竞争退化到 2-3 个的有效吞吐
脆弱
持锁时崩溃、重复加锁、不加锁就改文件

改用乐观并发控制（自由读、状态变了就写失败）后更简单、更健壮，但更深的问题浮现：没有层级，agent 集体变得风险规避——只做小而安全的改动，没人对难题或端到端实现负责，工作长期空转却无进展。

协调机制的技术形态（锁 vs 乐观并发）只是表层，真正杀死进度的是"无人担责"——结构缺失比机制缺陷更致命。

03解法是分层：Planner 出任务、Worker 埋头干、Judge 决定是否继续

用角色分工取代平权混战

Planner持续探索代码库、生成任务，还能为特定领域派生 sub-planner，让规划本身并行且递归。Worker领到任务就专注完成、不与他人协调、不操心全局，干完即推送。每轮结束由 Judge 判断是否继续，下一轮全新开始。

分层不是为了"管理"，而是为了同时消灭两个极端：平权带来的空转，和单 agent 的"隧道视野"。每轮 fresh start 是对抗漂移的关键设计。

"数百 agent 推同一分支冲突极少"颠覆了直觉——它说明当结构对了，并发规模本身不再是协作的敌人。

05模型不是越强越通用：按角色选模型

超长任务暴露出模型的真实差异

在数周级任务上，GPT-5.2 明显更擅长保持专注、避免漂移、实现得精确而完整；Opus 4.5 倾向更早停止、图方便走捷径、过快交回控制权。更反直觉的是：GPT-5.2 当 planner 优于专为编码训练的 GPT-5.1-Codex。于是 Cursor 改为"每个角色用最合适的模型"，而非一个通用模型打天下。

"为编码训练"≠"最会规划"。模型能力是分维度的，长时自主任务把这种差异放大到必须区别对待。

06最好的系统比你想的更简单：减法胜过加法

删掉 integrator，承认 prompt 比一切都重要

他们曾加一个 integrator 角色做质量控制与冲突解决，结果它制造的瓶颈比解决的还多——因为 worker 本就能自己处理冲突，于是砍掉。试图照搬分布式计算和组织设计的模型，发现并非都适用于 agent。结构的正确量在"中间"：太少则冲突、重复、漂移，太多则脆弱。

最关键的一句：harness 和模型都重要，但 prompt 更重要。让 agent 长期协调、不走偏、保持专注，靠的是大量 prompt 实验而非更多机制。

07仍未解决：协调依然是难题

乐观但清醒的结论

系统能用但远未最优：planner 应在任务完成时被唤醒去规划下一步；agent 偶尔会跑太久；仍需周期性 fresh start 对抗漂移和隧道视野。但核心问题——"能否靠堆更多 agent 来扩展自主编码"——答案比预期更乐观：数百个 agent 真能在同一代码库上协作数周，对雄心勃勃的项目取得真实进展。

最值得带走的 5 条洞见

  1. 规模可行性已被证明：数百 agent × 数周 × 百万行代码 × 数万亿 token，多 agent 自主编码不再是理论假设。
  2. 结构要"刚刚好"：平权自协调会让 agent 集体变胆小空转；过度结构又脆弱。Planner/Worker/Judge 的轻分层是当前最优解。
  3. prompt 比 harness 和模型更重要：真正决定协调质量、防漂移、保专注的是 prompt 设计，而非更复杂的机制。
  4. 按角色选模型：GPT-5.2 更适合长时自主与规划，Opus 4.5 易走捷径早停；"为编码训练"不等于最会规划。
  5. 减法常胜加法：删掉 integrator 反而更顺——能删掉的复杂度，往往比新增的机制更有价值。
