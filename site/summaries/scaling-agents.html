一句话总括

扩展长时间运行的自主编码

Scaling long-running autonomous coding

Cursor 的核心贡献是把多代理编码从“能否并行”推进到“如何长期组织”。实验显示，代理规模能带来真实进展，但前提是角色分工、周期性重启和提示设计足够克制。

数百并发代理在同一项目上工作
100万+浏览器实验生成代码行数
1000浏览器实验涉及文件数
数万亿投入到单一目标的 token 量级

平级协作不是自然答案

文章先否定了一个直觉：让代理共享状态并自行抢任务，并不会自动形成高吞吐团队。

锁机制让二十个代理退化到两三个代理的有效吞吐。问题不只是工程实现粗糙，而是等待、遗忘释放锁和绕过锁会不断放大系统脆弱性。

瓶颈：锁让代理排队，等待时间吞掉并行收益。
脆弱：代理可能持锁失败、重复申请锁，或直接跳过协调协议。
避险：没有层级时，代理倾向挑小任务，困难任务无人负责。

有效结构来自角色拆分

1

规划者探索代码库

规划者持续发现工作，并能派生子规划者。规划本身也被并行化。

2

工人只完成任务

工人不负责全局协调。它们领取任务、实现、处理冲突，然后推送结果。

3

评审者决定续跑

每轮结束后由 judge agent 判断是否继续。新一轮可以减少漂移和视野收窄。

实验规模改变了判断标准

实验 关键数据 读者应如何理解
从零构建浏览器 接近一周，超过 100 万行代码，跨 1000 个文件。 重点不是浏览器已经可替代成熟产品，而是新代理仍能理解大代码库并继续推进。
Solid 到 React 迁移 超过三周，+266K/-193K 修改，通过 CI 和早期检查。 长周期代理可以承担真实迁移任务，但仍需要严肃人工审查。
视频渲染优化 Rust 版本让渲染快 25 倍，并加入缩放、平移和运动模糊。 多代理并不只会生成大量代码，也可能交付可合并的性能改进。
仍在运行的项目 Java LSP 7.4K commits、550K LoC；Windows 7 emulator 14.6K commits、1.2M LoC；Excel 12K commits、1.6M LoC。 这些数字显示实验已进入软件工程尺度，而不是短 demo 尺度。

真正的变量不是代理数量

结构太少会冲突、重复和漂移；结构太多会变脆。长期自主编码需要的是中等约束，而不是完全自治。
模型差异会累积。Cursor 认为 GPT-5.2 更适合长时间自主工作，尤其是在遵守指令、保持焦点和完整实现上。
复杂角色未必有用。他们曾加入 integrator 做质量控制和冲突解决，但它反而制造瓶颈，最后被移除。

最值得带走的洞见

并行不是免费午餐。共享文件和锁看似简单，但会把代理群变成等待队列。
层级解决责任问题。规划者承担拆解和探索，工人承担交付，困难任务才不容易被绕开。
提示是系统设计的一部分。文章强调，协调、聚焦和避免病态行为，很大程度依赖反复打磨提示。
长期运行需要刷新机制。当前系统仍要周期性 fresh start，避免代理运行过久后漂移或陷入局部视角。
AI 对软件工程的影响更像组织问题。模型能力重要，但把代理组织成可持续推进的流程同样关键。
