Warp Decode：把并行轴从“专家”翻转到“输出”

来源：Cursor · 2026 年 4 月 6 日 · 作者：Less Wright、Federico Cassano、Zhiyuan Zhang
一句话总括：在 MoE 模型“每次只产出一个 token”的解码阶段，传统按专家组织的内核有一大半步骤只是在搬数据；Cursor 把并行轴翻转成按输出组织——每个 warp 只算一个输出值——从而在 Blackwell GPU 上同时拿到 1.84x 吞吐和 1.4x 更高的精度。这是一个罕见的“既更快又更准”的内核。

传统路径中纯“记账”的阶段

1问题：解码阶段，专家中心的内核在“空转”

关键数据与趋势

现代 MoE 每层把每个 token 路由到一小撮专家（例如从 128 个里选 8 个）。主流实现围绕专家组织计算：收集每个专家要处理的 token、批量算、再重组。这在 prefill 和大批量下很划算，因为同一专家被很多 token 共享，整理数据的开销被真实计算摊薄。

但在自回归解码阶段，一次只产出一个 token，专家间没有足够的共享计算量。结果是：传统八阶段流水线里有五个阶段纯粹在为“专家视图”整理数据布局，不做任何实际运算。

结论：解码阶段的瓶颈不是算力，而是为专家中心布局付出的、无法摊薄的记账开销。

2做法：每个 warp 只负责一个输出标量

关键数据与趋势

GPU 以 32 条并行 lane 为一个 warp 执行。新做法让每个 warp 只算一个输出值：直接从显存流式读取所需权重，把八个路由专家的贡献在私有寄存器里累加成一个 FP32 running total，最后只写一次结果。整个 MoE 计算层被压成两个内核——moe_gate_up_3d_batched 和 moe_down_3d_batched。

down 内核里，每个 warp 负责一个 token 的一个输出维度，遍历 top-k 专家、把路由权重直接折进累加器；最后用 __shfl_xor_sync（编译成单条 shfl.sync.bfly 硬件指令）在 lane 之间做蝶形归约，完全绕开 shared memory——没有 L1 往返、没有 bank conflict、没有显式 barrier。

结论：“按输出组织”让每个 warp 在整个生命周期里只有一个稳定任务，从根上消除了暂存、跨 warp 同步和中间缓冲。

3提速来源：删阶段、删缓冲、再加上 warp 独立

关键数据与趋势

传统路径

  • 把每个专家的 token 列表 padding 到 2 的幂 / 128 字节边界
  • scatter：八份中间结果写回显存
  • combine：再跑一遍归约把它们合起来
  • 激活 gather 缓冲（batch=1 时是已有数据的整份拷贝）
  • 每 token 的专家输出缓冲：8×2048×2B = 32KB

Warp Decode

  • 从不形成“每专家批次”，padding 直接消失
  • 路由权重折进累加器，八份中间结果不再落地
  • 省下 combine 这一整趟读写
  • 无需 gather 缓冲
  • 去掉 32KB+ 的中间缓冲流量，腾出 L2 给真正决定性能的权重行

更妙的是保留下来的计算也更快：内核“天生 embarrassingly parallel”——warp 之间没有共享可变状态。调度器可以任意顺序发射任何 warp，某个 warp 等访存时立刻切到别的；B200 的 148 个 SM 上同时有成千上万 warp 在飞，访存延迟几乎被其他 warp 的有用计算完全掩盖。输出维度或批大小翻倍，独立 warp 数线性翻倍，无需额外同步。

结论：大部分吞吐增益来自“删阶段”，而 warp 独立则让剩余计算的延迟被调度自然隐藏——两者叠加才有 1.84x。

4为什么还更准：少了一次 MXFP8 来回

关键数据与趋势

传统路径把 BF16 激活转成 MXFP8 再转回，会引入一个随层数累积的舍入误差底噪。Warp decode 全程保持激活为 BF16、累加器为 FP32，归约从不在被降级的输入上进行。结果：输出比传统路径与 32 位真值接近 1.4x。正确性也很扎实——最小余弦相似度 > 0.999996，最大绝对误差 0.001953。

结论：这次提速不是用精度换来的；恰恰相反，去掉中间量化步骤让速度和精度同向改善。

5离硬件极限有多近，以及它的适用边界

关键数据与趋势

B200 连续读（0,1,2,3）实测峰值 6.8 TB/s；warp decode 在 B=32 时持续 3.95 TB/s，即峰值的 58%。剩下的差距主要来自专家路由带来的随机访存——一个 token 可能路由到 5、8、14、19 这种不相邻的专家。吞吐增益在所有上下文长度桶上保持一致，证明这是纯粹的生成期收益，与 prompt 长度无关。

但作者明确：warp decode 不是专家中心执行的通用替代品。prefill 和大批量推理里很多 token 共享同一专家，整理开销能被足够的真实计算摊薄，专家中心打包仍然更优。warp decode 只在“每专家共享工作量不足以摊薄开销”时才赢——而这恰是 MoE 解码的常态。

结论：它是为特定场景（小批量解码）量身打造的工具，价值在于让 Composer 的迭代更快、输出更准地抵达开发者。

最值得带走的洞见

  1. 换并行轴，而不是优化老流水线。把“按专家”改成“按输出”，五个记账阶段直接消失——有时最大的优化是重组问题，而非加速旧步骤。
  2. 解码 ≠ prefill。同一个 MoE，在批量大时该用专家中心打包，在一次只出一个 token 时该用 warp decode；没有放之四海皆准的内核。
  3. “更快”和“更准”可以同源。去掉中间 MXFP8 量化既省了一趟数据搬运，又消除了累积舍入误差——1.84x 吞吐与 1.4x 精度来自同一个简化。
  4. warp 独立是性能的杠杆。无共享可变状态、单一稳定任务，让调度器用海量在飞 warp 把访存延迟藏起来，这是它“embarrassingly parallel”的本质。
  5. 推理投资和训练投资互补。预训练数据与 RL 决定模型输出的质量，而 warp decode 这类推理优化决定这些输出能多快、多准地送到用户手里。
