用 warp decode 改进 MoE 模型推理

Better MoE model inference with warp decode

一句话总括：warp decode 把 MoE 小批量解码的并行单位从专家翻转为输出值，用更少的数据搬运同时换来更高吞吐量和更小数值误差。

1.84×Blackwell 上的吞吐量提升
1.4×输出更接近完整 FP32 参考结果
3.95 TB/sB=32 时的持续内存吞吐量
32+ KB每个 token 省去的中间流量

问题不在计算量，而在组织方式

传统 MoE 路径以专家为中心。它先收集每位专家需要的 token，再执行计算，最后重新组合结果。

这种方法适合 prefill 和大批量推理。但在一次只生成一个 token 的解码阶段，组织数据的成本无法被摊薄。

传统八阶段流程中有 五个阶段不执行实际计算。它们只为专家中心的数据布局服务。

结论：小批量解码的主要机会不是加速同一套流程，而是改变并行轴。

传统路径：专家拥有工作

系统形成专家批次，并执行 padding、scatter、combine 和中间缓冲区读写。单 token 解码无法有效利用这些准备工作。

风险：数据编排本身会吞掉带宽和延迟预算。

warp decode：输出值拥有工作

每个 warp 在整个生命周期内只负责一个输出标量。它直接读取所需权重，并把八位专家的贡献累加到 FP32 寄存器中。

机会：独立 warp 为调度器提供大量无依赖任务，可用其他计算隐藏内存等待。

两枚 kernel 如何完成整层计算

  1. Gate/up 融合
    激活向量只读取一次。gate 与 up 两个点积在私有寄存器中累计，再计算 SiLU(gate) × up。
  2. Down 直接聚合
    每个 warp 遍历所有 top-k 专家。路由权重直接折入同一个 FP32 累加器。
  3. Warp 内归约
    __shfl_xor_sync 直接交换 lane 寄存器。流程不再需要共享内存往返和显式 barrier。
精度提升来自删除中间激活量化：激活始终保持 BF16，累加使用 FP32，因此不会反复经历 BF16 → MXFP8 → BF16 带来的舍入误差。

速度主要来自“删除”

warp decode 删除 padding、scatter 和 combine。它也不再创建 activation gather buffer 和 per-expert output buffer。

当隐藏维度为 2048、路由到 8 位专家时，传统 BF16 专家输出缓冲区达到 8 × 2048 × 2 = 32 KB。这些数据会被分配、写入、立即读取一次，然后丢弃。

省下的 L2 容量可以留给真正决定性能的权重行。由此可见，优化重点是减少中间状态，而不只是融合算术操作。

硬件效率仍有边界

B200 的连续内存读取实测峰值为 6.8 TB/s。warp decode 在 B=32 时达到 3.95 TB/s，即峰值的 58%。

剩余差距可能来自专家路由造成的随机访问。token 可能跳到 5、8、14、19 等不相邻的专家，而峰值测试读取的是连续地址。

这说明 kernel 已显著逼近硬件上限，但路由本身仍会带来不可忽略的内存延迟。

最值得带走的洞见

选择与负载匹配的并行轴专家中心布局并非普遍最优。单 token 解码更适合围绕输出标量组织工作。
减少数据搬运可以同时改善速度和精度删除中间量化与缓冲区后，系统既少读写数据，也少引入舍入误差。
独立 warp 是延迟隐藏的基础没有跨 warp 依赖后，B200 的 148 个 SM 可以在某个 warp 等待内存时调度其他任务。
优化有清晰适用范围warp decode 适合共享工作不足的小批量解码。prefill 和大批量推理仍可受益于专家中心打包。

多数 MoE 推理系统都以专家为中心来组织 token 生成路径。这种方式符合路由的工作方式，也一直是大规模系统的标准方案。

不过，我们发现，在 Blackwell GPU 上进行小批量解码时，围绕输出而不是专家来组织 kernel，效果更好。我们把这种方法称为“warp decode”。

我们最初思考的是，Blackwell 上的 MoE 解码最多能达到多高的内存带宽。这个问题促使我们彻底翻转并行轴。

我们不再把 warp 分配给专家，而是让每个 warp 负责一个输出值，也就是一个神经元。

能够同时提升性能和准确性的 kernel 并不多见，warp decode 正是其中之一。

在 Blackwell 上，它将吞吐量提升了 1.84 倍。与此同时，它生成的输出与完整 FP32 参考结果之间的距离缩小了 1.4 倍。

这项技术加快了 Composer 的研究和训练流程，让我们可以更快地改进模型，并更频繁地发布新版本。

传统 MoE 路径

现代 MoE 模型会让每个 token 经过一部分专门的专家网络。例如，模型可能在某一层的 128 位专家中选择 8 位。

标准实现围绕这些专家组织全部计算。系统先收集每位专家需要的 token，再执行计算，最后重新组装结果。

这种方式很适合 prefill 和大批量任务。此时，多项任务会共享同一位专家，因此组织数据的开销可以被摊薄。

但在自回归解码阶段，系统一次只生成一个 token。此时没有足够多的共享工作来抵消这些开销。

传统路径一共有八个阶段。其中五个阶段只是为了管理专家中心视角下的数据布局，并不执行任何实际计算。

我们做了哪些改变

warp decode 不再以专家为中心，而是围绕输出重新组织并行计算。这样便能删除上述五个“记账”步骤。

现代 GPU 会以 warp 为单位执行指令。每个 warp 包含 32 条并行处理通道。

在新方法中，每个 warp 只负责计算一个输出值。它直接从内存流式读取所需的权重数据，把八位已路由专家的结果聚合到同一个累计值中，最后写出一个结果。

这种 warp 独立性让 warp decode 不再需要暂存、交接、跨 warp 同步点或中间缓冲区。

整个 MoE 计算层被压缩为两枚 kernel：moe_gate_up_3d_batched 和 moe_down_3d_batched。

两枚 kernel 如何工作

在 gate/up kernel 中，每个协作线程阵列（cooperative thread array，CTA）包含八个 warp。

对于 token 与已路由专家的每一种配对，每个 warp 都负责一个中间神经元。

warp 先载入已路由专家的 ID，再读取该神经元对应的 gate 和 up 权重行，并流式遍历输入激活向量。

MXFP8 权重会即时转换为 FP32。两个点积都在私有寄存器中累计。

两个 kernel 的相关操作被融合到同一次处理中，因此激活向量只需读取一次，并立即供两个投影复用。整个过程不需要通过共享内存暂存数据。

完成 warp 级归约后，warp 会计算 SiLU(gate) × up，并写入一个中间值。

在 down kernel 中，每个 warp 负责一个 token 的一个输出维度。

它会遍历所有 top-k 已路由专家。处理过程中，它载入对应的下投影权重行，并流式遍历中间激活。

每位专家的路由权重也会直接折入同一个持续更新的 FP32 累加器。

处理完全部专家后，我们使用 __shfl_xor_sync 执行 warp 级蝶形归约，把 32 条 lane 上的局部和归约到一起。

该操作会直接编译为 PTX 的 shfl.sync.bfly 指令。这条硬件原语可以在同一 warp 的不同 lane 之间交换寄存器，完全绕过共享内存。

这样做的好处是，我们不再需要经过 L1 往返，也不会遇到共享内存 bank 冲突或显式 barrier。

同步已经通过 lane mask 内置于指令中。最终的加权 top-k 组合也不再作为独立的收尾阶段执行，而是直接成为投影的一部分。

warp decode 中的每个 warp 都彼此独立，并且在整个生命周期内承担一项固定任务：生成一个输出标量。

这种 warp 独立性删除了传统路径所需的共享内存暂存、跨 warp 同步和中间缓冲区。

简化并加速流水线

warp decode 主要通过两种机制提升性能。

第一种机制是删除传统路径所需的阶段和缓冲区。第二种机制是建立 warp 独立性，从而改善调度和延迟隐藏能力。

删除阶段

删除阶段贡献了大部分吞吐量提升。我们去掉了 padding、scatter 和 combine 步骤。

要删除这些阶段，必须从头改变并行计算的组织方式。仅仅融合传统流水线中的现有阶段并不足够。

删除 padding

传统路径：为了满足分组 kernel 的要求，系统会把每位专家的 token 列表填充到 2 的幂次边界或 128 字节边界。

在单 token 解码时，这项开销无法被摊薄。

warp decode 路径：它从不形成每位专家各自的批次，因此完全避开了这项开销。

删除 scatter 和 combine

传统路径：每位专家完成计算后，会把八个中间结果写入 GPU 内存。系统随后运行单独的归约步骤，将这些结果组合起来。

warp decode 路径：每位专家的路由权重会直接折入 warp 内持续更新的累加器。

八个中间结果不会真正出现在内存中，因此省去了写入成本，也省去了后续归约过程的读取成本。

删除缓冲区

新的组织方式还删除了传统路径需要的两块中间内存缓冲区。这两块缓冲区都是专家中心布局的产物。

第一块是 activation gather buffer。系统会把输入激活向量复制一份，并重新排列成以专家为主的布局。

当 batch size 为 1 时，这相当于完整复制一份已经存在的数据。

第二块是 per-expert output buffer。

当系统使用 8 位专家、隐藏维度为 2048 时，这块 BF16 缓冲区对每个 token 的大小是 8 × 2048 × 2 字节 = 32 KB。

系统会分配并写入这块缓冲区，随后立即读取一次，然后将其丢弃。

warp decode 把八位专家的贡献折入分布在 32 条 warp lane 上的寄存器累加器，因此删除了这两块缓冲区。

在最终的单标量写入发生前，不会有任何内容进入全局内存。

每个 token 减少 32 KB 以上的中间缓冲区流量后，L2 缓存便能为真正决定性能的权重行腾出更多空间。

Warp 独立性

新的组织方式也让保留下来的计算更快。因为从设计上看，这枚 kernel 属于“易并行”任务，每个 warp 都与其他 warp 完全独立。

每个 warp 只负责一个输出标量，并且只读取自己所需的权重行，因此 warp 之间不存在共享的可变状态。

在单个 warp 内部，这种独立性是彻底的。

输入激活是只读的。累加器位于私有寄存器中。输出会写入唯一的地址。

从硬件调度器的角度看，整个输出维度就是一组扁平、相互独立的工作项。

GPU 的 warp 调度器可以在任意时刻、以任意顺序发出任何 warp，而不会受到正确性约束。

当一个 warp 因等待内存载入而停顿时，调度器可以立即切换到另一个 warp。

B200 拥有 148 个流式多处理器。数千个 warp 可以同时处于执行状态，因此内存延迟几乎都能被其他 warp 的有效计算隐藏起来。

这枚 kernel 还可以线性扩展。

把输出维度扩大一倍，就会让独立 warp 的数量增加一倍，同时不增加任何同步操作。

token 的 batch 维度同样如此，因此调度器看到的是一个扁平的工作命名空间，其中的节点之间没有依赖边。

传统路径则不同。它使用面向专家的 GEMM kernel，需要在 block 内进行协调。

结果

大规模端到端解码吞吐量

我们在内部推理系统上进行了测试。该系统在 NVIDIA B200 GPU 上运行一个 Qwen-3 风格的模型。

测试结果显示，warp decode 带来了稳定的吞吐量提升。

各个上下文长度区间的吞吐量增幅都基本持平。这证明它带来的是纯粹的生成阶段改进，不依赖提示词长度。

准确性提升

删除中间激活量化步骤对质量产生了可测量的影响。

把 BF16 激活转换为 MXFP8，再转换回来，会引入一个舍入误差下限。该误差会在模型的各层之间不断累积。

warp decode 全程把激活保持为 BF16，并使用 FP32 累加器。因此，归约过程不会使用已经降质的输入。

最终，warp decode 的输出与完整 32 位基准真值之间的距离，比传统路径缩小了 1.4 倍。

硬件效率

我们在开发 warp decode 时，首先提出了一个问题：它能在多大程度上接近硬件的最高吞吐量？

使用 copy kernel 测得的结果显示，B200 连续内存读取的峰值为 6.8 TB/s。

warp decode 在 B=32 时可以持续达到 3.95 TB/s，相当于该峰值的 58%。

剩余差距可能来自专家路由产生的随机访问模式所带来的内存延迟。

每个 token 可能被路由到 5、8、14、19 等互不相邻的专家。

相比之下，峰值吞吐量是通过连续的 0、1、2、3 内存读取测得的。

在所有 batch size 下，与参考实现相比的正确性都非常接近：最低余弦相似度大于 0.999996，最大绝对差为 0.001953。

warp decode 与 Composer 训练

warp decode 并不能全面替代以专家为中心的执行方式。

prefill 和大批量推理等高吞吐负载仍然适合使用专家中心的打包方式。

在这些场景中，许多 token 会共享同一位专家。组织这些 token 的成本可以由足够多的实际计算摊薄，因此仍然划算。

当每位专家没有足够多的共享工作，无法抵消组织开销时，warp decode 更有优势。MoE 解码往往就是这种情况。

因此，warp decode 成为我们持续改进 Composer 的重要组成部分。

预训练数据和强化学习（RL）方面的投入决定模型输出的质量。warp decode 等推理技术方面的投入，则决定这些输出能以多快的速度、多高的准确性送达开发者。

这样做的好处是，我们不再需要经过 L1 往返，也不会遇到共享内存 bank 冲突或显式 barrier。

同步已经通过 lane mask 内置于指令中。最终的加权 top-k 组合也不再作为独立的收尾阶段执行，而是直接成为投影的一部分。

warp decode 中的每个 warp 都彼此独立，并且在整个生命周期内承担一项固定任务：生成一个输出标量。

这种 warp 独立性删除了传统路径所需的共享内存暂存、跨 warp 同步和中间缓冲区。

简化并加速流水线

warp decode 主要通过两种机制提升性能。

第一种机制是删除传统路径所需的阶段和缓冲区。第二种机制是建立 warp 独立性，从而改善调度和延迟隐藏能力。

删除阶段

删除阶段贡献了大部分吞吐量提升。我们去掉了 padding、scatter 和 combine 步骤。

要删除这些阶段，必须从头改变并行计算的组织方式。仅仅融合传统流水线中的现有阶段并不足够。

删除 padding

传统路径：为了满足分组 kernel 的要求，系统会把每位专家的 token 列表填充到 2 的幂次边界或 128 字节边界。

在单 token 解码时，这项开销无法被摊薄。

warp decode 路径：它从不形成每位专家各自的批次，因此完全避开了这项开销。

删除 scatter 和 combine

传统路径：每位专家完成计算后，会把八个中间结果写入 GPU 内存。系统随后运行单独的归约步骤，将这些结果组合起来。

warp decode 路径：每位专家的路由权重会直接折入 warp 内持续更新的累加器。

八个中间结果不会真正出现在内存中，因此省去了写入成本，也省去了后续归约过程的读取成本。

删除缓冲区

新的组织方式还删除了传统路径需要的两块中间内存缓冲区。这两块缓冲区都是专家中心布局的产物。

第一块是 activation gather buffer。系统会把输入激活向量复制一份，并重新排列成以专家为主的布局。

当 batch size 为 1 时，这相当于完整复制一份已经存在的数据。

第二块是 per-expert output buffer。

当系统使用 8 位专家、隐藏维度为 2048 时，这块 BF16 缓冲区对每个 token 的大小是 8 × 2048 × 2 字节 = 32 KB。

系统会分配并写入这块缓冲区，随后立即读取一次，然后将其丢弃。

warp decode 把八位专家的贡献折入分布在 32 条 warp lane 上的寄存器累加器，因此删除了这两块缓冲区。

在最终的单标量写入发生前，不会有任何内容进入全局内存。

每个 token 减少 32 KB 以上的中间缓冲区流量后，L2 缓存便能为真正决定性能的权重行腾出更多空间。

Warp 独立性

新的组织方式也让保留下来的计算更快。因为从设计上看，这枚 kernel 属于“易并行”任务，每个 warp 都与其他 warp 完全独立。

每个 warp 只负责一个输出标量，并且只读取自己所需的权重行，因此 warp 之间不存在共享的可变状态。

在单个 warp 内部，这种独立性是彻底的。

输入激活是只读的。累加器位于私有寄存器中。输出会写入唯一的地址。

从硬件调度器的角度看，整个输出维度就是一组扁平、相互独立的工作项。

GPU 的 warp 调度器可以在任意时刻、以任意顺序发出任何 warp，而不会受到正确性约束。

当一个 warp 因等待内存载入而停顿时，调度器可以立即切换到另一个 warp。

B200 拥有 148 个流式多处理器。数千个 warp 可以同时处于执行状态，因此内存延迟几乎都能被其他 warp 的有效计算隐藏起来。

这枚 kernel 还可以线性扩展。

把输出维度扩大一倍，就会让独立 warp 的数量增加一倍，同时不增加任何同步操作。

token 的 batch 维度同样如此，因此调度器看到的是一个扁平的工作命名空间，其中的节点之间没有依赖边。

传统路径则不同。它使用面向专家的 GEMM kernel，需要在 block 内进行协调。

结果

大规模端到端解码吞吐量

我们在内部推理系统上进行了测试。该系统在 NVIDIA B200 GPU 上运行一个 Qwen-3 风格的模型。

测试结果显示，warp decode 带来了稳定的吞吐量提升。

各个上下文长度区间的吞吐量增幅都基本持平。这证明它带来的是纯粹的生成阶段改进，不依赖提示词长度。

准确性提升

删除中间激活量化步骤对质量产生了可测量的影响。

把 BF16 激活转换为 MXFP8，再转换回来，会引入一个舍入误差下限。该误差会在模型的各层之间不断累积。

warp decode 全程把激活保持为 BF16，并使用 FP32 累加器。因此，归约过程不会使用已经降质的输入。

最终，warp decode 的输出与完整 32 位基准真值之间的距离，比传统路径缩小了 1.4 倍。

硬件效率

我们在开发 warp decode 时，首先提出了一个问题：它能在多大程度上接近硬件的最高吞吐量？

使用 copy kernel 测得的结果显示，B200 连续内存读取的峰值为 6.8 TB/s。

warp decode 在 B=32 时可以持续达到 3.95 TB/s，相当于该峰值的 58%。

剩余差距可能来自专家路由产生的随机访问模式所带来的内存延迟。

每个 token 可能被路由到 5、8、14、19 等互不相邻的专家。

相比之下，峰值吞吐量是通过连续的 0、1、2、3 内存读取测得的。

在所有 batch size 下，与参考实现相比的正确性都非常接近：最低余弦相似度大于 0.999996，最大绝对差为 0.001953。

warp decode 与 Composer 训练

warp decode 并不能全面替代以专家为中心的执行方式。

prefill 和大批量推理等高吞吐负载仍然适合使用专家中心的打包方式。

在这些场景中，许多 token 会共享同一位专家。组织这些 token 的成本可以由足够多的实际计算摊薄，因此仍然划算。

当每位专家没有足够多的共享工作，无法抵消组织开销时，warp decode 更有优势。MoE 解码往往就是这种情况。

因此，warp decode 成为我们持续改进 Composer 的重要组成部分。

预训练数据和强化学习（RL）方面的投入决定模型输出的质量。warp decode 等推理技术方面的投入，则决定这些输出能以多快的速度、多高的准确性送达开发者。
