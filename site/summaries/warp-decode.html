用 warp decode 改进 MoE 模型推理

Better MoE model inference with warp decode

Cursor 的贡献不是单纯写了更快的 kernel，而是把 MoE 解码从“围绕专家组织”改成“围绕输出值组织”，从源头减少数据搬运，并让每个 warp 独立完成一个标量输出。

1.84xBlackwell 上的端到端解码吞吐提升
1.4x输出更接近完整 FP32 参考
3.95 TB/sB=32 时持续内存带宽
32+ KB每 token 可移除的中间缓冲流量

传统路径的问题是组织成本

MoE 通常先按专家收集 token，再运行专家计算，最后把结果重新组合。这个设计适合 prefill 和大 batch。

但自回归 decode 一次只生成一个 token。共享工作太少，填充、scatter、combine 等步骤变成不可摊销的开销。

warp decode 改变并行对象

新路径让每个 warp 负责一个输出值。它直接读取所需权重，把 8 个路由专家的贡献合并到同一个 FP32 累加器。

这改变了读者对 MoE 推理的理解：瓶颈不只在算力，也在是否为小 batch 保留了大 batch 的数据组织方式。

两段 kernel 把路径压短

1

gate/up kernel

每个 CTA 有 8 个 warp。每个 warp 负责一个中间神经元，读取 gate 和 up 权重行。

2

一次读取激活

输入激活只读一次，并立即复用于两个投影。MXFP8 权重动态转为 FP32。

3

down kernel

每个 warp 负责一个输出维度，遍历 top-k 专家，并把路由权重折入累加器。

4

warp 内归约

__shfl_xor_sync 直接在 lane 间交换寄存器，避开共享内存和显式 barrier。

传统路径与 warp decode 的差异

维度传统专家中心路径warp decode
并行轴按专家组织 token 和计算。按输出标量组织 warp。
中间数据需要 activation gather buffer 和 per-expert output buffer。专家贡献保留在寄存器累加器中，最终才写回。
调度形态专家级 GEMM 需要块内协调。warp 之间互不依赖，调度器可用其他 warp 隐藏内存延迟。
适用场景prefill 和大 batch 仍然有优势。小 batch decode 更合适，尤其是共享专家工作不足时。
性能提升主要来自删除阶段，而不是把旧流水线简单融合。准确率提升来自避免 BF16 激活在中间被量化到 MXFP8 后再还原。 这说明推理系统优化会影响模型输出质量，而不只是影响响应速度。

最值得带走的洞见

数据布局决定吞吐

1.84 倍提升来自重新组织工作，而不是只调参。MoE decode 的真实成本包含大量数据搬运。

传统路径不总是通用最优

专家中心执行适合共享工作足够多的场景。小 batch decode 会放大它的管理成本。

独立 warp 更容易隐藏延迟

B200 有 148 个 SM。大量独立 warp 让调度器在内存等待时切换到其他有用工作。

速度和精度可以同时改善

warp decode 保持 BF16 激活和 FP32 累加，使输出比传统路径接近 FP32 参考 1.4 倍。推理工程因此成为模型质量迭代的一部分。

边界同样重要

作者明确说它不是专家中心执行的全面替代。prefill 和大 batch 仍然适合传统打包方式。
