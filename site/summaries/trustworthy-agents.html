可信 agent 的关键不只是模型，而是整套运行系统

原文：Trustworthy agents in practice

一句话总括：文章的真正贡献，是把“agent 如何值得信任”拆成可操作的产品、训练、安全和标准问题，而不是停留在原则宣言。

Agent 的风险来自自主循环

Anthropic 把 agent 定义为能自行决定流程和工具使用的 AI：它会规划、行动、观察结果、调整，再重复执行，直到任务完成或需要人类输入。

关键变化 是它不再只是聊天机器人，而能写代码、执行代码、管理文件、跨应用完成任务。自主性带来生产力，也放大了误解意图和误操作的风险。

因此，agent 治理的核心不是阻止自主性，而是让自主性始终处在可理解、可限制、可接管的边界内。

治理对象有四层，不是只有模型

文章把 agent 拆成 4 个组件：模型、harness、工具和环境。模型提供能力，harness 提供指令和护栏，工具连接外部服务，环境决定它能访问哪些文件、网站和系统。

一个训练良好的模型，仍可能被过度授权的工具、配置不当的 harness 或暴露的运行环境拖入风险。

这改变了对 AI 安全的常见理解：评估模型能力还不够，必须评估模型被放进什么产品结构里。

人类控制要从逐步批准升级为策略批准

Claude.ai 和 Claude Desktop 允许用户为工具和动作设置权限，例如 always allow、needs approval、block。这适合简单任务，但当任务包含几十个动作时，逐项确认会制造疲劳。

Claude Code 的 Plan Mode 让 Claude 先展示整体计划，用户可以审阅、编辑并批准，再让它执行，同时保留中途干预能力。

这里的洞见是：可信控制不等于更多弹窗，而是把人的判断放在最有价值的位置，也就是目标和策略层。

好的 agent 必须知道何时停下来问

文章承认目标理解仍是 agent 的难题：有些缺口可以自己查资料解决，有些则涉及偏好、授权或真实意图，只能由用户决定。

Anthropic 的做法包括构造模糊训练场景，强化 Claude 在不确定时暂停询问；Claude Constitution 也偏向“提出担忧、寻求澄清或拒绝继续”，而不是凭假设行动。

关键数据：复杂任务中，用户打断 Claude 的频率只比简单任务略高，但 Claude 主动 check in 的频率约翻倍。这说明更强的 agent 不应更莽撞，而应更会识别哪些决定需要交还给人。

Prompt injection 需要多层防线

Prompt injection 是藏在邮件、网页或文件里的恶意指令，例如要求模型忽略原指令并转发最近 10 封邮件给攻击者。agent 环境越开放、工具越多，攻击入口和后果就越大。

Anthropic 描述的防线包括训练模型识别注入模式、监控生产流量拦截真实攻击，以及让外部 red team 测试系统。

但文章也明确说这些不是保证。真正的安全取决于开发者、企业和用户共同限制工具、数据、权限与运行环境。

行业基础设施决定 agent 能否大规模可信

Anthropic 认为单家公司无法独自解决 agent 的可靠性和安全问题，生态还需要 3 类基础设施：标准化 benchmark、使用证据共享和开放标准。

文章特别指出，目前还没有严谨、标准化、独立验证的方法来比较 agent 抵抗 prompt injection 的能力，或比较它们发现不确定性并上报的可靠性。NIST 等标准机构可以在这里发挥作用。

Model Context Protocol 被描述为开放标准案例：把模型连接外部工具和数据源的规则开放出来，可以让安全属性嵌入基础设施，而不是每次部署再临时修补。

最值得带走的洞见

  • Agent 的治理对象是“模型 + 产品结构 + 工具权限 + 运行环境”，不能只看模型能力。
  • 有效的人类控制要减少无意义确认，把用户判断集中到计划、授权和例外处理上。
  • 复杂任务中，可靠 agent 应更频繁主动询问，而不是把不确定性伪装成自主完成。
  • Prompt injection 没有单点解法，必须用训练、监控、权限设计、环境隔离和红队测试共同防守。
  • 标准化 benchmark、证据共享和开放协议会影响 agent 行业是走向可审计基础设施，还是碎片化的私有集成。
