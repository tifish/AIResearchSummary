我们在构建云端智能体过程中学到的经验

What we’ve learned building cloud agents

一句话总括：Cursor 的核心发现是，云端智能体的上限不只由模型决定，更取决于围绕模型建设的完整开发环境、持久化执行系统和可恢复的编排层。

>99%迁移到 Temporal 后超过“两个 9”
5000 万+Temporal 每天处理的操作次数
40%+Cursor 内部由云端智能体贡献的 PR

环境缺陷会伪装成模型缺陷

本地智能体天然继承开发者的工具、依赖和配置。云端智能体没有这种便利。环境缺少一环时，系统往往不会直接报错，只会让结果质量悄悄下降。

随着模型变强，模型越来越能利用外部环境。于是，完整开发环境从辅助条件变成了决定智能体能否发挥全部能力的关键因素。

云端智能体需要一层新的基础设施

Cursor 为此建设了虚拟机休眠与恢复、镜像检查点、分叉机制、客户端集成和受控网络访问。密钥脱敏、网络策略和凭据管理也成为必要能力。

这改变了问题的性质。团队建设的已不再是“服务器上的本地智能体”，而是面向智能体的企业 IT 和运行层。

可靠性架构经历了三次关键转变

1

直接移植本地循环

最初采用工作窃取架构。工作节点领取智能体并持续运行，但节点和服务中断会破坏执行。早期可靠性只有一个 9。

2

迁移到 Temporal

重试、跨机器调度和节点故障恢复交给持久化执行系统。迁移后可靠性超过两个 9，任务可以持续数天甚至数周。

3

拆分状态与工作流

“永久”工作流被拆成完成单项任务后退出的短工作流。智能体循环、虚拟机和对话流也各自解耦，升级与恢复因此更容易。

更好的 harness 不是替智能体做更多决定，而是在模型尚未成熟的地方提供脚手架，并把其余能力交还给智能体。

Cursor 已把提交、推送、多仓库处理和 CI 日志分析等固定逻辑逐步移入工具。智能体可以自行选择调用方式。计算机操作仍需要专用子智能体、模型路由和录屏机制，但调用时机已经由智能体控制。

最值得带走的洞见

01

环境就是产品

云端智能体的体验不能只靠模型升级。工具、依赖、网络和凭据共同决定结果质量。环境问题还很容易被误判为模型能力不足。

02

长任务必须可恢复

任务从几分钟延长到数小时、数天甚至数周后，推理服务和计算节点的短暂故障就会成为常态。持久化执行不是附加功能。

03

状态解耦带来弹性

把智能体循环、机器状态和对话状态分开，才能自由迁移 Pod、启用预热虚拟机，并正确处理流式输出后的重试。

04

下一步是环境自愈

理想系统会让智能体识别密钥缺失、网络受限等阻塞，并主动修复。这样就能越过“处处管控”和“完全放手”的二选一。

一年前，我们首次推出云端智能体。当时，它看起来只是本地智能体很自然的一次延伸。此后，云端智能体的能力有了大幅扩展。

如今，云端智能体会在各自专用的虚拟机上运行。它们拥有独立的环境、依赖和网络访问权限。它们可以并行工作，也可以在无人看管时继续执行。相比运行在笔记本电脑上的本地智能体，它们还能承担耗时更长的任务。

这些能力也带来了环境配置、可靠性和编排方面的挑战。当智能体只在你的笔记本电脑上运行时，这些问题不会如此突出。

本文将分享我们在构建云端智能体过程中得到的最重要经验。我们也会解释，为什么这项工作越来越不像是把本地智能体搬到服务器上，而更像是在智能体周围建设一层操作系统式的运行基础设施。

开发环境就是产品

过去一年里，我们发现，影响云端智能体输出质量的首要因素，是它能否获得与开发者相当的完整开发环境。

在本地使用智能体时，你不太需要考虑这个问题。因为本地智能体会直接继承笔记本电脑上已经可用的开发环境。在云端，你必须从零开始重建这一切。更棘手的是，环境没有完全搭好时，往往很难察觉。

系统可能不会崩溃，也不会弹出错误消息。唯一的迹象往往只是输出质量略有下降。你一开始可能注意不到。即使发现了，也可能把问题归因于模型。

但我们一次又一次追查后，都得出了同一个诊断结果：云端智能体缺少执行工作或验证结果所需的环境。一年前，这个问题还没有那么重要，因为当时的模型本来也无法充分利用周围环境。随着模型变得更聪明，环境配置已经成为决定它们能否发挥全部潜力的关键因素。

如今，要实现所谓的“完整环境”，就需要重建数量惊人的基础设施：

  • 为用户提供更好的工具，用来构建智能体环境。
  • 提供高效的方法，让智能体虚拟机可以在两次消息之间休眠并恢复。
  • 建设能够快速、可靠地创建检查点、恢复和分叉虚拟机镜像的流水线。
  • 让 harness 与客户端紧密集成，使智能体和人类都能理解环境，并与它交互。

随着云端智能体承担更多工作，它们还需要受控的网络访问权限，以便创建 PR、拉取依赖和开展研究。久而久之，我们实际上为智能体建设了一套企业 IT 系统，其中包括密钥脱敏、网络策略和凭据管理。

长时间运行的智能体需要持久化执行

云端智能体面临的可靠性挑战与本地智能体不同。它们不会和笔记本电脑上的其他程序争抢本地资源，而是在各自隔离的虚拟机中运行。这样一来，开发者更容易并行启动多个智能体，也可以把长任务委派出去。这类任务通常需要数小时，而不是几分钟。

不过，在虚拟机中运行也意味着系统会受到各种中断影响。例如，推理服务提供商可能发生故障，Pod 可能需要替换，EC2 节点也可能宕机。

我们最初采用工作窃取架构来构建云端智能体。工作节点可以领取智能体，并持续运行循环直至任务完成。这相当于把本地有效的方案直接移植到服务器上，但整体十分脆弱。早期云端智能体测试版往往只有“一个 9”的可靠性。

随着云端智能体逐渐成熟，我们发现自己几乎要重新实现 Temporal 已经解决的许多持久化执行基础能力。例如重试机制、跨机器调度任务，以及节点发生故障时保持执行状态。因此，我们决定迁移到 Temporal。

现在，我们运行在 Temporal 上的智能体循环可以承受推理服务的短暂不稳定，也能应对 Pod 的休眠与恢复。即使一次运行持续数天甚至数周，它也能继续执行。仅这次迁移，就让我们的可靠性超过了“两个 9”。如今，Temporal 每天处理超过 5000 万次操作，覆盖逾 700 万个独立工作流。在 Cursor 内部，超过 40% 的 PR 来自云端智能体，而且这一比例还在增长。

随着时间推移，我们也学会了如何更合理地设计 Temporal 工作流。我们不再使用“永久运行”的智能体工作流，而是改为多个较短的工作流。每个工作流完成一项任务后便退出，这让版本升级变得更容易。我们还拆分了各项活动，以便更准确地处理超时和重试。异步工具调用、子智能体和推理服务中断已经改变了我们最初对系统的假设。

将智能体、机器与对话状态解耦

云端智能体不再只是运行在一台机器上的单个循环。一个智能体可能先在一台机器上运行，再把异步子智能体分派到另外几台机器上。它也可能从本地启动，随后把工作委派给云端。子智能体甚至可能比父智能体存活得更久，或者运行在完全不同类型的 Pod 上。

为了实现这些能力，我们发现，最好将智能体循环、机器状态和对话状态作为彼此解耦的组件。智能体循环存在于 Temporal 中，而不是虚拟机本身。因此，我们可以独立管理 Pod 的生命周期，也可以让智能体跨不同类型的 Pod 运行。这还支持只读虚拟机和预热虚拟机等优化方案。

在对话方面，我们把存储和流式传输层从智能体核心工作流中分离出来。我们构建了一种高效的仅追加存储机制，用来把对话更新实时发送到网页端和桌面客户端。

这一层也会处理重试。如果智能体循环中的某一步在已经发送部分输出后失败，随后又被重试，客户端就能识别这种情况。它会回退数据流，并显示重试产生的新数据，而不是继续展示旧数据。

懂得何时放手

构建云端智能体 harness，意味着要持续重新评估两件事：哪些行为应该由确定性逻辑控制，哪些行为应该交给智能体决定。

早期，我们不太信任智能体。因此，每项任务完成后，harness 都会复查它的工作，强制创建提交并推送。随着模型变得更聪明，我们开始把逻辑从 harness 中移出，转化成由智能体控制的工具。一年前，多仓库环境需要 harness 内置硬编码行为。现在，我们可以告诉智能体仓库布局，为它提供分支和 PR 工具，再让它自行决定如何完成工作。

CI Autofix 也经历了同样的变化。早期版本的云端智能体 harness 包含一套逻辑，用来获取任务失败日志并把日志写入虚拟机。现在，我们只需要让智能体访问 GitHub CLI，并自动把大段输出写入它可以搜索的文件。发送给智能体的通知因此简单了许多。我们预计，这一趋势还会继续。

harness 并不会消失，只是其中包含的内容正在变化。计算机操作就是当前很有代表性的例子。我们的云端智能体 harness 为计算机操作提供了一种专用子智能体。它拥有自己的模型路由、自定义提示词和屏幕录制功能。VNC 和 Chrome 属于运行环境，并由父智能体和子智能体共享。因此，父智能体也能直接使用它们。例如，它可以运行 Playwright 脚本。我们采用这套脚手架，是因为模型还没有完全准备好独立处理计算机操作。不过，是否调用这项能力，仍由智能体决定。

云端智能体在 harness 中需要的提示方式也不同于本地智能体。我们会鼓励它们更加自主，因为阻塞的代价要高得多。在本地，你会知道智能体已经停止，正在等待许可。但在云端，它可能就这样停留数小时，直到你再次回来查看。

可自愈的智能体环境

展望未来，我们希望越过“全程手把手指导”和“完全放手不管”之间的二元选择。更好的模式，是为智能体提供理解周围系统的工具。

我们希望云端智能体能够报告密钥缺失、网络访问受阻等问题。环境妨碍任务推进时，它也应该能够发现原因，随后以自愈方式采取行动。我们在最近的一篇研究博客中介绍过一条可能的实现路径，并把它称为“autoinstall”。

仅仅在过去几个月里，云端智能体就取得了巨大进步。我们预计，接下来的变化还会进一步加速。Cursor 云端智能体让团队可以利用这一广阔的能力空间，而不必自行构建或维护底层基础设施。

懂得何时放手

构建云端智能体 harness，意味着要持续重新评估两件事：哪些行为应该由确定性逻辑控制，哪些行为应该交给智能体决定。

早期，我们不太信任智能体。因此，每项任务完成后，harness 都会复查它的工作，强制创建提交并推送。随着模型变得更聪明，我们开始把逻辑从 harness 中移出，转化成由智能体控制的工具。一年前，多仓库环境需要 harness 内置硬编码行为。现在，我们可以告诉智能体仓库布局，为它提供分支和 PR 工具，再让它自行决定如何完成工作。

CI Autofix 也经历了同样的变化。早期版本的云端智能体 harness 包含一套逻辑，用来获取任务失败日志并把日志写入虚拟机。现在，我们只需要让智能体访问 GitHub CLI，并自动把大段输出写入它可以搜索的文件。发送给智能体的通知因此简单了许多。我们预计，这一趋势还会继续。

harness 并不会消失，只是其中包含的内容正在变化。计算机操作就是当前很有代表性的例子。我们的云端智能体 harness 为计算机操作提供了一种专用子智能体。它拥有自己的模型路由、自定义提示词和屏幕录制功能。VNC 和 Chrome 属于运行环境，并由父智能体和子智能体共享。因此，父智能体也能直接使用它们。例如，它可以运行 Playwright 脚本。我们采用这套脚手架，是因为模型还没有完全准备好独立处理计算机操作。不过，是否调用这项能力，仍由智能体决定。

云端智能体在 harness 中需要的提示方式也不同于本地智能体。我们会鼓励它们更加自主，因为阻塞的代价要高得多。在本地，你会知道智能体已经停止，正在等待许可。但在云端，它可能就这样停留数小时，直到你再次回来查看。

可自愈的智能体环境

展望未来，我们希望越过“全程手把手指导”和“完全放手不管”之间的二元选择。更好的模式，是为智能体提供理解周围系统的工具。

我们希望云端智能体能够报告密钥缺失、网络访问受阻等问题。环境妨碍任务推进时，它也应该能够发现原因，随后以自愈方式采取行动。我们在最近的一篇研究博客中介绍过一条可能的实现路径，并把它称为“autoinstall”。

仅仅在过去几个月里，云端智能体就取得了巨大进步。我们预计，接下来的变化还会进一步加速。Cursor 云端智能体让团队可以利用这一广阔的能力空间，而不必自行构建或维护底层基础设施。
