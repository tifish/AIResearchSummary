我们从构建云端智能体中学到的经验

What we’ve learned building cloud agents

Cursor 的核心贡献是把云端智能体重新定义为一套运行层工程，而不只是本地智能体的远程版本。

1 年 云端智能体从本地扩展思路，演化成独立虚拟机与运行系统。
two 9s+ 迁移到 Temporal 后，可靠性从 early beta 的 one 9 提升到超过 two 9s。
5000 万+ Temporal 每天处理的 actions 数量，说明 agent workflow 已进入基础设施规模。
40%+ Cursor 内部 PR 来自云端智能体，显示它已不只是实验工具。

开发环境就是产品

云端智能体的质量瓶颈，经常不是模型本身，而是它有没有像开发者一样完整的环境。

本地智能体天然继承笔记本上的依赖、凭据、网络和工具。云端必须重新构建这一切。
环境不完整时，问题未必表现为崩溃。更常见的是输出质量悄悄下降。
因此，VM 休眠恢复、镜像 checkpoint、secret redaction、网络策略和凭据管理，都变成智能体产品的一部分。

长任务需要可持久执行

云端任务可能运行数小时、数天甚至数周。一次节点故障不该让整段工作消失。

早期用 work-stealing 架构，让 worker 把 agent loop 跑到完成。

状态必须拆开

组件 为什么要解耦 带来的理解变化
Agent loop 放在 Temporal 中，而不是绑死在某台 VM 上。 智能体可以跨 pod 生存，任务不再等同于一台机器上的一个进程。
Machine state VM 可以休眠、恢复、预热、只读化或被替换。 环境成为可调度资源，而不是静态背景。
Conversation state 追加式存储和流式输出要能处理 retry 后的回滚。 用户看到的对话流，也需要分布式系统级一致性设计。

随着模型变强，harness 不应替智能体做所有决定。更好的方向是把硬编码流程变成智能体能主动使用的工具。

让路，不是放任

Cursor 早期会在每个任务后强制检查、提交和推送。现在更多逻辑被移到 agent 可控工具里。

多仓库任务从硬编码 harness，变成提供 repo layout、branch 和 PR 工具。
CI Autofix 从自动抓日志写入 VM，变成给 agent GitHub CLI，并把大输出保存为可搜索文件。
computer use 仍需要专门 subagent、模型路由、prompt 和录屏。模型还没准备好完全独立处理它。

最值得带走的洞见

环境决定上限 当模型足够聪明后，缺依赖、缺权限和缺网络会直接压低输出质量。
可靠性是核心能力 云端智能体越能跑长任务，就越需要 workflow、retry 和跨节点恢复。
工具优于硬编码 让 agent 自己调用分支、PR、CLI 和浏览器工具，比把每个流程写死更可扩展。
