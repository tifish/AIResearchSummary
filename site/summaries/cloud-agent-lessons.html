我们在构建云端 Agent 中学到的事

作者：Josh Ma　·　来源：Cursor Blog
一句话总括：构建长时运行的云端编程 agent，真正的难点不是把本地 agent 移植到服务器，而是要在 agent 周围重建一整套"企业级 IT"——完整的开发环境、持久化执行、状态解耦——决定成败的往往是这些看不见的基础设施，而不是模型本身。

环境，才是真正的产品

关键趋势：决定云端 agent 输出质量的最大单一因素，是它是否拥有和真实开发者一样完整的开发环境。

本地 agent 免费继承了你笔记本上现成的环境，云端却要从零重建，而且很难判断有没有做对。环境不全时不会崩溃也不会报错，只表现为输出质量"悄悄变差"——人们往往误以为是模型不行。模型越聪明，这个差距越致命，因为强模型本该更充分地利用环境。

为达到"完整环境"，Cursor 重建了大量基础设施：VM 的休眠/恢复、镜像的 checkpoint/restore/fork、受控网络访问，以及密钥脱敏、网络策略、凭证管理。

结论：在云端，"搭好环境"本身就是产品力，它比换模型更能决定结果。

长时运行的 agent 需要"持久化执行"

关键数据：可靠性从"一个 9"跃升到"两个 9"，靠的是迁移到 Temporal。

1×9 → 2×9
可靠性从 ~90% 跨越到 99%+
5000 万+
Temporal 每天处理的 action
700 万+
独立 workflow 数量
数天~数周
单次任务可跨越的时长

云端 VM 会遭遇推理服务中断、pod 替换、EC2 节点宕机。最初的"work-stealing"架构只是把本地逻辑搬上服务器，非常脆弱。当团队发现自己快要重造一遍 Temporal 已解决的重试、跨机调度、节点容错时，便直接迁移过去。架构上也从"永生"workflow 改为"完成单个任务即退出"的短 workflow，便于版本升级。

结论：长时 agent 的可靠性不是靠模型，而是靠借用成熟的持久化执行引擎。

把 agent、机器、对话状态解耦

关键趋势：一个 agent 已不再是"一台机器上的一个循环"。

它可能在一台机上运行、向多台机派生异步子 agent，或本地起步再委派到云端——子 agent 甚至能比父进程活得更久。因此 Cursor 让 agent loop（放在 Temporal）、机器状态、对话状态三者独立，可以单独管理 pod 生命周期，使用只读 VM 或预热 VM 等优化。对话侧则用 append-only 存储流式推送到客户端，并能在某步重试后让客户端"回退流"、用新数据覆盖旧数据。

结论：解耦让同一个 agent 能横跨多种机器与重试场景而不丢状态。

学会"适时让开"

关键趋势：模型变强后，逻辑从 harness 不断移交给 agent 自己掌控。

早期不信任 agent，harness 会在每个任务后复查、强制提交并推送。如今多仓库操作只需把仓库布局和分支/PR 工具交给 agent 自行决定；CI Autofix 也从"harness 抓日志"简化为"给 agent 一个 GitHub CLI"。但 harness 不会消失，只是内容在变——例如 computer use 仍保留专用子 agent（独立模型路由、提示、录屏，共享 VNC/Chrome），因为模型还没准备好独立处理。云端还需要更"自主"的提示词，因为在云端阻塞等待的代价远高于本地。

结论：harness 的角色从"替 agent 做"转向"给 agent 工具，让它自己做"。

未来：自愈式环境

关键趋势：超越"牵着手"与"完全放手"的二元选择。

目标是让 agent 能理解周围系统：当密钥缺失、网络被封或环境阻碍进展时，能主动上报并自我修复。Cursor 把其中一条路径称为 autoinstall。当前内部已有 超过 40% 的 PR 来自云端 agent 且仍在增长。

结论：下一步的竞争力在于 agent 能否自己诊断并修好自己的运行环境。

最值得带走的洞见

  1. 环境质量 > 模型选择。云端 agent 输出变差，最常见的根因是环境不完整，而非模型——而它从不报错，只悄悄退化。
  2. 可靠性是工程问题。从一个 9 到两个 9 的飞跃来自迁移到 Temporal（每天 5000 万 action、700 万 workflow），而不是更强的提示词。
  3. 把状态解耦。agent loop、机器、对话三者分离，才能支撑子 agent 跨机、长达数周、可重试的运行。
  4. 随模型变强而"让开"。把确定性逻辑持续交还给 agent 控制，harness 不消失但越来越薄。
  5. 云端的成本结构不同。阻塞等待代价极高，因此云端 agent 必须更自主，并最终走向能自我诊断、自我修复的环境。
