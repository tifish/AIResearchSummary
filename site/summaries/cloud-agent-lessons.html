一句话总括

云端 agent 的真正产品，是让模型像开发者一样拥有可靠、可恢复、可操作的工作环境。

Cursor 的核心贡献不是宣布 cloud agents 更强，而是把它们背后的工程条件讲清楚：环境完整性、耐久执行、状态解耦和工具边界，正在决定 agent 能不能稳定完成真实开发任务。

5000 万+Temporal 每天处理的 actions
700 万+Temporal 覆盖的 unique workflows
40%+Cursor 内部 PR 来自 cloud agents
两 9+迁移 Temporal 后达到的可靠性水平

完整开发环境比提示词更像产品本体

本地 agent 自动继承开发者电脑上的依赖、网络、工具链和认证，而云端 agent 必须从零重建这些条件。Cursor 发现，环境不完整时常常不是直接报错，而是输出质量微妙下降，容易被误判成模型能力不足。

这改变了对 AI 编程影响的理解：模型越强，越能利用环境，也越受环境缺陷限制。真正的云端 agent 产品需要 VM 休眠与恢复、镜像 checkpoint、环境交互 harness、受控网络访问、密钥脱敏和凭证管理。

长任务的关键不是跑得久，而是失败后还能继续

云端 agent 可以并行运行、无人值守，并承担从数小时到数天甚至数周的任务，但 VM、pod、EC2 节点和推理服务都会中断。Cursor 早期采用 work-stealing 架构，可靠性大约只有“一 9”，说明把本地循环直接搬到服务器并不够。

迁移到 Temporal 后，agent loop 可以跨推理波动、pod 休眠恢复和长时间运行保持耐久，可靠性超过“两 9”。这说明云端 agent 的价值不只来自一次生成，而来自可调度、可重试、可恢复的执行系统。

agent、机器和对话状态必须拆开

一个云端 agent 不再等于一台机器上的一个循环：它可能启动异步 subagent，跨多种 pod 工作，甚至本地开始、云端继续。Cursor 因此把 agent loop 放在 Temporal 中，让机器生命周期可以单独管理。

对话层也被拆出核心 workflow，使用 append-only 存储向 Web 和桌面客户端流式发送更新，并在重试时让客户端回退旧输出、显示新结果。这个设计让读者看到，agent 体验的稳定性依赖后端状态模型，而不只是前端聊天窗口。

harness 的职责正在从替 agent 做事，转向给 agent 工具

早期 harness 会检查任务、强制 commit 和 push，也会硬编码多仓库或 CI Autofix 的处理逻辑。随着模型变强，Cursor 把更多决策移给 agent：提供 repo 布局、branch 和 PR 工具、GitHub CLI，以及可搜索的大输出文件。

这不是取消 harness，而是改变边界。比如 computer use 仍需要专门 subagent、模型路由、提示和录屏，但父 agent 能决定何时调用它。结论是，优秀的编排层应该减少硬编码流程，同时保留模型尚不能可靠独立完成的支架。

下一阶段是自愈环境，而不是更多人工托管

Cursor 认为更好的模式不是在“牵着 agent 走”和“完全放手”之间二选一，而是让 agent 能理解系统状态。它应能报告密钥缺失、网络受限或环境阻塞，并通过类似 autoinstall 的能力主动修复。

这会改变团队采用 AI 开发工具的成本结构：企业不只是买一个会写代码的模型，而是在租用一套可诊断、可管控、可扩展的 agent 运行平台。

最值得带走的洞见

  • 云端 agent 的核心竞争力正在从模型输出，转向完整开发环境和运行基础设施。
  • 长任务 agent 必须默认会失败，并把重试、恢复、调度和版本升级作为基础能力。
  • 对话状态、机器状态和 agent loop 解耦后，agent 才能跨 pod、跨时间、跨子任务稳定运行。
  • harness 不应永远写死流程，而应逐步变成 agent 可调用、可组合的工具层。
  • 未来更强的 cloud agents 会主动诊断和修复环境问题，这比单纯提高模型能力更接近真实生产需求。
