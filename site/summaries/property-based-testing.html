用 Claude 和基于性质的测试在 Python 生态系统中发现缺陷

Finding bugs across the Python ecosystem with Claude and property-based testing
一句话总括：这项工作把 LLM 的代码理解能力和基于性质的测试结合起来，让模型不只生成测试用例，而是主动提出“代码应始终满足的性质”，并在真实 Python 生态中找到可验证、可修复的缺陷。

核心贡献是让模型寻找“性质”，不是只补测试

普通单元测试验证几个具体例子。它容易漏掉开发者没有想到的边界条件。

基于性质的测试换了一个层次。开发者或代理先描述代码应该一直满足的规律，再让测试框架自动生成输入去寻找反例。

Claude 的价值在于读上下文。它会看类型标注、文档字符串、函数名、注释和调用方式，然后写出 Hypothesis 测试。

这改变了对 AI 编程影响的理解。

模型不只是代码生成器。它可以参与软件质量保障中的语义判断：这个函数到底承诺了什么，什么输入会打破承诺。

代理的工作流程是一个反复校验的闭环

1

先阅读目标代码和相关文档，理解函数、模块或文件在项目中的位置。

2

提出应当成立的性质。性质必须来自代码和文档，而不是凭空猜测。

3

用 Hypothesis 写基于性质的测试，并运行测试寻找反例。

4

测试失败后，代理会反思是发现了真实 bug，还是测试写错了。这个环节用于降低误报。

5

当代理认为问题可信时，它输出结构化 bug 报告，供人工和维护者继续验证。

关键洞见：如果 AI 能更准确地描述代码的正确性边界，那么发现 bug 和生成补丁都会变得更接近工程流程，而不是一次性猜答案。

真实项目验证显示：有效性来自筛选和人工复核

阶段 做法 结果 对 AI 影响的含义
第一阶段 Claude Opus 4.1 在每个包上运行，收集全部报告。 984 份报告中人工抽查 50 份，56% 有效，32% 有效且值得报告。 模型可以大规模发现候选问题，但原始输出仍需要过滤。
排序阶段 用 15 分规则对报告打分，优先查看高分项。 最高分报告中 86% 有效，81% 有效且值得报告。 “生成加筛选”比单纯生成更接近可用开发工具。
第二阶段 Sonnet 4.5 在 10 个重要包上重复运行，并加入评估代理和专家审查。 重点转向高严重性 bug 的正确性和影响评估。 AI 测试工具要进入真实维护流程，必须解释严重性而不只是报错。

已确认案例说明它能抓住真实边界错误

NumPy 的 numpy.random.wald 有时返回负数，而 Wald 分布样本应为正数。团队追踪到数值不稳定，并提交了更稳定的实现。

NumPy 维护者显示，新算法相对误差降低了接近 10 个数量级。这说明代理发现的不是格式瑕疵，而是会影响数值正确性的深层问题。

失败案例说明语义仍是难点

python-dateutil 的 easter() 在 Julian calendar 下返回非星期日日期。维护者认为这是预期行为，因为历法语义更细微。

这暴露了边界：当正确性依赖隐含领域知识时，模型推断的性质可能看似合理，却不是维护者认可的契约。

为什么这比传统自动测试更有潜力

传统测试自动化经常停留在生成更多例子。例子多了，仍可能没有覆盖真正的异常输入。

这项工作把重点放在性质上。性质一旦写对，就能让测试框架系统性地搜索反例。

LLM 正适合做前半段工作：从上下文中推断不变量。Hypothesis 负责后半段工作：高强度生成输入并缩小失败样本。

实践意义：

未来的软件质量工具可能不是单独的 linter、fuzzer 或聊天助手，而是能读代码、提假设、运行实验、过滤误报并生成报告的代理。

最值得带走的洞见

AI 的测试价值来自语义抽象真正有价值的部分不是写测试语法，而是提出可验证的不变量。
误报控制决定工具能否被维护者接受Anthropic 采用多轮人工复核，是为了避免把不确定报告转嫁给开源维护者。
筛选比海量生成更关键原始报告有效率有限，但高分报告的有效率提升到 86%。
领域语义仍需要人类裁决dateutil 案例说明，模型能提出合理性质，但不能替代维护者定义真实契约。
下一步可能是补丁生成如果性质能接近完整描述正确性，自动修复会更容易被验证，也更可能进入真实开发流程。
