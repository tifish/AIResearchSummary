代理如何完成一次缺陷搜索

  1. 阅读目标代码及相关文档。
  2. 提出有明确依据的代码性质。
  3. 编写 Hypothesis 测试。
  4. 运行测试并自我反思。
  5. 确认后生成规范的缺陷报告。

这套流程刻意追求低误报，但成本很高。当代码依赖隐含假设、历法差异或复杂领域语义时，维护者仍是最终裁判。

基于属性的测试让模型在更高的抽象层工作：先问清楚“什么必须始终成立”，再让工具寻找能够推翻它的输入。这比盲目增加测试数量更接近缺陷的本质。

完整译文

作者：Muhammad Maaz1,2、Liam DeVoe3、Zac Hatfield-Dodds2、Nicholas Carlini2

1 MATS，2 Anthropic，3 Northeastern University

我们开发了一种代理，能够高效识别大型软件项目中的缺陷。代理会推断代码通常应当满足的性质。随后，我们应用基于属性的测试。这是一种与模糊测试相似的技术。借助这种方法，我们在 NumPy、SciPy 和 Pandas 等主流 Python 包中发现了缺陷。

经过大量人工验证后，我们正在向开发者报告这些问题。其中几个缺陷已经得到修复。

如需了解更多信息，可以阅读完整论文、查看 GitHub 仓库，或在我们的网站上浏览发现的缺陷。

引言

确保程序没有缺陷，是软件工程中最具挑战性的工作之一。即使开发者付出了最大努力，缺陷仍常常潜伏在代码中。

最常见的代码测试方法是基于示例的测试（example-based tests）。开发者先写出一个具体用例，再验证实际输出是否符合预期。例如，开发者可以检查：对列表 [2, 10, 5, 4] 调用排序函数后，是否会返回 [2, 4, 5, 10]。

然而，仅靠这类测试很难覆盖整个程序。缺陷经常藏在开发者没有测试的边界情况中。毕竟，如果开发者没有想到要测试某个边界情况，那么在实现代码时，他也很可能没有考虑过这种情况。

相比之下，基于属性的测试（property-based testing）关注的是：某项代码性质能否对全部输入，或大多数输入成立。

开发者需要为程序指定一种性质或不变量。例如，JSON 反序列化应当是序列化的逆过程。开发者还需要描述该性质接受哪些输入，例如任何可以序列化为 JSON 的对象。

基于属性的测试框架随后会自动寻找反例。它使用类似模糊测试的技术生成有效输入，并把这些输入作为测试用例。开发者只需指定总体输入域，无须逐个列出测试用例。这样一来，他们就不用设法想到每一种边界情况，可以在更高的抽象层上工作。

在一项源自 MATS 项目的研究中，我们开发了一个能够为现有代码自主编写基于属性测试的 AI 代理。相关论文已在 2025 年 NeurIPS Deep Learning for Code Workshop 上发表。

我们要求代理通过阅读类型标注、文档字符串、函数名和注释来发现代码性质。随后，代理使用 Hypothesis 编写对应的基于属性测试。

这项工作关注的是一般意义上的缺陷识别，而不只是安全漏洞。许多可能造成安全漏洞的逻辑缺陷，都适合通过基于属性的测试发现。

例如，我们最近在一篇博客文章中重点介绍了如何识别智能合约中的缺陷。那些漏洞几乎全都源于逻辑错误。我们设想，未来可以使用本文介绍的技术，在软件部署前主动发现缺陷。

我们使用这一代理，在 NumPy、SciPy 和 Pandas 等热门开源 Python 仓库中发现了数百个潜在缺陷。为了负责任地披露这些问题，同时避免给维护者带来不必要的负担，我们仔细审核了每一个缺陷。

我们的审核流程比内部代码审查通常采用的流程更费力，但我们更希望尽量减少误报。

具体流程如下。首先，我们只选择优先级最高的缺陷进行审核。然后，我们把每个潜在缺陷交给 3 名人类专家审核。每个缺陷平均需要 1 小时。如果 3 名审核者中有任何一人无法确认问题有效，我们就会将其舍弃。

最后，我们这些博客作者还会亲自审核每个候选缺陷。只有当我们也确信判断正确时，才会手动向对应仓库的维护者提交问题。目前，我们已经提交了几份缺陷报告，并正在继续报告更多问题。

我们公开了全部数据。其中包括尚未验证的缺陷，也包括经判断无效的缺陷，以便维护者在网站上查看。未来几周，我们计划在完成进一步验证后，继续报告许多剩余问题。我们还将把项目扩展到更多 PyPI 项目。

图 1。测试代码的两种方式。基于示例的单元测试会验证人工指定的具体输入。基于属性的测试则指定一种通用性质，例如有序列表的定义，再由框架自动构造可能违反该性质的输入。

我们的基于属性测试代理

我们的基于属性测试代理以自定义 Claude Code 命令的形式实现。代理只接受一个参数，用来指定测试目标。

目标可以是单个 Python 文件，例如 normalizers.py；也可以是模块，例如 numpy 或 scipy.signal；还可以是函数，例如 requests.get 或 json.loads。

随后，代理按照以下流程寻找潜在缺陷：

  1. 阅读并理解目标。代理会阅读代码、获取相关文档，并探索目标与代码库其他部分的关系。
  2. 根据这些发现提出有明确依据的性质。
  3. 使用 Hypothesis 编写对应的基于属性测试。
  4. 运行测试并进行反思。如果测试失败，它会判断自己是否真的发现了缺陷，还是测试本身需要调整。如果测试成功，它会判断测试是否真正有价值，还是只验证了一个无关紧要的性质。
  5. 如果代理确信自己发现了真实缺陷，就会生成一份格式规范的缺陷报告。

为了帮助代理完成长距离、多步骤推理，我们要求它使用待办事项列表跟踪进度。

设计代理时，我们优先考虑的是减少误报。根据我们的亲身经验，真正有用的开发工具，应当尽量减少呈现给开发者的错误报告。

自我反思循环有助于减少误报。要求所有性质都以目标的明确用法和文档为依据，也能达到同样的效果。

例如，在一次运行中，代理编写的性质测试最初通过了。但经过自我反思，代理意识到自己用 try-catch 块包住了整个测试。移除这个结构后，测试失败了，代理由此发现了一个缺陷。

我们还观察到，与 Sonnet 4 相比，Opus 4.1 和 Sonnet 4.5 的自我反思能力有明显提升。

在真实 PyPI 包中寻找缺陷

为了在真实环境中测试代理的能力，我们整理了一组多样化的流行 Python 包，总数超过 100 个。这些库覆盖多个领域，包括数值计算、解析和数据库。

我们让代理分别处理每个包，并收集它生成的全部缺陷报告。

评估代理

在论文介绍的第一阶段评估中，我们使用 Claude Opus 4.1，让代理分别处理每个软件包，并收集生成的全部缺陷报告。

评估这些报告时，我们采用了两项标准。第一项是：“这是不是一个有效缺陷？”第二项是：“它是否既是有效缺陷，又值得我们向库维护者报告？”

第二项标准比第一项更严格。例如，一个缺陷可能确实存在，但问题过于轻微，不值得专门提交报告。

代理总共生成了 984 份缺陷报告。我们从中人工选出 50 份进行审核。结果显示，其中 56% 是有效缺陷，32% 既是有效缺陷，也是我们愿意报告的问题。

根据这次人工审核，我们制定了一套满分 15 分的评分标准。其目标是优先呈现最有可能真实存在、也最值得开发者修复的缺陷。

我们使用 Opus 4.1，按照这套标准为所有缺陷报告评分。结果表明，排序步骤相当有效。在得分最高的缺陷报告中，86% 是有效缺陷，81% 既有效又值得报告。

在第二阶段评估中，我们使用 Sonnet 4.5 测试了 10 个重要软件包。每个软件包都运行了多次。

我们还开发了一个评估代理。它使用 Sonnet 4.5 阅读代码和缺陷报告，再检查问题是否正确，并判断严重程度。与第一阶段的评分标准相比，这个评估代理更加复杂。

最后，我们付费请 3 名人类专家审核高严重性缺陷，判断这些问题是否成立。

如需阅读代理发现的全部缺陷报告，请访问 https://mmaaz-git.github.io/agentic-pbt-site/。

维护者验证

评估任何代码缺陷发现工具的有效性都很困难。虽然我们会尽力验证缺陷报告是否正确，但软件包维护者才是最终的裁判。

为了确认代理发现的问题确实会被维护者视为有效且值得修复，我们选出了 5 个特别有意思的缺陷。我们把这些问题连同建议补丁一起，手动报告到了对应项目的 GitHub 仓库。

未来几周，我们计划在完成验证后继续报告其他缺陷。

numpy

numpy.random.wald 有时会返回负数。这是一个缺陷，因为从 Wald 分布采样时，只应该得到正数。

这正是前面运行示例中展示的问题。Claude 知道 Wald 分布应当满足这一性质，因此编写了一个简单的基于属性测试，检查所有生成的样本是否都为正。

我们将错误追溯到代码中发生的灾难性消减（catastrophic cancellation）。提交拉取请求时，我们设计了一种数值上更稳定的表达式。

正如 NumPy 维护者在拉取请求中展示的那样，新表达式的相对误差比旧算法低近 10 个数量级。

补丁已合并：https://github.com/numpy/numpy/pull/29609

aws-lambda-powertools

slice_dictionary() 会反复返回第一个分块，因为代码没有递增迭代器。

代理发现，字典经过切分再重新组合后，应该还原为原始字典。它通过这一性质捕获了问题。

补丁已合并：https://github.com/aws-powertools/powertools-lambda-python/pull/7246

cloudformation-cli-java-plugin

item_hash() 对所有列表都会生成相同的 hash(None) 值。原因是代码使用了原地执行的 .sort() 方法，而该方法返回 None。

代理通过检查“不同输入的哈希值应当不同”这一性质发现了问题。

补丁已提交：https://github.com/aws-cloudformation/cloudformation-cli/pull/1106

tokenizers

EncodingVisualizer.calculate_label_colors() 缺少一个右括号，因此会返回无效的 HSL CSS。

代理通过测试输出是否符合 HSL 颜色代码的正则表达式发现了问题。

补丁已合并：https://github.com/huggingface/tokenizers/pull/1853

python-dateutil

使用儒略历时，easter() 会在部分年份返回一个不是星期日的日期。维护者认为这种行为符合设计，因为不同历法系统之间存在差异。他们也承认其中的语义十分微妙。

问题无效：https://github.com/dateutil/dateutil/issues/1437

向 python-dateutil 提交的这份报告揭示了代理的一项重要局限：当代码具有微妙或复杂的语义时，很难推导出正确的性质。

如果代码依赖某项隐含假设，那么只有库维护者能够决定真正应该测试什么性质。

结论

随着语言模型持续进步，我们认为，代理式基于属性测试可以成为人工编写测试日益重要的补充。

基于属性的测试关注高层语义保证，因此天然适合在开发过程中与语言模型配合。我们发现，LLM 尤其擅长根据上下文识别一段代码应当满足的性质。相关上下文包括函数名、文档字符串，以及其他函数调用它的方式。

借助这些信息，LLM 可以有效编写高质量的基于属性测试。

展望未来，我们认为，将 LLM 用于测试和缺陷发现是一项重要研究方向。尤其是当 LLM 越来越善于利用漏洞时，我们有必要抢在使用 LLM 发动攻击的攻击者前面。

这项工作并未重点研究自动生成补丁，但这是一个明确的未来方向。

如果我们能够近乎完整地描述一段代码的正确性性质，修复缺陷就会容易得多。我们相信，在不久的将来，LLM 将能够有效提出高质量补丁。这些补丁将值得维护者认真考虑。
