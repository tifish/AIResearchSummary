用 Claude + 属性测试，在 Python 生态里批量抓 bug

来源 Anthropic ｜ 作者 Muhammad Maaz、Liam DeVoe、Zac Hatfield-Dodds、Nicholas Carlini（MATS / Anthropic / 东北大学）｜ 2026-01-14

一句话总括：研究者把"基于属性的测试"（property-based testing）这件高门槛的活儿交给了一个 Claude 智能体——它不靠人写测试用例，而是从函数名、文档和类型注解里推断"代码本应满足的一般性属性"，再用 Hypothesis 像模糊测试那样自动生成海量输入去找反例，从而在 NumPy、SciPy、Pandas 等顶级库中翻出了数百个候选 bug，部分补丁已被官方合并。真正的看点不是"AI 会找 bug"，而是它能把一种强大却难用的测试范式自动化到可被维护者采纳的质量。

核心方法不写用例，而是推断"应当成立的属性"

传统测试是"举例式"的：开发者写死一个输入和期望输出，但没想到的边界往往也没被实现。属性测试反过来——只描述一个不变量（例如"JSON 反序列化是序列化的逆运算"）和输入域，框架自动搜索反例。门槛在于"想出正确的属性"，而论文发现 LLM 恰好擅长从上下文归纳属性：读代码、拉文档、看它如何被调用，就能提出该成立的不变式。

结论：智能体把属性测试里最难的"人类创造性"环节自动化了，让测试在更高抽象层运作。

智能体怎么跑一个自带自我反思的 Claude Code 命令

智能体是一个自定义 Claude Code 命令，接受单个目标（一个文件 / 模块 / 函数，如 numpy、json.loads），用待办清单做多步推理，按五步闭环工作：

①读懂目标：阅读代码、拉文档、理清它与代码库其余部分的关系
②基于发现提出属性（grounding 在真实用法与文档上，减少臆测）
③用 Hypothesis 写出对应的属性测试
④运行并反思：失败了——是真 bug 还是测试该改？通过了——是否只是平凡的测试？
⑤确信是真 bug 才写出格式化的bug 报告

设计的第一优先级是压低误报。有一次智能体写的测试一开始"通过"，自我反思后才发现自己把整个测试包进了 try-catch；去掉之后测试失败，bug 现身。研究者还观察到：Opus 4.1 与 Sonnet 4.5 的自我反思能力明显强于 Sonnet 4。

结论：让模型"质疑自己刚写的测试"，是把噪声压到开发者愿意看的关键一步。

真实评测984 份报告，排序后命中率大幅抬升

研究者在 100+ 个流行 Python 包（数值计算、解析、数据库等）上运行智能体。第一阶段用 Claude Opus 4.1 生成 984 份 bug 报告，人工抽查 50 份：

56%
抽查报告中是真实 bug
32%
真实且"值得上报"
86%
按 15 分制排序后，最高分报告中真实
81%
最高分报告中真实且可上报

也就是说，用 Opus 4.1 按一套 15 分制 rubric 给报告排序，能把原始 56% 的有效率拉到顶部的 86%——排序这一步本身就极大提升了交付质量。第二阶段改用 Sonnet 4.5 在 10 个重要包上多次运行，并引入一个 Sonnet 4.5 评估智能体读代码与报告核验严重性，再请 3 名专家人工复核高危 bug。

结论：发现 bug 之外，"自动排序 / 分诊"是让 AI 找 bug 工具真正可用的隐形一半。

不滥发报告每个 bug 平均一小时、三名专家把关

为了不给维护者添负担，上报流程刻意比自家代码评审更严：只挑最高优先级 → 交 3 名专家评审、平均每个 bug 约 1 小时 → 任一人存疑即丢弃 → 作者再亲自复核 → 仍有把握才手动提 issue。所有数据（含未验证、甚至已判定无效的 bug）都公开，供维护者自查。

结论：负责任披露的成本很高，但"宁可漏报不可误报"才是开发者愿意采用此类工具的前提。

战果清单5 个精选 bug，4 个已被官方接受

项目问题状态

每个 bug 都对应一条朴素属性：Wald 样本应恒正、切片重组应还原、不同输入的哈希应不同、颜色字符串应匹配 HSL 正则。而 dateutil 的反例揭示了边界——当代码有微妙/隐含语义时，只有维护者能裁定"正确的属性"是什么。

结论：智能体擅长把"显而易见但没人测"的不变量变成测试；对语义争议，它无法替代领域判断。

为什么重要测试、找 bug，是攻防赛跑的前哨

作者把这条线索放进更大的图景：很多导致安全漏洞的其实是逻辑 bug，而逻辑 bug 正适合属性测试（此前他们也用类似思路找智能合约漏洞）。随着 LLM 越来越会利用漏洞，必须抢先用它来发现并修漏洞。本文未做自动修补，但作者认为：一旦能较完整地刻画正确性属性，让 LLM 提出高质量补丁会是顺理成章的下一步。

结论：把属性测试自动化，是"在攻击者之前用 AI 加固代码"这场赛跑里的一步先手。

最值得带走的洞见

  1. 难点从"写用例"转移到"想属性"，而这正是 LLM 的强项。智能体从函数名、文档、调用关系里归纳出应当成立的不变量，使高门槛的属性测试得以自动化。
  2. "自动排序"和"发现"同样重要。原始报告仅 56% 有效，用 Opus 4.1 按 15 分制排序后，顶部报告有效率升到 86%、可上报率 81%——分诊是工具可用性的隐形一半。
  3. 自我反思直接决定误报率。智能体能发现"自己把测试包进了 try-catch"这类自欺；Opus 4.1 / Sonnet 4.5 在这点上明显强于 Sonnet 4。
  4. 真实战果而非演示。NumPy（误差降近 10 个数量级）、aws-lambda-powertools、tokenizers 等补丁已被官方合并，维护者是最终裁判。
  5. 边界很清楚：语义争议归人类。dateutil 的"无效"报告说明，对隐含/微妙语义，只有维护者能定义正确属性；自动找 bug 是人工测试的补充，不是替代。
